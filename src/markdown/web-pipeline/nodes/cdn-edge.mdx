## 概念定位

CDN 边缘节点是用户请求最先接触的共享缓存层，目标是在就近节点直接返回内容，减少回源和长距离传输成本。

## 工作机制

1. DNS 解析阶段通过 GSLB（Global Server Load Balancing）将请求调度到合适边缘节点。
2. 边缘节点按缓存键判断是否命中可用副本。
3. 命中时直接返回；未命中时回源获取并回填缓存。
4. 根据缓存规则决定副本生命周期和失效策略。

## 关键知识点

- **缓存键设计**：URL、查询参数、请求头是否参与键会直接影响命中率。
- **TTL（Time To Live） 策略**：决定缓存收敛速度与回源压力。
- **回源保护**：超时、重试、熔断和限流避免源站雪崩。
- **边缘一致性**：多地域节点失效传播速度影响版本生效体验。

## 对页面性能的影响

- 高命中率可显著降低 RTT（Round-Trip Time）和首包等待。
- 缓存键碎片化会导致命中率下降，回源抖动加剧。
- 边缘回源链路异常会直接体现在用户 TTFB（Time To First Byte） 波动。

## 观测与排障

### 核心指标

1. 命中率（整体与关键资源分层）。
2. 回源率与回源带宽。
3. 边缘 4xx/5xx 比例与地区分布。
4. 首包分位数（P75/P95）和区域波动。

### 排障步骤

1. 命中率下降先看缓存键是否被参数污染。
2. 再看 TTL 是否过短导致频繁回源。
3. 再看是否频繁全量 purge 造成缓存雪崩。
4. 结合源站日志确认回源峰值是否超容量。
5. 回归验证命中率、回源率和用户侧首包指标。

## 常见误区

- “接了 CDN 就自然会快”：不准确，规则错配会让 CDN 形同透传。
- “动态内容不能走 CDN”：不准确，可用短 TTL 和分层键策略做加速。
- “命中率高就一定没问题”：不准确，还要看关键资源命中和地区稳定性。

## 面试补充

### 30 秒说明

CDN 节点是请求链路中的共享缓存中枢。性能收益不来自“有 CDN”本身，而来自缓存键、TTL、回源保护和失效策略协同。优化目标是高命中、低波动、可控回源。

### 高频追问

1. 命中率突然下降，第一步查什么？

先查缓存键维度是否意外扩大（参数、Header 或 Cookie 参与键导致碎片化）。

2. 为什么全量清缓存容易引发事故？

边缘副本同时失效会造成回源风暴，源站可能瞬时过载，导致整体可用性下降。

3. 如何兼顾更新速度和命中率？

采用分层 TTL、版本化资源和精准失效策略，避免全量短 TTL 或全量强制失效。
