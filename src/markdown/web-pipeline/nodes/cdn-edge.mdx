## 概念定位

CDN 边缘节点是“离用户更近”的缓存与分发入口。该节点表示请求先到边缘站点判断是否可直接返回内容。

## 工作机制

1. 通过 DNS/GSLB（Global Server Load Balancing） 把用户调度到合适边缘节点。
2. 边缘节点按缓存键检查资源是否存在且新鲜。
3. 命中则直接返回；未命中则回源拉取并回填缓存。
4. 依据规则决定是否缓存、缓存多久、何时失效。

## 关键知识点

- **缓存键设计**：URL、查询参数、请求头维度会影响命中率。
- **回源策略**：回源超时、重试、熔断会影响稳定性。
- **失效策略**：主动 purge 与 TTL（Time To Live） 自然过期要配套。

## 对页面性能的影响

- 命中边缘缓存可显著缩短 RTT（Round-Trip Time），降低源站压力。
- 命中率低会导致频繁回源，首屏波动大。
- 缓存键过细会“碎片化”，缓存价值下降。

## 观测与排障

### 浏览器侧

- 检查响应头中的 `x-cache`、`age`、`via` 等字段。
- 对比不同地域访问时的 TTFB（Time To First Byte） 差异。

### 服务端侧

- 监控 CDN 命中率、回源率、边缘 5xx 比例。
- 关注热点资源的缓存穿透与回源峰值。

## 常见误区

- “上了 CDN 就一定快”：不准确，规则错误会让命中率很差。
- “动态接口不能用 CDN”：不准确，可用短 TTL + 键策略做加速。

## 面试补充

### 30 秒说明

CDN 边缘节点是请求链路上的就近缓存层。命中时直接返回，未命中回源并回填。性能收益核心取决于缓存键、TTL 与回源策略是否设计正确。

### 高频追问

1. CDN 命中率低怎么排查？

先看缓存键是否过细（参数/头过多），再看 TTL 是否太短、是否频繁主动清缓存，最后看是否有大量不可缓存响应头。

2. 动态内容如何在 CDN 上做加速？

对可缓存片段设置短 TTL 与明确键维度；对强个性化内容走回源或拆分成静态壳 + 动态数据。
