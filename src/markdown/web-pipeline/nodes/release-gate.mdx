## 概念定位

发布回归与性能门禁节点把“性能治理”从事后排障前移到发布流程，确保每次上线都可量化、可拦截、可回滚。

核心目标：

- 发布前：发现风险并阻断
- 发布中：控制影响面
- 发布后：快速定位和恢复

## 工作机制

1. 发布前生成基线（上一个稳定版本指标）。
2. 在 CI/CD 阶段执行门禁检查（体积、性能、错误率）。
3. 灰度发布并实时观测关键指标分位数。
4. 超阈值触发自动化动作（降级、回滚、熔断）。
5. 复盘结果回写动作库和预算配置。

## 关键知识点

### 性能预算（Performance Budget）

预算应至少覆盖：

- 体验指标：LCP、INP、CLS
- 链路指标：TTFB、失败率
- 产物指标：关键路由 JS/CSS 体积
- 稳定性指标：错误率、超时率、重连率

### 门禁分层

| 层级 | 示例 | 触发动作 |
| --- | --- | --- |
| 构建门禁 | 包体积超预算 | 阻断合并/发布 |
| 预发布门禁 | 关键路径压测回退 | 阻断上线 |
| 灰度门禁 | P75/P95 异常抬升 | 暂停放量 |
| 全量后门禁 | 错误率超阈值 | 自动回滚 |

### 灰度策略

- 按流量比例逐步放量，而不是直接全量。
- 按维度分桶：地区、网络类型、设备等级。
- 每一步放量前必须看指标稳定窗口。

### 回滚策略

- 明确回滚阈值，避免“靠感觉”。
- 预置一键回滚路径，减少人工链路。
- 回滚后必须验证恢复，不是回滚即结束。

## 对页面性能的影响

- 把回归问题在小流量阶段止损，避免全量事故。
- 缩短异常发现到恢复时间，降低用户受损时长。
- 通过长期预算约束抑制“版本越迭代越慢”。

## 观测与排障

### 发布窗口必看指标

1. LCP/INP/CLS 分位数
2. TTFB 与关键请求失败率
3. 前端 JS 错误率与资源加载错误率
4. CDN 命中率与回源率波动

### 排障剧本

1. 发现异常后先确认影响面（全量还是局部桶）。
2. 对比新旧版本变更项：资源体积、关键依赖、接口行为。
3. 使用动作库执行定位（网络层、渲染层、运行时层）。
4. 若超阈值持续，立即回滚并锁定发布窗口。
5. 复盘并追加防回归规则。

## 实战治理策略

### 流程规范

- 每个发布单必须附带性能基线对比。
- 关键变更（脚本体积、第三方引入、缓存策略）必须额外审批。
- 线上异常自动创建复盘任务，不允许“口头关闭”。

### 自动化策略

- CI 自动跑关键路由实验室指标。
- 灰度期间自动看板比较旧/新版本分位数差异。
- 回滚阈值触发自动化流水线。

### 组织协同

- 前端、后端、网关、运维共享同一门禁定义。
- 指标 owner 和动作 owner 明确到角色。
- 复盘结果回写 `metrics-playbook` 节点内容。

## 常见误区

- “测试环境没问题就能发”：错误，真实流量分布更复杂。
- “门禁会拖慢效率”：错误，长期看是减少返工与事故的提效手段。
- “回滚代表失败”：错误，快速止损是工程成熟度体现。

## 面试补充

### 30 秒说明

发布门禁的本质是把性能风险前移治理。通过预算、灰度、告警、回滚形成闭环，保证每次上线都在可控范围内。关键不是写文档，而是门禁自动化与阈值可执行，异常时能在分钟级止损。

### 高频追问

1. 性能门禁最小可行集是什么？

至少包含 LCP/INP/CLS、关键资源体积、错误率和回滚阈值。

2. 为什么要看分位数而不是均值？

均值会掩盖弱网和低端设备，分位数更接近真实用户体验。

3. 回滚阈值如何设定？

基于历史稳定版本分位数和业务可承受波动范围，按层级定义。

4. 灰度阶段最怕什么？

只看总量不看分桶，导致局部异常被掩盖后放大到全量。

5. 如何防止“回滚后再次发布又复发”？

复盘后必须把根因转为门禁规则或自动检测项。

6. 门禁只靠前端可以完成吗？

不行。需要后端、网关、CDN、运维共担指标和动作链。


## 进阶补充（深挖）

### 与上下游节点的因果关系

- 上游输入（常见）：前端工程化实践、性能监控指标体系。
- 本节点职责：将“发布回归与性能门禁”的结果转化为可执行的后续分支决策。
- 下游影响（常见）：缓存策略优化、资源预提示优化。

### 线上故障案例

- 现象：新版本全量后性能回退，回滚耗时长导致影响扩大。
- 根因：灰度窗口不足、回滚阈值不清、自动化动作缺失。
- 处置：强化分桶灰度、固化阈值触发回滚并缩短决策链路。
- 回归标准：关键指标恢复到基线区间，且连续观测窗口无二次抖动。

### 高频追问）

1. 门禁为什么不能只看一次压测？

真实流量设备和网络分布更复杂，需灰度验证。

2. 回滚后为什么还要复盘？

要把根因转为规则，避免重复事故。

3. 性能门禁会不会影响交付速度？

短期多一步检查，长期显著减少事故返工。


### 补充记忆框架

- 先判断该节点是“前置等待”问题还是“运行时抖动”问题。
- 再判断问题来自“资源策略”还是“执行调度”。
- 最后把定位结论转为可复用规则，进入门禁与动作库。

### 高频追问

1. 这个节点优化后如何做回归验收？

至少做两组对比：关键指标分位数对比 + 上下游链路稳定性对比，避免局部优化导致成本转移。

2. 为什么这个节点要单独做发布观察窗口？

因为它对主链路有放大效应，短时间异常会快速传导到首屏和交互体验。
