## 概念定位

HTTP 协议演进本质是三件事：

- 降低握手成本
- 提高并发传输效率
- 降低队头阻塞与弱网损耗

从 HTTP/1.0 到 HTTP/3 的升级，不是“版本号更新”，而是网络传输模型、连接复用策略和浏览器调度能力的系统升级。

## 工作机制

### HTTP/1.0 到 HTTP/1.1

HTTP/1.0 默认短连接，请求多时反复建连。HTTP/1.1 引入持久连接（Keep-Alive）后，多个请求可复用同一个 TCP（Transmission Control Protocol）连接，显著减少三次握手开销。

HTTP/1.1 还补齐了缓存与部分内容请求能力（如 `Range`、`206`），让大资源传输和断点续传可用。

### HTTP/2（h2）

HTTP/2 的核心是“二进制分帧 + 流（Stream）多路复用”：

- 一个连接上并发多个流
- 每个流拆成帧交错传输
- 用头部压缩降低重复头开销（HPACK）

它解决了 HTTP 应用层串行队头问题，但底层仍是 TCP，出现丢包时仍可能触发 TCP 层阻塞放大。

### HTTP/3（h3）

HTTP/3 运行在 QUIC（Quick UDP Internet Connections）之上：

- 基于 UDP 构建可靠传输
- 内置 TLS 1.3（Transport Layer Security 1.3）
- 支持连接迁移（Connection Migration）
- 多流之间更少互相阻塞（尤其弱网收益明显）

HTTP/3 的优势通常在高 RTT、抖动、丢包、移动网络切换场景更明显。

## 关键知识点

### 版本差异总览

| 维度 | HTTP/1.1 | HTTP/2 | HTTP/3 |
| --- | --- | --- | --- |
| 传输基座 | TCP | TCP | QUIC(UDP) |
| 报文形态 | 文本 | 二进制帧 | 二进制帧 |
| 并发模型 | 多连接并发 | 单连接多流复用 | 单连接多流复用 |
| 头部压缩 | 无统一机制 | HPACK | QPACK |
| 队头阻塞 | 应用层明显 | 应用层缓解，TCP 层仍可能放大 | 进一步缓解 |
| 协商方式 | 明确版本/升级 | TLS ALPN 协商 | TLS ALPN 协商 |

### 握手与 RTT 成本

- 首次访问成本通常包含 DNS、TCP/QUIC、TLS、请求发送。
- 连接复用越充分，后续请求边际成本越低。
- 协议升级收益要看分位数（P75/P95），不能只看平均值。

### ALPN 与真实协议落地

应用层协议协商（Application-Layer Protocol Negotiation, ALPN）决定连接最终跑 `http/1.1`、`h2` 还是 `h3`。前端看到“服务端支持 h3”不代表用户端都走 h3，仍受浏览器版本、网络路径、CDN/网关配置影响。

### 连接复用边界

复用通常受这些因素限制：

- 协议、域名、端口是否一致
- 证书覆盖范围与可复用策略
- 代理层是否拆连接
- 浏览器连接池与优先级调度

很多“明明上了 h2/h3 但还是慢”的根因，是复用实际没生效。

### 域名分片在 h2/h3 下的副作用

HTTP/1.1 时代用域名分片提升并发是常规手段；但在 h2/h3 下，过多域名会增加 DNS、TLS、连接池开销，常常适得其反。现代优化更强调“关键域收敛 + 合理优先级”。

## 对页面性能的影响

- 首屏关键链路通常受协议协商和首连接质量影响最大。
- h2/h3 能提升并发传输效率，但不能替代资源优先级治理。
- 若关键资源与低优先级资源争抢带宽，协议升级收益会被抵消。
- 弱网场景下 h3 常见收益是失败率降低与尾延迟收敛，而非每次都更快。

## 观测与排障

### 浏览器侧核心观测

- Network 协议列：确认 `http/1.1` / `h2` / `h3`。
- 看连接阶段耗时是否重复出现，判断复用是否生效。
- 看关键资源是否被排队、阻塞、低优先级拖慢。

### 指标联动

建议把协议观察与这些指标联动：

- 首包时间（Time To First Byte, TTFB）分位数
- 首次内容绘制（First Contentful Paint, FCP）
- 最大内容绘制（Largest Contentful Paint, LCP）
- 交互到下一次绘制（Interaction to Next Paint, INP）
- 资源失败率与重试率

### 排障步骤

1. 先确认关键域真实协商协议，不要只看服务端宣称。
2. 再看连接复用是否生效，是否频繁新建连接。
3. 再看关键资源优先级是否正确，避免关键资源被挤压。
4. 对比 `h2` 与 `h3` 在弱网下的 P75/P95 和失败率。
5. 若收益不稳定，检查 CDN、网关、证书、探针回退策略。

## 实战优化策略

### 协议与连接层

- 开启并验证 h2/h3，但保留稳定回退路径。
- 避免盲目域名分片，优先收敛关键静态域。
- 合理使用预连接（`preconnect`）与 DNS 预解析（`dns-prefetch`）。

### 资源与调度层

- 关键资源优先加载，非关键资源延后。
- 减少首屏关键请求数量和体积。
- 对大资源结合 `Range`/分块策略，避免长尾拖累。

### 缓存与传输层

- 静态资源配合长期缓存与内容哈希。
- 文本资源启用 Brotli/Gzip 压缩。
- 结合 CDN 边缘缓存缩短跨地域 RTT。

## 常见误区

- “HTTP/2 一上就不用做性能优化”：错误，调度与资源治理仍决定体验。
- “HTTP/3 一定比 HTTP/2 快”：错误，收益依赖网络质量与端到端配置。
- “域名分片永远提升并发”：在 h2/h3 下常会增加额外成本。
- “只看平均 TTFB 就能评估协议收益”：要看分位数和失败率。
- “服务器推送一定有收益”：很多场景收益有限，维护成本更高。

## 面试补充

### 30 秒说明

HTTP 演进的核心是降低连接与并发成本。HTTP/1.1 通过持久连接减少重复握手，HTTP/2 通过多路复用提升同连接并发，HTTP/3 通过 QUIC 改善弱网下的阻塞与重传体验。真实收益取决于协议协商是否落地、连接复用是否生效、关键资源优先级是否正确。

### 高频追问

1. 为什么用了 HTTP/2 页面还是慢？

因为“协议并发能力”不等于“业务调度正确”。关键资源可能被非关键资源挤压，或复用没有真正生效。

2. HTTP/2 的多路复用是否彻底解决队头阻塞？

只解决了 HTTP 应用层串行问题；底层仍是 TCP，丢包时仍会在 TCP 层放大等待。

3. HTTP/3 的主要收益点是什么？

在高延迟、丢包、抖动、移动网络切换场景，尾延迟和失败率通常更可控。

4. 如何判断连接复用没有生效？

后续请求持续出现完整连接阶段耗时，且同域请求频繁新建连接，就是典型信号。

5. 什么时候不建议继续做域名分片？

在 h2/h3 主导的现代链路中，多域会增加 DNS 与握手成本，往往不如域收敛收益高。

6. 协议升级上线如何做灰度？

按地域/运营商/浏览器分桶，观察 P75/P95 TTFB、失败率、回退率，设置可回滚阈值再逐步放量。
