## 概念定位

TCP/TLS 建连是网络请求真正可发送前的传输准备阶段。它决定链路是否可靠、安全以及是否能快速进入应用层请求。

## 工作机制

1. 先完成 TCP 三次握手，建立可靠字节流连接。
2. HTTPS 场景继续进行 TLS 握手，协商证书、加密套件和会话密钥。
3. 通过 ALPN（Application-Layer Protocol Negotiation）协商应用协议版本（HTTP/1.1、HTTP/2、HTTP/3）。
4. 握手成功后进入 HTTP 请求发送阶段。

## 关键知识点

- **RTT（Round-Trip Time）**：往返时延，直接决定握手成本。
- **SNI（Server Name Indication）**：在 TLS 握手中指明目标域名。
- **会话复用**：通过会话恢复减少重复握手开销。
- **连接复用**：减少重复建连，降低请求前等待。

## 对页面性能的影响

- 冷连接成本高，常是首访首屏链路中的关键延迟来源。
- 弱网下 RTT 放大，建连成本会明显拖慢 TTFB（Time To First Byte）。
- 跨域资源过多会产生多条连接链，增加总握手成本。

## 观测与排障

### 浏览器侧

- Network 查看 `DNS`、`Initial connection`、`SSL` 时长。
- 对比首请求与复用连接请求，确认复用是否生效。

### 服务端与链路侧

- 检查证书链完整性和握手失败率。
- 分地区看握手耗时分位数，识别弱网或跨境链路问题。

### 排障步骤

1. 优先确认是否存在不必要的跨域关键资源。
2. 对关键跨域提前 `preconnect`，减少首请求建连等待。
3. 检查证书与中间证书配置，避免握手失败重试。
4. 评估连接复用与会话恢复是否生效。
5. 回归查看 TTFB 与 LCP 是否同步下降。

## 常见误区

- “HTTPS 只影响安全，不影响性能”：不准确，握手成本是真实存在的。
- “HTTP/2 就没有连接成本”：不准确，首次建连仍需成本，只是后续复用更好。
- “只要带宽够大就不会慢”：不准确，握手时延与带宽不是一回事。

## 面试补充

### 30 秒说明

TCP/TLS 建连是请求前必经链路。TCP 保证可靠传输，TLS 保证安全与身份，RTT 决定握手时延。优化重点是减少不必要连接、提升复用率、为关键跨域提前建连。

### 高频追问

1. 为什么 `preconnect` 在跨域关键资源上经常有效？

因为它把 DNS、TCP、TLS 的准备成本前置了，用户真正触发请求时可直接发包。

2. 如何判断慢在网络还是慢在服务端？

若建连相关阶段（DNS/SSL）明显高，偏网络链路；若建连正常但 TTFB 仍高，常见是服务端处理或回源慢。

3. 为什么同一个站点有时快有时慢？

可能是连接是否复用、会话是否恢复、地区链路差异或瞬时网络抖动导致。
