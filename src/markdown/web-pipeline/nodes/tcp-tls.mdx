## 概念定位

TCP/TLS 建连是“拿到 IP 后真正开始通信”的阶段。这里决定了请求能否安全、稳定地进入 HTTP 传输。

## 工作机制

1. 先完成 TCP 三次握手，建立可靠连接。
2. HTTPS 场景下继续进行 TLS 握手，协商加密套件与会话密钥。
3. 协商 ALPN（Application-Layer Protocol Negotiation），确定走 HTTP/1.1、HTTP/2 或 HTTP/3（链路不同）。
4. 建连成功后进入 HTTP 请求发送阶段。

## 关键知识点

- **RTT（Round-Trip Time） 成本**：冷连接至少消耗多次往返。
- **TLS 会话复用**：可降低后续握手开销。
- **SNI（Server Name Indication）/证书校验**：保障连接到正确服务端。

## 对页面性能的影响

- 冷启动建连常是首屏关键路径中的大头。
- 高 RTT、丢包、握手失败会直接拖慢 TTFB（Time To First Byte）。
- `preconnect` 可提前支付建连成本，降低请求触发时延迟。

## 观测与排障

### 浏览器侧

- DevTools Network 查看 `Initial connection` 与 `SSL` 耗时。
- 对比首个请求与复用连接请求的耗时差异。

### 命令行

```sh
openssl s_client -connect example.com:443 -servername example.com
```

用于检查证书链、协议版本与握手状态。

## 常见误区

- “HTTPS 只影响安全，不影响性能”：不准确，握手有成本但可通过复用优化。
- “建连成功就代表请求一定成功”：不准确，后续仍可能超时或被限流。

## 面试补充

### 30 秒说明

TCP/TLS 建连是请求前的传输准备阶段。TCP 负责可靠传输，TLS 负责身份与加密。这个阶段的 RTT 成本会直接影响首包时间，是性能优化重点。

### 高频追问

1. 为什么 `preconnect` 往往比 `dns-prefetch` 优化更明显？

`preconnect` 不只做 DNS，还提前完成 TCP（以及 HTTPS 的 TLS）握手，能直接减少真正请求触发时的等待。

2. TLS1.3 相比旧版本的性能优势是什么？

握手往返次数更少，配合会话恢复可进一步降低建连延迟，对高 RTT 场景收益明显。
