## 概念定位

导航开始是浏览器把“用户意图”转换为“加载工作流”的入口节点。它不只是触发请求，而是决定后续是否走缓存复用、是否新建文档、是否可用历史快照恢复。

## 工作机制

1. 浏览器接收导航意图：地址栏输入、链接点击、前进后退、脚本跳转。
2. 解析 URL（协议、域名、路径、查询参数），判断导航类型。
3. 判断是否同文档导航（如 hash 变化）或跨文档导航。
4. 跨文档导航进入缓存/SW/网络链路；同文档导航通常不走完整网络。
5. 记录 Navigation Timing 并更新会话历史。

## 关键知识点

- **同文档导航**：更新 URL 片段和滚动位置，通常无需重建文档。
- **跨文档导航**：触发新文档加载和完整渲染流程。
- **BFCache（Back-Forward Cache）**：前进后退可能直接恢复页面快照。
- **重定向链**：导航初期重定向会吞噬首屏预算。

## 对页面性能的影响

- 导航起点阶段的额外跳转会推迟后续所有阶段。
- BFCache 命中可显著缩短返回路径体验。
- 导航前的主线程拥塞会拖慢请求真正发起时机。

## 观测与排障

### 浏览器侧

- Performance 看从 `navigationStart` 到首请求发起的时间。
- Network 检查是否存在多跳 301/302 重定向。
- 检查 back/forward 是否命中 BFCache。

### 排障步骤

1. 减少不必要重定向和跨域跳转。
2. 关键入口路由做 URL 规范化，避免循环跳转。
3. 检查前进后退路径是否被禁用 BFCache（事件监听、资源占用）。
4. 回归看首屏路径下的导航阶段耗时和 LCP（Largest Contentful Paint）。

## 常见误区

- “导航开始就等于请求发出”：不准确，中间还有多层判断与拦截。
- “返回上一页一定会重载”：不准确，BFCache 命中时可直接恢复。
- “重定向只影响 SEO”：不准确，用户首屏体验也会被直接拖慢。

## 面试补充

### 30 秒说明

导航开始是加载工作流的起点。浏览器先判断导航类型和可复用路径，再决定是否走完整请求链路。优化重点是减少重定向、提升 BFCache 命中、压缩导航到首请求发起的间隔。

### 高频追问

1. 地址栏回车后第一个关键性能风险是什么？

重定向链和主线程阻塞会让“真正发请求”明显后移，导致后续全部阶段被拖慢。

2. BFCache 与普通缓存有何不同？

普通缓存是资源级复用，BFCache 是页面快照级恢复，能跳过大量解析和渲染步骤。

3. 如何快速判断导航瓶颈是否在前置阶段？

看 `navigationStart -> requestStart` 区间是否异常偏大，再结合重定向数量和主线程活动定位。
