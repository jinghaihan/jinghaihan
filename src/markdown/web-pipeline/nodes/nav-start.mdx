## 概念定位

导航开始节点是 Web Pipeline 的入口，负责把“用户意图”变成“浏览器加载动作”。

它决定了后续是否走完整网络链路、是否可复用历史页面状态、是否触发重定向与缓存策略。

## 工作机制

1. 浏览器接收导航意图：地址栏回车、链接点击、前进后退、脚本跳转。
2. 解析 URL，判断协议、域名、路径、查询参数是否合法。
3. 判断导航类型：同文档导航或跨文档导航。
4. 若跨文档导航，进入缓存/SW/网络链路；若同文档，通常不走完整请求流程。
5. 记录性能时间线与会话历史，供监控与回溯使用。

## 关键知识点

### 导航类型对比

| 类型 | 是否重建文档 | 是否走完整网络链路 | 常见触发 |
| --- | --- | --- | --- |
| 同文档导航 | 否 | 通常否 | hash 变化、部分 SPA 路由 |
| 跨文档导航 | 是 | 通常是 | 新链接、地址栏输入、跨页跳转 |
| 前进后退恢复 | 可能否 | 可能否（BFCache 命中） | 浏览器后退/前进 |

### BFCache（Back-Forward Cache）

前进后退缓存（Back-Forward Cache, BFCache）命中时，可直接恢复页面快照，跳过大部分解析与渲染过程。

### 导航前置风险

- 多跳重定向
- 导航前主线程拥塞（脚本长任务）
- 非必要跨域跳转

## 对页面性能的影响

- 导航开始阶段出现额外跳转，会推迟后续所有阶段。
- BFCache 命中可显著改善“返回上一页”的体感。
- 导航到请求发起的间隔过大，通常意味着前置阶段有阻塞。

## 观测与排障

### 核心观测

1. 看 `navigationStart -> requestStart` 区间是否异常偏大。
2. 看重定向链长度和跨域跳转次数。
3. 看前进后退路径 BFCache 命中率。

### 排障步骤

1. 减少不必要 301/302/307/308 跳转。
2. 做入口 URL 规范化，避免循环跳转。
3. 排查阻断 BFCache 的行为（如不当生命周期监听）。
4. 回归对比导航阶段耗时与 LCP（Largest Contentful Paint）。

## 实战优化清单

- 关键入口避免多域名串跳。
- 首页路由和登录态跳转保持单跳可达。
- 对返回路径优先保障 BFCache 命中。
- 把导航前长任务纳入性能门禁。

## 常见误区

- “导航开始就等于请求已发出”：错误，前面还有判断与调度。
- “后退一定会重新加载页面”：错误，BFCache 命中时可直接恢复。
- “重定向只影响 SEO”：错误，也会直接消耗用户性能预算。

## 面试补充

### 30 秒说明

导航开始阶段是请求链路和渲染链路的起点。浏览器会先判断导航类型和可复用路径，再决定是否走完整网络流程。优化重点是减少重定向、提升 BFCache 命中、压缩导航到请求发起的前置耗时。

### 高频追问

1. 如何快速判断导航慢在前置阶段？

看 `navigationStart -> requestStart` 是否异常，若异常再查重定向和主线程占用。

2. BFCache 和普通缓存有什么本质区别？

普通缓存是资源级复用，BFCache 是页面快照级恢复，能跳过大量解析与执行步骤。

3. SPA 场景下导航阶段还重要吗？

重要。即使不整页重载，路由切换前后仍有资源、状态和渲染调度成本。

4. 入口跳转策略如何做治理？

建立单跳目标规则，限制跨域串跳，并把跳转链长度纳入发布检查。

5. 导航优化应先动哪一层？

先处理最前置可控项：URL 规范、重定向策略、前置长任务。


## 补充视角（标准节点）

### 与上下游节点边界

- 上游节点（高频）：HTML 加载与语义策略。
- 本节点重点：聚焦“导航开始”本身的机制与决策，不替代下游渲染/监控职责。
- 下游节点（高频）：Service Worker 拦截、刷新模式与缓存判定。

### 落地动作清单

1. 先定义该节点的关键输入和关键输出，避免职责漂移。
2. 再绑定至少 1 个可观测指标，确保问题可量化定位。
3. 将该节点的异常处理写入动作库，确保值班时可执行。

### 高频追问

1. 这个节点优化应该先看什么？

先看它的关键输入是否稳定，再看关键输出是否满足下游依赖和时延预算。

2. 如何避免“优化该节点却拖慢其他节点”？

做上下游联动回归，对比关键链路分阶段指标，防止成本转移。

3. 这个节点如何进入发布门禁？

把它的核心指标加入灰度对比阈值，超阈值时触发暂停放量或回滚。
