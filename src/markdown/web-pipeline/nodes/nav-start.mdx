## 概念定位

导航开始是“用户意图”真正进入浏览器加载管线的起点。常见触发包括地址栏回车、点击链接、前进后退、`location` 跳转。

## 工作机制

1. 浏览器接收导航意图，解析 URL（协议、域名、路径、查询参数）。
2. 判断是否同文档导航（如 hash 变化）或完整文档导航。
3. 命中新文档导航后，进入缓存/SW/网络请求分支。
4. 同时记录新的 Navigation Timing，更新会话历史栈。

## 关键知识点

- **同文档导航**：只更新 URL 片段和滚动位置，不走完整网络链路。
- **跨文档导航**：会创建新文档上下文，通常触发完整加载流程。
- **BFCache（Back-Forward Cache）**：前进后退可能直接从页面快照恢复，而不是重新请求。

## 对页面性能的影响

- 重定向链过长会在“真正发请求前”消耗掉首屏预算。
- 导航触发时主线程繁忙，会推迟后续解析和渲染阶段。
- 过度依赖全量跳转而非局部更新，会增加重复建连和重复解析成本。

## 观测与排障

### 浏览器侧

- Performance 面板查看 `navigationStart` 到首个请求发起的间隔。
- Network 关注是否出现多次 301/302 跳转。
- 观察返回/前进是否命中 BFCache（不是每次都会重载）。

## 常见误区

- “导航开始就等于网络请求已经发出”：不准确，中间仍有缓存/SW/策略判断。
- “前进后退一定会重新加载页面”：不准确，可能直接命中 BFCache。

## 面试补充

### 30 秒说明

导航开始是浏览器从“用户操作”进入“加载管线”的起点。它会先判断是不是同文档跳转，再决定是否走缓存、Service Worker 或网络请求链路，并为整个加载过程打时间戳。

### 高频追问

1. 地址栏输入 URL 回车后，最先发生什么？

浏览器会先做 URL 解析与导航类型判断，不是直接发网络请求；只有确认要加载新文档后，才会进入缓存/SW/DNS/建连链路。

2. 普通刷新和强制刷新在导航起点有什么差异？

普通刷新会尽量复用缓存策略（可能触发 304）；强制刷新会更激进地绕过本地缓存，导致后续链路更偏向真实网络请求。
