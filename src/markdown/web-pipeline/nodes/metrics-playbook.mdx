## 概念定位

指标异常排障动作库节点的目标是：当监控报警出现时，团队能直接执行标准动作，而不是临时讨论。

它是“指标体系”到“工程动作”的中间层，强调可执行、可复用、可复盘。

## 工作机制

1. 每个核心指标绑定对应的异常阈值。
2. 指标异常时进入预定义排障路径（网络、渲染、运行时、服务）。
3. 执行动作后记录结果和耗时。
4. 将有效动作沉淀为标准剧本，更新发布门禁与监控规则。

## 关键知识点

### 动作库结构模板

每条动作模板建议包含：

- 触发条件（阈值、持续时间、影响面）
- 快速诊断（先看什么）
- 定位路径（分层排查顺序）
- 临时止损（灰度暂停、回滚、降级）
- 根因修复（代码/配置/架构）
- 回归验证（看哪些指标恢复）

### 指标到动作映射示例

| 指标异常 | 常见根因 | 第一动作 |
| --- | --- | --- |
| LCP 上升 | 关键资源阻塞、图片体积回退、缓存失效 | 检查关键请求链和资源体积变化 |
| INP 上升 | 长任务、强制布局、事件处理过重 | 抓交互火焰图定位长任务调用栈 |
| CLS 上升 | 图片/广告位无尺寸、异步插入 | 检查布局稳定约束是否丢失 |
| TTFB 上升 | 回源慢、建连抖动、网关拥塞 | 分层看 DNS/TCP/TLS/服务耗时 |
| 5xx 升高 | 上游异常、容量不足、配置变更 | 降级并按链路分层止损 |
| 429 升高 | 限流过紧、重试风暴 | 收敛重试并检查限流策略 |

### 动作优先级

- 先止损，再定位。
- 先影响面最大链路，再处理局部问题。
- 优先可回滚动作，避免在线上做高风险试验。

## 对页面性能的影响

- 动作库成熟可显著缩短 MTTR（Mean Time To Recovery）。
- 避免故障期间“多人重复排查”导致响应延迟。
- 让性能治理从“个人经验”升级为“团队能力”。

## 观测与排障

### 实战排障步骤

1. 确认报警真实性（采样、上报、基线）
2. 判断影响范围（全量/局部、地区/机型/网络）
3. 选择对应动作模板执行第一动作
4. 判断是否触发止损（暂停灰度/回滚）
5. 根因修复并回归验证
6. 复盘沉淀并升级动作模板

### 关键日志与证据

- 前端 RUM 指标与错误事件
- Network HAR 与关键请求链
- 服务端 trace 与网关日志
- 发布记录与配置变更记录

## 实战动作模板（可直接套用）

### 模板 A：LCP 突增

1. 看关键路由 LCP 元素是否变化。
2. 对比关键资源体积、加载顺序、缓存命中率。
3. 若伴随 TTFB 抬升，联动网络层排查。
4. 超阈值持续 10 分钟触发降级或回滚。

### 模板 B：INP 突增

1. 锁定高频交互场景（点击、输入、滚动）。
2. 采样慢交互并定位长任务调用栈。
3. 检查是否新增同步计算、强制布局、第三方脚本。
4. 临时止损：关闭高成本交互特性或降级动画。

### 模板 C：5xx / 429 异常

1. 按状态码拆分（500/502/503/504/429）。
2. 检查是否由发布、限流策略或依赖抖动引发。
3. 执行退避重试与熔断策略，避免放大故障。
4. 超阈值持续触发回滚并冻结发布。

## 常见误区

- “有监控就够了”：错误，没有动作模板等于没有响应能力。
- “每次故障临时分析更灵活”：错误，临时方案难复用且容易遗漏关键步骤。
- “动作库写一次就完”：错误，需要随系统演进持续更新。

## 面试补充

### 30 秒说明

指标异常排障动作库是性能治理的执行层。它把每个指标异常映射成标准动作：先止损、再定位、后修复、最后回归。价值在于缩短故障恢复时间，并把经验沉淀成团队可复用能力，而不是依赖个人临场发挥。

### 高频追问

1. 动作库和普通监控文档有什么区别？

动作库强调“触发条件 + 执行步骤 + 止损阈值 + 回归标准”，可直接执行。

2. 动作库如何避免过时？

每次故障复盘后更新模板，并把新规则写入门禁和告警系统。

3. 为什么要先止损再定位？

因为线上影响在持续扩大，先控影响面再深挖根因是更稳妥策略。

4. 如何定义动作库是否有效？

看 MTTR、重复故障率、告警到执行动作的时延是否持续下降。

5. 动作库要不要细分到业务线？

要。基础模板统一，业务差异通过子模板扩展。

6. 谁来维护动作库？

建议由性能 owner 牵头，前端/后端/运维共同维护并评审。


## 进阶补充（深挖）

### 与上下游节点的因果关系

- 上游输入（常见）：性能监控指标体系。
- 本节点职责：将“指标异常排障动作库”的结果转化为可执行的后续分支决策。
- 下游影响（常见）：缓存策略优化、强制布局与回流治理、前端工程化实践。

### 线上故障案例

- 现象：同类性能报警反复出现，但处理方式每次都不同。
- 根因：缺少标准动作模板和证据留存，经验无法复用。
- 处置：把高频告警固化成动作模板，并纳入发布与值班流程。
- 回归标准：关键指标恢复到基线区间，且连续观测窗口无二次抖动。

### 高频追问）

1. 动作库模板最小要素是什么？

触发条件、排查顺序、止损动作、回归标准。

2. 如何证明动作库有效？

看 MTTR、重复故障率和告警处置时长。

3. 动作库是否需要按业务拆分？

基础模板统一，业务场景做扩展。


### 补充记忆框架

- 先判断该节点是“前置等待”问题还是“运行时抖动”问题。
- 再判断问题来自“资源策略”还是“执行调度”。
- 最后把定位结论转为可复用规则，进入门禁与动作库。

### 高频追问

1. 这个节点优化后如何做回归验收？

至少做两组对比：关键指标分位数对比 + 上下游链路稳定性对比，避免局部优化导致成本转移。

2. 为什么这个节点要单独做发布观察窗口？

因为它对主链路有放大效应，短时间异常会快速传导到首屏和交互体验。
