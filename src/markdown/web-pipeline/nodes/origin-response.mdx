## 概念定位

该节点表示源站完成请求处理后返回状态码、响应头和响应体。它决定后续缓存行为与渲染入口时机。

## 工作机制

1. 服务端处理请求并生成响应。
2. 返回状态码（如 `200`、`304`、`206`）与缓存/压缩等响应头。
3. 响应体可能一次性返回，也可能分块流式返回。
4. 浏览器根据响应头决定缓存和解析策略。

## 关键知识点

- **状态码语义**：`304` 复用缓存，`206` 支持范围传输。
- **缓存头控制**：`Cache-Control`、`ETag` 直接影响后续命中率。
- **压缩策略**：`br` / `gzip` 影响传输体积与解压开销。

## 对页面性能的影响

- 高 TTFB（Time To First Byte） 常由后端处理慢或回源链路慢引起。
- 响应头策略错误会造成“每次都重下”或“资源过旧”。
- 缺少压缩会显著增加下载体积。

## 观测与排障

### 浏览器侧

- Network 关注 TTFB、状态码和响应头。
- 比较 `200` 与 `304` 场景下的总耗时。

### 服务端侧

- 结合日志看应用耗时、数据库耗时、回源耗时。
- 使用 `Server-Timing` 暴露关键阶段分解。

## 常见误区

- “304 越多越好”：不绝对，304 仍有 RTT（Round-Trip Time） 成本。
- “压缩率越高越好”：不绝对，压缩 CPU 成本也要平衡。

## 面试补充

### 30 秒说明

源站响应节点决定了请求的“结果质量”：状态码、缓存头、压缩头都在这里形成。它不仅影响本次加载，还会影响后续复访的缓存命中行为。

### 高频追问

1. 为什么有些资源 `304` 了页面还是慢？

因为 304 只是省了响应体下载，但请求往返、队列等待、主线程竞争仍然存在。

2. `200`、`304`、`206` 在性能上怎么理解？

`200` 是完整返回，`304` 是条件命中复用本地副本，`206` 是范围返回，常见于大文件分段加载与断点续传。
