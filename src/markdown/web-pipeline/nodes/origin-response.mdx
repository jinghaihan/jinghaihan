## 概念定位

源站返回响应节点是“请求处理结果”进入浏览器的入口。它不仅决定本次展示速度，也决定后续缓存命中和重复访问成本。

## 工作机制

1. 服务端完成业务处理，生成状态码、响应头和响应体。
2. 响应头声明缓存、压缩、传输与安全策略。
3. 响应体可一次性返回，也可流式分块返回。
4. 浏览器依据状态码与头部决定缓存复用和解析策略。

## 关键知识点

- **状态码语义**：`200` 完整返回，`304` 协商命中，`206` 范围传输。
- **缓存控制**：`Cache-Control`、`ETag`、`Last-Modified` 影响复访开销。
- **传输压缩**：`br` / `gzip` 影响下载体积与解压成本。
- **服务耗时暴露**：`Server-Timing` 可帮助拆解后端阶段耗时。

## 对页面性能的影响

- 后端处理慢会直接推高 TTFB。
- 缓存头错误会让浏览器无法复用副本，导致重复下载。
- 压缩与分块策略不当会拖慢首包到首屏的转换效率。

## 观测与排障

### 浏览器侧

- Network 重点看 TTFB、状态码分布、响应头策略。
- 对比 200 与 304 场景，确认缓存复用是否真正生效。

### 服务端侧

- 监控接口 P95/P99 耗时、错误率、慢查询比例。
- 使用 `Server-Timing` 拆解应用、数据库、下游依赖耗时。

### 排障步骤

1. 先判断慢在首包还是慢在下载。
2. 若首包慢，优先排查服务端计算和下游依赖。
3. 若下载慢，检查压缩、分块、资源体积和链路质量。
4. 校验缓存头，避免“每次都 200 完整返回”。
5. 回归看 TTFB、LCP、回源率是否同时改善。

## 常见误区

- “304 一定说明性能很好”：不准确，304 仍有往返成本。
- “只压缩就能解决响应慢”：不准确，首包慢常在服务端处理阶段。
- “后端指标和前端体验无关”：不准确，TTFB 和 LCP 强相关。

## 面试补充

### 30 秒说明

源站响应阶段决定了请求质量：返回多快、是否可缓存、体积多大。性能优化要分层处理：服务端降首包、传输层控体积、缓存层提命中，最后用 TTFB 和 LCP 回归。

### 高频追问

1. 为什么 TTFB 高时，LCP 往往也会差？

因为首字节到达晚会推迟后续解析和关键资源调度，导致主要内容可见时间整体后移。

2. 304 与强缓存命中的体验差异是什么？

强缓存命中通常不发请求，成本最低；304 需要一次条件请求再复用副本，仍有网络往返。

3. 如何快速定位是“后端慢”还是“前端慢”？

先看 TTFB 与下载耗时分段，再结合 `Server-Timing` 与前端主线程性能数据，判断瓶颈所在层级。
