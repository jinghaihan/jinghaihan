## 概念定位

资源预提示优化是把未来要做的网络动作提前到更早时机，目的是在用户真正触发请求时减少等待。它属于“调度优化”，不是“盲目预加载”。

## 工作机制

1. 浏览器读取 HTML 中的资源提示标签。
2. 按提示类型执行不同粒度前置：解析、建连、下载。
3. 真正请求发生时复用前置成果，缩短关键路径。
4. 若提示资源未被使用，会产生额外带宽和连接开销。

## 关键知识点

- **`dns-prefetch`**：提前做域名解析。
- **`preconnect`**：提前完成 DNS、TCP、TLS 等连接准备。
- **`preload`**：高优先级下载当前页面必需资源。
- **`prefetch`**：低优先级预取未来可能用到的资源。
- **优先级控制**：预提示必须服务关键路径，不应和关键资源抢通道。

## 对页面性能的影响

- 正确使用可明显缩短首屏关键资源到达时间。
- 误用会导致带宽争用，反而拖慢真正关键请求。
- 跨域关键资源场景中，`preconnect` 常比单纯解析前置收益更大。

## 观测与排障

### 浏览器侧

- Network 观察提示资源发起时机与优先级是否符合预期。
- 对比加提示前后关键资源下载开始时间与完成时间。
- 检查是否存在“预加载未使用”的浪费警告。

### 排障步骤

1. 梳理首屏关键资源清单，先定“必须前置”的最小集合。
2. 跨域关键资源优先尝试 `preconnect`。
3. 仅对首屏必需资源使用 `preload`，避免泛化。
4. 对未来路由资源使用 `prefetch`，并控制数量。
5. 用 LCP、TTFB、关键请求链长度回归验证。

## 常见误区

- “把全部资源都 `preload` 最快”：错误，会挤压真正关键请求。
- “有了 HTTP/2 就不需要预提示”：不准确，连接和调度前置仍有价值。
- “预提示配置一次就不用管”：不准确，页面结构变化后需要持续校正。

## 面试补充

### 30 秒说明

资源预提示是把解析、建连、下载前置到更早时机。`dns-prefetch`、`preconnect`、`preload`、`prefetch` 各自解决不同层级问题，关键是精准服务首屏关键路径。

### 高频追问

1. 什么时候优先选 `preconnect`？

当关键首屏资源来自跨域且连接成本高时，应优先用 `preconnect`，因为它能提前支付连接开销。

2. `preload` 和 `prefetch` 的本质区别？

`preload` 面向当前页面关键资源，优先级高；`prefetch` 面向未来可能用到的资源，优先级低。

3. 如何判断提示策略是否有效？

看关键资源瀑布图是否前移，并结合 LCP、首屏链路长度、未使用预加载比例做综合判断。
