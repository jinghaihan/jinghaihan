## 概念定位

资源预提示是把未来可能发生的网络动作前置到更早时机，典型手段包括 `dns-prefetch`、`preconnect`、`preload`、`prefetch`。

## 工作机制

1. 浏览器在空闲或早期阶段读取资源提示标签。
2. 按提示类型执行不同粒度的预热：DNS、建连、下载。
3. 请求真正触发时尽量复用已完成的前置结果。
4. 未被使用的提示会形成额外资源开销。

## 关键知识点

- **`dns-prefetch`**：只做 DNS。
- **`preconnect`**：DNS + TCP（HTTPS 常含 TLS）。
- **`preload`**：当前导航高优先级必需资源。
- **`prefetch`**：未来导航的低优先级预取。

## 对页面性能的影响

- 合理使用可压缩关键路径等待时间。
- 误用会占用带宽与连接，反而影响关键请求。
- 预提示收益高度依赖“命中未来真实请求”的概率。

## 观测与排障

### 浏览器侧

- Network 查看提示资源的发起时机与优先级。
- 比较加提示前后关键资源到达时间。
- 排查是否存在“预加载但未使用”的浪费。

## 常见误区

- “所有资源都应该 preload”：错误，会挤占关键请求通道。
- “preconnect 越多越好”：错误，连接本身有成本，滥用会浪费。

## 面试补充

### 30 秒说明

资源预提示是把 DNS、建连或资源下载提前做掉。`dns-prefetch`、`preconnect`、`preload`、`prefetch` 目标不同，关键是只对真正关键且高概率使用的资源做前置。

### 高频追问

1. `preload` 和 `prefetch` 最大区别是什么？

`preload` 服务当前页面关键路径，优先级高、期望尽快使用；`prefetch` 面向未来可能访问资源，优先级低且可被浏览器延后。

2. 什么时候优先用 `preconnect`？

当关键首屏资源来自跨域且建连成本明显时，优先考虑 `preconnect`，它比仅做 DNS 预热更直接。
