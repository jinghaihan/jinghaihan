## 概念定位

合成（Composite）节点负责把多个渲染层按顺序合并成最终帧，交由 GPU（Graphics Processing Unit）显示。

它是渲染链路中“最后一公里”，直接关系到动画和滚动流畅度。

## 工作机制

1. 浏览器根据绘制结果和层信息生成合成输入。
2. 合成线程按层级关系执行合成。
3. 输出最终帧并提交显示。
4. 新一轮更新重复执行。

## 关键知识点

### 合成层价值

- 把部分更新从主线程布局/绘制中解耦。
- 高频动效可在合成阶段完成，降低主线程压力。

### 常见触发层提升场景

- `transform`
- `opacity`
- 特定 `will-change` 使用

### 层管理取舍

| 策略 | 收益 | 风险 |
| --- | --- | --- |
| 合理层提升 | 提升动画流畅度 | 管理复杂度上升 |
| 过度层提升 | 局部看似流畅 | 内存占用和合成负担上升 |

## 对页面性能的影响

- 合成优化到位可显著降低掉帧。
- 过多图层会增加 GPU 内存压力和合成开销。
- 图层策略不当会出现“某些机型更卡”的问题。

## 观测与排障

### 观测方法

1. 查看合成层数量与变化频率。
2. 看动画期间主线程与合成线程负载。
3. 对比不同设备上的帧稳定性。

### 排障步骤

1. 标记高频动画区域并确认是否走合成友好属性。
2. 清理不必要 `will-change` 和无效图层提升。
3. 控制大面积图层和复杂叠层效果。
4. 回归对比 FPS 和卡顿率。

## 实战优化清单

- 高频动画优先使用 `transform/opacity`。
- `will-change` 仅在短时高频变化场景启用。
- 监控图层数量与显存占用趋势。
- 对低端机做差异化效果降级。

## 常见误区

- “图层越多越流畅”：错误，过度分层会增加合成成本。
- “用了 `will-change` 就一定更快”：错误，不当使用会带来副作用。
- “合成阶段不占性能预算”：错误，GPU/合成线程同样有预算限制。

## 面试补充

### 30 秒说明

Composite 阶段负责把图层合成为最终帧。优化目标是在不增加过多图层成本的前提下，让高频变化尽量在合成阶段完成。关键实践是用 `transform/opacity` 做动画、谨慎使用 `will-change`、并持续监控帧率和图层负载。

### 高频追问

1. 合成优化为什么对动画流畅度很关键？

因为能减少主线程布局/绘制参与，让帧生成更稳定。

2. `will-change` 的风险是什么？

长期开启会增加图层和内存占用，反而降低整体性能。

3. 如何判断图层策略是否过度？

看图层数量、显存占用和不同设备上的帧率变化。


## 补充视角（标准节点）

### 与上下游节点边界

- 上游节点（高频）：绘制（Paint）、性能监控指标体系。
- 本节点重点：聚焦“合成（Composite）”本身的机制与决策，不替代下游渲染/监控职责。
- 下游节点（高频）：后续解析或渲染节点。

### 落地动作清单

1. 先定义该节点的关键输入和关键输出，避免职责漂移。
2. 再绑定至少 1 个可观测指标，确保问题可量化定位。
3. 将该节点的异常处理写入动作库，确保值班时可执行。

### 高频追问

1. 这个节点优化应该先看什么？

先看它的关键输入是否稳定，再看关键输出是否满足下游依赖和时延预算。

2. 如何避免“优化该节点却拖慢其他节点”？

做上下游联动回归，对比关键链路分阶段指标，防止成本转移。

3. 这个节点如何进入发布门禁？

把它的核心指标加入灰度对比阈值，超阈值时触发暂停放量或回滚。
