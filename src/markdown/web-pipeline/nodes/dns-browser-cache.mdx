## 概念定位

浏览器 DNS 缓存是 DNS 链路的第一层本地命中点。若命中域名到 IP 的映射，请求可直接跳过后续 DNS 查询。

## 工作机制

1. 浏览器根据域名查找本地 DNS 缓存条目。
2. 判断 TTL（Time To Live） 是否有效。
3. 命中且未过期则直接返回 IP，进入建连。
4. 未命中则继续查系统 DNS 缓存。

## 关键知识点

- **TTL 驱动**：缓存是否可用由生存时间决定。
- **域名粒度**：不同子域名通常分别缓存。
- **预提示影响**：`dns-prefetch` 可提前触发解析。

## 对页面性能的影响

- 命中后可减少一次或多次 DNS 往返。
- 多域名页面收益更明显。
- TTL 过短会导致频繁重复解析。

## 观测与排障

### 浏览器侧

- DevTools Network 观察 `DNS Lookup` 时间是否接近 0。
- 对比首访/复访同域名资源的建连前耗时。

## 常见误区

- “清空页面缓存就等于清空 DNS 缓存”：不一定。
- “一个域名只会解析一次”：不准确，TTL 过期后会重新解析。

## 面试补充

### 30 秒说明

浏览器 DNS 缓存是最靠前的域名解析复用层。命中后可直接进入 TCP/TLS，避免重复查 DNS；是否命中主要取决于 TTL 与访问模式。

### 高频追问

1. `dns-prefetch` 实际优化的是哪一层？

它主要把“DNS 查询”前置到浏览器空闲时机，目标就是提升浏览器 DNS 缓存命中概率。

2. 为什么有时二次访问 DNS 仍然很慢？

可能 TTL 已过期、域名不同、缓存被回收，或请求走了不同网络环境导致复用失败。
