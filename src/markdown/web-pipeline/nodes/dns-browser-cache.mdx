## 概念定位

浏览器 DNS 缓存是 DNS 链路最前置命中点，负责复用近期解析结果，减少重复递归查询。

## 工作机制

1. 浏览器收到域名请求后，先查询本地 DNS 缓存。
2. 校验记录是否在 TTL（Time To Live） 有效期内。
3. 命中时直接返回 IP，进入建连阶段。
4. 未命中则继续查询系统 DNS 缓存。

## 关键知识点

- **首层命中价值高**：命中后可直接跳过后续 DNS 分层查询。
- **TTL 驱动复用**：过短 TTL 会显著增加重复解析。
- **域名粒度隔离**：子域通常独立缓存，跨域资源越多越容易失配。
- **资源提示协同**：`dns-prefetch` 可提升潜在命中概率。

## 对页面性能的影响

- 多域名首屏页面中，命中收益会被倍数放大。
- 首层未命中会把耗时传导到系统与递归链路。
- 记录波动会引发建连路径不稳定。

## 观测与排障

### 浏览器侧

- Network 观察 DNS Lookup 时长和重复域名请求行为。
- 对比首访/复访同域资源的 DNS 开销。

### 排障步骤

1. 统计首屏关键域名数量和复用情况。
2. 检查是否存在短 TTL 导致频繁重解析。
3. 优化域名收敛，减少非必要跨域首屏依赖。
4. 用 `dns-prefetch` 前置高概率域名。
5. 回归看 DNS 阶段耗时分布变化。

## 常见误区

- “浏览器缓存清了就等于 DNS 清了”：不准确，系统层可能仍命中。
- “同域资源一定只解析一次”：不准确，TTL 到期会重新解析。
- “DNS 阶段不重要”：不准确，它在建连前，慢一步会拖慢全链路。

## 面试补充

### 30 秒说明

浏览器 DNS 缓存是解析链路第一道加速层。命中即可跳过后续查询，直接进入建连。优化重点是控制域名数量、治理 TTL、用好资源提示策略。

### 高频追问

1. 为什么二次访问 DNS 仍可能慢？

TTL 到期、缓存淘汰、域名变化或网络环境切换都会导致再次解析。

2. DNS 优化第一优先级是什么？

先减少首屏关键域名数量，再做提示和缓存策略优化。

3. `dns-prefetch` 的真实收益场景？

适合“很可能访问但不确定立即请求”的域名，可把解析前置到浏览器空闲时段。
