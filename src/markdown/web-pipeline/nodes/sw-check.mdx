## 概念定位

Service Worker 拦截是请求进入浏览器默认缓存/网络链路前的可编程代理层。它可以改写请求流向、离线兜底、缓存命中策略，是性能与可用性的重要控制点。

## 工作机制

1. 浏览器判断当前作用域是否存在已激活 Service Worker。
2. 若命中作用域，触发 `fetch` 事件，交给 Service Worker 决策。
3. Service Worker 可直接返回缓存、回源请求、或自定义响应。
4. 若未命中或放行，请求进入内存/磁盘缓存与网络链路。

## 关键知识点

- **HTTPS 要求**：生产环境下 Service Worker 需安全上下文。
- **生命周期**：`install` -> `activate` -> `fetch`，生效时机需要管理。
- **缓存策略模式**：Cache First、Network First、Stale While Revalidate。
- **更新接管**：`skipWaiting` 与 `clientsClaim` 决定新版本接管速度。

## 对页面性能的影响

- 离线或弱网下，合理策略可大幅提升可用性与首屏稳定性。
- 冷启动和脚本逻辑过重会增加首请求延迟。
- 策略设计错误会出现旧资源不更新或缓存污染。

## 观测与排障

### 浏览器侧

- Application 面板检查 SW 状态（installing/waiting/active）。
- Network 查看资源是否由 Service Worker 提供。
- 检查更新后是否存在 waiting 卡住不接管问题。

### 排障步骤

1. 先核对作用域是否覆盖目标请求路径。
2. 再核对缓存策略是否与资源类型匹配。
3. 检查更新流程是否及时激活并接管。
4. 结合版本号和缓存键排查旧资源残留。
5. 回归看离线可用率、首屏成功率和命中比例。

## 常见误区

- “注册成功就会立即接管页面”：不准确，通常需下一次导航稳定生效。
- “所有资源都应缓存到 SW”：不准确，个性化数据和高变更资源需谨慎。
- “SW 只用于离线”：不准确，它也是请求调度与性能优化工具。

## 面试补充

### 30 秒说明

Service Worker 是可编程网络代理层。它能在请求到达网络前决定“命中缓存还是回源”，并提升离线与弱网体验。关键在于生命周期管理和缓存策略与资源类型匹配。

### 高频追问

1. 为什么新版本 SW 经常不立即生效？

新 Worker 常先进入 waiting 状态，只有满足接管条件才会替换旧 Worker，需要正确处理更新接管逻辑。

2. 哪种缓存策略更适合 HTML 文档？

通常 Network First 更稳，能优先拿到新内容，网络失败再回退缓存。

3. SW 性能优化最容易踩的坑是什么？

把所有请求都统一套同一种缓存策略，导致个性化数据错缓存或静态资源更新失控。
