## 概念定位

Service Worker（服务工作线程）拦截节点用于说明：请求在到达网络前，可能先经过离线代理层处理。

这会改变缓存命中路径、离线能力和刷新行为，是前端可控网络层的重要扩展点。

## 工作机制

1. 浏览器检测当前作用域是否有已激活的 Service Worker。
2. 命中后，请求进入 `fetch` 事件回调。
3. Service Worker 按策略决定：读缓存、走网络、或两者组合。
4. 返回响应后，浏览器继续后续解析与渲染流程。

## 关键知识点

### 生命周期

- `install`：安装并可预缓存核心资源。
- `activate`：接管页面并清理旧缓存。
- `fetch`：拦截请求并执行策略。

### 常见缓存策略

| 策略 | 特点 | 适用场景 | 风险 |
| --- | --- | --- | --- |
| Cache First | 快速返回缓存 | 静态资源 | 版本更新滞后 |
| Network First | 优先网络，失败回缓存 | 动态内容 | 弱网体验波动 |
| Stale While Revalidate | 先旧后新 | 可容忍短暂旧数据场景 | 一致性管理复杂 |

### 与普通缓存的区别

- 浏览器缓存是内建策略。
- Service Worker 是应用可编排策略。

## 对页面性能的影响

- 命中 SW 缓存可显著改善复访速度。
- 策略错误会导致“刷新不更新”或旧资源长期滞留。
- SW 回调逻辑过重也会引入额外延迟。

## 观测与排障

### 观测方法

1. DevTools Application 查看 SW 状态与版本。
2. Network 观察请求来源是否被 SW 拦截。
3. 检查缓存命名与版本清理是否正确。

### 排障步骤

1. 先确认当前页面是否已被 SW 接管。
2. 检查 `fetch` 策略是否符合资源类型。
3. 验证新版本激活与旧缓存清理流程。
4. 对比“禁用 SW”与“启用 SW”性能差异。

## 实战优化清单

- 按资源类型定义缓存策略，不做一刀切。
- 发布时确保 SW 版本升级与旧缓存清理联动。
- 对关键页面提供强制更新机制。
- 把 SW 命中率与错误率纳入监控。

## 常见误区

- “有 SW 就一定快”：错误，策略设计决定真实收益。
- “SW 只用于离线”：错误，它也是在线性能治理工具。
- “刷新后一定拿到最新资源”：错误，可能仍被 SW 缓存策略控制。

## 面试补充

### 30 秒说明

Service Worker 让前端拥有可编排的请求拦截层。它可通过缓存策略提升复访和离线体验，但需要严格版本管理和缓存清理，否则容易出现更新滞后和调试困难。关键是按资源类型选策略并建立监控回归。

### 高频追问

1. SW 和浏览器缓存有什么差异？

浏览器缓存是内建机制，SW 是应用层可编排拦截逻辑。

2. 为什么线上常出现“SW 导致更新不生效”？

通常是激活时机、缓存命名或旧缓存清理策略不完整。

3. 如何降低 SW 引入的风险？

先小范围灰度，明确回滚开关，并监控命中率与错误率。

4. 哪类资源最适合 Cache First？

内容稳定、版本可指纹化的静态资源。

5. SW 会影响性能排障吗？

会。排障时需要区分是网络问题还是 SW 策略命中问题。


## 补充视角（标准节点）

### 与上下游节点边界

- 上游节点（高频）：导航开始。
- 本节点重点：聚焦“Service Worker 拦截”本身的机制与决策，不替代下游渲染/监控职责。
- 下游节点（高频）：响应流接收、内存缓存检查。

### 落地动作清单

1. 先定义该节点的关键输入和关键输出，避免职责漂移。
2. 再绑定至少 1 个可观测指标，确保问题可量化定位。
3. 将该节点的异常处理写入动作库，确保值班时可执行。

### 高频追问

1. 这个节点优化应该先看什么？

先看它的关键输入是否稳定，再看关键输出是否满足下游依赖和时延预算。

2. 如何避免“优化该节点却拖慢其他节点”？

做上下游联动回归，对比关键链路分阶段指标，防止成本转移。

3. 这个节点如何进入发布门禁？

把它的核心指标加入灰度对比阈值，超阈值时触发暂停放量或回滚。
