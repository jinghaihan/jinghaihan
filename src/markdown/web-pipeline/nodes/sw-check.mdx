## 概念定位

Service Worker 是浏览器中的可编程网络代理层。该节点表示请求在进入浏览器常规缓存/网络前，先经过 SW 的 `fetch` 拦截判断。

## 工作机制

1. 浏览器判断当前页面作用域内是否存在已激活的 Service Worker。
2. 若命中作用域，触发 SW `fetch` 事件。
3. SW 可通过 `respondWith()` 返回缓存响应或自定义响应。
4. 若未拦截或未命中缓存，请求继续走内存缓存/磁盘缓存/网络分支。

## 关键知识点

- **HTTPS 要求**：SW 仅在安全上下文下可用（本地开发例外）。
- **生命周期**：`install` -> `activate` -> `fetch`，首次安装后不一定立刻接管。
- **作用域**：只拦截其 scope 范围内请求。

## 对页面性能的影响

- 命中 SW 缓存可显著缩短首包等待，离线场景也可用。
- SW 脚本逻辑过重会增加主流程延迟（尤其冷启动）。
- 错误缓存策略可能导致“旧资源长期不更新”。

## 观测与排障

### 浏览器侧

- DevTools Application -> Service Workers 检查注册、激活状态。
- Network 面板看资源来源是否来自 Service Worker。
- 观察更新流程是否卡在 `waiting` 阶段。

## 常见误区

- “注册了 SW 就会立刻拦截当前页面”：不准确，通常要下次导航才稳定接管。
- “SW 一定比浏览器缓存快”：不一定，脚本执行与序列化也有开销。

## 面试补充

### 30 秒说明

Service Worker 是请求链路上的可编程拦截层。命中时可直接返回缓存，未命中则放行到浏览器缓存和网络。它提升离线能力和可控性，但生命周期和更新策略必须设计好。

### 高频追问

1. 为什么首次访问经常看不到 SW 缓存命中？

首次访问一般还在安装/激活流程，且旧页面未被新 SW 接管，通常从下一次导航开始才稳定命中拦截。

2. `skipWaiting` 和 `clientsClaim` 解决什么问题？

用于加速新 SW 生效：`skipWaiting` 让 waiting 直接进入激活，`clientsClaim` 让激活后的 SW 立即接管当前客户端，减少“更新了但不生效”的窗口期。
