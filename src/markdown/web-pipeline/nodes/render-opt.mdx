## 概念定位

渲染阶段优化是对 `layout -> paint -> composite` 的系统治理，目标不是“某一帧变快”，而是让页面在真实交互场景下长期保持稳定帧率和可预测响应。

## 工作机制

1. 先定位瓶颈属于布局、绘制还是合成。
2. 对布局阶段做范围隔离，减少回流扩散。
3. 对绘制阶段缩小无效区域，降低重绘面积。
4. 对合成阶段合理图层化，避免过度提升层级。
5. 用指标回归验证，防止优化后回退。

## 关键知识点

- **`contain`**：限制样式/布局/绘制影响范围。
- **`content-visibility`**：推迟非视区渲染，减轻首屏压力。
- **图层化策略**：适量图层能提速，过量图层会增加内存与调度成本。
- **合成友好属性**：`transform`、`opacity` 更容易绕开 Layout/Paint 瓶颈。

## 对页面性能的影响

- 能显著改善滚动稳定性、动画流畅度和交互响应。
- 错误优化会出现“指标变好但体验变差”的副作用（闪动、占位异常）。
- 渲染优化收益在长列表、复杂组件树、低端设备上最明显。

## 观测与排障

### 浏览器侧

- Performance：看 Layout/Paint/Composite 三段占比变化。
- Rendering：开启 FPS、Paint Flashing、Layer Borders。
- 与关键交互动作绑定录制，避免只看空闲页数据。

### 实战检查清单

1. 是否存在高频布局读写交错。
2. 是否有大面积重绘区域可局部化。
3. 动画是否大量使用几何属性（`top/left/width/height`）。
4. 是否滥用 `will-change` 导致层数膨胀。
5. 回归验证 INP、LCP、FPS、Long Task 是否同步改善。

## 常见误区

- “只要加 `will-change` 就会更快”：不准确，滥用会带来显存和调度负担。
- “渲染优化只影响动画”：不准确，滚动、输入和列表交互都受益。
- “合成阶段不占成本”：不准确，图层管理和栅格化同样有预算。

## 面试补充

### 30 秒说明

渲染优化要分阶段治理：布局减扩散、绘制减面积、合成减抖动。核心是把高频交互路径里的主线程压力降下来，并用 INP、FPS、Long Task 等指标回归验证。

### 高频追问

1. 为什么很多场景推荐 `transform` 动画？

它更容易在合成阶段处理，通常能减少布局与绘制参与，主线程压力更小，动画更稳定。

2. `content-visibility` 的收益与风险是什么？

收益是减少非视区渲染成本；风险是需要正确占位与可见时机管理，否则可能出现跳动或闪现。

3. 渲染优化为什么容易“越改越复杂”？

因为它跨越布局、绘制、合成多个阶段，单点优化可能把成本转移到另一个阶段，所以必须以指标和场景回归为准。
