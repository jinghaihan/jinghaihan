## 概念定位

当浏览器 DNS 缓存、系统 DNS 缓存、`hosts` 都未命中时，请求会进入递归 DNS 解析器。它负责替客户端完成完整的 DNS 寻址过程，并返回最终可用记录。

## 工作机制

1. 客户端向递归 DNS 发起查询，请求“查到底并把结果返回我”。
2. 递归 DNS 先查自己的缓存，命中则直接返回。
3. 缓存未命中时，递归 DNS 发起迭代查询：根 DNS -> TLD DNS -> 权威 DNS。
4. 拿到 A/AAAA/CNAME 等记录后返回客户端，并按 TTL（Time To Live） 写入缓存。

## 关键知识点

- **递归查询**：调用方只关心最终结果，不关心中间跳转。
- **迭代查询**：每一跳只返回“下一步去哪问”。
- **TTL**：控制缓存生存时间，影响命中率与变更收敛速度。

## 对页面性能的影响

- DNS 在 `TCP/TLS` 建连前执行，DNS 慢会直接推迟连接建立。
- 关键域名越多，DNS 开销越容易叠加到首屏链路。
- TTL 配置不合理会导致频繁回源解析。

## 观测与排障

### 浏览器侧

- DevTools Network 中关注 `DNS Lookup` 时间。
- 对比首访与二访，判断缓存是否有效命中。

### 命令行

```sh
dig example.com
nslookup example.com
```

关注响应耗时、TTL、返回记录是否稳定。

## 常见误区

- “DNS 只影响首次访问”：不准确，TTL 短或多域场景会反复触发解析。
- “公共 DNS 一定更快”：不一定，和地域、运营商链路、时段都相关。

## 面试补充

### 30 秒说明

递归 DNS 是客户端和权威 DNS 体系之间的“代理查询层”。它先查缓存，未命中再走根到权威的迭代查询，返回最终记录并缓存。这个阶段会影响连接建立前的总等待时长。

### 高频追问

1. `dns-prefetch` 与 `preconnect` 区别是什么？

`dns-prefetch` 只提前做 DNS 解析；`preconnect` 会进一步建立 TCP，HTTPS 场景还会做 TLS 握手。`preconnect` 更重，但能显著缩短真正发请求时的建连等待，适合关键跨域资源；`dns-prefetch` 更轻，适合潜在会访问但不一定马上请求的域名。

2. TTL 如何在“命中率”和“变更生效速度”之间取舍？

TTL 越长，缓存命中率通常越高、权威 DNS 压力越小，但切流或故障切换生效更慢；TTL 越短，变更收敛更快，但解析请求会更多。常见做法是平时用中长 TTL 保命中，发布或迁移前提前把 TTL 降下来，变更完成后再升回去。

3. 为什么有时候换了 DNS 记录，用户还是访问旧地址？

因为缓存是分层的，浏览器、系统、递归 DNS、甚至中间网络设备都可能缓存旧记录。即使权威 DNS 已更新，只要某一层缓存未过期，就可能继续返回旧值。
