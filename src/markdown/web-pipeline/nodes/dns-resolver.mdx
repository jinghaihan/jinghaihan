## 概念定位

递归 DNS 解析器是客户端与 DNS 分层体系之间的代理查询层。当前置缓存（浏览器、系统、hosts）都未命中时，请求会进入这里完成完整寻址。

## 工作机制

1. 客户端把“查到底并返回结果”的请求交给递归 DNS。
2. 递归 DNS 先查本地缓存，命中则直接返回。
3. 未命中时按分层执行迭代查询：根 -> 顶级域 -> 权威 DNS。
4. 拿到 A/AAAA/CNAME 等记录后回传客户端，并按 TTL（Time To Live） 缓存。

## 关键知识点

- **递归查询与迭代查询**：客户端看递归，服务器之间看迭代。
- **TTL 治理**：影响命中率、故障切换收敛速度、解析压力。
- **CNAME 链路**：链条越长，解析跳数与失败点越多。
- **分层缓存**：浏览器、系统、递归层共同决定最终命中表现。

## 对页面性能的影响

- DNS 发生在建连前，慢解析会直接拖慢首包。
- 多域名首屏场景中，DNS 开销会叠加放大。
- TTL 设置失衡会带来“频繁解析”或“切流慢”的两难。

## 观测与排障

### 浏览器侧

- Network 关注 DNS Lookup 耗时和域名数量。
- 对比首访与复访确认缓存是否生效。

### 命令行

```sh
dig example.com
dig +trace example.com
nslookup example.com
```

### 排障步骤

1. 先确认是否是单域慢还是多域慢。
2. 检查关键域是否存在过短 TTL 或 CNAME 过长。
3. 检查递归 DNS 服务提供方与地域链路质量。
4. 验证解析结果与权威记录是否一致。
5. 回归对比 TTFB 与 DNS 阶段耗时变化。

## 常见误区

- “DNS 只影响首次访问”：不准确，TTL 过期后会持续触发。
- “公共 DNS 一定更快”：不准确，地域与运营商链路差异明显。
- “域名越多越好分流”：不准确，首屏多域常会增加解析与建连成本。

## 面试补充

### 30 秒说明

递归 DNS 是解析链路的执行核心。缓存命中时可快速返回，未命中时需完成根到权威的查询链。性能优化关键在域名治理、TTL 策略和链路质量，而不只是换 DNS 服务商。

### 高频追问

1. TTL 如何在性能和切流之间平衡？

平时用中长 TTL 保命中，变更窗口前下调 TTL，加速收敛，变更完成后再回调。

2. 为什么有时改了 DNS 用户仍访问旧地址？

因为缓存是分层的，任一层缓存未过期都可能继续返回旧记录。

3. 解析慢一定是权威 DNS 慢吗？

不一定，递归层缓存命中、地域路径、运营商链路都可能是主因。
