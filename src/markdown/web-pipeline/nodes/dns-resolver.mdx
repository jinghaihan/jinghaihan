## 概念定位

递归 DNS 解析器是“帮客户端查到底”的执行者。浏览器只提需求，递归解析器负责沿 DNS 分层体系追到最终答案。

它在 Web Pipeline 中决定了“域名到 IP”这一步的稳定性与速度。

## 工作机制

1. 客户端将查询请求交给递归 DNS。
2. 递归 DNS 先查本地缓存，命中即返回。
3. 未命中则按层级迭代：根 -> 顶级域 -> 权威 DNS。
4. 获取 A/AAAA/CNAME 等记录后回传，并按 TTL 缓存。

## 关键知识点

### 递归与迭代区别

- 客户端视角：递归查询（我问一次，你查到底）。
- DNS 服务器间：迭代查询（我给你下一跳线索）。

### 记录类型

- `A` / `AAAA`：IPv4 / IPv6 地址
- `CNAME`：别名映射
- `NS`：权威服务器信息

### TTL 治理取舍

- TTL 长：命中高、解析压力小，但变更收敛慢。
- TTL 短：收敛快、灵活，但解析流量和抖动更高。

### 安全与隐私扩展

- DoH（DNS over HTTPS）
- DoT（DNS over TLS）

这些方案在隐私和链路安全上有价值，但会带来实现和链路复杂度权衡。

## 对页面性能的影响

- DNS 在建连前，慢解析会直接拖慢首包。
- 多域名首屏会放大解析成本。
- CNAME 链过长会增加跳数与失败点。

## 观测与排障

### 观测方法

- 浏览器看 DNS Lookup 耗时和分位数。
- 命令行用 `dig`、`dig +trace`、`nslookup` 做逐层验证。
- 对比首访与复访确认缓存命中效果。

### 排障步骤

1. 先判断是单域问题还是全局解析问题。
2. 检查关键域 TTL、CNAME 链长度和变更策略。
3. 核对权威记录与递归结果是否一致。
4. 分运营商和地域比较解析时延。

## 实战优化清单

- 收敛首屏关键域名数量。
- 关键域采用稳态 TTL + 变更窗口降 TTL 策略。
- 避免不必要 CNAME 串联。
- 解析异常纳入网络性能告警。

## 常见误区

- “DNS 只影响首次访问”：错误，TTL 到期后会持续触发。
- “换公共 DNS 一定更快”：错误，地域路径差异很大。
- “只看浏览器 DNS 就够了”：错误，系统和递归层都可能是瓶颈。

## 面试补充

### 30 秒说明

递归 DNS 是域名解析链路的执行核心。客户端发一次请求，递归解析器会沿根、顶级域、权威链路迭代查到结果并缓存。优化关键是域名收敛、TTL 策略和链路质量，而不是单纯更换 DNS 服务商。

### 高频追问

1. TTL 如何平衡性能与切流速度？

稳态使用中长 TTL，变更前临时下调 TTL，变更完成后回调。

2. 为什么改 DNS 后用户还访问旧地址？

因为浏览器、系统、递归层都可能还在命中旧缓存。

3. `dig +trace` 的实战价值是什么？

可逐跳定位卡在根、顶级域还是权威层，快速缩小故障范围。

4. CNAME 链过长有什么风险？

增加解析跳数、失败点和整体不确定性。

5. DNS 优化第一步应做什么？

先减少首屏关键域数量，再优化 TTL 和解析链路。


## 补充视角（标准节点）

### 与上下游节点边界

- 上游节点（高频）：Hosts 文件匹配。
- 本节点重点：聚焦“递归 DNS 解析器”本身的机制与决策，不替代下游渲染/监控职责。
- 下游节点（高频）：根域名服务器。

### 落地动作清单

1. 先定义该节点的关键输入和关键输出，避免职责漂移。
2. 再绑定至少 1 个可观测指标，确保问题可量化定位。
3. 将该节点的异常处理写入动作库，确保值班时可执行。

### 高频追问

1. 这个节点优化应该先看什么？

先看它的关键输入是否稳定，再看关键输出是否满足下游依赖和时延预算。

2. 如何避免“优化该节点却拖慢其他节点”？

做上下游联动回归，对比关键链路分阶段指标，防止成本转移。

3. 这个节点如何进入发布门禁？

把它的核心指标加入灰度对比阈值，超阈值时触发暂停放量或回滚。
