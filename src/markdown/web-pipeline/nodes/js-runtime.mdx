## 概念定位

JavaScript 执行与调度是 `cssom-js -> forced-layout -> perf-metrics` 的汇总节点。它关注的不是语法本身，而是脚本如何占用主线程、如何影响解析与渲染时机。

## 工作机制

1. 脚本下载完成后进入解析与执行，任务进入事件循环调度。
2. 同步任务会独占主线程，异步任务进入宏任务/微任务队列。
3. 脚本读写 DOM 与样式时，会触发样式重算、布局或重绘。
4. 长任务和不合理调度会推迟输入处理与下一帧渲染。
5. 通过分块执行、延迟执行、空闲调度减少主线程拥塞。

## 关键知识点

- **事件循环（Event Loop）**：决定任务执行顺序与可打断性。
- **长任务（Long Task）**：超过 50ms 的主线程任务。
- **微任务挤占**：微任务队列过长会推迟渲染机会。
- **强制布局触发**：写样式后立刻读几何信息会触发同步布局。
- **交互优先级**：输入响应应优先于非关键计算任务。

## 对页面性能的影响

- 脚本执行策略直接决定 INP（Interaction to Next Paint）表现。
- 任务过大或调度不当会造成明显卡顿与掉帧。
- DOM 操作与样式操作交错会放大布局和绘制链路成本。

## 观测与排障

### 浏览器侧

- Performance 定位 Script Long Task 和主线程阻塞段。
- 观察输入事件到下一帧渲染的时间差。
- 结合调用栈识别热函数与高频调度点。

### 排障步骤

1. 对长任务做切片，拆成更小批次执行。
2. 非关键计算延后到空闲时段或后台任务。
3. 高频交互路径优先保证输入处理和渲染机会。
4. 合并 DOM 更新，避免读写交错触发强制布局。
5. 回归验证 INP、TBT、Long Task 数量是否下降。

## 常见误区

- “异步就一定不卡”：不准确，异步任务仍运行在主线程。
- “只要代码量小就没执行问题”：不准确，执行时机和调用频率同样关键。
- “微任务越多越快”：不准确，微任务链过长会饿死渲染。

## 面试补充

### 30 秒说明

JS 性能问题本质是主线程调度问题。要控制长任务、合理切片、优化事件循环中的任务优先级，并避免 DOM 读写交错引发布局抖动，最终用 INP 和 TBT 验证优化效果。

### 高频追问

1. 为什么 INP 能反映 JS 调度质量？

INP 衡量交互触发到下一次可见更新的延迟，主线程被脚本任务占满时，INP 会直接变差。

2. 如何处理一个必须执行但很重的计算任务？

优先拆分计算批次并让出主线程时隙，必要时转到 Worker（Web Worker）避免阻塞 UI 主线程。

3. 微任务和宏任务对渲染有什么影响？

微任务会在当前宏任务结束后立刻清空，若微任务链过长会推迟渲染时机，导致交互和动画卡顿。
