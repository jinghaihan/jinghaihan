## 概念定位

根域名服务器是 DNS 分层体系的顶层入口。它不负责返回业务域名最终 IP，而是告诉递归 DNS 下一步该去哪个顶级域服务器。

## 工作机制

1. 递归 DNS 向根服务器查询目标域名。
2. 根服务器返回对应 TLD（如 `.com`、`.cn`）的 NS 线索。
3. 递归 DNS 根据线索继续查询下一级。

## 关键知识点

- **转介（Referral）**：返回“下一跳”而不是最终记录。
- **Anycast 部署**：全球就近访问，保证高可用。
- **缓存价值高**：根和 TLD 线索通常可被长期复用。

## 对页面性能的影响

- 正常情况下根查询开销被缓存摊薄，不会每次都触发。
- 冷启动或缓存失效时会增加 DNS 查询链路长度。
- 根链路异常会放大整网解析抖动。

## 观测与排障

### 命令行

```sh
dig +trace example.com
```

通过 `+trace` 可以看到从根到权威的完整转介路径。

## 常见误区

- “根服务器保存了所有域名的 A 记录”：错误，它主要保存顶级域委派信息。
- “每次解析都要访问根服务器”：不准确，递归 DNS 会缓存中间结果。

## 面试补充

### 30 秒说明

根域名服务器是 DNS 查询的顶层路由节点，职责是返回顶级域的委派线索，而不是最终 IP。实际性能影响常被递归 DNS 缓存显著削弱。

### 高频追问

1. 为什么根服务器不是性能瓶颈？

因为大量查询结果在递归 DNS 缓存命中，且根服务采用 Anycast 分布式部署，真正直接打到根的请求比例相对可控。

2. 什么是 Glue 记录？

当返回 NS 时，如果 NS 本身也在被委派域内，响应会附带 Glue A/AAAA，避免“查 NS 的地址又要再查一次”的循环依赖。
