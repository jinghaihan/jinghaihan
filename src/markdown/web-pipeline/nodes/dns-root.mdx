## 概念定位

根域名服务器是 DNS 分层体系的顶层路由入口，职责是返回顶级域的委派线索，而不是直接给业务域名最终 IP。

## 工作机制

1. 递归 DNS 向根服务器查询目标域名。
2. 根服务器返回对应顶级域的 NS（Name Server） 线索。
3. 递归 DNS 根据线索继续向顶级域服务器查询。

## 关键知识点

- **委派机制**：根层返回“下一跳”，不返回终态业务记录。
- **Anycast 架构**：全球就近接入提升可用性与抗压能力。
- **中间结果可缓存**：根层查询结果通常被递归层复用。
- **异常放大效应**：根链路异常会影响广泛域名解析。

## 对页面性能的影响

- 冷启动或缓存失效时会增加 DNS 链路延迟。
- 正常情况下根查询成本常被缓存摊薄。
- 若根链路异常，用户会表现为“解析全面变慢”。

## 观测与排障

### 命令行

```sh
dig +trace example.com
```

### 排障步骤

1. 先确认是否为单域问题还是全域解析异常。
2. 用 `+trace` 观察是否在根层或后续层中断。
3. 检查递归 DNS 缓存是否异常失效。
4. 结合地域和运营商维度查看异常分布。

## 常见误区

- “根服务器保存所有网站 IP”：错误，它保存的是顶级域委派信息。
- “每次解析都要访问根服务器”：不准确，递归缓存会大幅减少访问。
- “根层问题对业务影响小”：不准确，影响范围常是系统级。

## 面试补充

### 30 秒说明

根域名服务器是 DNS 查询链路的顶层转发节点，负责把请求引导到正确顶级域服务器。它不直接返回业务 IP，但在冷解析链路中是关键起点。

### 高频追问

1. 为什么根层不是常态性能瓶颈？

因为递归 DNS 会缓存大量中间结果，实际直接打到根层的比例可控。

2. 根层异常时用户会看到什么现象？

常表现为多个不相关域名同时解析变慢或失败。

3. `dig +trace` 的价值是什么？

它能逐跳展示解析路径，快速定位卡在根层、顶级域还是权威层。
