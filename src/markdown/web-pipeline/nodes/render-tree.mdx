## 概念定位

构建渲染树节点把 DOM 与 CSSOM（CSS Object Model）融合为可渲染对象模型，是布局阶段的直接输入。

## 工作机制

1. 浏览器以 DOM 为结构基础，结合样式计算结果。
2. 过滤不参与渲染的节点（如 `display: none`）。
3. 生成可绘制对象并建立层级关系。
4. 将渲染树交给布局阶段计算几何信息。

## 关键知识点

- **DOM 与渲染树非一一对应**：不可见节点通常不进入渲染树。
- **样式继承与层叠**：决定每个渲染对象的最终样式。
- **失效与重建**：DOM/CSS 变化会触发局部或全局渲染树更新。
- **复杂度传导**：节点规模和样式复杂度都会向布局阶段传导成本。

## 对页面性能的影响

- 渲染树构建成本高会推迟布局和首屏绘制。
- 大范围样式失效会触发渲染树重建，造成卡顿。
- 结构不稳定会在交互中放大后续布局和绘制开销。

## 观测与排障

### 浏览器侧

- Performance 关注 Style Recalculation 与后续 Layout 耗时。
- 检查高频交互路径是否触发大范围样式失效。
- 结合调用栈定位触发渲染树重建的组件和样式点。

### 排障步骤

1. 降低样式作用范围，减少全局失效。
2. 控制动态节点插入规模和频率。
3. 优化组件结构，避免深层嵌套与重复层。
4. 回归验证 Layout/Paint 占比与 INP 变化。

## 常见误区

- “DOM 稳定就不会有渲染树成本”：不准确，样式变更同样会触发重算。
- “渲染树只在首次构建”：不准确，交互中会持续更新。
- “只优化 JS 就能解决渲染树问题”：不准确，样式策略是同等关键因素。

## 面试补充

### 30 秒说明

渲染树是 DOM 和 CSSOM 融合后的可绘制模型，它决定后续布局和绘制的输入质量。优化重点是减少样式失效范围、控制结构复杂度、降低重建频率。

### 高频追问

1. 为什么 `display: none` 和 `visibility: hidden` 影响不同？

`display: none` 通常不进入渲染树，而 `visibility: hidden` 仍参与布局，仅不显示。

2. 什么行为最容易引发渲染树大规模更新？

大范围类名切换、全局样式覆盖、深层结构批量更新都会触发更高成本重建。

3. 渲染树优化如何验证？

看交互路径下 Style Recalculation 与 Layout 时长是否下降，并结合 INP 和帧率变化。
