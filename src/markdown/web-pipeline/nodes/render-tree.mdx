## 概念定位

渲染树（Render Tree）构建节点用于把 DOM 与 CSSOM 合并成“可渲染对象集合”。

它是从“文档结构”走向“视觉输出”的关键桥梁。

## 工作机制

1. 浏览器拿到 DOM 与 CSSOM。
2. 过滤不可见节点（如 `display: none`）。
3. 计算每个可见节点的最终样式。
4. 生成渲染树，为后续布局计算提供输入。

## 关键知识点

### 渲染树与 DOM 的区别

- DOM 是文档结构树，包含所有节点。
- 渲染树只保留参与绘制的节点和样式信息。

### 样式计算开销来源

- 规则数量和复杂度
- 继承与层叠冲突
- 动态 class/style 高频变更

### 与上下游关系

- 上游依赖 `html-parser` 与 `cssom-js`。
- 下游直接进入 `layout`（布局）阶段。

## 对页面性能的影响

- 渲染树生成慢会推迟首次布局和首屏绘制。
- 样式规则复杂会放大后续重算成本。
- 高频样式变化可能导致渲染树频繁失效重建。

## 观测与排障

### 观测要点

1. 看 `Recalculate Style` 时长和频次。
2. 看样式变更是否集中在交互高频区域。
3. 看规则规模与命中效率。

### 排障步骤

1. 简化高频区域样式结构。
2. 降低全局选择器和深层选择器使用。
3. 将频繁变化样式局部化，减少全局重算。
4. 回归验证样式重算与首屏指标变化。

## 实战优化清单

- 控制样式规则规模和优先级复杂度。
- 减少跨组件的隐式样式耦合。
- 高频区域样式更新尽量局部化。
- 把样式重算时长接入监控。

## 常见误区

- “渲染树就是 DOM”：错误，渲染树是可视子集。
- “样式复杂只影响开发效率”：错误，也会影响运行时重算成本。
- “只要页面能显示就行”：错误，重算成本会在交互阶段持续放大。

## 面试补充

### 30 秒说明

渲染树是 DOM 和 CSSOM 合并后的可视结构，决定后续布局输入。优化重点是降低样式复杂度和重算频率，避免在高频交互区域触发全局样式重算，从而改善首屏和交互性能。

### 高频追问

1. 为什么 `display:none` 节点不在渲染树里？

因为它不参与绘制，不需要进入可视渲染流程。

2. 渲染树重建频繁通常由什么引起？

高频样式修改、复杂选择器和全局样式联动。

3. 如何判断样式问题是否拖慢渲染树阶段？

看 `Recalculate Style` 的时长、触发次数和调用栈位置。


## 补充视角（标准节点）

### 与上下游节点边界

- 上游节点（高频）：HTML 解析与 DOM、CSSOM 构建与脚本执行。
- 本节点重点：聚焦“构建渲染树”本身的机制与决策，不替代下游渲染/监控职责。
- 下游节点（高频）：布局计算（Layout）。

### 落地动作清单

1. 先定义该节点的关键输入和关键输出，避免职责漂移。
2. 再绑定至少 1 个可观测指标，确保问题可量化定位。
3. 将该节点的异常处理写入动作库，确保值班时可执行。

### 高频追问

1. 这个节点优化应该先看什么？

先看它的关键输入是否稳定，再看关键输出是否满足下游依赖和时延预算。

2. 如何避免“优化该节点却拖慢其他节点”？

做上下游联动回归，对比关键链路分阶段指标，防止成本转移。

3. 这个节点如何进入发布门禁？

把它的核心指标加入灰度对比阈值，超阈值时触发暂停放量或回滚。
