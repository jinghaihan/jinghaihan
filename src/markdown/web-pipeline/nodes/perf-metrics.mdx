## 概念定位

性能监控指标体系是贯穿网络链路、渲染链路、交互链路的汇总节点。目标是把“哪里慢、慢多少、为什么慢”映射到可观测指标上，并可持续回归。

## 工作机制

1. 在网络阶段采集 DNS、连接、首包、下载等耗时。
2. 在渲染阶段采集首屏与核心体验指标（CWV）。
3. 在运行时采集长任务、主线程占用、帧率等交互指标。
4. 在服务端与 CDN 侧补充命中率、回源率、接口耗时。
5. 通过统一面板和告警阈值形成闭环优化。

## 关键知识点

- **TTFB（Time To First Byte）**：请求发出到收到首字节的时间。
- **CWV（Core Web Vitals）**：LCP（Largest Contentful Paint）、INP（Interaction to Next Paint）、CLS（Cumulative Layout Shift）。
- **TBT（Total Blocking Time）**：主线程阻塞总时长。
- **Long Task（长任务）**：单次任务时长超过 50ms 的主线程任务。
- **P75/P95**：分位数比均值更能反映真实用户体验分布。

## 对页面性能的影响

- 只有建立指标体系，优化才能从“感觉快”变成“可量化快”。
- 单看实验室数据容易失真，必须结合真实用户监控（RUM）。
- 无阈值与告警时，性能回退通常会在上线后才被动发现。

## 观测与排障

### 建议的最小指标集

1. 网络链路：DNS、TCP、TLS、TTFB、下载耗时。
2. 渲染体验：FCP（First Contentful Paint）、LCP、CLS、INP。
3. 运行时：Long Task 数量、TBT、FPS（Frames Per Second）。
4. 缓存与服务：CDN 命中率、304 比例、回源率、接口 P95。

### 排障方法

- 先看分层指标：网络慢、服务慢、渲染慢、交互慢分别定位。
- 再看细分人群：地区、设备、网络类型、版本对比。
- 最后做变更关联：上线前后指标变化和错误日志联动。

## 常见误区

- “只看 Lighthouse 分数就够了”：不准确，实验室数据不能替代真实流量。
- “均值变好了就代表整体变好”：不准确，长尾用户常藏在 P95/P99。
- “指标越多越好”：不准确，先保证最小指标集稳定、可解释、可行动。

## 面试补充

### 30 秒说明

性能监控不是单一指标，而是分层体系：网络阶段看 TTFB 与连接耗时，渲染阶段看 CWV，运行时看 Long Task 与 TBT，服务侧看命中率和 P95。核心是建立“采集-告警-定位-回归”的闭环。

### 高频追问

1. 为什么要同时看 FCP 和 LCP？

FCP 反映“页面开始有内容”，LCP 反映“主要内容可见”。两者一起看能区分“看起来有东西”和“核心内容真的到了”。

2. INP 为什么比旧的 FID 更有价值？

INP 覆盖整个页面生命周期中的交互延迟，不只看首次输入，能更真实反映复杂页面的交互体验。

3. 监控体系落地第一步做什么？

先定义最小指标集和阈值，再打通前端 RUM、后端日志、CDN 统计，确保每个指标都能追到具体责任链路。
