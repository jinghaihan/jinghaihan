## 概念定位

`CSSOM 构建与脚本执行` 是渲染前最容易形成瓶颈的复合阶段。这里既有样式系统（CSS 解析、样式计算），也有脚本系统（JS 执行、DOM 改写），二者互相影响。

## 工作机制

1. 浏览器下载样式表并解析成 CSSOM（CSS Object Model）。
2. 主线程执行 JavaScript，可能创建/修改 DOM，也可能读取样式信息。
3. 当脚本依赖样式计算结果时，脚本执行会等待 CSSOM 关键部分完成。
4. 脚本执行产生的 DOM/CSS 变化会触发样式重算（Style Recalculation）。
5. 当 DOM 与 CSSOM 状态稳定后，进入渲染树构建与布局阶段。

## 关键知识点

- **样式重算（Style Recalculation）**：选择器匹配和层叠计算的成本中心。
- **长任务（Long Task）**：单次主线程任务超过 50ms，会拉高交互延迟。
- **主线程阻塞**：脚本、样式、布局共享主线程预算，会互相抢占。
- **执行顺序治理**：脚本加载策略和 CSS 拆分策略共同决定首屏速度。

## 对页面性能的影响

- 大体积 CSS 与重 JS 叠加时，常见首屏慢与交互卡顿同时出现。
- 主线程长任务会拖高 TBT（Total Blocking Time）并恶化 INP。
- 不合理选择器和频繁 DOM 改写会导致样式重算放大。

## 观测与排障

### 浏览器侧

- Performance 看 `Recalculate Style`、`Script`、`Layout` 的时间占比。
- Coverage 检查首屏未使用 CSS/JS 体积，识别可拆分部分。
- Long Task 视图定位超过 50ms 的主线程热点函数。

### 排障步骤

1. 首先拆分首屏关键 CSS，减少首屏必须解析的样式体积。
2. 把非关键脚本后移，避免在首屏窗口抢占主线程。
3. 优化选择器复杂度，控制大范围样式失效。
4. 合并 DOM 批量更新，降低样式重算触发频率。
5. 回归关注 TBT、INP、LCP 是否同步下降。

## 常见误区

- “CSS 不占主线程”：不准确，CSS 解析和样式计算都在渲染主线程预算里。
- “只要打包体积小就不会卡”：不准确，执行时机和执行方式同样关键。
- “框架会自动解决所有样式性能问题”：不准确，业务层读写和样式策略仍是主要变量。

## 面试补充

### 30 秒说明

CSSOM 与脚本执行是首屏和交互性能的共同瓶颈。CSS 解析、样式重算、脚本执行都在主线程竞争。优化核心是关键资源前置、非关键后移、减少样式失效和长任务。

### 高频追问

1. 为什么会出现“改了一点 DOM 却很卡”？

如果改动触发了大范围选择器重新匹配或样式失效，就会触发高成本样式重算，再连带布局和绘制，卡顿会被放大。

2. 如何判断瓶颈在 CSS 还是 JS？

看 Performance 分段占比：`Script` 高说明执行热点偏 JS；`Recalculate Style` 高说明样式重算是主因；两者都高说明是复合瓶颈。

3. 哪个指标最能反映这段链路的用户体感？

通常看 INP 与 TBT。INP 体现交互延迟，TBT 体现主线程阻塞总量，两者一起看更准确。
