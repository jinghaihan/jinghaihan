## 概念定位

该节点表示样式系统与脚本系统并行推进但相互制约：浏览器一边构建 CSSOM，一边执行 JS，并协调它们对渲染的影响。

## 工作机制

1. 浏览器下载并解析 CSS，生成 CSSOM。
2. JS 引擎执行脚本，可能读写 DOM/CSSOM。
3. 若脚本依赖样式计算结果，浏览器需等待 CSSOM 就绪。
4. CSSOM 与脚本状态共同决定渲染树能否继续构建。

## 关键知识点

- **CSS 不是完全“非阻塞”**：会间接阻塞脚本与渲染推进。
- **长任务影响大**：重 JS 执行会推迟渲染主线程时隙。
- **关键 CSS 优先**：减少首屏阻塞。

## 对页面性能的影响

- 大体积 CSS 与重脚本叠加会拉长首屏关键路径。
- 不必要的同步脚本会放大主线程拥塞。
- 首屏 CSS 拆分不当会引发白屏时间变长。

## 观测与排障

### 浏览器侧

- Performance 看 Main Thread 的长任务分布。
- Coverage 检查首屏未使用 CSS/JS 体积。
- 看渲染前是否出现明显 Script 阻塞段。

## 常见误区

- “CSS 不会阻塞任何东西”：不准确，在脚本依赖样式时会产生阻塞链。
- “脚本异步化后就没有渲染问题”：不准确，异步脚本执行时仍会占主线程。

## 面试补充

### 30 秒说明

CSSOM 构建与脚本执行是渲染前最常见的瓶颈组合。CSS 与 JS 不只是并行关系，还会互相等待。优化核心是控制关键 CSS 体积和脚本长任务。

### 高频追问

1. 为什么说 CSS 会“阻塞 JS”？

当脚本需要读取样式信息时，浏览器必须先确保 CSSOM 可用，否则脚本会拿到不正确结果，因此会延迟脚本执行。

2. 首屏阶段怎么平衡 CSS 与 JS？

先保证关键 CSS 尽快可用，再把非关键脚本延后或拆分，避免在首屏窗口内占满主线程。
