## 概念定位

CSSOM（CSS Object Model）构建与脚本执行节点关注的是：样式系统和 JavaScript 在主线程上的相互等待关系。

很多“明明 JS 不大却卡首屏”的问题，根因不是下载量，而是 JS 与 CSSOM 时序耦合。

## 工作机制

1. 浏览器下载并解析 CSS，构建 CSSOM。
2. 浏览器执行脚本时，如果脚本需要读取样式/布局信息，需等待 CSSOM 就绪。
3. CSSOM 与 DOM 准备完成后，进入渲染树构建和后续布局。
4. 若脚本频繁触发样式读取与写入，后续会放大重排重绘成本。

## 关键知识点

### CSSOM 为什么重要

- DOM 决定“有什么”，CSSOM 决定“怎么显示”。
- 缺失 CSSOM 时，很多样式相关 API 无法给出稳定结果。
- CSSOM 不是单纯样式文本，而是可计算、可匹配、可继承的规则系统。

### JS 与 CSSOM 的阻塞关系

- JS 如果只做纯计算，不一定等待 CSSOM。
- JS 一旦读取 `getComputedStyle`、布局尺寸等信息，浏览器需要确保样式计算有效。
- 这会形成“脚本等待样式 -> 样式下载解析完成 -> 脚本继续”的链路。

### 常见高成本组合

- 大型全量 CSS + 首屏同步脚本
- 动态注入样式 + 立即读取布局信息
- 大量 class 切换 + 同步查询几何属性

### CSS 复杂度对 JS 的反作用

- 选择器复杂、规则多、层叠关系复杂会提高样式计算成本。
- JS 每次触发样式相关读取，可能都要付出更高重新计算代价。

## 对页面性能的影响

- 首屏阶段：CSSOM 慢会拖慢首屏渲染和脚本推进。
- 交互阶段：JS 与样式读写交错会造成卡顿和掉帧。
- 弱设备下：样式计算成本更容易成为瓶颈，INP（Interaction to Next Paint）恶化明显。

## 观测与排障

### 浏览器侧观测

1. Performance 查看 `Recalculate Style`、`Evaluate Script` 占比。
2. 检查首屏脚本是否在样式就绪前被触发。
3. 排查是否存在频繁 `getComputedStyle` 与几何读取。

### 排障剧本

1. 拆出首屏关键 CSS，减少样式可用前等待。
2. 首屏脚本按依赖分级，避免无意义同步执行。
3. 把“读样式”和“写样式”分批，避免交错触发。
4. 对复杂样式区域做隔离（减少全局重算范围）。
5. 回归验证样式重算时长与 INP 分位数。

## 实战优化策略

### 样式侧

- 控制首屏样式体积，避免一次性加载大量非关键规则。
- 减少深层复杂选择器，优先可预测的类选择器组织。
- 避免过多 `@import` 串联样式文件。

### 脚本侧

- 首屏脚本仅做关键初始化。
- 样式读取集中到单一读阶段，样式写入集中到单一写阶段。
- 高频动画更新优先用 transform/opacity，减少触发布局。

### 工程侧

- 建立样式重算与脚本执行的联合指标。
- PR 阶段对首屏样式体积和同步脚本数做门禁。
- 第三方样式/脚本按优先级分层加载。

## 常见误区

- “CSS 只影响视觉，不影响脚本性能”：错误，脚本读取样式时直接受 CSSOM 影响。
- “JS 卡顿一定是 JS 太重”：错误，样式重算和布局同步也会占用主线程。
- “把 CSS 拆得越碎越好”：错误，碎片化会增加请求和管理复杂度，需要按关键路径拆分。

## 面试补充

### 30 秒说明

CSSOM 与 JS 在主线程存在强耦合。脚本一旦读取样式和布局信息，就可能等待 CSSOM 并触发样式重算。优化核心是控制首屏样式体积、拆分脚本执行时机、统一读写批次，避免样式与脚本互相阻塞。

### 高频追问

1. 为什么脚本会等待 CSSOM？

因为脚本读取样式/布局 API 时，浏览器必须保证样式计算结果有效。

2. 如何减少 `Recalculate Style` 成本？

简化选择器、限制影响范围、减少高频样式变更、批量处理读写。

3. CSSOM 慢会影响哪些指标？

首屏常影响 FCP/LCP，交互期常影响 INP 与帧稳定性。

4. 如何判断瓶颈在样式还是脚本？

看火焰图中 `Recalculate Style` 与 `Evaluate Script` 的占比及触发顺序。

5. 为什么高频读取 `offsetWidth` 会卡？

它可能触发布局同步计算，若与写操作交错，成本会持续放大。

6. CSS 架构对运行时性能有影响吗？

有。规则组织和选择器复杂度会影响匹配与重算成本，最终反映到交互延迟。


## 进阶补充（深挖）

### 与上下游节点的因果关系

- 上游输入（常见）：响应流接收、解析阻塞治理、强制布局与回流治理。
- 本节点职责：将“CSSOM 构建与脚本执行”的结果转化为可执行的后续分支决策。
- 下游影响（常见）：构建渲染树。

### 线上故障案例

- 现象：交互场景中样式重算激增，页面出现明显卡顿。
- 根因：脚本高频读取样式/布局信息，与样式写入交错触发重算。
- 处置：读写分离、缓存几何数据、降低高频区域样式联动范围。
- 回归标准：关键指标恢复到基线区间，且连续观测窗口无二次抖动。

### 高频追问）

1. 为什么 getComputedStyle 会拖慢交互？

在样式未稳定时读取会触发同步样式计算。

2. 如何降低 Recalculate Style 占比？

减少复杂选择器、降低全局联动、合并样式变更。

3. CSS 架构是否影响运行时？

会，规则组织复杂度直接影响匹配和重算成本。


### 补充记忆框架

- 先判断该节点是“前置等待”问题还是“运行时抖动”问题。
- 再判断问题来自“资源策略”还是“执行调度”。
- 最后把定位结论转为可复用规则，进入门禁与动作库。

### 高频追问

1. 这个节点优化后如何做回归验收？

至少做两组对比：关键指标分位数对比 + 上下游链路稳定性对比，避免局部优化导致成本转移。

2. 为什么这个节点要单独做发布观察窗口？

因为它对主链路有放大效应，短时间异常会快速传导到首屏和交互体验。
