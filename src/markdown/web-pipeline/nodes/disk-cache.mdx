## 概念定位

磁盘缓存是浏览器对 HTTP 资源的持久化缓存层。内存缓存未命中后，请求会在这里尝试复用本地副本。

## 工作机制

1. 依据缓存键在磁盘索引中定位历史响应。
2. 若资源仍新鲜，直接返回本地副本（强缓存）。
3. 若资源过期但可协商，带条件头发起验证（`If-None-Match` / `If-Modified-Since`）。
4. 服务器返回 `304` 时复用本地副本，返回 `200` 时更新缓存。

## 关键知识点

- **强缓存**：有效期内不发请求。
- **协商缓存**：过期后发条件请求，可能返回 `304`。
- **验证优先级**：通常 `ETag` 精度高于 `Last-Modified`。

## 对页面性能的影响

- 强缓存命中能显著减少网络流量与等待。
- 协商缓存虽省带宽，但仍有一次往返成本。
- 错误缓存头会导致“缓存不生效”或“资源过旧”。

## 观测与排障

### 浏览器侧

- Network 观察 `Status`（`200` / `304`）与资源来源。
- 检查响应头中的 `Cache-Control`、`ETag`、`Last-Modified`。
- 对比普通刷新与强刷行为差异。

## 常见误区

- “304 说明没有任何网络开销”：不准确，仍有请求往返。
- “设置了缓存头就一定命中”：不准确，还受 Vary、URL 指纹、刷新方式影响。

## 面试补充

### 30 秒说明

磁盘缓存是持久化缓存层。先走强缓存判断，过期后走协商缓存。协商命中返回 304 并复用本地副本，未命中返回 200 并覆盖旧缓存。

### 高频追问

1. `Cache-Control` 和 `Expires` 应该怎么选？

优先使用 `Cache-Control`，语义更清晰、控制更细。`Expires` 受客户端时间影响大，通常作为兼容补充。

2. `ETag` 和 `Last-Modified` 应该怎么配合？

可同时使用；通常以 `ETag` 作为更精确校验，`Last-Modified` 作为兜底与兼容信号。
