## 概念定位

状态码决策链节点用于回答一个核心问题：服务端返回了什么状态码，浏览器接下来会做什么。

它不是“响应结果标签”，而是请求链路的分流开关：

- 是否继续下载响应体
- 是否复用本地缓存
- 是否重定向到新地址
- 是否触发鉴权、限流、降级、重试

## 工作机制

1. 浏览器发起请求，携带条件头（如 `If-None-Match`、`If-Modified-Since`）或范围头（`Range`）。
2. CDN / 网关 / 源站根据资源状态与策略返回状态码和响应头。
3. 浏览器根据状态码决定后续动作：复用缓存、处理重定向、下载全部或部分内容。
4. 运行时再把结果映射到渲染路径、错误处理、重试策略与监控上报。

## 关键知识点

### 常见状态码全景（前端高频）

- **1xx 信息状态**：`100`、`101`、`103`
- **2xx 成功状态**：`200`、`201`、`202`、`204`、`206`
- **3xx 重定向与缓存**：`301`、`302`、`303`、`304`、`307`、`308`
- **4xx 客户端问题**：`400`、`401`、`403`、`404`、`405`、`408`、`409`、`410`、`412`、`413`、`414`、`415`、`422`、`429`、`431`
- **5xx 服务端问题**：`500`、`501`、`502`、`503`、`504`

### 状态码与浏览器动作对照

| 状态码 | 常见含义 | 浏览器/前端常见动作 |
| --- | --- | --- |
| `200` | 完整成功 | 读取并解析响应体 |
| `204` | 成功但无体 | 不解析 body，常用于更新确认 |
| `206` | 部分内容 | 分段下载/续传 |
| `301/308` | 永久重定向 | 更新跳转路径，可能被缓存 |
| `302/303/307` | 临时重定向 | 跟随跳转，语义依方法而定 |
| `304` | 协商命中 | 复用本地副本，通常不下实体 |
| `401` | 未认证 | 进入登录/刷新 token 流程 |
| `403` | 无权限 | 显示权限不足或降级 UI |
| `404` | 资源不存在 | 展示兜底页，检查路由/发布清单 |
| `429` | 限流 | 延迟重试，遵循 `Retry-After` |
| `5xx` | 服务异常 | 熔断、降级、告警、回滚 |

### 缓存链路里的关键状态码

- `200`：完整响应，可能携带新的缓存策略。
- `304`：协商命中，省流量但不省 RTT（Round-Trip Time）。
- `206`：范围请求命中，适合大文件、媒体和断点续传。
- `412`：前提条件失败，常见于条件写冲突控制。

### 重定向语义差异

- `301/308` 都是永久重定向，`308` 更严格保持方法和请求体。
- `302` 在历史兼容中常被“改写方法”处理。
- `307` 明确要求不改写方法和请求体，适合语义敏感链路。

## 对页面性能的影响

- 重定向链会拉长关键请求链，直接推迟 FCP（First Contentful Paint）和 LCP（Largest Contentful Paint）。
- `304` 比例高不代表快，弱网下往返仍贵。
- `206` 可降低单次等待，但会引入分段调度与并发管理复杂度。
- `429` 与 `5xx` 波动会触发重试风暴，放大拥塞与卡顿。

## 观测与排障

### 浏览器与日志侧最小检查集

1. Network 看关键资源状态码分布与耗时。
2. 按资源类型统计 `200/304/206` 比例。
3. 监控重定向跳数（含跨域跳转）。
4. 统计 `4xx/5xx` 分位数与峰值时段。

### 状态码到动作映射（排障剧本）

1. `301/302/307/308` 升高：检查跳转链是否冗余、是否循环。
2. `304` 高但页面慢：检查 RTT、并发队列、连接复用质量。
3. `401` 升高：检查 token 过期策略、刷新并发控制、Cookie 传递。
4. `403` 升高：检查权限模型、网关 ACL、跨域凭证配置。
5. `404/410` 升高：检查静态资源发布清单、路由映射、灰度一致性。
6. `429` 升高：检查限流阈值、重试间隔、客户端突发请求模式。
7. `502/503/504` 升高：分层定位网关、应用、下游依赖与超时配置。

### 线上故障示例

- **现象**：首屏变慢，关键资源大多 `304`。
- **定位**：协商命中但 RTT 高，且连接未复用导致往返叠加。
- **处理**：静态资源改长缓存 + 指纹；关键域做连接预热；降低协商请求频次。

## 实战策略

### 资源分层策略

- HTML：短缓存或协商，确保可控更新。
- 指纹静态资源：强缓存长 TTL（Time To Live）。
- 大文件/媒体：合理启用 `206` 分段。
- API：按幂等性与一致性设计状态码语义，不混用错误码。

### 错误与重试策略

- `429`：遵循 `Retry-After`，指数退避，限制并发重试。
- `5xx`：小流量重试 + 快速失败 + 熔断降级。
- `401`：单飞刷新（single flight）避免并发刷新风暴。

### 发布治理策略

- 发布门禁检查：重定向跳数、`404` 峰值、`5xx` 分位数。
- 灰度期间对比新旧版本状态码分布。
- 超阈值自动回滚并锁定版本。

## 常见误区

- “`304` 越高越好”：错误，往返时延仍是成本。
- “`200` 一定慢”：错误，小资源直接 `200` 可能更稳。
- “`4xx` 都是前端问题”：错误，鉴权与网关策略常是主因。
- “`5xx` 都是代码 bug”：错误，依赖超时、网关配置、容量抖动也会导致。
- “重定向只是 SEO 问题”：错误，它直接吃掉性能预算。

## 面试补充

### 30 秒说明

状态码决定请求分支。`200` 是完整返回，`304` 是协商复用，`206` 是分段传输。优化时要把状态码策略与资源类型、缓存策略、重试策略绑定起来，并通过真实网络指标验证收益，不能只看某个状态码占比。

### 高频追问

1. 为什么 `304` 命中后页面仍可能慢？

因为协商请求仍有 RTT 和队列等待，弱网或连接复用差时成本依旧高。

2. `307` 与 `302` 的关键差异是什么？

`307` 保证请求方法和请求体不改写；`302` 在历史兼容中可能发生方法改写。

3. 何时使用 `206` 最有价值？

大文件、媒体流和断点续传场景，能降低单次等待并提升可恢复性。

4. `401` 和 `403` 在前端处理上如何区分？

`401` 偏向“先认证再访问”，`403` 偏向“已认证但无权限”，处理路径不同。

5. `429` 到来时重试策略怎么设计？

先读 `Retry-After`，采用指数退避与抖动，限制并发，避免放大雪崩。

6. 状态码治理如何进入发布流程？

把重定向跳数、`404` 和 `5xx` 分位数作为门禁，灰度阶段对比新旧分布并设置回滚阈值。


## 进阶补充（深挖）

### 与上下游节点的因果关系

- 上游输入（常见）：源站返回响应。
- 本节点职责：将“状态码决策链（200/206/304 与常见状态码）”的结果转化为可执行的后续分支决策。
- 下游影响（常见）：响应流接收、响应流接收、磁盘缓存检查。

### 线上故障案例

- 现象：关键静态资源 `304` 占比高，但页面首屏仍明显变慢。
- 根因：协商请求往返成本高 + 连接复用不足，导致网络排队放大。
- 处置：将稳定资源切换为长缓存 + 指纹，减少协商请求数量，并检查关键域连接预热。
- 回归标准：关键指标恢复到基线区间，且连续观测窗口无二次抖动。

### 高频追问）

1. 为什么不把所有资源都改成长缓存？

因为入口 HTML 与动态资源需要可控更新，否则会出现版本错配。

2. 状态码治理如何进入发布门禁？

对重定向跳数、4xx/5xx 分位数、关键资源 304 比例设置阈值并自动阻断。

3. 429 出现时前端优先做什么？

遵循 Retry-After、指数退避和并发限流，避免重试风暴。
