## 概念定位

权威 DNS 服务器是域名记录的最终来源，负责返回业务域的真实资源记录（A/AAAA/CNAME 等）。

## 工作机制

1. 递归 DNS 向权威服务器请求目标主机记录。
2. 权威服务器返回记录集及 TTL。
3. 递归 DNS 缓存后回传给客户端。
4. 客户端据此进入建连与请求阶段。

## 关键知识点

- **权威性**：权威层是记录真实值来源。
- **记录结构**：A/AAAA/CNAME 组合影响建连路径和稳定性。
- **TTL 策略**：决定命中率、切换速度与解析压力。
- **多记录负载分配**：返回记录顺序和策略会影响流量分布。

## 对页面性能的影响

- 权威响应慢会推高冷解析耗时。
- CNAME 过长会增加查询跳数和失败点。
- TTL 失衡会在“性能稳定”和“切换灵活”之间失衡。

## 观测与排障

### 命令行

```sh
dig @<authoritative-ns> www.example.com
```

### 排障步骤

1. 先直查权威确认“源事实”是否正确。
2. 再对比递归结果，判断是否缓存延迟造成偏差。
3. 检查 CNAME 链路长度与跨域依赖。
4. 按业务场景调整 TTL（平稳期与切换期分离）。

## 常见误区

- “权威 DNS 一定最快”：不准确，权威是正确性来源，不是最短路径。
- “只改 A 记录就够了”：不准确，可能还受 CNAME、证书和回源策略影响。
- “TTL 越短越好”：不准确，会显著增加解析频率和不稳定性。

## 面试补充

### 30 秒说明

权威 DNS 是解析链路中的最终事实源。性能与稳定性的关键在记录设计和 TTL 策略：平时保命中，切换窗口保收敛。

### 高频追问

1. 为什么权威正确但用户仍访问旧地址？

通常是递归或客户端缓存未过期，仍在复用旧记录。

2. CNAME 设计对性能有什么影响？

链路越长，查询跳数越多，冷解析时延和失败点越高。

3. TTL 怎么做分阶段策略？

平稳期中长 TTL，变更前下调 TTL，完成后再升回，兼顾稳定和收敛。
