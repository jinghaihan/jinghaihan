## 概念定位

权威 DNS 服务器是域名最终记录的“事实来源”。该节点负责返回域名对应的 A/AAAA/CNAME 等实际可用记录。

## 工作机制

1. 递归 DNS 带着目标主机名请求权威服务器。
2. 权威服务器按 zone 数据返回最终记录或 CNAME 链。
3. 返回记录附带 TTL（Time To Live），供递归层和客户端缓存。
4. 递归 DNS 把结果回传给发起方。

## 关键知识点

- **权威性**：该域记录以权威服务器数据为准。
- **记录类型**：A、AAAA、CNAME、TXT、MX 等。
- **TTL 策略**：直接影响切流和故障恢复速度。

## 对页面性能的影响

- 权威服务器响应慢会直接拉长冷解析耗时。
- CNAME 链过长会叠加查询跳数。
- TTL 过短会增加查询频率，过长会降低变更灵活性。

## 观测与排障

### 命令行

```sh
dig @<authoritative-ns> www.example.com
```

直接查权威可快速判断“源事实”和“递归缓存”是否不一致。

## 常见误区

- “权威 DNS 一定最快”：不一定，权威是正确性来源，不代表网络路径最短。
- “只改 A 记录就够了”：不一定，CNAME 链路和证书域名也要匹配。

## 面试补充

### 30 秒说明

权威 DNS 服务器是域名记录的最终来源。递归 DNS 最终要到这里拿到 A/AAAA/CNAME 等记录，并按 TTL 缓存后返回客户端。

### 高频追问

1. CNAME 链过长有什么影响？

每增加一层都可能增加解析查询与失败点，最终体现在建连前延迟和稳定性风险上。

2. TTL 在故障切换里怎么用？

平时可用中长 TTL 保命中；切流或迁移前提前下调 TTL，变更完成后再回调，平衡性能与收敛速度。
