## 概念定位

内存缓存检查是浏览器缓存链路中最快的一层。它在请求触发后优先尝试从进程内存直接复用响应副本，目标是极低延迟返回。

## 工作机制

1. 浏览器按请求键匹配可复用响应（URL、方法、Vary 维度等）。
2. 判断响应是否仍在可用生命周期内。
3. 命中且新鲜时直接返回副本，不进入磁盘和网络。
4. 未命中或失效则继续下探到磁盘缓存或后续链路。

## 关键知识点

- **近零延迟**：省去磁盘 IO（Input/Output）和网络往返。
- **生命周期短**：受页面、进程、内存回收策略影响。
- **可用性受策略约束**：并非所有资源都可驻留内存缓存。
- **与 BFCache 区别**：内存缓存是资源级，BFCache 是页面快照级。

## 对页面性能的影响

- 复访或重复请求场景下可显著降低等待。
- 内存压力波动会导致命中率不稳定。
- 大资源滥用可能挤占可用内存，影响整体性能稳定性。

## 观测与排障

### 浏览器侧

- Network 关注资源来源是否显示 memory cache。
- 对比同会话首次与二次请求耗时。
- 结合 Performance 看主线程是否仍被后续解析阻塞。

### 排障步骤

1. 先确认响应头是否允许合理缓存。
2. 观察是否存在频繁进程重启或页面重建。
3. 检查资源体积是否过大导致缓存淘汰。
4. 回归看重复访问路径下的请求耗时分布。

## 常见误区

- “命中内存缓存就没有性能问题”：不准确，后续解析和渲染仍可能慢。
- “清除磁盘缓存就等于清内存缓存”：不总是等价。
- “内存缓存命中率越高越好”：不绝对，要平衡内存占用和稳定性。

## 面试补充

### 30 秒说明

内存缓存是请求链路里最快的复用层。命中后可以跳过磁盘和网络，显著降低等待。但它受生命周期和内存压力影响，不能替代完整缓存策略设计。

### 高频追问

1. 为什么同页面里二次请求经常极快？

因为命中了进程内的响应副本，省去磁盘与网络阶段。

2. 什么时候会突然失去内存缓存收益？

页面重建、进程回收、内存压力或缓存策略变化都会导致命中率下降。

3. 如何把内存缓存收益稳定下来？

核心是资源分层与缓存头治理，避免大资源污染缓存空间，并保持请求键稳定。
