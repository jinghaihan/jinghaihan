## 概念定位

解析阻塞治理是一个跨阶段汇总专题，核心目标是把“HTML 解析、样式计算、脚本执行”之间的互相等待链路拆开，减少关键渲染路径上的硬阻塞。

## 工作机制

1. 浏览器在 `response-stream` 阶段边收边解析，进入 `html-parser`。
2. 解析器遇到脚本、样式、资源提示时，会触发不同阻塞行为。
3. `cssom-js` 节点中，脚本可能等待样式就绪，样式也可能推迟脚本可执行时机。
4. 通过脚本加载策略、关键 CSS 策略、预加载策略，缩短“可渲染内容出现前”的等待链。

## 关键知识点

- **解析阻塞（Parsing Blocking）**：HTML 解析线程被同步脚本或依赖链暂停。
- **预加载扫描器（Preload Scanner）**：在主解析外并行发现资源引用，提前发请求。
- **关键渲染路径（Critical Rendering Path）**：从拿到字节到首屏可见的最短必要链路。
- **首屏阻塞资源（Render Blocking Resource）**：未就绪会直接推迟首屏绘制的资源。

## 对页面性能的影响

- 同步脚本过多会直接推迟首次内容绘制时间。
- 关键 CSS 体积过大，会把脚本和渲染一起压在主线程前半段。
- 预加载配置不当（漏配或误配）会导致关键资源晚到。
- 阻塞链路一旦叠加高延迟网络，用户侧体感会被放大。

## 观测与排障

### 浏览器侧

- Performance 面板定位 `Parse HTML` 与 `Evaluate Script` 的交错关系。
- Network 观察关键 CSS/JS 的到达顺序，确认是否按预期前置。
- Lighthouse 查看“移除阻塞渲染资源”与“关键请求链”提示。

### 排障步骤

1. 先确认首屏依赖清单：首屏到底必须哪些 CSS/JS。
2. 把非关键脚本改为 `defer` 或按路由延迟加载。
3. 把首屏关键 CSS 内联或拆小，非关键样式后置加载。
4. 用 `preload` 精准提前关键字体/样式/首屏脚本，避免全量滥用。
5. 回看 LCP（Largest Contentful Paint）与首屏瀑布图，验证收益。

## 常见误区

- “把所有脚本都加 `async` 就能提速”：不准确，`async` 会打乱执行顺序，依赖脚本会出错。
- “CSS 不会阻塞渲染”：不准确，关键 CSS 未就绪时首屏通常无法稳定渲染。
- “只要有 `preload` 就一定更快”：不准确，错配会抢占带宽，反而拖慢关键资源。

## 面试补充

### 30 秒说明

解析阻塞治理的本质是拆解关键渲染路径上的等待链。重点在三件事：脚本加载策略（`defer/async`）、关键 CSS 控制、`preload` 精准前置。目标是让首屏可见内容尽早出现。

### 高频追问

1. `defer` 和 `async` 在阻塞治理上的本质区别是什么？

`defer` 会并行下载、按顺序执行，通常在 HTML 解析结束后执行，适合有依赖关系的脚本；`async` 下载完成即执行，顺序不保证，适合独立脚本。

2. 为什么“CSS 会阻塞脚本执行”这个说法成立？

当脚本读取样式信息时，浏览器必须保证样式计算结果可靠，因此会等待相关样式资源与 CSSOM（CSS Object Model）完成后再执行脚本。

3. 阻塞治理最小可行落地方案是什么？

先清理首屏关键资源清单，再做三步：非关键脚本 `defer` 化、关键 CSS 拆分前置、关键字体/首屏脚本 `preload`，最后用指标回归验证。
