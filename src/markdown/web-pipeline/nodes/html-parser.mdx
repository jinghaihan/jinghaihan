## 概念定位

该节点是浏览器把 HTML 字节流转换为 DOM 树的过程，是渲染链路中的结构构建阶段。

## 工作机制

1. 词法/语法解析 HTML token。
2. 按规范进行树构建，逐步生成 DOM 节点。
3. 解析过程中可并行进行预加载扫描。
4. 遇到阻塞脚本时可能暂停主解析流程。

## 关键知识点

- **DOM 构建是增量过程**：不是一次性完成。
- **脚本阻塞**：同步脚本可能暂停 HTML 解析。
- **文档复杂度**：DOM 节点规模会影响后续渲染成本。

## 对页面性能的影响

- 过大的 HTML 与深层 DOM 会拉长解析时间。
- 阻塞脚本放在关键位置会推迟首屏可见内容。
- 结构不合理会放大后续布局与重绘开销。

## 观测与排障

### 浏览器侧

- Performance 面板查看 Parse HTML 与 Script 执行交替。
- Coverage 检查首屏阶段是否加载了过多无用脚本。

## 常见误区

- “HTML 解析只和网络速度有关”：不准确，文档复杂度和脚本同样关键。
- “把 script 放到底部就万事大吉”：不绝对，仍应结合 `defer/async` 与依赖关系。

## 面试补充

### 30 秒说明

HTML 解析与 DOM 构建是渲染的结构基础。浏览器边收边解析并生成 DOM，但同步脚本会阻塞解析，因此脚本加载策略直接影响首屏速度。

### 高频追问

1. `async` 和 `defer` 对 DOM 解析有什么差异？

`async` 下载完成就执行，可能打断解析；`defer` 会延迟到 HTML 解析完成后按顺序执行，更适合有依赖关系的脚本。

2. 为什么“减少 DOM 节点数”是有效优化？

DOM 规模会影响解析、样式计算、布局和重绘多个阶段，节点越多，整条渲染链路成本越高。
