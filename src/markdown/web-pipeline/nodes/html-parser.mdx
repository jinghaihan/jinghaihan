## 概念定位

`HTML 解析与 DOM` 是浏览器把响应字节流转换成文档结构树的阶段。它是渲染链路里最早的结构化步骤，后续样式计算、布局、绘制都依赖这棵树。

## 工作机制

1. 浏览器读取字节流并做词法拆分，产生 token（标签、文本、属性等）。
2. 解析器根据 HTML 规范执行树构建算法，逐步生成 DOM（Document Object Model）。
3. 解析过程中会并行触发预加载扫描器，提前发现 CSS、JS、字体、图片等外部资源。
4. 当遇到同步脚本时，主解析会暂停，等待脚本下载并执行后继续。
5. 解析完成后，DOM 进入后续样式合并与渲染树构建阶段。

## 关键知识点

- **DOM 增量构建**：不是一次性生成，而是随着字节流持续生长。
- **同步脚本阻塞**：未加 `defer` / `async` 的脚本会阻塞解析推进。
- **预解析能力**：浏览器可提前发现外链资源，但不改变主 DOM 构建顺序。
- **脚本与样式依赖**：脚本读取样式信息时，可能等待 CSSOM（CSS Object Model）就绪。

## 对页面性能的影响

- HTML 体积、DOM 规模、同步脚本密度会直接影响首屏时间。
- 首屏关键结构到达越晚，FCP（First Contentful Paint）和 LCP（Largest Contentful Paint）越容易恶化。
- 解析阶段若主线程被脚本长任务占满，会连带拖慢后续样式和布局阶段。

## 观测与排障

### 浏览器侧

- Performance 看 `Parse HTML` 与 `Evaluate Script` 的交替关系。
- Network 检查关键 CSS/JS 的下载顺序，确认是否前置到首屏窗口。
- Lighthouse 看“阻塞渲染资源”和“关键请求链”提示。

### 排障步骤

1. 列清首屏必需资源：先知道哪些 CSS/JS 真正阻塞首屏。
2. 非关键脚本改为 `defer`，无依赖脚本再评估 `async`。
3. 拆分关键 CSS，避免整站样式把首屏解析拖慢。
4. 用 `preload` 前置首屏关键样式或字体，验证是否改善 LCP。
5. 回归检查 FCP/LCP/INP（Interaction to Next Paint）是否同步改善。

## 常见误区

- “HTML 解析只和网速有关”：不准确，脚本阻塞与文档复杂度同样重要。
- “script 放底部就一定最优”：不绝对，现代策略应以 `defer/async` 和依赖关系为准。
- “预加载扫描器能解决所有阻塞”：不准确，它只能提前发现资源，不能消除错误依赖链。

## 面试补充

### 30 秒说明

HTML 解析阶段负责把字节流转成 DOM。它本身是增量过程，但同步脚本会打断解析。性能优化重点是减少首屏阻塞资源、合理使用 `defer/async`、把关键 CSS 和关键资源前置。

### 高频追问

1. `defer` 和 `async` 对解析阻塞的差异是什么？

`defer` 会并行下载并在 HTML 解析完成后按顺序执行，适合有依赖脚本；`async` 下载完成就执行，顺序不保证，可能打断解析，适合独立脚本。

2. 为什么有时脚本会“被 CSS 卡住”？

当脚本依赖样式计算结果时，浏览器需要先确保 CSSOM 可用，所以会等待相关样式加载并构建后再执行脚本。

3. DOM 节点数量为什么会影响首屏？

节点规模会影响解析、样式计算、布局和绘制多个阶段，节点越多，关键路径总成本越高。
