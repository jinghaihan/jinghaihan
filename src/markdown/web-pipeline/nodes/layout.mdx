## 概念定位

布局计算阶段负责为渲染树中的每个可见对象确定几何信息：位置、尺寸、盒模型关系。

## 工作机制

1. 浏览器基于渲染树计算各节点几何约束。
2. 结合文档流、定位、弹性/网格规则完成布局。
3. 生成每个渲染对象的最终几何结果。
4. 结果传递给 Paint 阶段生成绘制指令。

## 关键知识点

- **回流（Reflow）**：会触发布局重算。
- **强制同步布局**：读写交错可能立即触发布局。
- **布局作用域**：全局布局和局部布局成本差异大。

## 对页面性能的影响

- 高频触发布局会显著占用主线程。
- 读写 DOM 混用容易导致布局抖动（layout thrashing）。
- 大列表、复杂网格在低端设备上成本更高。

## 观测与排障

### 浏览器侧

- Performance 查看 Layout 事件时长与调用栈。
- 关注是否存在连续的 style read/write 交替。

## 常见误区

- “只改一个元素样式就不会触发布局”：不一定，可能影响祖先/兄弟布局。
- “`transform` 一定触发布局”：多数情况下不会，它更偏向合成阶段处理。

## 面试补充

### 30 秒说明

Layout 是把“要渲染什么”变成“渲染到哪里、多大”的阶段。性能优化重点是减少不必要回流，避免读写交错导致强制同步布局。

### 高频追问

1. 哪类操作最容易触发回流？

修改影响几何的属性（宽高、定位、字体、盒模型）或频繁读取布局信息（如 `offsetHeight`）后立刻写样式，最容易触发回流。

2. 如何避免 layout thrashing？

把读操作和写操作分批处理，合并 DOM 更新，必要时用 `requestAnimationFrame` 统一调度。
