## 概念定位

布局（Layout）节点负责计算渲染树中每个可视对象的几何信息：位置、尺寸、层级关系。

这是“样式已知”到“像素可绘制”之间的几何求解阶段。

## 工作机制

1. 浏览器根据渲染树和视口尺寸计算盒模型。
2. 按格式化上下文确定元素位置与尺寸。
3. 产出布局结果供绘制阶段消费。
4. 若后续样式/结构变化，可能触发重新布局。

## 关键知识点

### 布局触发来源

- DOM 结构变化
- 尺寸相关样式变化（宽高、边距、字体）
- 视口变化（窗口 resize）

### 全局布局与局部布局

- 全局布局影响范围大、成本高。
- 局部布局可控性强，适合高频交互场景。

### 与强制布局关系

运行时脚本读写交错会触发强制同步布局，显著增加交互延迟。

## 对页面性能的影响

- 布局阶段成本高会直接影响首屏与交互流畅度。
- 高频布局变更会造成掉帧和滚动卡顿。
- 布局范围越大，INP（Interaction to Next Paint）越容易恶化。

## 观测与排障

### 观测方法

1. Performance 看 `Layout` 事件时长与次数。
2. 定位高频交互中的布局触发点。
3. 分析是否存在全局范围连锁更新。

### 排障步骤

1. 确认是否存在布局抖动（读写交错）。
2. 将频繁更新区域局部化，减少影响范围。
3. 减少高频修改尺寸类属性。
4. 回归验证 INP、帧率与 Layout 总时长。

## 实战优化清单

- 高频动画优先 `transform/opacity`。
- 控制复杂布局嵌套与深层联动。
- 滚动/resize 监听加节流。
- 把布局次数和时长纳入交互监控。

## 常见误区

- “布局只在首屏发生一次”：错误，运行时交互会持续触发布局。
- “改一点样式不会有影响”：错误，可能触发范围很大的连锁布局。
- “掉帧一定是绘制慢”：错误，布局阶段也常是主因。

## 面试补充

### 30 秒说明

布局阶段负责计算元素几何信息，是渲染链中的高成本环节。优化要点是减少不必要布局、限制影响范围、避免读写交错触发强制同步布局，并通过 Layout 事件和 INP 指标持续验证。

### 高频追问

1. 布局和重绘的区别是什么？

布局计算几何信息，重绘负责像素填充，布局通常成本更高。

2. 为什么 `transform` 常用于动效优化？

它通常不触发布局，可减少几何重算成本。

3. 如何定位布局范围过大问题？

看火焰图里 Layout 的调用路径和影响节点范围。


## 补充视角（标准节点）

### 与上下游节点边界

- 上游节点（高频）：构建渲染树、渲染阶段优化、强制布局与回流治理。
- 本节点重点：聚焦“布局计算（Layout）”本身的机制与决策，不替代下游渲染/监控职责。
- 下游节点（高频）：绘制（Paint）。

### 落地动作清单

1. 先定义该节点的关键输入和关键输出，避免职责漂移。
2. 再绑定至少 1 个可观测指标，确保问题可量化定位。
3. 将该节点的异常处理写入动作库，确保值班时可执行。

### 高频追问

1. 这个节点优化应该先看什么？

先看它的关键输入是否稳定，再看关键输出是否满足下游依赖和时延预算。

2. 如何避免“优化该节点却拖慢其他节点”？

做上下游联动回归，对比关键链路分阶段指标，防止成本转移。

3. 这个节点如何进入发布门禁？

把它的核心指标加入灰度对比阈值，超阈值时触发暂停放量或回滚。
