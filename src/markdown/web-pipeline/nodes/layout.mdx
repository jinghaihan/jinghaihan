## 概念定位

布局计算（Layout）阶段负责把渲染对象转成几何结果：位置、尺寸、盒模型关系。它决定元素“画到哪里”和“占多大空间”。

## 工作机制

1. 浏览器基于渲染树和样式结果计算几何约束。
2. 结合文档流、定位、弹性布局、网格布局等规则生成最终几何信息。
3. 布局结果输出给 Paint 阶段形成绘制指令。
4. 后续 DOM 或样式变化若影响几何，可能再次触发布局。

## 关键知识点

- **回流（Reflow）**：几何变化触发布局重算。
- **强制同步布局（Forced Synchronous Layout）**：写样式后立即读布局导致立刻重算。
- **布局抖动（Layout Thrashing）**：读写交错反复触发布局。
- **局部与全局布局**：影响范围不同，成本差异很大。

## 对页面性能的影响

- 高频回流会占用主线程，拉低交互流畅度。
- 大列表和复杂嵌套布局中，回流扩散成本更高。
- 布局不稳定会影响 CLS（Cumulative Layout Shift）表现。

## 观测与排障

### 浏览器侧

- Performance 查看 `Layout` 事件次数、总时长和调用栈。
- 重点定位交互路径里是否出现密集布局事件簇。
- 配合 Paint Flashing 判断是否伴随大面积重绘。

### 排障步骤

1. 梳理触发布局的代码路径，识别高频读写交错点。
2. 批量处理 DOM 更新，分离“先读后写”的执行节奏。
3. 高频动画优先走 `transform/opacity`，避开布局链路。
4. 用 `contain` 与 `content-visibility` 控制布局影响范围。
5. 回归验证 INP 与 CLS 是否改善。

## 常见误区

- “回流只影响当前元素”：不准确，可能连带祖先和兄弟节点。
- “重绘比回流更重”：不绝对，通常回流成本更高，但要看范围。
- “只要减少 JS 体积就能解决布局卡顿”：不准确，执行节奏和样式策略同样关键。

## 面试补充

### 30 秒说明

Layout 是渲染链路中把样式结果变成几何结果的关键步骤。性能问题常来自频繁回流和强制同步布局。核心治理是读写分离、批量更新、限制影响范围、动画尽量走合成阶段。

### 高频追问

1. 哪些读取会触发布局计算？

在样式变更后读取 `offsetWidth`、`offsetHeight`、`scrollTop`、`getBoundingClientRect()` 等，浏览器可能被迫同步刷新布局。

2. 为什么布局问题会影响交互响应？

布局计算占用主线程时，输入事件处理和下一帧渲染都会排队，直接拉高交互延迟。

3. CLS 和布局阶段有什么关系？

CLS 反映可视区域元素的非预期位移，布局不稳定和资源晚到引发的重排是 CLS 变差的常见根因。
