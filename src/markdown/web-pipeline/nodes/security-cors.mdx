## 概念定位

同源策略（Same-Origin Policy, SOP）是浏览器安全模型的基础。它限制脚本读取跨源资源，防止站点间任意窃取数据。跨源资源共享（Cross-Origin Resource Sharing, CORS）是在这套限制下的“白名单放行机制”，不是绕过安全，而是有条件授权。

这个节点的重点不是“怎么让请求发出去”，而是“请求能不能被前端读取、会不会触发预检、如何兼顾性能和安全”。

## 工作机制

### 源（Origin）判定

同源要求三元组完全一致：

- 协议（Protocol）
- 域名（Host）
- 端口（Port）

三者任一不同就是跨源。跨源并不等于请求一定失败，常见情况是“请求发出去了，但响应被浏览器拦读”。

### 简单请求路径

满足以下条件通常走简单请求，不触发预检：

- 方法是 `GET`、`HEAD`、`POST`
- 自定义头较少，且头部满足简单头约束
- `Content-Type` 属于简单类型（如 `text/plain`、`application/x-www-form-urlencoded`、`multipart/form-data`）

流程：

1. 浏览器直接发送请求，自动带 `Origin`。
2. 服务端返回 `Access-Control-Allow-Origin` 等响应头。
3. 浏览器校验通过则脚本可读响应；校验失败则控制台报 CORS 错误。

### 预检请求路径

非简单请求先发预检（Preflight）：

1. 浏览器先发 `OPTIONS`。
2. 请求里带：
- `Origin`
- `Access-Control-Request-Method`
- `Access-Control-Request-Headers`
3. 服务端返回是否允许：
- `Access-Control-Allow-Origin`
- `Access-Control-Allow-Methods`
- `Access-Control-Allow-Headers`
- 可选 `Access-Control-Max-Age`
4. 预检通过后，浏览器再发真正业务请求。

### 凭证请求路径（Cookie / Session）

跨域携带凭证需要同时满足：

- 前端显式允许携带凭证（如 `fetch(..., { credentials: 'include' })`）
- 响应头 `Access-Control-Allow-Credentials: true`
- `Access-Control-Allow-Origin` 不能是 `*`，必须是明确源
- Cookie 自身策略允许跨站发送（如 `SameSite=None; Secure`）

任一环节不满足，都可能出现“接口 200，但前端拿不到数据 / Cookie 不生效”。

## 关键知识点

### CORS 头部矩阵

| 阶段 | 关键头 | 作用 |
| --- | --- | --- |
| 请求 | `Origin` | 声明请求来源 |
| 预检请求 | `Access-Control-Request-Method` | 告知将要使用的方法 |
| 预检请求 | `Access-Control-Request-Headers` | 告知将要发送的自定义头 |
| 预检响应 | `Access-Control-Allow-Origin` | 允许的来源 |
| 预检响应 | `Access-Control-Allow-Methods` | 允许的方法 |
| 预检响应 | `Access-Control-Allow-Headers` | 允许的头 |
| 预检响应 | `Access-Control-Max-Age` | 预检结果缓存时间 |
| 正式响应 | `Access-Control-Allow-Origin` | 再次声明来源许可 |
| 正式响应 | `Access-Control-Allow-Credentials` | 是否允许凭证 |
| 正式响应 | `Access-Control-Expose-Headers` | 允许前端读取的额外响应头 |

### 同源策略限制的边界

同源策略核心限制的是“读取能力”，不是“发送能力”。

- 图片、脚本、样式等资源很多情况下可跨域加载。
- 但脚本读取跨域接口响应，需要 CORS 许可。
- 这也是“Network 看到请求成功，业务代码却报错”的根源。

### CORS 与 CSRF 的关系

跨站请求伪造（Cross-Site Request Forgery, CSRF）与 CORS 是两类问题：

- CORS 解决“能不能读响应”。
- CSRF 解决“请求是不是用户真实意图”。

常见误区是“开了 CORS 就防 CSRF”。这是错误的。CSRF 防护仍要靠 Token、`SameSite`、Origin/Referer 校验等机制。

### 代理并不等于安全

开发环境常用本地代理规避跨域；生产环境仍应做服务端最小授权。

- 代理能减少前端跨域复杂度。
- 但安全边界仍在网关和服务端，不在浏览器。

## 对页面性能的影响

- 预检会引入额外往返（RTT），高延迟网络下影响明显。
- 不必要的自定义头会把简单请求升级为预检请求。
- 跨域域名过多会叠加 DNS、建连、TLS、预检开销。
- `Access-Control-Max-Age` 配置合理可显著减少重复预检。

## 观测与排障

### 浏览器侧检查顺序

1. 先看 Console 报错类型：
- `No 'Access-Control-Allow-Origin' header`
- `Credentials flag is true but 'Access-Control-Allow-Origin' is '*'`
- `Request header field xxx is not allowed by Access-Control-Allow-Headers`
2. 再看 Network：有没有 `OPTIONS`，预检状态码多少，响应头是否完整。
3. 最后对比成功请求与失败请求的 `Origin`、方法、请求头差异。

### 网关与 CDN 常见坑

- 只在源站加头，网关层没透传，导致线上头部丢失。
- CDN 命中缓存但没 `Vary: Origin`，把 A 站响应错发给 B 站。
- 预检 `OPTIONS` 被网关拦截或返回 405。
- 多环境域名白名单不一致（测试可用，生产失败）。

### 排障剧本

1. 先判定失败发生在预检还是正式请求。
2. 若预检失败，逐项核对 `Allow-Origin/Methods/Headers`。
3. 若凭证失败，核对三件套：`credentials`、`Allow-Credentials`、非 `*` 的 `Allow-Origin`。
4. 若 Cookie 未带上，核对 Cookie 属性（`SameSite`、`Secure`、`Domain`、`Path`）。
5. 若仅线上失败，优先检查网关/CDN 是否覆盖头部。
6. 复测时记录 HAR，对比改动前后预检次数与耗时。

## 实战配置建议

### 最小授权原则

- 不要默认 `Access-Control-Allow-Origin: *`。
- 按环境、业务域、接口分组配置白名单。
- 对敏感接口限制方法与头部，不做“全开”。

### 降低预检成本

- 能用简单请求就不用复杂请求。
- 收敛自定义头，避免无意义 `X-*` 头。
- 合理配置 `Access-Control-Max-Age`，降低重复预检。
- 合并小请求，减少跨域请求总数。

### 安全协同策略

建议把 CORS 与以下策略协同：

- 内容安全策略（Content Security Policy, CSP）
- 跨源资源策略（Cross-Origin Resource Policy, CORP）
- 跨源隔离（Cross-Origin-Embedder-Policy, COEP / Cross-Origin-Opener-Policy, COOP）
- Fetch Metadata 头（`Sec-Fetch-Site` 等）

## 常见误区

- “服务端返回 200 就代表跨域没问题”：错误，浏览器可能仍拦截读取。
- “把 `Access-Control-Allow-Origin` 配成 `*` 最省事”：凭证场景下不可用，且风险高。
- “跨域问题只在前端改”：多数线上问题在网关/服务端白名单和透传链路。
- “CORS 就是安全策略全貌”：它只覆盖跨源读取控制，不等于完整 Web 安全体系。

## 面试补充

### 30 秒说明

同源策略限制的是脚本跨源读取能力，CORS 是在此基础上的受控放行。判断跨域问题要先分“请求是否发出”“响应是否可读”。优化上关注预检次数和缓存，安全上坚持最小授权与凭证约束，避免一把梭放开所有源。

### 高频追问

1. 为什么接口状态码是 200，前端仍报 CORS 错误？

因为浏览器会在拿到响应后再做 CORS 校验。校验失败时，响应对脚本不可见，即使网络层是 200。

2. 什么时候会触发预检请求？

当请求不满足简单请求条件，比如方法是 `PUT/DELETE`，或带了非简单头，或 `Content-Type` 不是简单类型。

3. 为什么跨域带 Cookie 经常失败？

通常是三件套不一致：前端没开凭证、服务端没 `Allow-Credentials`、`Allow-Origin` 配成了 `*`，或 Cookie `SameSite` 配置不允许跨站发送。

4. 如何减少 OPTIONS 预检数量？

收敛方法和请求头、尽量保持简单请求、配置 `Access-Control-Max-Age`、减少无意义跨域调用。

5. CORS 能防 CSRF 吗？

不能。CORS 管“读响应”，CSRF 管“伪造请求”。防 CSRF 要用 Token、`SameSite`、Origin/Referer 校验等手段。

6. 线上 CORS 偶发失败怎么定位？

重点看网关/CDN 是否覆盖或丢失 CORS 头，是否缺失 `Vary: Origin`，以及多机房/多环境白名单是否一致。
