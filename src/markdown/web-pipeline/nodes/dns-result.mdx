## 概念定位

该节点表示 DNS 解析链路收敛完成：客户端获得目标 IP（或记录集），并将结果写入缓存，准备进入 TCP/TLS 建连。

## 工作机制

1. 递归 DNS 返回最终记录给客户端。
2. 浏览器/系统按 TTL（Time To Live） 写入各自缓存层。
3. 客户端根据策略选择可用地址（IPv4/IPv6、多 A 记录等）。
4. 把选定地址交给传输层开始建连。

## 关键知识点

- **多记录返回**：一个域名可能对应多个 IP。
- **缓存回填**：同一次解析结果会在多层缓存中复用。
- **地址选择策略**：受协议栈与网络环境影响。

## 对页面性能的影响

- 解析结果稳定时后续访问会显著受益于缓存命中。
- 不合理的地址分配可能导致连到“远端机房”而变慢。
- 频繁切换 IP 可能引入连接抖动与失败重试。

## 观测与排障

### 浏览器侧

- 观察同域名资源是否频繁出现 DNS Lookup。
- 关注连接阶段失败是否集中在某个返回 IP。

### 命令行

```sh
dig www.example.com
```

查看返回记录、TTL 与是否多 IP。

## 常见误区

- “一个域名只会返回一个 IP”：不准确，常见负载均衡会返回记录集。
- “拿到 IP 就一定能连上”：不一定，还要经过 TCP/TLS 与网络可达性验证。

## 面试补充

### 30 秒说明

DNS 结果返回节点是解析阶段的收口。客户端拿到最终记录后会写入缓存，并选择地址进入 TCP/TLS 建连，这一步直接连接了解析与传输阶段。

### 高频追问

1. 为什么同域名有时连得快有时慢？

可能是同域返回多个 IP，客户端每次命中的地址路径不同，导致 RTT（Round-Trip Time） 和丢包率差异。

2. TTL 过短会带来什么问题？

解析请求频率上升、缓存命中下降，最终会增加建连前等待和递归 DNS 压力。
