## 概念定位

HTML 加载与语义策略是 `nav-start -> response-stream -> html-parser` 链路的汇总节点，目标是同时优化解析效率、首屏呈现和可访问性语义质量。

## 工作机制

1. 先设计首屏结构优先级：哪些区域必须最先进入 HTML。
2. 通过语义标签组织文档结构，减少无效嵌套和复杂度。
3. 控制阻塞脚本与阻塞样式在文档中的位置和加载策略。
4. 配置资源提示，让关键 HTML 相关资源尽早到达。
5. 对可访问性和 SEO（Search Engine Optimization）结构做一致治理。

## 关键知识点

- **语义化标签**：`header/main/section/article/nav/footer` 帮助结构清晰和可访问性。
- **关键内容前置**：首屏骨架和主要内容应尽量靠前输出。
- **阻塞控制**：同步脚本和关键样式是解析链路主要阻塞源。
- **可访问性映射**：语义结构影响读屏器路径和交互体验。
- **结构稳定性**：避免首屏后期大范围插入导致布局跳动。

## 对页面性能的影响

- 文档结构越清晰，解析和渲染链越稳定。
- 首屏关键内容输出顺序直接影响 FCP（First Contentful Paint）/LCP（Largest Contentful Paint）。
- 结构不稳定会放大 CLS（Cumulative Layout Shift）风险。

## 观测与排障

### 关注指标

- 首屏结构到达时间（首屏 HTML 是否早到）。
- FCP、LCP、CLS 的发布前后变化。
- 文档节点规模、嵌套深度和关键资源阻塞链。

### 排障步骤

1. 先看首屏区域是否被延后输出。
2. 检查脚本是否在关键结构前阻塞解析。
3. 检查关键样式是否过大或过晚加载。
4. 检查首屏后是否有大块 DOM 注入导致跳动。
5. 回归验证 LCP、CLS 与可访问性检查结果。

## 常见误区

- “语义化只影响 SEO，不影响性能”：不准确，结构质量也影响解析与渲染效率。
- “HTML 先随便输出，后面再修”：不准确，首屏内容顺序一旦错误，后续链路都会被拖慢。
- “骨架屏等于性能优化”：不准确，骨架只改善感知，不等于真实链路变短。

## 面试补充

### 30 秒说明

HTML 策略要同时解决三件事：首屏结构前置、解析阻塞控制、语义可访问性。把关键内容与关键资源组织好，能直接改善 FCP/LCP/CLS，并提升维护性。

### 高频追问

1. 为什么首屏 HTML 顺序很重要？

浏览器是流式解析，关键内容越早到达越早可见，顺序错误会把可见时间整体后移。

2. 语义化与性能如何关联？

语义化通常伴随更清晰结构和更低复杂度，有助于减少无效节点与后续渲染负担。

3. 如何避免首屏后期大跳动？

提前预留尺寸、控制异步内容插入策略，并确保首屏关键布局信息尽早确定。
