## 概念定位

该节点表示浏览器在连接可用后，正式把请求报文发送给服务端，进入应用层传输阶段。

## 工作机制

1. 组装请求行、请求头、Cookie、请求体。
2. 根据协议版本选择传输方式（如 HTTP/2 多路复用）。
3. 请求发出后等待服务端响应头与响应体。
4. 若连接可复用，请求可能与其他资源并发传输。

## 关键知识点

- **方法与幂等性**：GET/POST 等语义影响缓存与重试策略。
- **头部大小**：Cookie 过大时会增加每次请求负担。
- **优先级与复用**：HTTP/2 下多个请求共享连接。

## 对页面性能的影响

- 请求头冗余会放大上行开销。
- 请求调度不合理会阻塞关键资源先到达。
- 高并发下不合理接口拆分会增加链路竞争。

## 观测与排障

### 浏览器侧

- DevTools Headers 查看请求头体积与关键字段。
- Network 瀑布图查看请求队列与阻塞关系。

## 常见误区

- “GET 一定比 POST 快”：不绝对，关键看负载、缓存和链路。
- “HTTP/2 下就没有调度问题”：不准确，资源优先级仍很重要。

## 面试补充

### 30 秒说明

发送 HTTP 请求是应用层正式出站阶段。浏览器会组装方法、头、体并按协议传输。性能上要关注请求数量、头部冗余和关键资源优先级。

### 高频追问

1. 为什么要控制 Cookie 大小？

同域请求会重复携带 Cookie，Cookie 过大等于每次都增加带宽和延迟，尤其在高并发资源请求时影响明显。

2. HTTP/2 多路复用已经很好了，还要做资源拆分吗？

仍要按关键路径做拆分与优先级管理。多路复用解决的是连接层瓶颈，不会自动解决应用层依赖顺序。
