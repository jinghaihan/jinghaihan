## 概念定位

发送 HTTP 请求是连接阶段之后的应用层入口。它把业务意图封装为协议报文并进入传输链路，是后续服务端响应和前端渲染的前提。

## 工作机制

1. 浏览器组装请求行、请求头、Cookie、请求体。
2. 根据协商结果选择 HTTP/1.1、HTTP/2 或 HTTP/3 传输路径。
3. 请求在连接上发送，等待服务端返回首字节与响应头。
4. 若连接可复用，请求可能与其他资源并发复用通道。

## 关键知识点

- **请求头治理**：冗余头（尤其大 Cookie）会放大每次请求成本。
- **协议差异**：HTTP/2 多路复用改善队列瓶颈，但不替代资源优先级治理。
- **幂等与重试**：请求方法语义影响超时重试和容错策略。
- **优先级调度**：关键资源请求应优先于非关键请求。

## 对页面性能的影响

- 请求数量和请求体积会直接影响总等待时间。
- 关键请求排队会拖慢首屏和主内容可见时间。
- 请求调度错误会造成网络带宽与主线程处理双拥塞。

## 观测与排障

### 浏览器侧

- Network 瀑布图看关键请求是否被非关键请求阻塞。
- Headers 检查请求头体积和重复字段。
- Performance 联动看请求返回后是否被主线程长任务延迟消费。

### 排障步骤

1. 先识别关键请求链（文档、关键 CSS、关键 JS、首屏数据）。
2. 降低非关键请求优先级或延后时机。
3. 裁剪 Cookie 和冗余头字段。
4. 控制接口扇出，减少首屏并发风暴。
5. 回归验证 TTFB、LCP 与首屏请求链长度。

## 常见误区

- “HTTP/2 下不用关注请求顺序”：不准确，业务优先级仍关键。
- “请求并发越高越快”：不准确，过高并发会放大竞争和拥塞。
- “只优化接口响应时间就够”：不准确，请求发起和调度同样影响体感。

## 面试补充

### 30 秒说明

发送请求阶段的优化关键是“少而准”：减少不必要请求、压缩头和体积、保证关键请求优先级。HTTP/2 改善了连接利用率，但不替代应用层调度策略。

### 高频追问

1. 为什么要控制 Cookie 大小？

同域请求会反复携带 Cookie，头部变大意味着每次请求都有额外成本，首屏并发场景影响更明显。

2. 如何判断是请求调度问题而不是服务端慢？

若关键请求在客户端排队很久才发出，多是调度问题；若发出后首包慢，再排查服务端链路。

3. 请求优化优先级怎么排？

先保关键路径请求，再减冗余请求，再做细粒度并发和优先级调优。
