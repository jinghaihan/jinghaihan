## 概念定位

发送 HTTP 请求节点描述的是：浏览器在连接就绪后，如何组织请求并把它送入服务端链路。

这一步影响的不只是“能不能拿到数据”，还影响优先级调度、缓存命中、跨域策略和后续渲染时序。

## 工作机制

1. 浏览器组装请求行、请求头、请求体。
2. 根据缓存和跨域策略附加条件头与安全头。
3. 在可用连接上调度请求发送（复用或新建连接）。
4. 服务端返回响应后进入状态码分支与响应流处理。

## 关键知识点

### 请求组成

- 方法（GET/POST/PUT/DELETE...）
- URL（路径与查询参数）
- 请求头（鉴权、缓存、压缩、内容协商）
- 请求体（JSON、FormData、二进制）

### 请求调度维度

- 连接是否复用
- 请求优先级（关键资源 vs 非关键资源）
- 是否触发预检（CORS 复杂请求）
- 是否命中条件请求路径（`If-None-Match` 等）

### 前端常见策略差异

| 维度 | 方案 A | 方案 B | 影响 |
| --- | --- | --- | --- |
| 鉴权传递 | Cookie | Header Token | 跨域、缓存、网关策略不同 |
| 数据提交 | GET 参数 | POST Body | 可缓存性、语义与长度限制不同 |
| 重试策略 | 立即重试 | 退避重试 | 稳定性和限流风险不同 |

## 对页面性能的影响

- 请求头冗余会增加传输成本。
- 非关键请求抢占连接会拖慢关键资源到达。
- 不合理重试会放大拥塞和失败。

## 观测与排障

### 观测要点

1. 看关键请求是否在首屏窗口优先发出。
2. 看请求头体积和重复字段。
3. 看错误重试是否集中在短时间爆发。

### 排障步骤

1. 区分关键请求与次级请求，确认优先级是否正确。
2. 精简请求头，减少无效字段传输。
3. 检查跨域请求是否引入额外预检。
4. 优化重试策略，避免瞬时重试风暴。

## 实战优化清单

- 关键首屏请求应优先并尽量复用连接。
- 对接口做幂等标记，支持安全重试。
- 统一错误码和超时处理策略，减少重复逻辑。
- 把请求级指标接入性能看板和发布门禁。

## 常见误区

- “请求发出就算完成优化”：错误，请求调度顺序和头部策略同样关键。
- “重试越积极越稳”：错误，可能引发限流和雪崩。
- “GET 一定比 POST 快”：错误，速度由网络和服务链路共同决定。

## 面试补充

### 30 秒说明

HTTP 请求阶段的关键是“正确发送 + 正确调度”。前端要关注请求组成、优先级、跨域预检、缓存条件头和重试策略。真正的优化不是单次请求快，而是整条关键链路稳定且可控。

### 高频追问

1. 请求优先级为什么会影响首屏？

连接和带宽是共享资源，非关键请求抢占会推迟关键资源返回。

2. 为什么要做幂等重试？

避免网络抖动下重复提交引发数据不一致，同时保证失败恢复能力。

3. 如何减少请求头开销？

去除无效自定义头，收敛鉴权字段，避免重复传输大字段。

4. 预检请求和业务请求的关系是什么？

预检是权限探测，只有通过后业务请求才会发送。

5. 请求优化如何与后端协同？

统一状态码语义、超时配置、重试边界和网关限流策略。


## 补充视角（标准节点）

### 与上下游节点边界

- 上游节点（高频）：HTTP 协议演进与连接复用、TCP/TLS 建连。
- 本节点重点：聚焦“发送 HTTP 请求”本身的机制与决策，不替代下游渲染/监控职责。
- 下游节点（高频）：同源与跨域安全策略、实时通道分支（WebSocket/SSE）、源站返回响应。

### 落地动作清单

1. 先定义该节点的关键输入和关键输出，避免职责漂移。
2. 再绑定至少 1 个可观测指标，确保问题可量化定位。
3. 将该节点的异常处理写入动作库，确保值班时可执行。

### 高频追问

1. 这个节点优化应该先看什么？

先看它的关键输入是否稳定，再看关键输出是否满足下游依赖和时延预算。

2. 如何避免“优化该节点却拖慢其他节点”？

做上下游联动回归，对比关键链路分阶段指标，防止成本转移。

3. 这个节点如何进入发布门禁？

把它的核心指标加入灰度对比阈值，超阈值时触发暂停放量或回滚。
