## 概念定位

CSS 架构与渲染策略节点用于回答：如何让样式系统既可维护，又不拖慢渲染和交互。

它关注的不只是“样式能不能写出来”，而是样式如何影响 CSSOM 构建、样式重算和布局稳定性。

## 工作机制

1. 样式资源加载并构建 CSSOM。
2. CSS 规则与 DOM 匹配，生成计算样式。
3. 样式变化触发重算、布局或重绘。
4. 持续演进中通过架构规范降低复杂度。

## 关键知识点

### 架构层策略

- 设计 token 与变量体系，减少硬编码。
- 按模块拆分样式，控制作用域。
- 约束选择器复杂度，避免过深嵌套。

### 渲染层策略

- 首屏关键样式优先加载。
- 高频动效优先使用 `transform/opacity`。
- 避免频繁修改触发布局的属性。

### 常见方案取舍

| 方案 | 优点 | 风险 |
| --- | --- | --- |
| 全局样式集中 | 管理直观 | 易污染、耦合高 |
| 模块化样式 | 作用域清晰 | 规范不统一时易碎片化 |
| 原子类策略 | 复用高、体积可控 | 可读性和心智负担需治理 |

## 对页面性能的影响

- 规则复杂度高会拉高样式重算成本。
- 非关键样式阻塞首屏会拖慢 FCP/LCP。
- 高频样式写入不当会恶化 INP 和帧率。

## 观测与排障

### 观测方法

1. 看 CSS 体积与首屏关键样式占比。
2. 看 `Recalculate Style` 在交互中的耗时。
3. 看布局抖动和样式更新频率。

### 排障步骤

1. 优先拆分关键与非关键样式。
2. 简化高频区域选择器与层级。
3. 读写分离，避免频繁触发布局。
4. 回归验证 INP、长任务和样式重算时长。

## 实战优化清单

- 建立样式命名和层级规范。
- 限制选择器复杂度与文件体积。
- 对大页面做样式分块和按需加载。
- 把样式重算指标纳入性能看板。

## 常见误区

- “CSS 只影响美观”：错误，样式系统直接影响运行时性能。
- “选择器写得花不影响性能”：错误，复杂匹配会放大重算开销。
- “动效越多越高级”：错误，超出渲染预算会明显掉帧。

## 面试补充

### 30 秒说明

CSS 策略要兼顾工程可维护和运行时性能。架构上要模块化和约束复杂度，渲染上要优先关键样式并减少高成本属性更新。真正有效的 CSS 优化应体现在样式重算时长、INP 和帧稳定性上。

### 高频追问

1. 为什么 CSS 会影响交互延迟？

因为高频样式变更会触发重算与布局，占用主线程。

2. 如何判断样式系统是否过于复杂？

看选择器深度、重复规则、样式重算耗时和维护成本。

3. 关键 CSS 应如何抽取？

以首屏可见内容为边界，抽出必须样式，其余延后加载。

4. 动画优化里为什么强调 `transform`？

通常可在合成阶段处理，减少布局和重绘负担。

5. CSS 架构如何防止长期劣化？

通过规范、lint、评审和指标门禁持续约束。


## 补充视角（标准节点）

### 与上下游节点边界

- 上游节点（高频）：当前链路入口节点。
- 本节点重点：聚焦“CSS 架构与渲染策略”本身的机制与决策，不替代下游渲染/监控职责。
- 下游节点（高频）：CSSOM 构建与脚本执行、布局计算（Layout）、绘制（Paint）。

### 落地动作清单

1. 先定义该节点的关键输入和关键输出，避免职责漂移。
2. 再绑定至少 1 个可观测指标，确保问题可量化定位。
3. 将该节点的异常处理写入动作库，确保值班时可执行。

### 高频追问

1. 这个节点优化应该先看什么？

先看它的关键输入是否稳定，再看关键输出是否满足下游依赖和时延预算。

2. 如何避免“优化该节点却拖慢其他节点”？

做上下游联动回归，对比关键链路分阶段指标，防止成本转移。

3. 这个节点如何进入发布门禁？

把它的核心指标加入灰度对比阈值，超阈值时触发暂停放量或回滚。
