## 概念定位

响应流接收是“网络传输”与“浏览器解析”之间的桥接节点。它决定浏览器是否能边接收边消费内容，而不是等全部下载完再开始解析。

## 工作机制

1. 浏览器收到响应头后开始按数据块接收响应体。
2. HTML 场景可边接收边解析，提前构建 DOM。
3. 预加载扫描器并行发现关键资源并发起请求。
4. 数据流结束后，文档进入完整状态并继续后续渲染。

## 关键知识点

- **流式消费**：下载与解析可并行，缩短可见内容等待。
- **TTFB（Time To First Byte） 与下载拆分**：首字节早到不代表主内容已可见。
- **关键块组织**：首屏关键结构输出顺序比总字节大小更影响体感。
- **背压关系**：接收速度、解析速度、主线程可用性互相耦合。

## 对页面性能的影响

- 正确流式输出能显著改善首屏起步速度。
- 若关键 CSS/JS 晚到，流式收益会被阻塞链抵消。
- 响应块组织不当会导致“有流但不快”的假优化。

## 观测与排障

### 浏览器侧

- Network 区分首包时间与下载阶段时长。
- Performance 观察首屏渲染是否早于下载完成。
- 检查关键资源是否在流式早期被发现并发起。

### 排障步骤

1. 确认首屏关键 HTML 是否在响应前段输出。
2. 检查关键 CSS/JS 是否被延后到后段数据块。
3. 检查服务端流式模板是否包含过多非关键内容。
4. 回归验证 FCP、LCP 与关键请求链长度。

## 常见误区

- “流式输出一定更快”：不准确，关键资源顺序错误会抵消收益。
- “TTFB 快就说明体验快”：不准确，后续解析和主线程仍可能慢。
- “只有 SSR 才能流式”：不准确，多种服务端输出模式都可分块。

## 面试补充

### 30 秒说明

响应流接收的价值是并行化：边下载边解析，尽早让首屏内容进入渲染链。真正收益取决于关键 HTML 与关键资源是否被放在正确的输出顺序。

### 高频追问

1. 为什么有时 TTFB 很好但 LCP 仍差？

因为首字节虽然早到，但关键资源可能晚发现、晚下载，导致主要内容仍迟到。

2. 流式策略如何验证有效？

看关键请求是否前移、首屏是否提前可见，并结合 FCP/LCP 变化确认。

3. 服务端分块时最先输出什么？

首屏关键结构和关键依赖引用应优先输出，非关键内容可后置。
