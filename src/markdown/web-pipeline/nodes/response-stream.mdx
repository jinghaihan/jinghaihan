## 概念定位

响应流接收表示浏览器开始持续接收字节流，而不是等待完整文件下载后才处理。它是“网络到解析”的桥接节点。

## 工作机制

1. 浏览器收到响应头后，按数据块持续接收响应体。
2. 对 HTML 场景可边接收边解析，逐步推进 DOM 构建。
3. 预加载扫描器可在主解析前发现关键资源并并行请求。
4. 数据流结束后进入完整内容状态。

## 关键知识点

- **TTFB（Time To First Byte） 与下载阶段拆分**：首字节到达不等于内容已就绪。
- **流式渲染**：服务端可通过分块早推关键内容。
- **背压机制**：接收与消费速度不匹配会影响吞吐。

## 对页面性能的影响

- 早到达的关键 HTML 可以提前触发首屏渲染。
- 若关键 CSS/JS 到达晚，流式收益会被抵消。
- 响应块设计不合理会导致首屏仍卡在后续资源。

## 观测与排障

### 浏览器侧

- 瀑布图区分 `TTFB` 与 `Content Download`。
- Performance 观察首屏时间是否明显早于下载完成时间。

## 常见误区

- “必须下载完整 HTML 才能开始解析”：不准确，现代浏览器支持流式解析。
- “只要开启流式就一定更快”：不一定，取决于关键资源的到达顺序。

## 面试补充

### 30 秒说明

响应流接收是浏览器边收边处理数据的阶段。它能把“下载”和“解析”并行化，核心价值是尽早把关键内容送到渲染链路。

### 高频追问

1. SSR 流式输出为什么能提升首屏体验？

它让浏览器更早拿到可解析的 HTML 片段，提前触发布局和绘制，而不必等完整页面一次性返回。

2. 流式输出的关键注意点是什么？

关键 CSS、关键脚本和首屏结构要尽早输出，否则虽然在流式传输，首屏仍会被后续阻塞资源拖慢。
