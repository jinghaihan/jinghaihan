## 概念定位

缓存策略优化节点用于统一浏览器缓存、CDN 缓存和发布策略，目标是在“快”和“准”之间取得稳定平衡：

- 快：尽量减少重复请求和回源。
- 准：版本更新可控生效，避免脏资源。

## 工作机制

1. 按资源类型定义缓存策略（HTML、静态资源、API）。
2. 设置强缓存与协商缓存头部组合。
3. CDN 按边缘策略执行命中与回源。
4. 发布时通过指纹与版本控制触发正确失效。
5. 通过命中率、回源率、状态码分布持续校正策略。

## 关键知识点

### 强缓存与协商缓存

- **强缓存**：在有效期内直接复用，不发请求。
- **协商缓存**：发条件请求，命中返回 `304 Not Modified`。

### 核心响应头

- `Cache-Control`：`max-age`、`s-maxage`、`public`、`private`、`no-cache`、`no-store`、`must-revalidate`、`stale-while-revalidate`
- `ETag` / `If-None-Match`
- `Last-Modified` / `If-Modified-Since`
- `Expires`（兼容历史）

### 分层缓存差异

| 层级 | 目标 | 关键关注点 |
| --- | --- | --- |
| 浏览器缓存 | 用户侧复用 | 首次与复访体感、304 往返成本 |
| CDN 边缘缓存 | 减少回源 | 命中率、回源带宽、区域一致性 |
| 源站缓存/应用缓存 | 降低计算成本 | 数据一致性、失效策略、穿透保护 |

### 版本发布与缓存一致性

- 指纹静态资源适合长缓存。
- HTML 和入口清单应短缓存或协商，保证版本可更新。
- 关键是“引用关系”一致：HTML 引用的静态资源必须可唯一定位新版本。

## 对页面性能的影响

- 缓存命中直接降低 TTFB（Time To First Byte）和总下载时间。
- 缓存策略错误会导致“首屏慢、回源高、发布后混乱”。
- 304 比例不合理可能暴露策略失衡（协商过多或强缓存不足）。

## 观测与排障

### 建议观测指标

- 浏览器层：Memory/Disk 命中率、`200/304` 比例。
- CDN 层：命中率、回源率、回源带宽、边缘 5xx。
- 服务层：热点接口 P95、缓存穿透率。

### 排障剧本

1. 按资源类型列策略矩阵，确认是否“一刀切”。
2. 抓关键响应头，核对 `Cache-Control` 与校验头是否一致。
3. 检查是否使用内容指纹、引用是否正确更新。
4. 检查 CDN key 设计是否过细导致命中碎片化。
5. 发布前后对比命中率、回源率、TTFB。

## 实战策略

### 推荐资源策略

- HTML：`no-cache` 或短 `max-age` + 协商。
- 指纹静态资源：长 `max-age` + `immutable` 思路。
- API：按业务一致性要求设短缓存或禁止缓存。
- 大文件：结合分段与边缘缓存策略。

### 防穿透与防雪崩

- 热点数据加多级缓存与随机过期抖动。
- 回源失败设置降级兜底和熔断。
- 重要接口限制并发回源与重试。

### 发布与回滚

- 新版本发布前预热关键静态资源。
- 灰度期间监控 `404` 与缓存错配。
- 回滚时保障入口资源和静态资源映射一致。

## 常见误区

- “`no-cache` 就是不缓存”：错误，它是“使用前必须验证”。
- “304 越高越好”：错误，协商请求仍有往返成本。
- “只管浏览器缓存就够”：错误，CDN 层常决定整体吞吐和稳定性。
- “缓存失效只靠删 CDN”：错误，版本引用链和头部策略同样关键。

## 面试补充

### 30 秒说明

缓存优化是分层治理问题。静态指纹资源走长缓存，入口 HTML 走短缓存或协商，CDN 与浏览器策略保持一致。上线时要关注命中率和回源率变化，防止出现“缓存命中了但用户没更新”或“全量回源打爆后端”。

### 高频追问

1. `no-store` 和 `no-cache` 有什么本质区别？

`no-store` 表示不落缓存；`no-cache` 允许落缓存但每次使用前需再验证。

2. 为什么发布后会有人看到旧页面？

通常是入口 HTML 缓存策略过长或 CDN/浏览器层策略不一致。

3. `ETag` 和 `Last-Modified` 如何取舍？

`ETag` 更精细但生成成本更高；`Last-Modified` 简单但粒度较粗，常结合使用。

4. 什么时候 `stale-while-revalidate` 有价值？

对可接受短暂旧数据的资源，可以先快返旧内容再后台更新，改善体感速度。

5. 如何判断缓存策略导致性能回退？

看命中率下降、304 上升、回源率抬升与 TTFB 变差是否同步出现。

6. 缓存策略如何纳入门禁？

把命中率、回源率、关键资源 `404` 与版本一致性纳入发布检查与告警。


## 进阶补充（深挖）

### 与上下游节点的因果关系

- 上游输入（常见）：刷新模式与缓存判定、指标异常排障动作库、前端工程化实践。
- 本节点职责：将“缓存策略优化”的结果转化为可执行的后续分支决策。
- 下游影响（常见）：内存缓存检查、磁盘缓存检查、CDN 边缘节点。

### 线上故障案例

- 现象：发布后出现部分用户旧资源、部分用户新资源的错配问题。
- 根因：入口与静态资源缓存策略不一致，版本引用链断裂。
- 处置：入口短缓存协商 + 指纹静态长缓存，并校验引用一致性。
- 回归标准：关键指标恢复到基线区间，且连续观测窗口无二次抖动。

### 高频追问）

1. 304 比例高却仍慢怎么解释？

协商仍有往返成本，弱网下收益可能不明显。

2. 为什么缓存策略要分资源类型？

不同资源在一致性和更新时效要求上不同。

3. CDN 命中率高是否一定好？

需结合业务正确性和回源稳定性综合判断。


### 补充记忆框架

- 先判断该节点是“前置等待”问题还是“运行时抖动”问题。
- 再判断问题来自“资源策略”还是“执行调度”。
- 最后把定位结论转为可复用规则，进入门禁与动作库。

### 高频追问

1. 这个节点优化后如何做回归验收？

至少做两组对比：关键指标分位数对比 + 上下游链路稳定性对比，避免局部优化导致成本转移。

2. 为什么这个节点要单独做发布观察窗口？

因为它对主链路有放大效应，短时间异常会快速传导到首屏和交互体验。
