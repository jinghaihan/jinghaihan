## 概念定位

缓存策略优化是网络与渲染性能的共同基础。它不是单一响应头配置，而是“浏览器缓存 + CDN 缓存 + 版本发布策略”的协同设计。

## 工作机制

1. 按资源类型拆分策略：HTML、静态资源、接口数据。
2. 静态不可变资源使用长缓存 + 指纹命名。
3. 变动资源使用协商缓存或短生命周期策略。
4. CDN 与浏览器层统一控制 `Cache-Control`、`ETag`、`s-maxage`。
5. 通过命中率和回源率回归，持续校正策略。

## 关键知识点

- **强缓存**：命中后不发请求，直接复用副本。
- **协商缓存**：发送条件请求，命中返回 `304`。
- **TTL（Time To Live）**：缓存有效时间，影响命中率与更新收敛。
- **ETag（Entity Tag）**：资源版本标识，常用于精细校验。
- **`stale-while-revalidate`**：先返回旧副本并后台更新，平衡速度与新鲜度。

## 对页面性能的影响

- 合理缓存策略可显著降低重复请求与首屏波动。
- 过短 TTL 会放大回源压力，过长 TTL 会拉慢版本生效。
- 缺少资源指纹会导致“缓存很长但更新不稳定”。

## 观测与排障

### 建议指标

- 浏览器层：内存/磁盘命中率、304 比例。
- CDN 层：命中率、回源率、回源带宽、边缘 5xx。
- 服务层：接口 P95、冷热点资源回源峰值。

### 排障步骤

1. 先按资源类别建策略矩阵，避免“一刀切”。
2. 查响应头是否与预期一致（强缓存/协商字段）。
3. 查 URL 是否指纹化，发布链路是否稳定变更版本。
4. 查 CDN 键是否过细导致命中率碎片化。
5. 用发布前后命中率与回源率对比验证策略有效性。

## 常见误区

- “`no-cache` 就是不缓存”：错误，`no-cache` 是使用前必须再验证。
- “304 越多越好”：不准确，304 仍有往返成本。
- “只优化浏览器缓存即可”：不准确，CDN 层常是整体吞吐关键。

## 面试补充

### 30 秒说明

缓存优化的核心是分层分资源治理。静态资源走长缓存+指纹，动态内容走协商或短缓存，CDN 与浏览器策略统一后，才能同时提升首屏速度和系统稳定性。

### 高频追问

1. `no-store` 和 `no-cache` 的区别是什么？

`no-store` 表示不保存副本；`no-cache` 允许保存但每次使用前都要向服务端验证。

2. 为什么上线后经常出现“有人看到新版本，有人没更新”？

通常是缓存层级策略不一致或指纹不完整，导致不同路径命中不同副本。

3. 如何给资源设置优先级缓存策略？

先按“是否可指纹、变更频率、业务关键性”分组，再为每组定义 TTL、校验头和 CDN 规则。
