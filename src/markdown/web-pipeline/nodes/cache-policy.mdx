## 概念定位

缓存策略优化节点是对“内存缓存、磁盘缓存、CDN 缓存”的统一策略设计。目标是在正确性前提下最大化命中率并降低回源成本。

## 工作机制

1. 按资源类型分层策略：HTML、JS/CSS、图片、接口响应。
2. 为静态不可变资源设置长缓存 + 文件指纹。
3. 为可变资源配置协商缓存或短 TTL（Time To Live）。
4. 在 CDN 与浏览器层协调 `Cache-Control`、`ETag`、`s-maxage`。

## 关键知识点

- **`max-age` / `s-maxage`**：分别面向私有缓存与共享缓存。
- **`immutable`**：适用于内容哈希命名资源。
- **协商字段**：`ETag`、`Last-Modified` 决定验证命中路径。

## 对页面性能的影响

- 策略合理时，复访加载速度和稳定性显著提升。
- 策略错误会导致“每次都回源”或“更新不生效”。
- 高频热点资源策略失衡会放大源站峰值压力。

## 观测与排障

### 浏览器侧

- 检查响应头中的缓存字段是否符合预期。
- 对比首次/复访是否出现明显命中收益。

### 服务端侧

- 监控 CDN 命中率、304 比例、回源带宽。
- 识别高频资源是否缺乏指纹与版本化。

## 常见误区

- “`no-cache` 就是完全不缓存”：错误，`no-cache` 是必须再验证。
- “只配浏览器缓存就够了”：不准确，CDN 与浏览器要协同设计。

## 面试补充

### 30 秒说明

缓存策略优化的本质是“分层分资源治理”：静态资源走长缓存 + 指纹，动态内容走短缓存或协商缓存，再通过 CDN 与浏览器策略协同实现性能与一致性的平衡。

### 高频追问

1. `no-cache` 和 `no-store` 的核心区别？

`no-cache` 可以缓存但每次使用前必须向服务端验证；`no-store` 是禁止存储，浏览器和代理都不应保留副本。

2. 为什么静态资源建议带 hash 文件名？

内容变更会直接映射到 URL 变更，能安全使用长缓存，避免“缓存很久但用户拿不到新版本”的冲突。
