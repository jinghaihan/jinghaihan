## 概念定位

前端工程化实践是把“工作流知识”落地成“可持续交付能力”的汇总节点。它解决的问题不是单次优化，而是让缓存、渲染、监控和质量门禁形成稳定闭环。

## 工作机制

1. 在构建阶段把产物按缓存策略拆分：长期缓存资源、可协商资源、实时资源。
2. 在交付阶段控制资源清单：chunk 拆分、关键资源标记、版本指纹。
3. 在发布阶段执行质量门禁：体积预算、性能预算、错误率阈值。
4. 在运行阶段回收真实指标：RUM（Real User Monitoring） + 日志 + CDN 数据。
5. 在回归阶段自动比对：版本前后指标趋势和异常告警。

## 关键知识点

- **构建产物治理**：文件命名、hash 策略、依赖边界直接影响缓存命中率。
- **性能预算（Performance Budget）**：对 JS/CSS 体积、LCP、INP 设置上线阈值。
- **发布回滚策略**：灰度、分批、快速回滚减少性能事故放大。
- **监控分层**：实验室数据 + 真实用户数据必须同时看。
- **工程与运行一体化**：CI 的通过不等于线上体验稳定。

## 对页面性能的影响

- 工程化不到位时，缓存策略和资源策略会频繁失效。
- 缺少预算和门禁，性能回退会在迭代中持续累积。
- 只有把监控接入交付链路，优化成果才能长期保留。

## 观测与排障

### 建议的工程化门禁

1. 产物预算：关键路由 JS、关键 CSS、字体总量阈值。
2. 指标预算：LCP、INP、CLS、TTFB（Time To First Byte） 上线阈值。
3. 缓存预算：CDN 命中率、回源率、304 比例。
4. 质量预算：错误率、白屏率、主线程长任务比例。

### 排障路径

1. 新版本上线后指标波动，先看构建产物 diff。
2. 再看缓存命中和回源变化，确认是否策略偏移。
3. 再看主线程长任务与路由加载链是否增加。
4. 最后回到代码层做依赖裁剪和执行时机调整。

## 常见误区

- “工程化就是构建速度优化”：不准确，核心是交付质量和可回归能力。
- “只要 Lighthouse 过线就可以发版”：不准确，真实流量和设备分层更关键。
- “性能优化是前端单点责任”：不准确，CDN、服务端、发布系统都在链路里。

## 面试补充

### 30 秒说明

前端工程化不是工具堆砌，而是把性能目标变成交付规则。通过产物治理、预算门禁、监控回归，把网络请求链路、渲染链路和性能优化链路真正固化成团队能力。

### 高频追问

1. 性能预算落地最小方案是什么？

先定义三类阈值：关键资源体积、核心体验指标、错误率阈值；再在 CI 增加检查并阻断超标发布。

2. 为什么要同时看实验室和真实用户数据？

实验室环境稳定但样本单一，真实用户环境复杂但可反映实际体验，二者结合才能避免误判。

3. 构建拆包和缓存策略有什么耦合？

拆包决定资源边界和复用粒度，缓存策略决定资源生命周期，两者不协同会造成“包拆得细但命中率不升”。
