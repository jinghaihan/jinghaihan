## 概念定位

刷新模式与缓存判定节点用于解释：用户“看起来都在刷新页面”，但浏览器实际走的是不同加载路径。

这直接影响缓存命中、请求数量、首屏耗时和问题复现方式。

## 工作机制

1. 浏览器接收刷新动作（地址栏回车、普通刷新、强制刷新）。
2. 根据刷新类型决定缓存使用策略。
3. 走内存缓存、磁盘缓存、协商请求或强制回源。
4. 页面重新解析渲染，并记录新的性能轨迹。

## 关键知识点

### 常见刷新类型对比

| 刷新类型 | 典型操作 | 缓存行为 | 适用场景 |
| --- | --- | --- | --- |
| 地址栏回车 | 回车同 URL | 常按普通导航策略 | 日常访问 |
| 普通刷新 | F5 / 浏览器刷新 | 常允许缓存并协商 | 一般更新验证 |
| 强制刷新 | Ctrl+F5 / Cmd+Shift+R | 更倾向忽略本地缓存 | 排查缓存疑难 |

### 判定差异来源

- 浏览器实现细节
- 资源缓存头策略
- 是否有 Service Worker（SW）拦截
- DevTools 禁用缓存开关状态

### 与排障的关系

很多“我这边复现不了”的问题，本质是不同同学采用了不同刷新路径，导致缓存状态不一致。

## 对页面性能的影响

- 普通刷新通常更快，但可能继续命中旧缓存策略。
- 强制刷新更接近冷启动，常更慢但排障价值高。
- 刷新模式差异会放大线上数据和本地复现差异。

## 观测与排障

### 观测方法

1. Network 看 `200/304/from memory cache/from disk cache` 分布。
2. 记录同一页面在不同刷新模式下的关键指标差异。
3. 联动 Service Worker 面板检查是否存在离线缓存干预。

### 排障步骤

1. 明确复现要求：指定刷新模式和缓存开关。
2. 先用强制刷新确认是否是缓存导致。
3. 再回到普通刷新看真实用户路径表现。
4. 结合缓存头和 SW 策略做根因定位。

## 实战优化清单

- 缺陷单必须记录刷新模式与缓存开关状态。
- 关键页面保留“冷启动”和“复访”两套监控基线。
- 发布验证至少覆盖普通刷新与强制刷新各一次。
- SW 场景增加版本切换与缓存失效检查。

## 常见误区

- “刷新就是重新请求全部资源”：错误，通常会触发多层缓存判定。
- “强制刷新就是用户真实路径”：错误，它更偏排障手段。
- “地址栏回车和 F5 完全一样”：错误，不同浏览器和上下文可能有差异。

## 面试补充

### 30 秒说明

刷新模式决定了请求和缓存分支。普通刷新通常会保留缓存协商路径，强制刷新更偏向回源验证。线上排障必须先统一刷新模式，否则很容易出现“同页面不同结果”的误判。

### 高频追问

1. 为什么同一个 bug 有人能复现有人不能？

常见是刷新模式、缓存开关或 Service Worker 状态不同。

2. 强制刷新应在什么时候用？

用于判断问题是否由缓存导致，或验证新版本资源是否真正生效。

3. 普通刷新排障价值低吗？

不低，它更接近真实用户路径，适合评估线上体感。

4. SW 为什么会影响刷新行为？

SW 可拦截请求并使用自身缓存策略，导致“刷新后仍旧资源”或“离线副本优先”。

5. 如何把刷新差异纳入测试流程？

把冷启动、普通刷新、强制刷新作为固定回归场景。

### 与上下游节点边界

- 上游节点（高频）：导航开始。
- 本节点重点：聚焦“刷新模式与缓存判定”本身的机制与决策，不替代下游渲染/监控职责。
- 下游节点（高频）：内存缓存检查、磁盘缓存检查、缓存策略优化。

### 落地动作清单

1. 先定义该节点的关键输入和关键输出，避免职责漂移。
2. 再绑定至少 1 个可观测指标，确保问题可量化定位。
3. 将该节点的异常处理写入动作库，确保值班时可执行。

### 高频追问

1. 这个节点优化应该先看什么？

先看它的关键输入是否稳定，再看关键输出是否满足下游依赖和时延预算。

2. 如何避免“优化该节点却拖慢其他节点”？

做上下游联动回归，对比关键链路分阶段指标，防止成本转移。

3. 这个节点如何进入发布门禁？

把它的核心指标加入灰度对比阈值，超阈值时触发暂停放量或回滚。
