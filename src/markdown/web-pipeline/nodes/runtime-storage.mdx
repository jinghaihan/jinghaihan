## 概念定位

运行时与存储关联节点用于解释：

- 浏览器主线程如何被存储访问影响
- 不同存储方案的生命周期、容量和性能差异
- 为什么“存在哪”会直接影响网络请求、渲染时序和稳定性

这不是单纯的数据持久化问题，而是请求链路、渲染链路和状态一致性的耦合问题。

## 工作机制

1. 页面初始化阶段读取本地状态（登录态、配置、草稿、缓存索引）。
2. 运行时根据状态决定是否发请求、发哪些请求、是否复用缓存。
3. 渲染过程中若频繁同步访问存储，会阻塞主线程。
4. 异步存储（如 IndexedDB）通过任务调度与事务机制落盘。
5. 存储结果反向影响下一轮请求、渲染和性能指标。

## 关键知识点

### 存储方案对比

| 方案 | 典型容量 | 生命周期 | 是否随请求发送 | API 特性 | 典型用途 |
| --- | --- | --- | --- | --- | --- |
| Cookie | 单条通常约 4KB；总量受浏览器限制 | 可设过期时间 | 是（同站请求头会携带） | 字符串、读写相对同步语义 | 会话标识、少量服务端协同状态 |
| localStorage | 常见约 5MB（不同浏览器有差异） | 持久，手动清除前长期存在 | 否 | 同步 API | 小体量本地配置、非敏感偏好 |
| sessionStorage | 常见约 5MB（不同浏览器有差异） | 会话级，标签页关闭失效 | 否 | 同步 API | 临时表单状态、单页会话缓存 |
| IndexedDB | 远大于 Web Storage，按配额策略动态分配 | 持久（可能在存储压力下被回收） | 否 | 异步 + 事务 + 索引 | 大体量结构化数据、离线数据 |
| Cache Storage | 与 IndexedDB 类似受配额管理 | 持久（可能被回收） | 否（由 Service Worker/Fetch 命中） | 异步，按 Request/Response 缓存 | 静态资源缓存、离线能力 |

### 它们的核心区别

1. **是否参与网络传输**

Cookie 会进入请求头，体积越大，每个请求成本越高；其余方案默认不随请求发送。

2. **是否阻塞主线程**

`localStorage/sessionStorage` 是同步 API，高频读写会卡主线程；IndexedDB/Cache Storage 是异步模型，更适合高频与大数据。

3. **生命周期与隔离范围**

- `sessionStorage` 是标签页级会话隔离。
- `localStorage` 是同源持久共享。
- Cookie 受 `Domain/Path/SameSite/Secure/HttpOnly` 等字段影响，跨子域与安全边界更复杂。

4. **数据模型能力**

- Cookie / Web Storage 适合轻量键值。
- IndexedDB 支持索引、事务、对象存储，适合复杂离线数据。

### 存储上限与配额策略

#### 1) Cookie

- 单条通常约 `4KB`。
- 每域可用数量和总容量受浏览器策略限制（不同浏览器不同，通常是“几十到数百条”的量级）。
- 因为每次请求都可能携带，工程上建议把 Cookie 控制在“必要且小体积”。

#### 2) localStorage / sessionStorage

- 常见经验值是每源约 `5MB` 量级（不同浏览器可能更高或更低）。
- 超出后会抛出 `QuotaExceededError`。
- 因为是同步 API，不建议作为高频更新存储。

#### 3) IndexedDB / Cache Storage

- 不采用固定 5MB 上限，通常按“源级配额”管理。
- 可用空间与设备磁盘、浏览器策略、是否持久化授权相关。
- 在存储压力下可能被回收，需要考虑数据重建与容灾。

#### 4) 如何在运行时查看容量

```ts
const estimate = await navigator.storage.estimate()
console.log('usage', estimate.usage)
console.log('quota', estimate.quota)
```

可结合 `navigator.storage.persist()` 申请更稳定的持久化策略（是否批准由浏览器与用户环境决定）。

### 运行时耦合点

- 初始化阶段同步读取过多：拉长启动时间、影响 FCP/LCP。
- 输入/滚动等交互中同步写入：抬高 INP，造成卡顿。
- 高频 JSON 序列化/反序列化：增加主线程 CPU 开销。
- 多标签并发写：引发状态覆盖、脏读和一致性问题。

## 对页面性能的影响

- Cookie 过大直接增加请求头体积，拖慢关键请求。
- `localStorage` 高频读写会形成长任务，影响交互响应。
- IndexedDB 事务设计不当会导致队列积压与 UI 更新延迟。
- 缓存命中策略混乱会出现“本地有数据但仍重复请求”的浪费。

## 观测与排障

### 浏览器侧

- `Application` 面板检查各存储占用、键数量、更新频率。
- `Performance` 关联长任务与同步存储调用位置。
- `Network` 观察请求头中的 Cookie 体积变化。
- `Storage` 估算查看当前 `usage/quota`。

### 排障步骤

1. 先查 Cookie 体积与冗余字段，剔除非必要项。
2. 把渲染关键路径中的同步存储读写迁移为异步或内存缓存。
3. 对高频写入增加合并与节流，避免每次输入都落盘。
4. 为大数据迁移到 IndexedDB，并设计索引与事务边界。
5. 处理 `QuotaExceededError`：降级、淘汰、压缩、分层存储。
6. 回归验证 INP（Interaction to Next Paint）、TBT（Total Blocking Time）、TTFB（Time To First Byte）和请求头体积。

## 实战选型建议

### 存储选型

- **Cookie**：只放会话必须字段，不放大对象和业务缓存。
- **localStorage**：低频、小体量、可丢失风险可接受的数据。
- **sessionStorage**：仅当前标签页有效的临时状态。
- **IndexedDB**：离线数据、列表索引、大体量结构化缓存。
- **Cache Storage**：静态资源与请求响应缓存，配合 Service Worker。

### 分层策略

建议按“热路径内存 + 异步持久层”分层：

1. 热数据放内存（最快、避免同步阻塞）。
2. 需要持久化再异步落到 IndexedDB/Cache Storage。
3. Cookie 仅承载服务端必需标识。

### 一致性策略

- 跨标签页同步可用 `storage` 事件或 `BroadcastChannel`。
- 数据结构加版本号，处理升级迁移。
- 关键写入保持幂等，避免重复提交与覆盖。

## 常见误区

- “本地存储不走网络就没有性能成本”：错误，主线程访问成本会直接影响交互。
- “localStorage 足够简单就应该优先使用”：错误，高频场景会造成明显卡顿。
- “Cookie 只多几 KB 没关系”：错误，请求频次高时开销会被放大。
- “IndexedDB 容量大就不会丢”：错误，浏览器在存储压力下可能回收。
- “所有状态都持久化最稳”：错误，过度持久化会带来一致性和迁移复杂度。

## 面试补充

### 30 秒说明

浏览器存储方案的差异主要在四个维度：是否随请求发送、是否阻塞主线程、生命周期范围、容量与配额策略。Cookie 适合极小的服务端协同状态；localStorage/sessionStorage 是同步且容量有限；大体量和高频数据应放 IndexedDB/Cache Storage，并通过异步批处理降低运行时压力。

### 高频追问

1. Cookie、localStorage、sessionStorage 的核心区别是什么？

Cookie 会随请求发送且容量小；localStorage 是同源持久存储；sessionStorage 是标签页会话级存储，关闭标签页即失效。

2. 它们的容量上限大概是多少？

Cookie 单条通常约 4KB；localStorage/sessionStorage 常见约 5MB 量级；IndexedDB/Cache Storage 由浏览器按源级配额动态分配，通常远高于 5MB。

3. 为什么 localStorage 可能引发卡顿？

因为它是同步 API，在主线程执行，高频读写会产生长任务并影响 INP。

4. 什么时候必须用 IndexedDB？

当数据量大、需要索引查询、需要事务一致性、或需要离线能力时应优先用 IndexedDB。

5. 如何处理存储超限？

捕获 `QuotaExceededError` 后进行分级降级：清理低价值缓存、压缩数据、把热数据放内存、保留关键数据并提示用户。

6. 如何避免跨标签页状态冲突？

通过版本号、时间戳和幂等写策略做冲突控制，并用 `BroadcastChannel` 同步状态变更。
