## 概念定位

HTTP 请求节点关注请求方法、头部、连接复用和协商机制，是网络链路的主干环节。

请求构造是否合理会直接影响延迟、缓存命中和后端压力。

## 工作机制

1. 构建请求报文：方法、URL、头部、主体。
2. 根据协议版本选择连接复用策略（HTTP/1.1 Keep-Alive、HTTP/2 多路复用）。
3. 带上缓存协商头和鉴权信息。
4. 处理重定向、跨域预检和错误码分支。
5. 返回响应后进入解析与缓存决策。

## 关键知识点

### 必须掌握

- GET/POST/PUT/PATCH/DELETE 的语义边界要清晰。
- OPTIONS 预检会放大跨域延迟，需优化触发频率。
- HTTP/2 头压缩和多路复用可减少队头阻塞。
- 请求头过大也会影响首字节时间。
- 同一接口要明确幂等语义，便于重试治理。
- 协议升级（HTTP/3）对弱网抖动场景收益明显。

### 选型与取舍

- 请求头字段越多可观测性越强，但传输开销更大。
- 激进复用连接可降时延，但连接管理复杂度上升。
- 预检收敛可提速，但需要更严格接口规范。

## 对页面性能与稳定性的影响

- 节点稳定时，Knowledge Graph 主线 的网络与渲染链路更可预测，性能治理收益更稳定。
- 节点失稳时，问题会向后续阶段扩散，常表现为首屏变慢、交互卡顿或版本一致性风险。
- 优化动作应结合上下游联动验证，避免局部优化把成本转移到其他节点。

## 观测与排障

### 核心指标

- 请求成功率和状态码分布
- TTFB（Time To First Byte）
- 预检请求占比
- 重定向次数

### 排障顺序

1. 先按状态码分层定位问题。
2. 检查请求头、响应头和协议版本。
3. 检查是否存在异常重定向与预检风暴。
4. 检查连接复用和并发策略。
5. 对关键接口做弱网回归。

### 常见故障场景

- 跨域接口偶发慢。
- 请求成功但首字节时间长。
- 接口重试导致副作用重复。
- 协议升级后个别设备兼容异常。

## 常见误区

- 只看接口耗时，不看协议与头部细节。
- 忽略 OPTIONS 预检成本。
- 接口语义不清导致错误重试。

## 题库深挖补充

### 高频对比与边界（基于题库）

- GET 偏读取、可缓存；POST 偏创建；PUT 常用于整体替换；PATCH 常用于局部更新。
- 幂等性是重试策略前提：GET/PUT/DELETE 通常幂等，POST 默认不幂等。
- 206（Partial Content）对应分段传输，常用于视频与断点续传。
- 307/308 与 302/301 的关键区别是是否保留原方法语义。
- 304 与缓存协商绑定，不能简单理解为“服务器没处理”。
- HTTP/1.1 Keep-Alive 与 HTTP/2 多路复用的优化点不在同一层。

### 线上排障剧本

1. 先按状态码族（2xx/3xx/4xx/5xx）分层定位。
2. 对关键请求检查方法语义与重试策略是否匹配。
3. 检查跨域请求中 OPTIONS 预检是否异常放大。
4. 检查重定向是否改变方法导致副作用。
5. 拉通前后端日志确认 traceId 全链路一致。

### 高频追问补充（附答案）

1. 为什么 302 可能引发业务副作用？

部分客户端在重定向时可能改变方法语义，导致行为偏差。

2. 304 的核心价值是什么？

复用本地副本，减少下载体积，但仍有请求往返成本。

3. 预检请求什么时候最容易失控？

自定义头过多、跨域策略复杂且无收敛时。

## 进阶追问（补充）

1. 如果网络指标正常但用户仍感知慢，优先看哪一层？

优先看主线程任务和渲染阶段（Layout/Paint/Composite）是否拥塞，网络快不代表交互快。

2. 缓存策略和发布策略为什么必须一起设计？

缓存决定资源复用路径，发布决定资源版本切换；两者脱节会出现版本错配。

3. 如何验证一次链路优化不是偶然收益？

至少做对照组、分位数对比，并覆盖冷缓存与弱网样本。

## 案例追问加深（第三轮）

### 真实故障片段

- 故障表现：同一版本在不同地区首屏耗时差异明显，且刷新方式不同结果不同。
- 排查结论：网络链路、缓存路径和渲染调度叠加导致体验分化，单点优化无法彻底解决。
- 治理动作：建立链路分层观察面板，按导航类型、缓存命中层和渲染阶段联合分析。

### 追问补充

1. 为什么链路问题经常不是单点根因？

Knowledge Graph 本身是多阶段因果链，一个阶段的波动会在后续阶段被放大。

2. 如何快速缩小排障范围？

先固定触发方式和样本维度，再按网络、缓存、渲染三层逐步排除。

## 面试补充

### 30 秒说明

HTTP 请求节点是网络性能和稳定性的核心控制点，方法语义、头部策略和协议版本都需要协同设计。

### 高频追问

1. 预检请求为什么会拖慢接口？

它引入额外往返，频繁触发会显著增加延迟。

2. HTTP/2 的核心收益是什么？

多路复用和头压缩可减少连接与传输开销。

3. TTFB 高通常先看哪里？

先分层看 DNS/TCP/TLS、服务端处理和重定向。

4. 为什么要强调接口幂等？

幂等是安全重试的前提。
