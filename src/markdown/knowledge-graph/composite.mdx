## 概念定义

- 本质定义：合成是浏览器将多个图层按顺序叠加并输出最终帧的阶段。
- 解决问题：在不重复重绘全部内容的前提下完成动画和滚动更新。
- 边界说明：它不负责布局计算，也不负责样式解析。

## 核心机制拆解

### 1) 机制/流程

1. 浏览器根据样式和渲染策略划分图层。
2. 绘制阶段产出的图层内容被提交到合成线程。
3. 合成线程按层级和透明度规则输出最终帧。
4. 帧结果提交给图形系统显示。

### 2) 关键条件与触发点

- 条件 A：动画是否可在合成层执行，决定是否绕开主线程重绘。
- 条件 B：图层数量是否过多，决定内存和调度负担。

### 3) 结果与影响

- 对性能 / 稳定性 / 可维护性的影响：合理分层可提升动画流畅度，过度分层会增加内存占用与合成开销。

## 相似概念对比

| 维度 | 合成层动画 | 主线程动画 | 结论 |
|---|---|---|---|
| 定义 | 主要在合成阶段更新 | 需要主线程参与布局或绘制 | 前者通常更稳定 |
| 触发时机 | 变换和透明度变化 | 布局属性变化 | 触发路径不同 |
| 性能影响 | 主线程压力更小 | 易受脚本和布局影响 | 动画优先选可合成属性 |
| 适用场景 | 位移、透明度动画 | 复杂布局驱动动画 | 需按视觉需求取舍 |

## 代码

```css
.box {
  transition: transform 240ms ease, opacity 240ms ease;
}
.box.active {
  transform: translateY(8px);
  opacity: 0.9;
}
```

### 示例说明

- 这段代码解决什么问题：优先使用可在合成阶段处理的属性提升动画稳定性。
- 关键语句或参数含义：`transform`、`opacity` 通常更容易走合成路径。
- 常见错误写法：动画直接修改 `top`、`left` 触发布局重算。

## 真实场景示例

- 场景描述：卡片悬浮动画在低端机掉帧。
- 问题表现：鼠标移入时明显卡顿。
- 处理方案：动画属性从布局属性改为 `transform` 和 `opacity`。
- 验证方式：对比动画帧稳定性和输入响应延迟。

## 常见误区

- 误区：图层越多越快。
  - 正解：图层过多会增加内存和合成调度成本。

## 关联知识点

- 依赖：依赖 `paint` 产出图层内容。
- 影响：影响 `render-opt` 指标表现。
- 对比：`composite` 聚焦图层叠加，`layout` 聚焦几何计算。
- 组合：常与 `forced-layout` 配合做动画性能治理。

## 相关面试题

1. 浏览器渲染优化
   - 标准答案：动画优化优先选择可走合成阶段的属性，如 `transform` 和 `opacity`，减少布局和重绘参与；同时控制图层数量避免内存膨胀。
   - 考察点：是否理解“可合成属性”和“过度分层”的双向取舍。
