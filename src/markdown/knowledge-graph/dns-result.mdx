## 概念定义

- 本质定义：DNS（Domain Name System）结果与缓存策略是解析结果返回后的地址选择与多级缓存复用机制。
- 解决问题：在保证地址可用的前提下，降低重复查询开销并控制切流收敛速度。
- 边界说明：它不负责权威记录配置发布流程，也不负责应用层重试策略。

## 核心机制拆解

### 1) 机制/流程

1. 递归解析器返回 A / AAAA / CNAME 记录和 TTL（Time To Live）。
2. 浏览器与系统按 TTL 写入本地缓存。
3. 客户端按地址族与可达性策略选择目标地址。
4. 地址被选中后进入 TCP（Transmission Control Protocol）/TLS（Transport Layer Security）建连。
5. 缓存到期或网络环境变化时重新解析。

### 2) 关键条件与触发点

- 条件 A：TTL 是否过期，决定复用缓存还是重新查询。
- 条件 B：多地址记录和网络质量差异，决定最终连接路径。

### 3) 结果与影响

- 对性能 / 稳定性 / 可维护性的影响：合理 TTL 可以兼顾命中率和切流速度；地址选择策略不佳会导致“解析成功但建连慢”。

## 相似概念对比

| 维度 | 强缓存复用 | 重新解析 | 结论 |
|---|---|---|---|
| 定义 | 结果仍在有效期，直接使用本地记录 | 结果过期或失效，重新进入解析链 | 两条路径的时延成本差异显著 |
| 触发时机 | TTL 未过期且缓存可用 | TTL 过期、缓存淘汰、切网或策略变更 | 线上抖动常出现在两种路径切换时 |
| 性能影响 | 少一次到多次查询往返 | 增加 DNS 查询成本 | 优化首访和复访要分别看 |
| 适用场景 | 稳定业务与日常访问 | 发布切流、故障恢复、网络变化 | 需要分阶段配置 TTL |

## 代码

```bash
dig www.example.com
```

```bash
curl --resolve www.example.com:443:1.2.3.4 https://www.example.com -I
```

### 示例说明

- 这段代码 / 配置解决什么问题：`dig` 查看记录和 TTL，`curl --resolve` 验证指定地址的真实连通和响应。
- 关键参数含义：`--resolve` 可把域名临时绑定到指定地址用于对照。
- 常见错误配置：只验证 DNS 时间，不验证建连和首包表现。

## 真实场景示例

- 场景描述：CDN 切流后，部分用户访问偶发超时。
- 问题表现：解析结果包含多个地址，但某地区命中某个地址时失败率高。
- 处理方案：降低切流窗口 TTL，按地区校正解析调度权重，并下线异常节点地址。
- 验证方式：观察各地址命中率、建连成功率和页面 `LCP` 回落情况。

## 常见误区

- 误区：DNS 返回 IP 就说明网络链路一定没问题。
- 正解：解析成功只代表“可寻址”，后续建连质量还受路径和节点负载影响。
- 误区：只要把 TTL 拉长就能提升稳定性。
- 正解：TTL 过长会导致故障切换慢，发布窗口内风险扩大。
- 误区：配了 `preconnect` 就不需要做 DNS 策略治理。
- 正解：`preconnect` 只能前置连接过程，不能替代 TTL、记录质量和调度策略治理。

## 关联知识点

- 依赖：`dns-resolver` 提供完整查询链和最终记录。
- 影响：`dns-result` 决定 `tcp-tls` 的目标地址和连接稳定性。
- 对比：`cache-policy` 处理 HTTP 缓存，`dns-result` 处理域名解析缓存。
- 组合：常与 `tcp-tls`、`status-code-path` 联合定位首访慢问题。

## 相关面试题

1. 当在浏览器中输入 Google.com 并且按下回车之后发生了什么？
- 标准答案：DNS 返回地址后会写入本地缓存，并按 TTL 决定后续是否复用；复用可减少查询开销，但切流时会带来收敛延迟。
- 考察点：是否理解“缓存命中收益”和“发布收敛速度”的取舍。
- 追问 1：为什么有时二次访问仍会走 DNS？
- 标准答案 1：缓存可能已过期、被淘汰或网络切换导致记录失效，因此会重新进入解析流程。
- 追问 2：如何验证是解析结果问题还是服务端问题？
- 标准答案 2：先固定域名并强制绑定不同地址对照请求，再比较建连与响应指标，能区分地址路径问题和服务端处理问题。

2. URL（Uniform Resource Locator）有哪些组成部分
- 标准答案：域名和端口决定 DNS 查询目标与后续连接端点，路径和参数属于应用层路由信息，不影响 DNS 结果本身。
- 考察点：是否能分清 URL 各部分在网络链路中的职责。
