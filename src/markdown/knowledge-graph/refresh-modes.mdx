## 概念定位

刷新模式节点关注普通刷新、强制刷新、地址栏回车等行为差异，核心是缓存策略是否被绕过。

它直接影响用户看到的是缓存内容还是最新资源。

## 工作机制

1. 普通刷新优先使用协商缓存与强缓存规则。
2. 强制刷新通常绕过部分缓存并触发完整资源拉取。
3. 地址栏回车更接近新导航行为，取决于缓存头策略。
4. 浏览器对不同资源类型有不同刷新处理。
5. 刷新行为需要与 Service Worker 策略协同。

## 关键知识点

### 必须掌握

- 304 不是“越多越好”，要看命中对象和请求成本。
- 强刷能解决旧资源问题，但会增加首屏等待。
- 普通刷新和回车行为在缓存层并非完全一致。
- 刷新行为和发布策略（指纹/入口缓存）强相关。
- Service Worker 可能改变刷新预期，需要明确策略。
- 问题排查要同时看请求头和响应头。

### 选型与取舍

- 偏向缓存命中体验快，但新鲜度风险更高。
- 偏向强刷一致性高，但流量和等待成本上升。
- 复杂刷新策略可控性高，但维护成本更高。

## 对页面性能与稳定性的影响

- 节点稳定时，Knowledge Graph 主线 的网络与渲染链路更可预测，性能治理收益更稳定。
- 节点失稳时，问题会向后续阶段扩散，常表现为首屏变慢、交互卡顿或版本一致性风险。
- 优化动作应结合上下游联动验证，避免局部优化把成本转移到其他节点。

## 观测与排障

### 核心指标

- 304 比例与命中对象分布
- 刷新后资源 200/304 比例
- 刷新行为下首屏耗时
- 缓存错配投诉率

### 排障顺序

1. 先确认用户使用的刷新方式。
2. 检查请求头中的缓存控制字段。
3. 检查入口与静态资源缓存头配置。
4. 检查 Service Worker 是否拦截刷新请求。
5. 按不同刷新方式复现并验证。

### 常见故障场景

- 普通刷新看到旧页面，强刷正常。
- 发布后部分用户刷新仍旧版本。
- 强刷后加载明显变慢。
- SW 场景刷新行为与预期不一致。

## 常见误区

- 把所有刷新行为当成同一种缓存策略。
- 只看状态码不看缓存头细节。
- 忽略 Service Worker 干预。

## 题库深挖补充

### 高频对比与边界（基于题库）

- 首次导航、reload、back_forward 三类导航在性能时间线上的语义不同。
- 地址栏回车并不总等于 F5，是否走缓存协商取决于浏览器与缓存头策略。
- 面试高频点：F5 与 Ctrl+F5 的区别，本质是缓存路径是否被强制绕过。
- 页面恢复（Back/Forward Cache, BFCache）命中时，链路会与完整冷启动明显不同。
- 导航阶段埋点字段缺失会让后续网络和渲染问题难归因。
- 重定向链路会增加总往返，尤其在移动网络下放大明显。

### 线上排障剧本

1. 先确认用户触发方式：新开、刷新、强刷、回退。
2. 对照 Navigation Timing 看 type 和关键耗时段。
3. 检查是否存在重定向或缓存策略与预期不一致。
4. 检查是否有 BFCache 命中导致行为差异。
5. 排障结论必须带“触发方式”维度，避免误判。

### 高频追问补充（附答案）

1. 为什么“同页面”在不同用户那里快慢差距很大？

导航类型、缓存命中和网络条件不同，链路不一样。

2. Ctrl+F5 能作为线上问题长期解决方案吗？

不能，它只是临时绕缓存手段，根因通常在发布与缓存策略。

3. BFCache 命中时为什么请求链路变短？

页面可能直接从快照恢复，不必完整重走网络与渲染流程。

## 进阶追问（补充）

1. 如果网络指标正常但用户仍感知慢，优先看哪一层？

优先看主线程任务和渲染阶段（Layout/Paint/Composite）是否拥塞，网络快不代表交互快。

2. 缓存策略和发布策略为什么必须一起设计？

缓存决定资源复用路径，发布决定资源版本切换；两者脱节会出现版本错配。

3. 如何验证一次链路优化不是偶然收益？

至少做对照组、分位数对比，并覆盖冷缓存与弱网样本。

## 案例追问加深（第三轮）

### 真实故障片段

- 故障表现：同一版本在不同地区首屏耗时差异明显，且刷新方式不同结果不同。
- 排查结论：网络链路、缓存路径和渲染调度叠加导致体验分化，单点优化无法彻底解决。
- 治理动作：建立链路分层观察面板，按导航类型、缓存命中层和渲染阶段联合分析。

### 追问补充

1. 为什么链路问题经常不是单点根因？

Knowledge Graph 本身是多阶段因果链，一个阶段的波动会在后续阶段被放大。

2. 如何快速缩小排障范围？

先固定触发方式和样本维度，再按网络、缓存、渲染三层逐步排除。

## 面试补充

### 30 秒说明

刷新模式节点的核心是理解不同用户操作对缓存链路的影响，这直接关系到版本一致性和体验。

### 高频追问

1. 为什么普通刷新有时拿不到最新资源？

普通刷新会优先走缓存协商，若策略配置不当会继续命中旧索引。

2. 304 多一定好吗？

不一定，要看是否命中真正可复用资源，且请求成本可接受。

3. 强刷适合长期解决问题吗？

不适合，强刷是临时手段，根因通常在缓存发布策略。

4. SW 为什么会影响刷新结果？

它可拦截请求并自定义响应来源。
