## 概念定位

Service Worker 检查节点决定请求是否被离线缓存与自定义策略拦截，是网络链路的重要分支。

策略正确可提速与离线可用，策略错误会导致缓存污染和版本错乱。

## 工作机制

1. 浏览器在受控范围内将请求交给 Service Worker。
2. 按策略选择 cache-first/network-first/stale-while-revalidate。
3. 命中缓存直接返回，未命中回源并可写回缓存。
4. 新 SW 安装激活后接管控制范围。
5. 版本更新时处理旧缓存清理与客户端同步。

## 关键知识点

### 必须掌握

- SW 生命周期（install/activate/fetch）必须理解透。
- 缓存 key 设计不当会导致脏数据。
- 更新策略要防止长期停留旧 worker。
- 离线兜底页面应明确能力边界。
- SW 调试要区分浏览器缓存与 SW 缓存。
- 发布策略要和 SW 缓存版本联动。

### 选型与取舍

- 激进缓存响应快，但一致性风险高。
- 网络优先一致性好，但离线能力弱。
- 自动接管更新快，但可能中断会话。

## 对页面性能与稳定性的影响

- 节点稳定时，Knowledge Graph 主线 的网络与渲染链路更可预测，性能治理收益更稳定。
- 节点失稳时，问题会向后续阶段扩散，常表现为首屏变慢、交互卡顿或版本一致性风险。
- 优化动作应结合上下游联动验证，避免局部优化把成本转移到其他节点。

## 观测与排障

### 核心指标

- SW 命中率
- 离线可用率
- 版本错配率
- SW 更新成功率

### 排障顺序

1. 先确认请求是否在 SW 控制范围。
2. 检查 fetch 策略分支命中路径。
3. 检查缓存 key 与版本前缀。
4. 检查更新激活和旧缓存清理。
5. 在无痕/新会话回归。

### 常见故障场景

- 发布后用户仍用旧资源。
- 离线可访问但数据长期过旧。
- SW 更新后偶发白屏。
- 调试时缓存来源判断错误。

## 常见误区

- 只实现缓存命中，不设计更新策略。
- SW 缓存与构建版本不关联。
- 忽视 SW 生命周期边界。

## 题库深挖补充

### 高频对比与边界（基于题库）

- 流式响应场景下，首段可见时间通常比整包返回更关键。
- SSE（Server-Sent Events）适合单向推送，WebSocket 适合双向高频交互。
- Chunk 太碎会增加解析与渲染调度成本，太大又会拉高首段等待。
- 背压（Backpressure）治理不到位会出现“网络没堵，前端先卡”的情况。
- Service Worker 介入后，流式链路可能被重写或缓存，需单独验证。
- 面试常问“流式为什么还会卡”，根因通常在前端逐条渲染和主线程拥塞。

### 线上排障剧本

1. 区分卡在网络到达、协议处理还是 UI 渲染。
2. 观察 chunk 到达节奏与前端消费节奏是否匹配。
3. 检查是否逐条触发状态更新导致重渲染风暴。
4. 检查断流重连是否有指数退避和幂等去重。
5. 针对弱网和高并发用户做专项压测。

### 高频追问补充（附答案）

1. SSE 与 WebSocket 怎么选？

单向连续输出优先 SSE，强双向交互优先 WebSocket。

2. 为什么流式场景更容易出现重复内容？

断线重连后若无 messageId 去重，历史片段会重复消费。

3. 前端如何减少流式渲染卡顿？

做批处理、按帧刷新、限制单次更新量。

## 进阶追问（补充）

1. 如果网络指标正常但用户仍感知慢，优先看哪一层？

优先看主线程任务和渲染阶段（Layout/Paint/Composite）是否拥塞，网络快不代表交互快。

2. 缓存策略和发布策略为什么必须一起设计？

缓存决定资源复用路径，发布决定资源版本切换；两者脱节会出现版本错配。

3. 如何验证一次链路优化不是偶然收益？

至少做对照组、分位数对比，并覆盖冷缓存与弱网样本。

## 案例追问加深（第三轮）

### 真实故障片段

- 故障表现：同一版本在不同地区首屏耗时差异明显，且刷新方式不同结果不同。
- 排查结论：网络链路、缓存路径和渲染调度叠加导致体验分化，单点优化无法彻底解决。
- 治理动作：建立链路分层观察面板，按导航类型、缓存命中层和渲染阶段联合分析。

### 追问补充

1. 为什么链路问题经常不是单点根因？

Knowledge Graph 本身是多阶段因果链，一个阶段的波动会在后续阶段被放大。

2. 如何快速缩小排障范围？

先固定触发方式和样本维度，再按网络、缓存、渲染三层逐步排除。

## 面试补充

### 30 秒说明

SW 节点本质是“可编程缓存层”，收益高但治理要求高，必须把版本和更新策略设计清楚。

### 高频追问

1. cache-first 适合什么资源？

低频变更的静态资源，且可容忍短时旧版本。

2. 为什么 SW 容易造成旧版本问题？

更新激活和缓存清理策略不当会长期命中旧缓存。

3. 如何判断请求是否被 SW 拦截？

在 DevTools 查看请求来源与 Service Worker 控制状态。

4. SW 更新为什么要谨慎 skipWaiting？

可能在用户会话中途切换版本，引发状态不一致。
