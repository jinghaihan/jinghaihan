## 概念定义

- 本质定义：Service Worker 拦截链是在浏览器请求发出前由独立线程接管、匹配和返回响应的机制。
- 解决问题：实现离线访问、缓存加速和可控回源策略。
- 边界说明：它不替代服务端缓存，也不能直接操作页面 DOM（Document Object Model）。

## 核心机制拆解

### 1) 机制/流程

1. 页面注册 Service Worker 并安装到本地。
2. 激活后拦截符合作用域的请求。
3. 先查命名缓存，命中则直接返回。
4. 未命中时回源拉取并按策略写回缓存。
5. 版本更新时通过激活流程切换新脚本。

### 2) 关键条件与触发点

- 条件 A：页面必须运行在 HTTPS（HyperText Transfer Protocol Secure）或本地安全环境。
- 条件 B：作用域与缓存版本策略是否正确，决定拦截是否生效。

### 3) 结果与影响

- 对性能 / 稳定性 / 可维护性的影响：合理策略可显著提升离线可用性和复访速度，策略失误会导致陈旧资源长期驻留。

## 相似概念对比

| 维度 | Service Worker 缓存 | 浏览器内置缓存 | 结论 |
|---|---|---|---|
| 定义 | 开发者可编排的请求拦截与缓存 | 浏览器按头字段自动缓存 | 前者可控性高，后者成本低 |
| 触发时机 | 命中作用域请求时 | 普通资源请求时 | 两者可能叠加生效 |
| 性能影响 | 可离线响应和自定义预热 | 命中成本低、实现简单 | 组合使用收益最高 |
| 适用场景 | 离线应用、精细缓存治理 | 通用静态资源缓存 | 面试常问策略差异 |

## 代码

```js
// sw.js
self.addEventListener('fetch', (event) => {
  event.respondWith(
    caches.match(event.request).then((cached) => {
      if (cached) return cached
      return fetch(event.request)
    }),
  )
})
```

### 示例说明

- 这段代码解决什么问题：请求先走缓存命中，未命中再回源。
- 关键语句或参数含义：`event.respondWith` 用于接管浏览器默认请求流程。
- 常见错误写法：只读缓存不做版本淘汰，导致老资源无法更新。

## 真实场景示例

- 场景描述：弱网地区用户首屏白屏率较高。
- 问题表现：静态资源下载超时导致页面不可用。
- 处理方案：关键资源预缓存，非关键资源回源后写回。
- 验证方式：断网和弱网条件下验证可访问率与加载时长。

## 常见误区

- 误区：注册后立即生效所有页面。
- 正解：Service Worker 受作用域和生命周期控制，激活与接管有时序。

## 关联知识点

- 依赖：依赖 `refresh-modes` 触发路径与网络状态。
- 影响：命中失败后会继续进入 `memory-cache` 与 `disk-cache`。
- 对比：`sw-check` 是可编排拦截层，`memory-cache` 是浏览器内部存储层。
- 组合：常与 `html-offline-storage` 节点联动设计离线体验。

## 相关面试题

1. 对 Service Worker 的理解
- 标准答案：Service Worker 是浏览器后台独立线程，可拦截请求并结合缓存策略返回响应，常用于离线能力和复访加速；前提是安全上下文和正确生命周期管理。
- 考察点：是否能说清“独立线程 + 请求拦截 + 生命周期”三要素。
- 追问 1：为什么 Service Worker 要求 HTTPS？
- 标准答案 1：它能拦截请求并影响返回内容，必须在可信链路下运行以避免被中间人篡改。
