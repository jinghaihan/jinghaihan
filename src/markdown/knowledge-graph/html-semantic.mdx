## 概念定位

HTML 语义结构与可访问性是所有渲染与交互的输入层。它不仅影响搜索引擎抓取，还直接决定屏幕阅读器、键盘导航、表单可用性和后续 CSS/JavaScript 的可维护性。

这个节点如果做对了，后续样式和脚本会更稳定；做错了，后续会出现“样式补丁越来越多、可访问性缺陷持续复发”的连锁问题。

## 工作机制

1. 先按信息层级确定语义骨架：`header/main/aside/footer/section/article/nav`，不要一上来就用 `div` 平铺。
2. 把内容语义和交互语义拆开：内容标题用 `h1~h6`，交互按钮用 `button`，跳转用 `a`。
3. 为媒体和表单补齐可访问信息：`alt`、`label`、`aria-*`、错误提示关联（`aria-describedby`）。
4. 在组件化阶段保持语义不丢失：封装组件时保留原生标签能力，不要把所有按钮封成 `div role="button"`。
5. 上线前做结构化校验：语义审计 + 键盘可达性 + 屏幕阅读器快速走查。

## 关键知识点

### 必须掌握

- `title`、`h1`、`meta description` 作用不同：`title` 影响页面主题与标签栏，`h1` 影响正文主标题语义。
- `label` 必须和表单控件关联，否则点击热区和读屏体验都变差。
- 空元素（void elements）如 `img`、`input` 不应写成成对闭合语义。
- 语义化不是“为了 SEO（Search Engine Optimization）”，更是为了“可读、可维护、可访问”。
- 避免用 `strong`/`em` 只做视觉样式，它们本质是语义强调。
- 复杂区域优先语义分块，再做样式布局，能显著降低后续 CSS 冲突。

### 选型与取舍

- 短期“纯 div + class”写法看起来快，但后续改版成本和无障碍改造成本显著更高。
- 语义标签略增加前期思考成本，但能减少组件层的补丁逻辑和选择器复杂度。
- ARIA（Accessible Rich Internet Applications）是补充手段，不应该替代原生语义标签。

## 对页面性能与稳定性的影响

- 节点稳定时，前端基础（HTML/CSS/JavaScript） 的链路可预测性会明显提升，发布和排障成本更可控。
- 节点失稳时，问题会向上下游扩散，常表现为性能回退、交互异常或交付风险叠加。
- 该节点优化应和上下游协同验证，避免“局部优化把成本转移给后续节点”。

## 观测与排障

### 核心指标

- 可访问性评分（Lighthouse Accessibility）
- 语义结构错误数（重复 `h1`、缺失 `label`、无 `alt`）
- 键盘可达率（核心交互是否可 Tab 到达并触发）
- SEO 基础项通过率（title/description/canonical）

### 排障顺序

1. 先看结构：检查标题层级、地标标签（landmarks）是否完整。
2. 再看交互：按钮是否可聚焦、可回车触发、状态是否可读。
3. 再看表单：错误提示是否和输入框建立可访问关联。
4. 最后看组件封装：是否在 UI 组件层丢失原生语义。
5. 修复后跑自动化审计，再用真实读屏工具做抽样验证。

### 常见故障场景

- “看起来可点但键盘无法操作”：通常是 `div onClick` 代替 `button`。
- “读屏读不出图片含义”：通常是 `img` 缺失 `alt` 或 `alt` 过于泛化。
- “SEO 抓取效果差”：常见是标题层级混乱、首屏语义信息不足。
- “表单可用性差”：常见是 `label` 未绑定导致点击与朗读脱节。

## 常见误区

- 把语义化理解成“加几个标签名”而不是“建立信息层级”。
- 过度依赖 ARIA 属性去补原生语义缺陷。
- 仅在上线前一次性修语义，不把它纳入日常评审。

## 题库深挖补充

### 高频对比与边界（基于题库）

- title、h1、meta description 作用不同：分别服务标签主题、正文主标题和摘要描述。
- label 与表单控件必须关联，否则键盘可达性和读屏体验都会下降。
- strong/em 是语义强调，b/i 更偏视觉；面试常考两者差异。
- iframe 需要权衡隔离性与性能开销，不应默认作为组件隔离手段。
- 语义化不仅是 SEO（Search Engine Optimization）问题，也是可维护性问题。
- HTML 结构一旦混乱，后续 CSS/JS 成本会持续放大。

### 线上排障剧本

1. 先做语义结构审查：地标标签、标题层级、表单关联。
2. 再做可访问性审查：键盘路径、焦点顺序、读屏朗读。
3. 再检查是否有过度 div 化导致结构信息丢失。
4. 组件层修复时保留原生标签语义，不做伪语义替代。
5. 把审查项加入评审清单与自动化检查。

### 高频追问补充（附答案）

1. label 的核心价值是什么？

扩大可点击区域并提供可访问语义，是表单可用性的基础。

2. title 和 h1 可以互相替代吗？

不能。两者面向的语义层级不同，应该各司其职。

3. 语义化为什么会影响性能治理？

结构清晰可减少后续样式和脚本补丁复杂度。

## 进阶追问（补充）

1. 为什么很多前端性能问题最终会回到“基础知识点”？

因为事件循环、布局模型、缓存与请求语义是上层框架和业务逻辑的底层约束。

2. 如何区分“语法会用”和“机制掌握”？

能解释边界、失效条件和排障顺序，才算机制掌握，而不只是 API 会写。

3. 性能优化里最容易被忽略的一步是什么？

对照验证和防回退机制，经常只做了修复动作没做持续守护。

## 案例追问加深（第三轮）

### 真实故障片段

- 故障表现：页面逻辑正确但体感卡顿，且在低端设备明显更差。
- 排查结论：基础层问题叠加，包括事件循环拥塞、样式重算成本和请求并发失控。
- 治理动作：按主线程任务、样式布局、请求治理三条线拆解并建立基线。

### 追问补充

1. 为什么基础知识点决定上层框架表现？

框架最终都运行在浏览器基础机制之上，底层约束没处理好，上层优化很难生效。

2. 如何避免“会写代码但不会排障”？

建立机制到现象的映射：看到症状能快速关联到可能的底层阶段。

## 面试补充

### 30 秒说明

HTML 语义结构的价值在于把“内容结构、交互语义、可访问能力”在输入层一次建对。对工程而言，这是后续样式和脚本复杂度的上限控制点。

### 高频追问

1. `label` 的价值到底是什么？

`label` 不只是显示文本，它让点击热区扩大并让读屏器正确播报表单含义，是表单可用性的基础。

2. `strong` 和 `b`、`em` 和 `i` 的区别是什么？

`strong`/`em` 带语义强调，`b`/`i` 更偏视觉表现。语义优先时应优先使用前者。

3. 为什么说语义化和性能也有关？

语义结构清晰可以减少后续“样式补丁 + 脚本修复”成本，间接降低样式重算与 DOM 操作复杂度。

4. ARIA 能完全替代语义标签吗？

不能。ARIA 是补充，不是替代。优先用原生语义，再用 ARIA 补充复杂交互语义。
