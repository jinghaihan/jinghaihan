## 概念定位

SFC（Single File Component）与模板编译节点关注“模板如何变成高效渲染函数”，核心是编译优化与运行时成本控制。

它是 Vue 渲染性能和可维护语法的关键连接层。

## 工作机制

1. SFC 经过 parse -> transform -> generate 转换为可执行渲染代码。
2. 编译阶段会做静态提升、补丁标记（Patch Flags）和事件缓存优化。
3. 运行时根据 patch flag 走最小更新路径，减少不必要 diff。
4. 模板指令（v-if/v-for/v-model）会转换为不同更新分支。
5. 复杂模板需配合组件拆分降低编译与渲染负担。

## 关键知识点

### 必须掌握

- 静态提升可减少重复 vnode 创建。
- v-if 和 v-show 的代价模型不同：前者切换重建，后者常驻切换显示。
- v-for 需要稳定 key 才能高效复用节点。
- 编译告警常提示可优化结构。
- 模板表达式过重会把业务计算放入渲染路径。
- SFC 作用域样式要关注选择器成本。

### 选型与取舍

- 模板语法可读性高，但过度复杂会拖慢编译和渲染。
- 内联计算写法方便，但会增加更新成本。
- 组件拆分提高复用，但拆太细会增加通信成本。

## 对页面性能与稳定性的影响

- 节点稳定时，Vue 工作流 的链路可预测性会明显提升，发布和排障成本更可控。
- 节点失稳时，问题会向上下游扩散，常表现为性能回退、交互异常或交付风险叠加。
- 该节点优化应和上下游协同验证，避免局部优化把成本转移给后续节点。

## 观测与排障

### 核心指标

- 编译告警数量
- 组件更新耗时
- 无效重渲染次数
- 首屏渲染耗时

### 排障顺序

1. 先定位高频重渲染组件。
2. 检查模板中是否存在高成本表达式。
3. 检查 key、条件分支和列表渲染策略。
4. 查看编译输出和告警提示。
5. 通过组件拆分和缓存策略回归验证。

### 常见故障场景

- 列表更新时整块重渲染。
- 条件切换频繁导致抖动。
- 模板里复杂计算导致卡顿。
- 作用域样式选择器过深导致重算慢。

## 常见误区

- 把模板当纯声明层，不关注运行时成本。
- 忽略 key 稳定性。
- 复杂页面不拆组件，导致单组件过重。

## 深入拆解

### 编译优化点

- 静态节点提升（hoist）减少重复创建。
- Patch Flag 标记动态字段，避免全量 diff。
- 事件处理器缓存减少重复函数创建。

### 指令成本模型

| 指令 | 主要成本 | 优化要点 |
| --- | --- | --- |
| v-if | 条件切换重建销毁 | 低频切换使用 |
| v-show | 常驻节点样式切换 | 高频切换使用 |
| v-for | 列表 diff 与复用 | key 稳定 + 控制列表规模 |

### 模板性能审查清单

1. 是否有复杂表达式放在模板中。
2. 是否有不稳定 key。
3. 是否可拆分重组件为子组件。

## 题库深挖补充

### 高频对比与边界（基于题库）

- 路由驱动请求要处理竞态：后发先至会覆盖新状态。
- Pinia 状态分层要区分持久状态、会话状态、派生状态。
- SFC 模板编译需理解 Patch Flag、静态提升和 key 稳定性。
- Vue 性能优化先看热点组件和更新频率，再决定是否缓存或拆分。
- 异步请求治理需覆盖超时、重试、取消和错误兜底。
- 面试常问“v-if/v-show、computed/watch、nextTick”，要能结合性能语义说明。

### 线上排障剧本

1. 先区分卡顿来自请求链路还是渲染链路。
2. 检查请求取消与去重是否生效。
3. 检查列表 key 和模板复杂表达式。
4. 检查 store 更新粒度是否过粗。
5. 通过灰度对照验证优化收益。

### 高频追问补充（附答案）

1. 为什么 v-for 一定要稳定 key？

否则节点复用错位，性能和状态正确性都会受影响。

2. computed 和 watch 的边界？

computed 用于派生值缓存，watch 用于副作用响应。

3. nextTick 解决的核心问题是什么？

在 DOM 更新批处理后读取最新视图状态。

## 进阶深挖（第二轮）

### 模板编译实战检查

1. 动态节点比例是否过高。
2. 列表渲染是否具备稳定 key。
3. 条件分支是否会频繁销毁重建。
4. 模板表达式是否承担了业务计算。

### 复杂页面优化顺序

- 先拆重组件，再看模板表达式成本。
- 先稳定 key，再做列表微优化。
- 先减少无效更新，再考虑细粒度缓存。

### 追问扩展

1. Patch Flag 为什么是高频考点？

它直接决定运行时 diff 的范围，是编译优化落地的关键证据。

2. 模板可读性和性能冲突时怎么取舍？

先保证语义清晰，再通过组件拆分和预计算平衡性能。

## 面试补充

### 30 秒说明

模板编译节点的核心是让声明式语法转成高效执行路径，理解编译优化点才能真正做出 Vue 性能优化。

### 高频追问

1. Patch Flag 的价值是什么？

告诉运行时哪些部分是动态的，从而减少全量 diff。

2. v-if 和 v-show 怎么选？

低频切换选 v-if，高频切换且需要快速响应常选 v-show。

3. 为什么 v-for 强调 key？

稳定 key 才能保证节点复用和状态正确映射。

4. 模板里写复杂计算有什么问题？

会把计算成本放到每次渲染更新路径中。
