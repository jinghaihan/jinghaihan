## 概念定义

- 本质定义：Webpack 产物与性能优化是围绕体积、加载时序和缓存稳定性对打包结果做系统治理。
- 解决问题：解决首屏慢、重复下载、缓存失效和产物膨胀问题。
- 边界说明：不替代后端接口优化和浏览器渲染层优化。

## 核心机制拆解

### 1) 机制/流程

1. 通过代码分割把首屏关键路径与低频模块拆开。
2. 对稳定依赖做公共包抽离，降低重复下载。
3. 利用 Tree Shaking 删除无效代码。
4. 产物文件使用 `contenthash`，建立长期缓存能力。
5. 结合资源分析报告持续校验优化收益。

### 2) 关键条件与触发点

- 条件 A：入口包体积持续增长时，必须回看拆包与依赖引入策略。
- 条件 B：发布后出现“新旧代码混用”时，通常是缓存和指纹策略不匹配。

### 3) 结果与影响

- 对性能的影响：可降低首包体积、减少解析执行时间并提高缓存命中率。
- 对稳定性的影响：合理指纹策略可减少发布后的版本错配风险。

## 相似概念对比

| 维度 | Tree Shaking | Code Splitting | 结论 |
|---|---|---|---|
| 定义 | 删除未使用导出代码 | 把代码拆为多个按需加载单元 | 目标不同但可叠加 |
| 触发时机 | 构建优化阶段 | 构建阶段和运行时加载阶段 | 一个“删”，一个“拆” |
| 性能影响 | 降低总包体积 | 降低首屏负担、提升缓存复用 | 常需组合使用 |
| 适用场景 | 库体积治理 | 路由和功能级按需加载 | 不应二选一 |

## 配置示例

```ts
import path from 'node:path'

export default {
  mode: 'production',
  output: {
    path: path.resolve(process.cwd(), 'dist'),
    filename: 'js/[name].[contenthash:8].js',
    chunkFilename: 'js/[name].[contenthash:8].js',
  },
  optimization: {
    splitChunks: {
      chunks: 'all',
      cacheGroups: {
        vendor: {
          test: /[\\/]node_modules[\\/]/,
          name: 'vendor',
          priority: 10,
        },
      },
    },
    usedExports: true,
  },
}
```

### 示例说明

- 这段配置同时覆盖了公共依赖抽离、按需分包和内容哈希命名。
- `usedExports: true` 为摇树优化提供静态分析前提。
- 常见错误写法是只配 `splitChunks`，但文件名没有哈希，导致缓存污染。

## 真实场景示例

- 场景描述：活动页版本迭代快，用户反馈“刚发布还是旧页面”。
- 问题表现：部分用户命中旧 JS，出现接口字段不匹配报错。
- 处理方案：主资源文件改用 `contenthash`，入口 HTML 缩短缓存，发布时先传资源再切入口。
- 验证方式：发布后多地区访问资源哈希一致，旧版本报错在 10 分钟内下降到基线。

## 常见误区

- 误区：体积越小就一定更快。
- 正解：还要看请求数、解析成本和执行时序。
- 误区：拆包越细越好。
- 正解：过细拆包会增加请求开销和调度成本。

## 相关面试题

1. 如何用 webpack 来优化前端性能？
- 标准答案：核心手段包括代码压缩、公共依赖抽离、按路由或功能拆包、Tree Shaking、资源指纹化和 CDN 路径治理。机制上要同时覆盖“体积、加载顺序、缓存复用”三条线，单做压缩通常不够。
- 考察点：是否具备系统化优化思路，而非单点技巧罗列。
