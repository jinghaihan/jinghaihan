## 概念定义

- 本质定义：小程序组件化用于封装视图和行为，事件系统则负责在组件和页面之间传递交互意图。
- 解决问题：解释“父子组件之间如何通信”，也解释“为什么同样是点击事件，`bindtap` 和 `catchtap` 行为不同”。
- 边界说明：本节点聚焦组件通信和事件传播，不展开复杂状态管理方案。

## 核心机制拆解

### 1) 机制/流程

1. 自定义组件通过 `properties` 接收父组件传入的数据。
2. 组件内部状态放在 `data` 中，通过 `setData` 更新视图。
3. 子组件需要把事件或结果通知父层时，通过 `triggerEvent` 向外派发自定义事件。
4. 视图事件可用 `bind` 系列让事件继续冒泡，也可用 `catch` 系列在当前层截断。

### 2) 关键条件与触发点

- 条件 A：父传子适合静态输入和配置，子传父适合交互结果回传。
- 条件 B：事件对象里的 `target` 和 `currentTarget` 语义不同，取错会导致数据错位。
- 条件 C：列表和嵌套组件场景里，是否让事件冒泡要提前设计清楚。

### 3) 结果与影响

- 对性能/稳定性/可维护性的影响：组件边界和事件流清晰后，页面复用和排障都会更轻松；设计混乱时，事件冲突和状态串用会很常见。

## 相似概念对比

| 维度 | `properties` | `triggerEvent` | 结论 |
|---|---|---|---|
| 通信方向 | 父传子 | 子传父 | 方向相反 |
| 适用场景 | 输入配置、初始化数据 | 交互结果、状态变化通知 | 组合使用最常见 |
| 数据来源 | 父组件 | 子组件 | 责任不同 |
| 风险点 | 过度依赖大对象透传 | 事件名和参数约定混乱 | 契约要清晰 |

## 代码

```js
Component({
  properties: {
    title: String,
  },
  methods: {
    handleTap() {
      this.triggerEvent('countchange', { count: 1 })
    },
  },
})
```

```xml
<my-counter title="计数器" bind:countchange="onCountChange" />
<view bindtap="handleTap"></view>
<view catchtap="handleTap"></view>
```

### 示例说明

- 这段代码解决什么问题：前一段展示父子组件通信，后一段展示同一类事件不同绑定方式的差异。
- 关键语句或参数含义：`properties` 定义输入；`triggerEvent` 派发自定义事件；`bindtap` 允许冒泡，`catchtap` 会截断。
- 常见错误写法：组件内部直接改父层数据、事件名没有统一规范、误把 `target` 当成绑定节点。

## 真实场景示例

- 场景描述：商品卡片组件要在点击“加入购物车”时通知列表页更新角标。
- 问题表现：如果组件和页面直接互相取状态，复用性会很差；如果事件冒泡设计不清，点击按钮还会误触父层跳详情。
- 处理方案：商品卡片通过 `properties` 接收商品信息，通过 `triggerEvent` 回传操作结果，必要处用 `catchtap` 截断冒泡。
- 验证方式：看组件复用是否稳定、事件是否只触发一次、父层状态是否按预期更新。

## 常见误区

- 误区：组件通信只能靠全局变量。
- 正解：大多数父子通信场景靠 `properties` 和 `triggerEvent` 就能解决。
- 误区：事件冒泡一定更方便。
- 正解：冒泡是默认协作方式，但按钮嵌套卡片这类场景常常需要明确截断。

## 关联知识点

- 依赖：数据绑定、生命周期和页面交互。
- 影响：组件复用性、交互准确性和状态边界。
- 对比：`properties` 与 `triggerEvent`、`bindtap` 与 `catchtap`。
- 组合：列表页、表单组件、弹层和卡片组件。

## 相关面试题

1. 小程序的组件化开发如何实现？
- 标准答案：小程序通过 `Component()` 创建自定义组件，用 `properties` 定义输入，用 `data` 管内部状态，用 `methods` 承载交互逻辑。父传子靠 `properties`，子传父靠 `triggerEvent`，页面里再通过 `usingComponents` 注册组件。结论是组件通信要靠明确契约，而不是互相直接取状态。
- 考察点：是否理解小程序组件化的基本组成和通信方式。

2. 小程序的事件系统是怎样的？
- 标准答案：小程序事件系统包括事件绑定、事件对象和事件传播。常见绑定方式有 `bind` 和 `catch`，前者允许冒泡，后者会阻止继续冒泡；事件对象里常用 `type`、`target`、`currentTarget` 和 `dataset`。场景上列表点击、表单输入和组件自定义事件都建立在这套机制上。
- 考察点：是否理解事件绑定方式和事件对象的区别。
