## 概念定义

- 本质定义：Webpack 模块图与构建流程是把入口文件递归解析为依赖图，再产出可部署资源的编译链路。
- 解决问题：解决多文件依赖管理、代码分包和产物输出一致性问题。
- 边界说明：不直接解决发布策略和运行时缓存命中策略。

## 核心机制拆解

### 1) 机制/流程

1. 读取配置并创建 `Compiler`，确定入口和输出规则。
2. 从 `entry` 出发解析依赖，逐个构建 `module`。
3. 根据依赖关系把 `module` 组装为 `chunk`。
4. 将 `chunk` 输出为 `bundle`，并写入文件系统。
5. 在生命周期钩子里执行扩展逻辑，生成额外元信息。

### 2) 关键条件与触发点

- 条件 A：入口越多且共享依赖越复杂，`chunk` 拆分策略影响越大。
- 条件 B：依赖图中存在动态导入时，会触发异步 `chunk` 生成。

### 3) 结果与影响

- 对性能的影响：合理 `chunk` 结构可降低首屏体积并提升缓存复用。
- 对可维护性的影响：清晰模块边界可降低改动影响面，便于定位构建异常。

## 相似概念对比

| 维度 | module | chunk / bundle | 结论 |
|---|---|---|---|
| 定义 | 源码中的文件模块单元 | `chunk` 是分组单元，`bundle` 是输出文件 | 三者处于不同层级 |
| 触发时机 | 依赖解析阶段 | 优化与输出阶段 | 先有 `module`，再有 `chunk` 和 `bundle` |
| 性能影响 | 决定可拆分粒度 | 决定加载请求和缓存策略 | 性能优化主要落在 `chunk` 设计 |
| 适用场景 | 代码组织和依赖表达 | 发布产物和加载策略设计 | 不能混为同一概念 |

## 配置示例

```ts
import path from 'node:path'

export default {
  mode: 'production',
  entry: {
    app: './src/main.ts',
  },
  output: {
    path: path.resolve(process.cwd(), 'dist'),
    filename: 'js/[name].[contenthash:8].js',
    clean: true,
  },
  optimization: {
    splitChunks: {
      chunks: 'all',
    },
  },
}
```

### 示例说明

- 这段配置演示了从单入口到按需拆分再到内容哈希输出的最小主链路。
- `splitChunks.chunks = 'all'` 会把同步和异步依赖都纳入拆分候选。
- 常见错误写法是只关心总包体积，不看拆分后请求瀑布和重复依赖。

## 真实场景示例

- 场景描述：电商主站一次上线后首屏 JS 从 280 KB 增长到 620 KB。
- 问题表现：首页 FCP 指标明显下降，低端机白屏时间变长。
- 处理方案：重构入口依赖，把编辑器和低频模块改为动态导入，并拆出稳定 vendor chunk。
- 验证方式：对比构建报告，确认首屏 chunk 降到 320 KB，首屏指标恢复。

## 常见误区

- 误区：`bundle`、`chunk`、`module` 是同义词。
- 正解：`module` 是源码单元，`chunk` 是分组单元，`bundle` 是输出文件。
- 误区：构建流程只有“打包”一步。
- 正解：至少包含参数初始化、模块编译、依赖组装、输出写入和钩子扩展。

## 关联知识点

- 依赖：构建工具选型对比。
- 影响：产物与性能优化、构建提速策略。
- 组合：Loader 与 Plugin 扩展机制。

## 相关面试题

1. bundle，chunk，module 是什么？
- 标准答案：`module` 是源码依赖图中的基本单元，通常对应一个文件；`chunk` 是多个 `module` 经过分组后的加载单元；`bundle` 是最终落盘或下发到浏览器的文件。结论是三者分别对应“源码层、分组层、产物层”。
- 考察点：是否真正理解 Webpack 的抽象层次，而不是只背术语。

2. webpack 的构建流程？
- 标准答案：核心流程是读取配置并初始化编译器、确定入口、递归编译模块、组装 `chunk`、输出 `bundle`。插件通过生命周期钩子介入流程并改变结果，所以流程既是串行主链路，也包含可扩展拦截点。
- 考察点：是否能把“依赖图构建”和“生命周期扩展”一起讲清楚。
