## 概念定位

Effect 生命周期治理节点关注 useEffect 的执行时机、依赖管理和清理语义，目标是避免副作用失控。

这是 React 面试和实战都高频的坑点节点。

## 工作机制

1. 渲染提交后才会执行 effect，不应在渲染阶段做副作用。
2. 依赖数组决定 effect 重跑条件，缺项会导致闭包旧值。
3. 清理函数在下一次执行前或组件卸载时触发。
4. 异步 effect 需要处理取消与竞态。
5. 不同 effect 目的分拆，避免单 effect 过载。

## 关键知识点

### 必须掌握

- Strict Mode 下开发环境可能触发双调用，需要幂等设计。
- 依赖项应完整，靠 eslint-plugin-react-hooks 保底。
- effect 不等同生命周期函数机械迁移。
- 订阅、定时器、监听器必须有清理。
- useLayoutEffect 与 useEffect 时机不同，要按场景选。
- 把可计算逻辑放回渲染层，effect 保持副作用职责。

### 选型与取舍

- 依赖写全更正确，但可能引发重跑频率上升。
- 依赖省略可暂时减少执行，但会埋时序 bug。
- 拆分 effect 可维护性更好，但代码组织更细。

## 对页面性能与稳定性的影响

- 节点稳定时，React 工作流 的链路可预测性会明显提升，发布和排障成本更可控。
- 节点失稳时，问题会向上下游扩散，常表现为性能回退、交互异常或交付风险叠加。
- 该节点优化应和上下游协同验证，避免局部优化把成本转移给后续节点。

## 观测与排障

### 核心指标

- effect 重跑次数
- 副作用泄漏告警数
- 竞态导致状态覆盖次数
- 卸载后更新告警数

### 排障顺序

1. 先确认问题是重跑过多还是未及时执行。
2. 检查依赖数组和闭包引用。
3. 检查清理函数是否完整。
4. 异步场景加取消标记或 AbortController。
5. 将大 effect 拆分为职责单一 effect。

### 常见故障场景

- 请求重复发送。
- 卸载后仍在 setState。
- 依赖漏写导致读取旧值。
- Strict Mode 下表现和生产不一致。

## 常见误区

- 把所有逻辑都塞进一个 effect。
- 为防重跑而故意漏依赖。
- 忽略清理函数。

## 深入拆解

### useEffect 与 useLayoutEffect

| Hook | 执行时机 | 适用场景 |
| --- | --- | --- |
| useEffect | 提交后异步执行 | 请求、订阅、日志 |
| useLayoutEffect | 提交后绘制前同步执行 | 需要读写布局 |

### 依赖数组排查清单

1. 是否遗漏闭包中使用的值。
2. 是否把不稳定对象/函数直接放依赖。
3. 是否可通过 useMemo/useCallback 稳定引用。

### 副作用拆分原则

- 请求、订阅、DOM 操作分开 effect。
- 每个 effect 只负责一类生命周期。

## 题库深挖补充

### 高频对比与边界（基于题库）

- useEffect 关键是依赖完整和清理完整，省依赖是高风险短路。
- Context 适合低频上下文，不适合高频大状态共享。
- 路由与数据获取策略要处理预取命中率和取消竞态。
- 性能优化应先看 Profiler 热点，再决定 memo/useMemo/useCallback。
- 状态管理选型要区分本地 UI 状态与服务端状态。
- 面试常问“Fiber/并发有什么价值”，实战落点是更好的交互优先级调度。

### 线上排障剧本

1. 先确认瓶颈是副作用重跑、状态扩散还是渲染热点。
2. 检查 effect 依赖和异步取消机制。
3. 检查 selector 粒度与订阅范围。
4. 检查路由切换期间数据请求竞态。
5. 用对照组验证优化前后 INP 和重渲染分布。

### 高频追问补充（附答案）

1. 为什么 useEffect 容易出事故？

依赖漏写、清理缺失和异步竞态会叠加放大问题。

2. React.memo 为什么不是万能优化？

它也有比较与维护成本，需命中稳定输入场景才有收益。

3. 状态管理何时该拆 store？

当更新频率和业务边界差异明显时，应拆分避免更新扩散。

## 进阶深挖（第二轮）

### Effect 设计清单

1. 这个 effect 是否真的是副作用。
2. 依赖是否完整且稳定。
3. 是否具备清理函数。
4. 异步场景是否可取消。

### useEffect 事故样例

- 现象：页面切换后仍持续请求。
- 根因：effect 清理漏掉订阅和请求取消。
- 修复：统一封装可取消请求并在 cleanup 执行。

### 追问扩展

1. useLayoutEffect 误用会带来什么问题？

它在绘制前同步执行，滥用会直接增加首帧阻塞。

2. Strict Mode 双调用的实战意义？

帮助提前暴露副作用幂等性问题，避免生产环境潜在风险。

## 面试补充

### 30 秒说明

Effect 治理的关键是时机、依赖、清理三件事。把这三点做对，副作用就能可控。

### 高频追问

1. 为什么 useEffect 会重复执行？

依赖变化会重跑，开发严格模式还会做额外检查执行。

2. 依赖数组能不能手动省略？

不建议，通常会引入闭包旧值 bug。

3. 什么时候用 useLayoutEffect？

需要在浏览器绘制前同步读取/写入布局时。

4. 异步 effect 如何防竞态？

用取消机制或请求版本号，只接收最新结果。
