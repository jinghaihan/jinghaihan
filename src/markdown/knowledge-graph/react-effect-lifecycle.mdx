## 概念定义

- 本质定义：该节点聚焦 副作用可清理可维护。
- 解决问题：把高频面试问题转成可复述、可落地的机制说明。
- 边界说明：不替代其他节点的专属机制域。

## 核心机制拆解

### 1) 机制/流程

1. 明确输入状态和触发条件。
2. 进入框架调度或编译阶段。
3. 执行更新并产出结果。
4. 用回归验证结果是否稳定。

### 2) 关键条件与触发点

- 条件 A：组件边界和数据来源必须清晰。
- 条件 B：触发时机与副作用清理要成对出现。

### 3) 结果与影响

- 对性能/稳定性/可维护性的影响：机制清晰时可预测，机制混乱时会出现渲染回退、状态错乱或协作成本上升。

## 代码

```tsx
import { useEffect, useState } from 'react'

export function Profile({ userId }: { userId: string }) {
  const [name, setName] = useState('')

  useEffect(() => {
    let canceled = false

    fetch(`/api/user/${userId}`)
      .then(res => res.json())
      .then(data => {
        if (!canceled) setName(data.name)
      })

    return () => {
      canceled = true
    }
  }, [userId])

  return <span>{name}</span>
}
```

### 示例说明

- 这段代码解决什么问题：防止组件切换时旧请求回包覆盖新状态。
- 关键语句或参数含义：`useEffect` 的清理函数在依赖变化或卸载时执行。
- 常见错误写法：没有清理副作用，导致竞态和内存泄漏。

## 真实场景示例

- 场景描述：核心页面迭代中出现跨组件状态与渲染时序问题。
- 问题表现：功能可用但偶发抖动、错渲染或发布回退。
- 处理方案：先收敛边界，再做最小修复，最后补自动化验证。
- 验证方式：通过关键路径回归、性能指标和线上监控确认生效。

## 相关面试题

1. 对 componentWillReceiveProps 的理解
- 标准答案：结论是该生命周期已被废弃，不建议新代码使用。机制上它在异步渲染下容易产生重复调用和竞态问题。场景上迁移时用 `getDerivedStateFromProps`、`componentDidUpdate` 或 Hook 方案替代。
- 考察点：是否能给出明确结论，并补充机制和业务场景。

2. React 的生命周期有哪些？
- 标准答案：结论是可按挂载、更新、卸载三阶段记忆。机制上类组件常见 `constructor/render/componentDidMount/componentDidUpdate/componentWillUnmount`，函数组件由 Hook 组合表达。场景上数据请求和订阅放挂载后，资源释放放卸载阶段。
- 考察点：是否能完整列举关键项，并说明每一项的适用场景。

3. React 废弃了哪些生命周期？为什么？
- 标准答案：结论是 `componentWillMount`、`componentWillReceiveProps`、`componentWillUpdate` 被标记为不安全。机制上它们在并发调度下可能重复触发导致副作用问题。场景上应迁移到安全生命周期或 Hook。
- 考察点：是否能说明结论背后的原因、成立条件和反例。

4. React 16.X 中 props 改变后在哪个生命周期中处理
- 标准答案：结论是类组件通常在 `componentDidUpdate` 处理 props 变化，函数组件在 `useEffect` 里处理。机制上要比较前后值避免无穷更新。场景上联动请求、重置局部状态都应加条件判断。
- 考察点：是否能给出明确结论，并补充机制和业务场景。

5. React 性能优化在哪个生命周期？它优化的原理是什么？
- 标准答案：结论是类组件常在 `shouldComponentUpdate`/`PureComponent` 做渲染剪枝。机制上通过浅比较阻断不必要渲染。场景上配合不可变数据结构才能稳定发挥收益。
- 考察点：是否能按阶段讲清核心流程、关键数据结构和边界条件。

6. state 和 props 触发更新的生命周期分别有什么区别？
- 标准答案：结论是两者最终都会进入同一更新流程。机制上差异在触发源不同：props 来自父组件渲染，state 来自组件自身调度。场景上排障要先区分更新源，再定位是父级传入还是本地状态引发。
- 考察点：是否能从定义、触发时机、性能影响和适用场景四个维度做对比。

7. React 中发起网络请求应该在哪个生命周期中进行？为什么？
- 标准答案：结论是类组件推荐 `componentDidMount`，函数组件推荐 `useEffect`。机制上确保首屏渲染后再执行副作用，并可在清理函数中取消请求。场景上避免在 render 阶段发请求导致重复触发。
- 考察点：是否能说明结论背后的原因、成立条件和反例。

8. React 16 新生命周期有哪些
- 标准答案：结论是新增 `getDerivedStateFromProps` 和 `getSnapshotBeforeUpdate`。机制上前者用于根据 props 推导状态，后者在 DOM 提交前捕获快照。场景上常见于滚动位置恢复和受控状态同步。
- 考察点：是否能完整列举关键项，并说明每一项的适用场景。

9. useEffect 与 useLayoutEffect 的区别
- 标准答案：结论是 `useEffect` 在浏览器绘制后执行，`useLayoutEffect` 在绘制前同步执行。机制上后者会阻塞绘制，适合必须同步读取布局的场景。场景上普通请求和订阅用 `useEffect`，测量 DOM 再修正位置用 `useLayoutEffect`。
- 考察点：是否能从定义、触发时机、性能影响和适用场景四个维度做对比。

10. React Hooks 和生命周期的关系？
- 标准答案：结论是 Hook 不是生命周期一一映射，而是把副作用按依赖表达。机制上 `useEffect` 可覆盖 mount/update/unmount 三阶段逻辑。场景上应按“功能聚合”组织 effect，而不是机械对照类生命周期。
- 考察点：是否能给出明确结论，并补充机制和业务场景。
