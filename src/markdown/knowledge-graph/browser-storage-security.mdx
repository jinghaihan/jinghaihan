## 概念定义

- 本质定义：浏览器存储与安全边界是指浏览器在本地数据持久化时的存储能力、作用域限制和安全约束。
- 解决问题：在提升用户体验的同时，避免敏感数据泄露和跨源滥用。
- 边界说明：本地存储不是权限系统，不能替代服务端鉴权与加密策略。

## 核心机制拆解

### 1) 机制/流程

1. 页面按数据类型选择 Cookie、LocalStorage、SessionStorage 或 IndexedDB。
2. 浏览器按同源规则隔离不同源存储空间。
3. 请求发送时仅 Cookie 可自动携带到服务端。
4. 本地数据读写受容量、生命周期和策略限制。

### 2) 关键条件与触发点

- 条件 A：数据是否需要随请求自动携带，决定是否使用 Cookie。
- 条件 B：数据是否包含敏感信息，决定是否必须加密和最小化持久化。

### 3) 结果与影响

- 对性能 / 稳定性 / 可维护性的影响：存储选型正确可减少重复请求和状态丢失，选型错误会带来安全风险和同步复杂度。

## 相似概念对比

| 维度 | LocalStorage | SessionStorage | 结论 |
|---|---|---|---|
| 定义 | 同源持久化键值存储 | 同源会话级键值存储 | 生命周期是核心差异 |
| 触发时机 | 页面读写本地配置和缓存数据 | 单标签会话状态维护 | 前者跨会话，后者随标签关闭失效 |
| 性能影响 | 读写简单但容量有限 | 同样简单且隔离更强 | 二者都不适合大规模结构化数据 |
| 适用场景 | 主题配置、非敏感偏好 | 临时表单态、一次性会话状态 | 大数据优先 IndexedDB |

## 代码

```js
localStorage.setItem('theme', 'dark')
sessionStorage.setItem('draft', 'v1')

const req = indexedDB.open('app-db', 1)
req.onupgradeneeded = () => {
  req.result.createObjectStore('kv', { keyPath: 'key' })
}
```

### 示例说明

- 这段代码解决什么问题：展示三种本地存储入口和典型用途。
- 关键语句或参数含义：`createObjectStore` 用于定义 IndexedDB（Indexed Database API）对象仓库。
- 常见错误写法：把访问令牌长期明文放在 LocalStorage。

## 真实场景示例

- 场景描述：后台系统需要保存筛选条件和临时编辑内容。
- 问题表现：刷新后状态丢失，跨标签页同步混乱。
- 处理方案：偏好配置放 LocalStorage，编辑草稿放 SessionStorage，大体量离线数据放 IndexedDB。
- 验证方式：刷新、重开标签页和跨标签页场景分别验证数据行为。

## 常见误区

- 误区：LocalStorage 更大就一定更适合存用户凭证。
- 正解：凭证应优先使用受策略保护的 Cookie，并配合 `HttpOnly`、`Secure` 和 `SameSite`。

## 关联知识点

- 依赖：依赖 `security-cors` 与同源策略约束。
- 影响：影响 `browser-web-security` 中 XSS 数据泄露风险面。
- 对比：`browser-storage-security` 关注本地数据边界，`memory-cache` 关注资源副本复用。
- 组合：常与 `sw-check` 结合实现离线能力。

## 相关面试题

1. 浏览器本地存储方式及使用场景
- 标准答案：常见方式有 Cookie、LocalStorage、SessionStorage、IndexedDB；Cookie 适合会话标识，LocalStorage 适合持久偏好，SessionStorage 适合标签页临时状态，IndexedDB 适合结构化大数据。
- 考察点：是否能按“能力边界 + 场景”完整列举。

2. Cookie 有哪些字段，作用分别是什么
- 标准答案：常见字段包括 `Name`、`Value`、`Domain`、`Path`、`Expires/Max-Age`、`HttpOnly`、`Secure`、`SameSite`；它们分别控制标识内容、作用域、有效期和安全策略。
- 考察点：是否能显式列出字段及作用，而不是只说“用于会话”。

3. Cookie、LocalStorage、SessionStorage 区别
- 标准答案：三者区别在容量、生命周期、是否自动随请求发送和作用域隔离；其中只有 Cookie 会自动随请求携带，LocalStorage 持久化，SessionStorage 随标签页关闭失效。
- 考察点：是否能按维度对比而非单点描述。

4. 前端储存的⽅式有哪些？
- 标准答案：可分为 Cookie、Web Storage（LocalStorage / SessionStorage）、IndexedDB、Cache API 和内存态存储，各自适配不同容量、持久化和安全要求。
- 考察点：是否能列举并区分轻量键值和结构化存储。

5. IndexedDB 有哪些特点？
- 标准答案：IndexedDB 具备较大容量、事务能力、索引查询和异步读写，适合离线数据和复杂对象存储，但使用复杂度高于键值存储。
- 考察点：是否理解它相对 LocalStorage 的能力提升和成本。

6. 如何安全地存储敏感信息？
- 标准答案：前端默认不应长期存放高敏感数据，尤其是密码和长期有效凭证。机制上应优先把会话凭证放在带 `HttpOnly`、`Secure`、`SameSite` 的 Cookie 中，必要时对本地缓存做短期化和加密，并设置过期与清理策略。场景上登录态和支付态都应采用短期令牌与服务端可撤销会话设计。
- 考察点：是否能在“可用性”和“暴露面”之间做出工程化取舍。

7. 浏览器存储方案全解
- 标准答案：存储选型应按“是否自动随请求发送、生命周期、容量、查询能力、安全暴露面”五个维度决策。Cookie 适合会话控制，LocalStorage / SessionStorage 适合轻量键值状态，IndexedDB 适合离线结构化数据。高敏感数据应减少前端持久化并优先服务端可撤销机制。
- 考察点：是否能给出面向场景的存储决策方法，而不只是罗列名词。

8. Cookie 有哪些安全属性？分别有什么作用？
- 标准答案：Cookie 的关键安全属性是 `HttpOnly`、`Secure`、`SameSite`。`HttpOnly` 防止脚本直接读取凭证，`Secure` 限制仅在 HTTPS 传输，`SameSite` 控制跨站请求携带策略。结论是三者应组合配置，而不是只开一个选项。
- 考察点：是否能把每个属性的作用边界和组合使用场景讲清楚。
