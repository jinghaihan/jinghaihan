## 概念定义

- 本质定义：小程序生命周期是一组按运行阶段触发的钩子，用来把初始化、展示、隐藏和销毁逻辑放到正确时机。
- 解决问题：解释“为什么同一段初始化代码放在不同生命周期里，页面表现会完全不同”。
- 边界说明：本节点聚焦 App 和 Page 生命周期，不展开组件生命周期和自定义渲染。

## 核心机制拆解

### 1) 机制/流程

1. 小程序冷启动时先触发 App 层的 `onLaunch`，只执行一次。
2. 小程序进入前台时触发 App 层的 `onShow`，退到后台时触发 `onHide`。
3. 某个页面首次打开时触发 `onLoad`，首次渲染完成后触发 `onReady`。
4. 页面切前台会触发 `onShow`，隐藏或卸载时分别触发 `onHide` 和 `onUnload`。

### 2) 关键条件与触发点

- 条件 A：只需要全局初始化一次的逻辑应放在 App 级生命周期，而不是每个页面重复执行。
- 条件 B：依赖页面参数的请求通常放在 `onLoad`，依赖节点渲染完成的逻辑再放 `onReady`。
- 条件 C：离开页面要清理定时器和监听器，否则容易残留状态。

### 3) 结果与影响

- 对性能/稳定性/可维护性的影响：生命周期放对位置后，请求、埋点和页面初始化会更稳定；放错位置会导致重复请求、白屏和状态残留。

## 相似概念对比

| 维度 | App 生命周期 | Page 生命周期 | 结论 |
|---|---|---|---|
| 作用范围 | 整个小程序 | 单个页面 | 粒度不同 |
| 典型钩子 | `onLaunch`、`onShow`、`onHide` | `onLoad`、`onShow`、`onReady`、`onUnload` | 关注点不同 |
| 常见用途 | 全局初始化、登录态恢复 | 页面请求、渲染和清理 | 不要混放 |
| 调用频率 | 全局少量触发 | 跟随页面切换频繁触发 | 成本不同 |

## 代码

```js
App({
  onLaunch() {
    console.log('app launch')
  },
  onShow() {
    console.log('app show')
  },
})

Page({
  onLoad(query) {
    console.log('page load', query)
  },
  onReady() {
    console.log('page ready')
  },
  onUnload() {
    console.log('page unload')
  },
})
```

### 示例说明

- 这段代码解决什么问题：把全局启动时机和页面渲染时机分开。
- 关键语句或参数含义：`onLaunch` 只触发一次；`onLoad` 用于接收页面参数；`onUnload` 适合做清理。
- 常见错误写法：把每次进入页面都要执行的逻辑放到 `onLaunch`，或者把清理逻辑遗漏到页面销毁之后。

## 真实场景示例

- 场景描述：首页需要恢复登录态，详情页需要根据路由参数请求数据。
- 问题表现：如果都写在页面 `onShow`，会出现重复请求；如果详情数据请求写在 `onReady`，用户会多等一轮渲染。
- 处理方案：全局登录态恢复放 App 生命周期，详情请求放 `onLoad`，依赖节点尺寸的逻辑放 `onReady`。
- 验证方式：观察页面切换时请求次数、首屏时间和离开页面后的资源释放情况。

## 常见误区

- 误区：页面每次显示都会重新触发 `onLoad`。
- 正解：`onLoad` 只在页面实例创建时触发一次，返回上一页再回来通常会走 `onShow`。
- 误区：所有初始化都放 `onReady` 最稳。
- 正解：`onReady` 适合依赖视图节点的逻辑，过多数据请求放这里会拖慢可见时间。

## 关联知识点

- 依赖：路由栈、数据更新和页面缓存策略。
- 影响：请求时机、埋点时机和资源清理。
- 对比：App 生命周期与 Page 生命周期。
- 组合：登录态恢复、页面请求、定时器清理。

## 相关面试题

1. 小程序的生命周期有哪些？
- 标准答案：小程序生命周期分成应用级和页面级两层。应用级常见有 `onLaunch`、`onShow`、`onHide`、`onError`，页面级常见有 `onLoad`、`onShow`、`onReady`、`onHide`、`onUnload`。结论是全局初始化和页面逻辑要分层放置，否则会出现重复执行和状态错位。
- 考察点：是否知道 App 和 Page 生命周期的职责差异。
