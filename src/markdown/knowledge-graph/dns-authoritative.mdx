## 概念定位

权威 DNS 服务器是域名记录的最终来源，负责返回业务域的真实资源记录（A/AAAA/CNAME 等）。

## 工作机制

1. 递归 DNS 向权威服务器请求目标主机记录。
2. 权威服务器返回记录集及 TTL。
3. 递归 DNS 缓存后回传给客户端。
4. 客户端据此进入建连与请求阶段。

## 关键知识点

- **权威性**：权威层是记录真实值来源。
- **记录结构**：A/AAAA/CNAME 组合影响建连路径和稳定性。
- **TTL 策略**：决定命中率、切换速度与解析压力。
- **多记录负载分配**：返回记录顺序和策略会影响流量分布。

## 对页面性能的影响

- 权威响应慢会推高冷解析耗时。
- CNAME 过长会增加查询跳数和失败点。
- TTL 失衡会在“性能稳定”和“切换灵活”之间失衡。

## 观测与排障

### 命令行

```sh
dig @<authoritative-ns> www.example.com
```

### 排障步骤

1. 先直查权威确认“源事实”是否正确。
2. 再对比递归结果，判断是否缓存延迟造成偏差。
3. 检查 CNAME 链路长度与跨域依赖。
4. 按业务场景调整 TTL（平稳期与切换期分离）。

## 常见误区

- “权威 DNS 一定最快”：不准确，权威是正确性来源，不是最短路径。
- “只改 A 记录就够了”：不准确，可能还受 CNAME、证书和回源策略影响。
- “TTL 越短越好”：不准确，会显著增加解析频率和不稳定性。

## 题库深挖补充

### 高频对比与边界（基于题库）

- 递归解析器（Recursive Resolver）代表客户端逐级查询：根域 -> 顶级域 -> 权威域。
- 根域不返回最终业务 IP，它只告诉你下一跳顶级域服务器位置。
- 顶级域（TLD, Top-Level Domain）负责指向该域名所属的权威服务器。
- 权威 DNS 才是最终记录来源（A/AAAA/CNAME/TXT 等）。
- CNAME 链过长会增加解析往返，面试常问“为什么 CDN 域名看起来层层跳转”。
- TTL（Time To Live）太短会放大查询压力，太长会导致切流慢，这是经典取舍题。

### 线上排障剧本

1. 用本地解析结果和公共 DNS 结果做第一轮对比。
2. 若不一致，沿根域/顶级域/权威域链路逐层确认返回记录。
3. 检查是否存在 CNAME 级联过深或记录冲突。
4. 检查 TTL 配置是否与切流策略匹配。
5. 对跨地区问题，分运营商和地区做样本对照。

### 高频追问补充（附答案）

1. 根域服务器为什么不直接给我 IP？

根域职责是指路，不维护所有业务域最终记录。

2. CNAME 一定不好吗？

不是。合理 CNAME 便于运维和 CDN 调度，问题在于链路过长和治理缺失。

3. TTL 怎么定才合理？

看业务切流频率和稳定性要求：稳定业务可适度拉长，高变更业务适度缩短。

### 术语速查与易错点

- 根域（Root）：解析链起点，不存业务最终记录。
- TLD（Top-Level Domain）：顶级域层。
- 权威 DNS（Authoritative DNS）：记录最终来源。
- 递归解析器（Recursive Resolver）：替客户端逐级查询。
- 迭代查询（Iterative Query）：每层返回下一跳线索。

常见易错点：

- 把递归与迭代概念混为一谈。
- 忽略 CNAME 链长度对时延影响。
- TTL 改完后不等待传播窗口就判定失败。

## 深度延展（案例对照）

### 权威记录一致性案例

- 场景：同域名在不同地区返回不同 A 记录，导致用户体验波动。
- 诊断：权威集群间记录同步延迟，SOA 序列号不一致。
- 处理：建立发布前后 SOA 校验流程，避免半数节点旧配置。

### 命令行速查

- dig SOA your-domain.com：检查序列号。
- dig A your-domain.com @ns1.example.com：逐个权威节点对照。
- dig AAAA your-domain.com @ns1.example.com：核对 IPv6 记录一致性。

## 进阶排障清单（补充）

### 现场处置顺序

1. 先固定样本：同地域、同网络、同设备，避免样本漂移。
2. 抓三组结果：本地解析、公共 DNS 解析、权威解析。
3. 比对 TTL 与记录版本，确认是否处于传播窗口。
4. 标记是否命中本地特殊配置（hosts、代理、VPN）。
5. 形成“可复现路径 + 影响范围 + 临时止损”三件套再升级处理。

### 面试追问强化

1. 为什么 DNS 问题经常“不可稳定复现”？

因为多层缓存、地域路由和传播窗口共同影响，同一时间不同样本可能命中不同路径。

2. 如何用最少步骤判断是不是 DNS 问题？

先看不同解析器结果是否分叉，再看业务域名能否稳定映射到预期接入点。

3. 何时应该优先止损而不是继续追根因？

当影响范围在扩大且已有可控切流手段时，应先保可用性再做根因定位。

## 速查补充（第四轮）

### 快速判定问答

1. 什么时候先怀疑缓存层而不是权威记录本身？

当不同客户端结果分叉明显且权威查询稳定时，优先检查本地与递归缓存状态。

2. DNS 类问题为什么要强调样本固定？

因为网络、地域、时间窗口会影响解析路径，不固定样本容易得到互相矛盾结论。

## 面试补充

### 30 秒说明

权威 DNS 是解析链路中的最终事实源。性能与稳定性的关键在记录设计和 TTL 策略：平时保命中，切换窗口保收敛。

### 高频追问

1. 为什么权威正确但用户仍访问旧地址？

通常是递归或客户端缓存未过期，仍在复用旧记录。

2. CNAME 设计对性能有什么影响？

链路越长，查询跳数越多，冷解析时延和失败点越高。

3. TTL 怎么做分阶段策略？

平稳期中长 TTL，变更前下调 TTL，完成后再升回，兼顾稳定和收敛。
