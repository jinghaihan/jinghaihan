## 概念定义

- 本质定义：垃圾回收与内存泄漏是浏览器运行时对无用对象回收和异常持有引用治理的机制。
- 解决问题：避免页面运行时间变长后内存持续增长导致卡顿或崩溃。
- 边界说明：它不直接优化网络时延，但会间接影响交互流畅度。

## 核心机制拆解

### 1) 机制/流程

1. 运行时分配对象并建立引用关系。
2. 垃圾回收器标记仍可达对象。
3. 不可达对象在回收阶段释放内存。
4. 长生命周期页面需持续清理无用监听和缓存。

### 2) 关键条件与触发点

- 条件 A：闭包、全局变量和缓存容器是否持续持有引用。
- 条件 B：定时器、事件监听、异步回调是否在组件卸载时清理。

### 3) 结果与影响

- 对性能 / 稳定性 / 可维护性的影响：回收压力过大时会出现停顿和延迟，泄漏持续增长会导致页面崩溃。

## 相似概念对比

| 维度 | 正常回收 | 内存泄漏 | 结论 |
|---|---|---|---|
| 定义 | 无用对象被及时释放 | 无用对象仍被引用无法释放 | 是否可达是关键差异 |
| 触发时机 | 对象失去可达路径后 | 引用链未断开时 | 泄漏通常来自生命周期管理错误 |
| 性能影响 | 内存曲线可回落 | 内存曲线持续上升 | 泄漏会放大回收停顿 |
| 适用场景 | 常规页面运行 | 长驻页面、后台系统、大列表 | 面试常考排查思路 |

## 代码

```js
const listeners = []

function bind() {
  const handler = () => {}
  window.addEventListener('resize', handler)
  listeners.push(handler)
}

function cleanup() {
  for (const handler of listeners) {
    window.removeEventListener('resize', handler)
  }
  listeners.length = 0
}
```

### 示例说明

- 这段代码解决什么问题：演示监听器注册与释放的成对管理，避免泄漏。
- 关键语句或参数含义：`removeEventListener` 必须使用同一函数引用。
- 常见错误写法：组件卸载后未清理定时器和全局监听器。

## 真实场景示例

- 场景描述：运营后台连续使用数小时后越来越卡。
- 问题表现：内存曲线持续上升且无法回落。
- 处理方案：排查闭包引用、事件监听、缓存集合和定时器清理。
- 验证方式：用内存快照对比优化前后对象保留路径。

## 常见误区

- 误区：只要用了框架就不会有内存泄漏。
- 正解：框架只提供生命周期钩子，资源清理仍需业务代码主动完成。

## 关联知识点

- 依赖：依赖 `js-event-loop` 的任务执行节奏。
- 影响：影响 `browser-renderer-threads` 主线程停顿。
- 对比：`browser-gc-memory` 关注对象生命周期，`memory-cache` 关注资源复用。
- 组合：常与 `performance-debug-methodology` 联合排障。

## 相关面试题

1. V8 的垃圾回收机制是怎样的
- 标准答案：V8（V8 JavaScript Engine）采用分代回收思路，短生命周期对象和长生命周期对象使用不同回收策略，通过标记与清理回收不可达对象。
- 考察点：是否能说出“分代 + 可达性判断”核心机制。

2. 哪些操作会造成内存泄漏？
- 标准答案：常见泄漏来源包括全局变量滥用、闭包长期引用、未清理定时器、未移除事件监听、缓存容器无限增长和脱离 DOM 的引用残留。
- 考察点：是否能显式列举泄漏来源并对应治理动作。
