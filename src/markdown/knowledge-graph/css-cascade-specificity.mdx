## 概念定位

CSS 层叠与优先级决定“最终样式是谁说了算”。在大型项目里，这个节点是样式可维护性的关键闸门。

优先级治理做不好，会导致 `!important` 滥用、选择器越来越长、改一个样式牵一片回归。

## 工作机制

1. 浏览器先按来源与层级（user agent/user/author）排序，再按优先级（specificity）与源码顺序决议。
2. 同层竞争时，ID > class/属性/伪类 > 标签/伪元素。
3. 引入 Cascade Layers（层叠层）可显式声明样式层次，避免靠选择器“打仗”。
4. 组件样式建议采用低耦合命名（BEM 或约定式前缀），降低跨模块干扰。
5. 设计系统层（token、utility、component）应有固定覆盖顺序。

## 关键知识点

### 必须掌握

- 优先级冲突应优先靠结构化分层解决，而不是加长选择器。
- `:where()` 不增加优先级，可用于写“低优先级基础样式”。
- `:is()` 使用不当会抬高优先级，需结合设计系统约束。
- 全局样式和组件局部样式要边界清晰，避免相互污染。
- `!important` 只能用于极少数“策略性兜底”，必须可追踪。
- 样式lint要覆盖选择器深度和 `!important` 数量。

### 选型与取舍

- 强约束命名增加前期规范成本，但显著降低后期维护成本。
- 原子化样式改动快，但若缺少设计令牌（Design Token）会产生语义噪声。
- CSS Modules 隔离强，但跨组件主题扩展需要额外机制。

## 对页面性能与稳定性的影响

- 节点稳定时，前端基础（HTML/CSS/JavaScript） 的链路可预测性会明显提升，发布和排障成本更可控。
- 节点失稳时，问题会向上下游扩散，常表现为性能回退、交互异常或交付风险叠加。
- 该节点优化应和上下游协同验证，避免“局部优化把成本转移给后续节点”。

## 观测与排障

### 核心指标

- 选择器平均深度
- `!important` 使用次数
- 样式冲突回归缺陷数
- 样式重算耗时（Style Recalculation）

### 排障顺序

1. 先用 DevTools 查看最终命中的样式来源与覆盖链。
2. 确认问题是优先级冲突还是加载顺序问题。
3. 评估是否可通过层叠层或命名约束一次性解决。
4. 清理历史补丁选择器，避免“修一次坏三处”。
5. 补充规则：lint + 评审清单 + 组件样式文档。

### 常见故障场景

- “改了 class 不生效”：通常是被更高优先级选择器覆盖。
- “线上主题错乱”：常见是样式加载顺序变化导致覆盖关系反转。
- “组件样式串染”：常见是全局选择器命名过宽。
- “每次修样式都要加 `!important`”：说明层叠体系已失控。

## 常见误区

- 把“样式生效”当目标，而不是“可持续维护”当目标。
- 忽略层叠层能力，仍靠选择器长度竞争。
- 在代码评审中不检查选择器复杂度。

## 题库深挖补充

### 高频对比与边界（基于题库）

- 层叠顺序由来源、优先级、后定义顺序共同决定，不只是“谁写在后面”。
- !important 能快速覆盖，但会破坏长期可维护性。
- :where() 不增加优先级，适合基础样式；:is() 继承内部最高优先级。
- link 与 @import 的加载行为差异会影响阻塞路径。
- 伪类与伪元素语义不同，面试常考两者区别与用途。
- 选择器复杂度会影响样式重算成本，不只是可读性问题。

### 线上排障剧本

1. 先看样式未生效是优先级问题还是加载顺序问题。
2. 检查是否存在过多 !important 补丁。
3. 检查全局样式与组件样式边界污染。
4. 收敛选择器深度并建立层叠层约定。
5. 对高风险页面做回归截图对比。

### 高频追问补充（附答案）

1. !important 什么时候可以用？

只在策略性兜底场景少量使用，并可追踪清理。

2. link 和 @import 主要差别？

link 更直接且可并行管理，@import 常增加加载不确定性。

3. 为什么优先级治理和性能有关？

复杂冲突会带来更重的样式重算与维护成本。

## 进阶追问（补充）

1. 为什么很多前端性能问题最终会回到“基础知识点”？

因为事件循环、布局模型、缓存与请求语义是上层框架和业务逻辑的底层约束。

2. 如何区分“语法会用”和“机制掌握”？

能解释边界、失效条件和排障顺序，才算机制掌握，而不只是 API 会写。

3. 性能优化里最容易被忽略的一步是什么？

对照验证和防回退机制，经常只做了修复动作没做持续守护。

## 案例追问加深（第三轮）

### 真实故障片段

- 故障表现：页面逻辑正确但体感卡顿，且在低端设备明显更差。
- 排查结论：基础层问题叠加，包括事件循环拥塞、样式重算成本和请求并发失控。
- 治理动作：按主线程任务、样式布局、请求治理三条线拆解并建立基线。

### 追问补充

1. 为什么基础知识点决定上层框架表现？

框架最终都运行在浏览器基础机制之上，底层约束没处理好，上层优化很难生效。

2. 如何避免“会写代码但不会排障”？

建立机制到现象的映射：看到症状能快速关联到可能的底层阶段。

## 面试补充

### 30 秒说明

CSS 优先级问题本质是治理问题。建立分层规则与命名边界，比一次次打补丁更关键。

### 高频追问

1. 为什么不建议经常用 `!important`？

`!important` 会破坏可预测性，后续只能用更强覆盖反制，形成恶性循环。

2. `where` 和 `is` 有什么实战差异？

`:where()` 优先级为 0，适合基础样式；`:is()` 继承内部最高优先级，适合组合匹配。

3. 优先级冲突和加载顺序冲突怎么区分？

DevTools 看规则来源：若低优先级后写仍不生效是优先级问题；同优先级则可能是顺序问题。

4. 如何在团队层面防止样式失控？

建立层叠层策略、命名规范和 lint 门禁，并在评审里强制检查。
