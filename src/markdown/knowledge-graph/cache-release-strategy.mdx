## 概念定位

缓存发布策略关注“让用户尽量命中缓存，同时确保版本一致可回滚”。核心是指纹、TTL（Time To Live）和失效规则联动。

这个节点直接影响首屏速度、回源成本和“新旧版本混用”事故概率。

## 工作机制

1. 静态资源采用内容哈希命名，保证内容变更即 URL 变更。
2. HTML 入口短缓存或协商缓存，静态资产长缓存（`immutable`）。
3. CDN（Content Delivery Network）和源站缓存策略一致化，避免双层冲突。
4. 发布时先上传新资源，再切换入口引用，保证原子切换。
5. 配置快速失效与回滚机制，支持故障时秒级止损。

## 关键知识点

### 必须掌握

- “缓存命中率高”不等于“用户拿到正确版本”。
- 入口 HTML 不宜长缓存，否则会长期引用旧资源。
- JS/CSS 资源适合强缓存 + 指纹策略。
- 协商缓存（ETag/Last-Modified）适合可复用但需校验资源。
- 发布顺序错误会导致 404 或资源版本错配。
- 缓存策略必须和构建产物命名策略配套。

### 选型与取舍

- 长缓存性能好，但失效控制复杂。
- 短缓存一致性好，但回源成本高。
- 强制刷新可快速修复，但用户体验受影响。

## 对页面性能与稳定性的影响

- 节点稳定时，前端工程化 的链路可预测性会明显提升，发布和排障成本更可控。
- 节点失稳时，问题会向上下游扩散，常表现为性能回退、交互异常或交付风险叠加。
- 该节点优化应和上下游协同验证，避免“局部优化把成本转移给后续节点”。

## 观测与排障

### 核心指标

- CDN 命中率与回源率
- 版本错配率
- 资源 404 比例
- 发布后首小时错误率

### 排障顺序

1. 先确认问题是缓存未命中还是命中了旧版本。
2. 检查入口 HTML 缓存策略与刷新策略。
3. 检查产物指纹与引用关系是否一致。
4. 检查 CDN 节点失效是否完成。
5. 修复后做多地区、多运营商验证。

### 常见故障场景

- “有人新有人旧”：常见是入口缓存策略不当。
- “发布后静态资源 404”：常见是发布顺序错误。
- “修复上线但用户仍报错”：常见是边缘缓存未失效。
- “回滚无效”：常见是旧版本资源已被清理。

## 常见误区

- 只配缓存头，不做发布原子性设计。
- 忽略入口与静态资源的差异化策略。
- 没有故障演练就直接上生产。

## 深入拆解

### 入口与静态资源策略拆分

- 入口 HTML：短缓存 + 协商缓存，保证索引快速更新。
- JS/CSS/字体：长缓存 + 指纹，追求高命中和低回源。
- API 响应：按业务一致性需求设置缓存控制。

### 发布原子性检查

1. 新资源上传完成并可访问。
2. 入口文件引用切换。
3. CDN 节点预热与失效策略验证。
4. 回滚版本可立即切回。

### 线上排障提示

- 同地区异常优先查边缘节点缓存。
- 全量异常优先查入口引用和资源清单一致性。

## 题库深挖补充

### 高频对比与边界（基于题库）

- 构建优化看三件事：速度、体积、可追踪，不是只看包大小。
- 质量门禁应分级阻断：P0 直接阻断，P1 评审放行，P2 持续治理。
- 缓存发布要做入口与静态资源差异化策略，避免版本错配。
- CI/CD 需要幂等和审计，失败重试不能污染环境状态。
- 灰度与监控联动是发布安全关键，不可分离。
- 回滚预案要可演练，不是“有命令就算有能力”。

### 线上排障剧本

1. 先定位问题发生在 build、verify、deploy、release 哪一段。
2. 检查门禁日志与放行记录，确认是否越权放行。
3. 检查缓存策略与发布顺序是否一致。
4. 灰度异常先止损，再做根因定位。
5. 回滚后验证核心链路并进入复盘。

### 高频追问补充（附答案）

1. 为什么说发布成功不等于上线成功？

真实流量下仍可能暴露性能或稳定性问题。

2. 灰度阶段最关键看什么？

错误率、核心性能分位数和关键业务转化。

3. 回滚后还要做什么？

确认恢复、记录时间线、沉淀治理规则。

## 进阶深挖（第二轮）

### 发布原子性四步法

1. 上传新指纹资源并验证可访问。
2. 切换入口引用到新版本。
3. 预热关键资源并检查边缘一致性。
4. 保留旧版本快速回切能力。

### 缓存事故高发根因

- 入口缓存过长导致新入口传播慢。
- 资源 URL 无指纹导致新旧资源混用。
- 发布顺序不一致引发短窗 404。

### 追问扩展

1. 为什么入口 HTML 不能长期强缓存？

入口负责索引资源版本，过长缓存会长期指向旧资源集合。

2. 缓存命中率高却故障的典型原因？

命中的是错误版本或错误缓存键，而不是正确新版本。

## 面试补充

### 30 秒说明

缓存发布不是“配个 max-age”这么简单，它是构建、分发、入口切换和回滚机制的系统工程。

### 高频追问

1. 为什么 HTML 和 JS/CSS 缓存策略要分开？

HTML 是入口索引，需更快更新；JS/CSS 带指纹可长期缓存。

2. 什么是版本错配？

入口指向了不匹配的静态资源版本，导致运行时错误或 404。

3. 如何实现安全回滚？

保留历史产物、支持入口快速切换、并有 CDN 失效能力。

4. 命中率高为什么还会出故障？

可能命中的是旧版本或错误版本，命中率要结合版本一致性一起看。
