## 概念定义

- 本质定义：重排与重绘描述样式变更后浏览器更新页面的成本路径，渲染抖动是其失控表现。
- 解决问题：减少高频交互中的卡顿、丢帧和布局跳动。
- 边界说明：动画 API 语法本身在“动画与渲染时序”节点。

## 核心机制拆解

### 1) 机制/流程

1. 修改几何属性（如 `width`、`top`）可能触发布局重排。
2. 重排后通常伴随重绘，再进入合成阶段。
3. 修改 `transform`、`opacity` 多数场景可绕过重排，直接进入复合。

### 2) 关键条件与触发点

- 条件 A：读写布局属性交错（layout thrashing）会放大重排次数。
- 条件 B：动画帧中频繁改几何属性会导致明显掉帧。

### 3) 结果与影响

- 对性能/稳定性/可维护性的影响：策略正确时动画流畅；策略错误时交互卡顿和能耗上升。

## 相似概念对比

| 维度 | 改 `top/left` | 改 `transform: translate` | 结论 |
|---|---|---|---|
| 布局影响 | 常触发重排 | 多为复合阶段 | 动效优先 `transform` |
| 性能稳定性 | 易抖动 | 通常更平滑 | 高频动画差异明显 |
| 占位行为 | 改变布局位置 | 不改文档流占位 | 语义不同 |
| 适用场景 | 真正布局调整 | 过渡动画与位移 | 按目标选择 |

## 代码

```css
.card {
  transition: transform .2s ease;
}
.card:hover {
  transform: translateY(-4px);
}
```

### 示例说明

- 这段代码解决什么问题：用 `transform` 完成悬浮位移动效，降低重排概率。
- 关键语句或参数含义：`transform` 通常走合成路径，减少布局计算。
- 常见错误写法：动画中持续修改 `top` 或 `margin`。

## 真实场景示例

- 场景描述：卡片列表 hover 动效导致滚动时明显掉帧。
- 问题表现：鼠标快速移动时页面卡顿。
- 处理方案：位移动画从 `top` 改为 `transform`，批量读写分帧处理。
- 验证方式：Performance 面板看 Layout 次数、Long Task 与 FPS（Frames Per Second）。

## 常见误区

- 误区：`transform` 一定没有成本。
  - 正解：仍有合成开销，滥用图层也会增加内存压力。
- 误区：只要不用定位属性就不会重排。
  - 正解：很多几何属性和读取布局信息同样会触发布局计算。

## 关联知识点

- 依赖：定位与层叠上下文、动画与渲染时序。
- 影响：性能优化体系中的渲染指标。
- 对比：布局变更与复合变更。
- 组合：`requestAnimationFrame` 调度。

## 相关面试题

1. 为什么有时候用 translate 来改变位置而不是定位？
   - 标准答案：结论是 `translate` 更适合动画位移。机制上它通常绕过重排，主要在复合阶段完成；而改定位常触发布局和重绘。场景上高频动效优先 `transform`，真正布局调整再用定位。
   - 考察点：是否能把“更快”讲成渲染链路差异。
