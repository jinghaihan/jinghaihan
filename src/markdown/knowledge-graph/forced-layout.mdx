## 概念定位

强制布局（Forced Synchronous Layout）是指浏览器在尚未完成批量渲染更新时，被脚本强制立即计算布局，导致主线程同步等待。

这是交互卡顿、滚动掉帧、动画抖动的高频根因之一。

## 工作机制

1. 脚本先修改样式或 DOM（写操作）。
2. 紧接着读取布局信息（读操作），如元素尺寸/位置。
3. 浏览器为返回准确值，必须立刻执行样式重算和布局。
4. 若读写交替频繁，反复触发强制布局，成本迅速放大。

## 关键知识点

### 典型触发 API

- `offsetWidth` / `offsetHeight`
- `clientWidth` / `clientHeight`
- `scrollTop` / `scrollHeight`（特定场景）
- `getBoundingClientRect()`
- `getComputedStyle()`（依赖布局属性时）

### 布局抖动（Layout Thrashing）

在循环里反复“写 -> 读 -> 写 -> 读”会造成持续同步布局，常见于列表动画、滚动监听、拖拽更新场景。

### 强制布局与普通布局区别

- 普通布局：浏览器在合适时机批量执行。
- 强制布局：脚本要求“现在就给我精确结果”，打断批处理。

### 成本放大路径

DOM 树越大、样式依赖越复杂、影响范围越广，强制布局每次成本越高。

## 对页面性能的影响

- 交互链路：点击、输入、滚动响应延迟明显。
- 渲染链路：帧预算被挤占，掉帧和动画撕裂增加。
- 指标层：INP（Interaction to Next Paint）与长任务显著恶化。

## 观测与排障

### 浏览器观测

1. Performance 中查看 `Layout` 与 `Recalculate Style` 高频穿插。
2. 定位输入/滚动事件回调中的布局读取。
3. 检查单帧内布局执行次数是否异常。

### 排障剧本

1. 查找高频事件中所有布局读取 API。
2. 把读操作前置、写操作后置，按批次执行。
3. 循环场景改成“先收集数据，再统一写入”。
4. 高频更新优先改 `transform`/`opacity`，减少布局参与。
5. 回归看 INP、帧率与布局执行时长。

## 实战优化策略

### 读写分离策略

- 一帧中先读后写，不交错。
- 读阶段缓存几何信息，避免重复查询。
- 写阶段批量提交 class/style 变更。

### 渲染友好策略

- 动画优先使用合成层友好属性（`transform`、`opacity`）。
- 限制会触发布局的属性高频更新（宽高、位置、字体等）。
- 对复杂区域做隔离，缩小重排影响范围。

### 事件治理策略

- 滚动/resize 监听加节流。
- 高频事件与布局读取分离到 `requestAnimationFrame` 节奏。
- 避免在同步输入回调里做大量布局测量。

## 常见误区

- “只是读尺寸，不会有成本”：错误，读操作可能触发同步布局。
- “布局问题只能靠减少 DOM”：错误，读写顺序同样关键。
- “只要用了动画库就不会抖动”：错误，底层若频繁布局仍会掉帧。

## 面试补充

### 30 秒说明

强制布局是脚本打断浏览器渲染批处理的典型问题。常见触发方式是样式写入后立刻读取布局属性。治理核心是读写分离、批量更新、减少会触发布局的高频属性，并结合帧节奏调度更新。

### 高频追问

1. 为什么 `getBoundingClientRect` 会导致卡顿？

在样式已变更但布局未结算时读取它，浏览器必须立即同步布局。

2. 如何快速发现布局抖动？

看 Performance 中布局事件是否在同一交互中频繁出现，并定位读写交错调用栈。

3. `transform` 为什么常被推荐？

它多数情况下可在合成阶段处理，减少布局和重绘成本。

4. 读写分离实践里最容易漏什么？

循环体内部的隐藏读取（第三方库/API）容易破坏分离策略。

5. 强制布局和重绘是什么关系？

强制布局发生在布局计算阶段，后续通常会触发重绘，二者常串联出现。

6. 如何验证优化有效？

比较优化前后单次交互的布局次数、Layout 总时长、INP 与掉帧率。


## 进阶补充（深挖）

### 与上下游节点的因果关系

- 上游输入（常见）：指标异常排障动作库、JavaScript 执行与调度。
- 本节点职责：将“强制布局与回流治理”的结果转化为可执行的后续分支决策。
- 下游影响（常见）：CSSOM 构建与脚本执行、布局计算（Layout）、绘制（Paint）。

### 线上故障案例

- 现象：滚动时偶发掉帧，且布局事件在单次滚动中频繁触发。
- 根因：滚动回调中读写交错触发强制同步布局。
- 处置：将布局读取前置，写入后置并节流滚动回调。
- 回归标准：关键指标恢复到基线区间，且连续观测窗口无二次抖动。

### 高频追问）

1. 为什么读属性也会触发布局？

浏览器需返回准确几何信息，可能立即结算布局。

2. 如何避免布局抖动？

同一帧中先读后写，避免循环内反复交错。

3. 哪些属性更适合高频动画？

transform/opacity，通常可减少布局参与。


### 补充记忆框架

- 先判断该节点是“前置等待”问题还是“运行时抖动”问题。
- 再判断问题来自“资源策略”还是“执行调度”。
- 最后把定位结论转为可复用规则，进入门禁与动作库。

### 高频追问

1. 这个节点优化后如何做回归验收？

至少做两组对比：关键指标分位数对比 + 上下游链路稳定性对比，避免局部优化导致成本转移。

2. 为什么这个节点要单独做发布观察窗口？

因为它对主链路有放大效应，短时间异常会快速传导到首屏和交互体验。

### 补充策略

- 对滚动与拖拽场景建立“只读帧”和“只写帧”约束。
- 对高频触发布局的组件建立专项性能基线。
