## 概念定义

- 本质定义：绘制是浏览器根据布局结果生成像素绘制指令的阶段。
- 解决问题：把几何和样式信息转成可被图形系统消费的绘图命令。
- 边界说明：它不负责图层最终叠加输出，叠加属于合成阶段。

## 核心机制拆解

### 1) 机制/流程

1. 浏览器遍历需要绘制的渲染对象。
2. 按绘制顺序生成文本、背景、边框等指令。
3. 把指令提交到后续图层合成流程。

### 2) 关键条件与触发点

- 条件 A：视觉样式变化会触发重绘。
- 条件 B：大面积阴影、滤镜和复杂渐变会增加绘制成本。

### 3) 结果与影响

- 对性能 / 稳定性 / 可维护性的影响：绘制压力高会造成帧时间超预算，导致动画卡顿和滚动不流畅。

## 相似概念对比

| 维度 | 绘制 | 合成 | 结论 |
|---|---|---|---|
| 定义 | 生成像素绘制指令 | 合并图层并输出屏幕帧 | 前者产出内容，后者产出结果 |
| 触发时机 | 样式或内容变更后 | 图层内容更新后 | 二者顺序固定 |
| 性能影响 | 复杂样式成本高 | 图层过多会增加管理成本 | 优化要平衡绘制与图层数量 |
| 适用场景 | 文本、背景、阴影渲染 | 动画和滚动图层合成 | 面试常考区别 |

## 代码

```css
.card {
  /* 阴影层级越多，绘制成本越高 */
  box-shadow: 0 8px 24px rgba(0, 0, 0, 0.18);
}
```

### 示例说明

- 这段代码解决什么问题：示例展示高成本样式的常见来源，便于做视觉与性能权衡。
- 关键语句或参数含义：`box-shadow` 的模糊半径和面积会影响绘制开销。
- 常见错误写法：在大面积滚动容器上叠加复杂阴影和滤镜。

## 真实场景示例

- 场景描述：卡片瀑布流滚动掉帧。
- 问题表现：布局耗时正常，但帧率仍低。
- 处理方案：降低阴影复杂度，减少大面积透明渐变。
- 验证方式：对比优化前后滚动帧率和绘制时间。

## 常见误区

- 误区：绘制一定比布局便宜。
- 正解：复杂视觉效果下，绘制同样可能成为主瓶颈。

## 关联知识点

- 依赖：依赖 `layout` 输出几何信息。
- 影响：影响 `composite` 的图层合并成本。
- 对比：`paint` 负责像素生成，`composite` 负责图层合并。
- 组合：常与 `render-opt` 联合做滚动性能治理。

## 相关面试题

1. 浏览器渲染优化
- 标准答案：绘制优化重点是减少高成本样式和大面积重绘区域，常见手段包括简化阴影滤镜、缩小重绘范围和合理分层。
- 考察点：是否能给出绘制层可执行优化动作。
