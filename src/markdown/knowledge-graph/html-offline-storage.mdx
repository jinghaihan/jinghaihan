## 概念定义

- 本质定义：离线存储与 Service Worker 节点关注“无网或弱网下如何保证页面可用”的资源策略。
- 解决问题：提升离线可访问率、弱网成功率和资源一致性。
- 边界说明：本节点聚焦离线资源管理，不展开媒体选型和线程模型。

## 核心机制拆解

### 1) 机制/流程

1. 识别离线核心资源：壳页面、核心静态资源、兜底页。
2. 区分历史 `.appcache` 机制与现代 `Service Worker` 机制。
3. 现代方案中在安装阶段预缓存关键资源。
4. 运行阶段拦截请求并按策略命中缓存或回源网络。
5. 发布阶段做版本更新与旧缓存淘汰。

### 2) 关键条件与触发点

- 条件 A：只有缓存没有版本策略，容易长期命中旧资源。
- 条件 B：离线兜底页缺失，断网访问会直接失败。

### 3) 结果与影响

- 对性能/稳定性/可维护性的影响：策略清晰时离线体验稳定；策略混乱时会出现缓存污染和版本错乱。

## 相似概念对比

| 维度 | `.appcache`（历史） | `Service Worker`（现代） | 结论 |
|---|---|---|---|
| 定义 | 基于 manifest 的离线缓存机制 | 基于事件拦截和可编程缓存机制 | 现代项目优先后者 |
| 更新机制 | 依赖 manifest 变化 | 可按资源类型自定义更新策略 | 可控性更高 |
| 调试成本 | 较高，行为不直观 | 可观测性和策略表达更清晰 | 工程化更友好 |
| 适用场景 | 历史题目认知 | 现代离线与弱网治理 | 认知历史，落地现代 |

## 代码

```js
self.addEventListener('fetch', (event) => {
  event.respondWith(
    caches.match(event.request).then((cached) => {
      return cached || fetch(event.request)
    }),
  )
})
```

### 示例说明

- 这段代码解决什么问题：断网或弱网时优先用缓存响应请求。
- 关键语句或参数含义：`fetch` 事件用于请求拦截，`caches.match` 用于缓存命中。
- 常见错误写法：不做缓存版本管理，发布后仍返回旧资源。

## 真实场景示例

- 场景描述：业务页面在地铁弱网环境频繁白屏。
- 问题表现：页面离线不可用，恢复联网后资源版本混乱。
- 处理方案：核心资源接入 `Service Worker` 缓存策略并增加版本淘汰机制。
- 验证方式：离线回访成功率、弱网失败率、版本一致性检查。

## 常见误区

- 误区：离线能力等于“把资源缓存起来”。
- 正解：还必须包含更新、淘汰和回退策略。
- 误区：`.appcache` 和 `Service Worker` 没区别。
- 正解：前者是历史机制，后者是现代可编程方案。

## 关联知识点

- 依赖：浏览器原理中的 Service Worker 拦截链。
- 影响：刷新模式与缓存判定、发布一致性。
- 对比：历史离线方案与现代离线方案。
- 组合：渐进增强与优雅降级。

## 相关面试题

1. HTML5 的离线储存怎么使用，它的工作原理是什么
- 标准答案：结论是题目原文主要描述 `.appcache` 历史机制。机制上通过 manifest 清单声明离线资源，浏览器按清单缓存并在离线时复用。场景上它有历史认知价值，但现代项目应优先使用 `Service Worker` 的可编程缓存策略。
- 考察点：是否能区分历史方案和现代方案。
- 追问 1：现代项目为什么优先 `Service Worker`？
- 标准答案 1：结论是策略更可控。机制上可按请求类型设计缓存、回源、兜底和更新规则。反例是 `.appcache` 更新和调试成本高、灵活性弱。

2. 浏览器是如何对 HTML5 的离线储存资源进行管理和加载？
- 标准答案：结论是在线时浏览器会检查资源和策略并更新缓存，离线时优先命中本地可用副本。机制上历史 `.appcache` 依赖 manifest 变化触发更新，现代 `Service Worker` 依赖安装/激活与运行时拦截策略。场景上应配合版本号和淘汰策略避免缓存污染。
- 考察点：是否理解离线资源生命周期管理。
