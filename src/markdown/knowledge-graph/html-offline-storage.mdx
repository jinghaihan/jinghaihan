## 概念定义

- 本质定义：离线存储与 Service Worker 节点关注“断网或弱网时，页面如何继续拿到资源、展示壳页并维持基础可用”。
- 解决问题：让页面在网络不稳定时仍然能打开、兜底、回源和更新，不至于直接白屏。
- 边界说明：本节点聚焦离线资源管理，不展开本地业务数据模型，也不展开 Worker 的计算线程能力。

## 核心机制拆解

### 1) 机制/流程

1. 先划分离线核心资源：壳页面、关键样式、关键脚本、兜底页和必要接口缓存。
2. 区分历史 `.appcache` 方案和现代 Service Worker 方案。
3. 现代方案里先注册 Service Worker，在 `install` 阶段预缓存关键资源，在 `activate` 阶段清理旧缓存。
4. 运行时通过 `fetch` 事件拦截请求，按“缓存优先”“网络优先”或“兜底回退”策略返回内容。
5. 发布时配合版本号或缓存名做更新，避免新旧资源交叉污染。

### 2) 关键条件与触发点

- 条件 A：只有缓存没有版本淘汰时，最容易命中旧资源。
- 条件 B：没有离线兜底页或失败回退策略时，断网访问会直接失败。
- 条件 C：Service Worker 运行条件不满足时，比如不是 HTTPS（HyperText Transfer Protocol Secure）环境，注册就不会成功。

### 3) 结果与影响

- 对性能/稳定性/可维护性的影响：策略清晰时，复访速度和离线体验都更稳；策略混乱时，最常见的问题就是“明明发版了，用户还在用旧文件”。

## 相似概念对比

| 维度 | `.appcache` | Service Worker | 结论 |
|---|---|---|---|
| 控制方式 | 基于 manifest 列表声明 | 基于事件和脚本可编程控制 | 现代项目优先后者 |
| 更新方式 | 依赖 manifest 变化 | 可按资源类型自定义策略 | 可控性更强 |
| 能力范围 | 主要是离线缓存 | 离线缓存、请求拦截、后台同步、推送 | 不在一个量级 |
| 工程价值 | 历史认知 | 现代离线和弱网治理基础 | 认知历史，落地现代 |

## 代码

```js
// main.js
if ('serviceWorker' in navigator)
  navigator.serviceWorker.register('/sw.js')
```

```js
// sw.js
const CACHE_NAME = 'app-shell-v1'

self.addEventListener('install', (event) => {
  event.waitUntil(
    caches.open(CACHE_NAME).then(cache => cache.addAll([
      '/',
      '/offline.html',
      '/assets/main.css',
      '/assets/main.js',
    ])),
  )
})

self.addEventListener('fetch', (event) => {
  event.respondWith(
    caches.match(event.request).then((cached) => {
      return cached || fetch(event.request)
    }),
  )
})
```

### 示例说明

- 这段代码解决什么问题：注册 Service Worker 并把壳页面和关键静态资源预缓存下来，离线时优先命中缓存。
- 关键语句或参数含义：`install` 负责预缓存，`fetch` 负责运行时拦截，`CACHE_NAME` 用于版本隔离。
- 常见错误写法：缓存名不变、发布后不清旧缓存、只缓存成功路径不做失败回退。

## 真实场景示例

- 场景描述：门店业务系统在地铁和仓库弱网场景下经常打不开，操作员刷新几次后偶尔又恢复。
- 问题表现：断网时直接白屏，恢复联网后又会混入旧版本文件。
- 处理方案：给核心页面接入 Service Worker，预缓存壳资源，失败时回退到离线页，并按版本清理旧缓存。
- 验证方式：断网访问成功率、复访速度、版本一致性和缓存命中日志。

## 常见误区

- 误区：离线能力就是“把资源缓存起来”。
- 正解：真正难的是更新、淘汰、兜底和作用域管理。
- 误区：`.appcache` 和 Service Worker 本质一样。
- 正解：前者是历史方案，能力弱且难调试；后者是现代可编程缓存和请求代理机制。
- 误区：只要注册了 Service Worker，页面就自动支持离线。
- 正解：不写缓存策略和回退逻辑，注册本身几乎没有价值。

## 关联知识点

- 依赖：浏览器原理中的 Service Worker 拦截链、渐进增强与优雅降级。
- 影响：复访速度、离线可用率、发布一致性。
- 对比：历史离线方案与现代离线方案。
- 组合：缓存策略、刷新模式与性能优化。

## 相关面试题

1. HTML5 的离线储存怎么使用，它的工作原理是什么
- 标准答案：如果按历史题来答，通常指 `.appcache`：通过 manifest 清单声明哪些资源要离线缓存，浏览器会按清单下载并在离线时复用。机制上它依赖 manifest 变化来触发更新。场景上这个方案今天主要是历史认知，现代项目应优先改用 Service Worker。
- 考察点：是否能分清历史题意和现代落地方案。

2. 浏览器是如何对 HTML5 的离线储存资源进行管理和加载？
- 标准答案：本质上就是“在线时更新，离线时复用”。机制上历史 `.appcache` 依赖清单变化管理资源，现代 Service Worker 依赖安装、激活和运行时 `fetch` 拦截来管理缓存和回源。场景上真正落地时必须带上缓存版本和失败兜底，否则很容易缓存污染。
- 考察点：是否理解离线资源的生命周期管理。

3. 什么是Service Worker？如何用于性能优化？
- 标准答案：Service Worker 是运行在浏览器后台的脚本，可以拦截作用域内请求，并结合 Cache API（Cache Application Programming Interface）做缓存命中、离线兜底和复访加速。机制上它独立于页面生命周期，常见流程是注册、安装、激活、拦截请求。场景上它既能提升复访速度，也能给弱网和离线场景提供基础可用能力。
- 考察点：是否知道 Service Worker 的核心不只是“离线”，还包括请求代理和性能收益。
