## 概念定义

- 本质定义：渲染树构建是把可见 DOM（Document Object Model）节点与对应样式规则合并成可布局对象的过程。
- 解决问题：为布局阶段提供“哪些元素可见、采用何种样式”的统一输入。
- 边界说明：它不计算最终坐标和尺寸，坐标计算属于布局阶段。

## 核心机制拆解

### 1) 机制/流程

1. 浏览器遍历 DOM 树筛选可见节点。
2. 为每个可见节点匹配样式并生成渲染对象。
3. 将渲染对象按层级组织成渲染树。
4. 把渲染树交给布局阶段计算几何信息。

### 2) 关键条件与触发点

- 条件 A：节点是否可见，决定是否进入渲染树。
- 条件 B：样式匹配复杂度，决定构建耗时。

### 3) 结果与影响

- 对性能 / 稳定性 / 可维护性的影响：渲染树越大，后续布局和绘制成本越高；可见内容收敛可显著降低首屏压力。

## 相似概念对比

| 维度 | DOM 树 | 渲染树 | 结论 |
|---|---|---|---|
| 定义 | 文档结构树 | 可见渲染对象树 | 渲染树不是 DOM 的一比一映射 |
| 触发时机 | HTML 解析阶段 | 样式可用后 | 构建顺序固定 |
| 性能影响 | 影响匹配规模 | 影响布局绘制成本 | 控制节点和样式复杂度都重要 |
| 适用场景 | 语义结构表达 | 视觉渲染执行 | 面试常考“哪些节点不进渲染树” |

## 代码

```js
// 评估可见元素规模
const visibleNodes = [...document.querySelectorAll('*')].filter((el) => {
  const style = getComputedStyle(el)
  return style.display !== 'none' && style.visibility !== 'hidden'
}).length
console.log('visible nodes', visibleNodes)
```

### 示例说明

- 这段代码解决什么问题：粗略估计可见节点规模，辅助判断渲染树复杂度。
- 关键语句或参数含义：`display` 与 `visibility` 共同影响可见性判断。
- 常见错误写法：把所有 DOM 节点都当作渲染树节点。

## 真实场景示例

- 场景描述：后台页面包含大量隐藏面板和预渲染区块。
- 问题表现：切换到目标面板时卡顿明显。
- 处理方案：拆分隐藏区块，延迟挂载非关键内容。
- 验证方式：对比渲染对象规模和布局耗时变化。

## 常见误区

- 误区：只要节点在 DOM 里就一定参与渲染。
- 正解：不可见节点可能不会进入渲染树。

## 关联知识点

- 依赖：依赖 `html-parser` 和 `cssom-js` 输出。
- 影响：直接影响 `layout` 和 `paint` 阶段成本。
- 对比：`render-tree` 负责渲染对象组织，`layout` 负责几何计算。
- 组合：常与 `parse-blocking` 一起优化首屏渲染路径。

## 相关面试题

1. 浏览器的渲染过程
- 标准答案：DOM 和样式数据准备好后会构建渲染树，渲染树只包含可见渲染对象，随后进入布局和绘制阶段。
- 考察点：是否能明确渲染树在整条渲染链中的位置。
