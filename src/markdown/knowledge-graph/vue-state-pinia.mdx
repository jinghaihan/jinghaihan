## 概念定位

Pinia 状态管理节点关注状态分层、更新粒度、持久化与可追踪性，核心是让共享状态可控而不泛滥。

它直接影响跨页面一致性和调试效率。

## 工作机制

1. 按业务域拆 store，避免全局大状态单点膨胀。
2. 区分持久状态、会话状态和派生状态。
3. 通过 actions 统一副作用入口，保证状态变更可追踪。
4. 高频更新数据与低频配置数据分层存储。
5. 必要时接入持久化插件并定义迁移策略。

## 关键知识点

### 必须掌握

- store 设计要围绕业务边界而不是页面边界。
- getter 适合派生状态，避免重复存储同一事实。
- 状态持久化必须考虑版本升级兼容。
- 避免在 store 中写过多视图层逻辑。
- 调试时应能追踪 action 到状态变更链路。
- 大对象状态更新要关注性能和序列化成本。

### 选型与取舍

- 全局状态集中管理易治理，但误用会扩大更新影响面。
- 局部状态灵活，但跨页面一致性更难保证。
- 持久化提升恢复体验，但会引入脏数据风险。

## 对页面性能与稳定性的影响

- 节点稳定时，Vue 工作流 的链路可预测性会明显提升，发布和排障成本更可控。
- 节点失稳时，问题会向上下游扩散，常表现为性能回退、交互异常或交付风险叠加。
- 该节点优化应和上下游协同验证，避免局部优化把成本转移给后续节点。

## 观测与排障

### 核心指标

- store 数量与平均体积
- 状态更新频率分布
- 状态同步相关缺陷数
- 持久化命中率与迁移失败率

### 排障顺序

1. 先定位问题是否由状态边界错误引起。
2. 检查 action 是否包含隐式副作用。
3. 检查持久化数据结构和版本兼容。
4. 分析高频更新是否导致不必要重渲染。
5. 调整 store 拆分和订阅粒度。

### 常见故障场景

- 小功能改动触发全局大量刷新。
- 刷新后状态恢复错误。
- 多页面共享状态互相覆盖。
- 调试时难以还原状态演进路径。

## 常见误区

- 把所有状态都放全局 store。
- getter 与 state 重复表达同一数据。
- 持久化无版本迁移策略。

## 深入拆解

### 状态分层建议

- Domain Store：业务核心状态。
- Session Store：会话态与页面态。
- UI Store：纯展示态（弹窗、面板状态）。

### 持久化治理

1. 只持久化必要字段。
2. 增加版本号和迁移函数。
3. 清理过期缓存，避免历史脏数据。

### 性能优化点

- 订阅粒度按 selector 收敛。
- 高频状态与低频状态拆 store。

## 题库深挖补充

### 高频对比与边界（基于题库）

- 路由驱动请求要处理竞态：后发先至会覆盖新状态。
- Pinia 状态分层要区分持久状态、会话状态、派生状态。
- SFC 模板编译需理解 Patch Flag、静态提升和 key 稳定性。
- Vue 性能优化先看热点组件和更新频率，再决定是否缓存或拆分。
- 异步请求治理需覆盖超时、重试、取消和错误兜底。
- 面试常问“v-if/v-show、computed/watch、nextTick”，要能结合性能语义说明。

### 线上排障剧本

1. 先区分卡顿来自请求链路还是渲染链路。
2. 检查请求取消与去重是否生效。
3. 检查列表 key 和模板复杂表达式。
4. 检查 store 更新粒度是否过粗。
5. 通过灰度对照验证优化收益。

### 高频追问补充（附答案）

1. 为什么 v-for 一定要稳定 key？

否则节点复用错位，性能和状态正确性都会受影响。

2. computed 和 watch 的边界？

computed 用于派生值缓存，watch 用于副作用响应。

3. nextTick 解决的核心问题是什么？

在 DOM 更新批处理后读取最新视图状态。

## 进阶深挖（第二轮）

### Store 设计反模式

- 单一超大 store 承载全部业务状态。
- getter 与 state 重复表达同一事实。
- action 内混杂大量视图逻辑。

### 持久化迁移模板

1. 状态结构增加版本号。
2. 旧版本数据提供迁移函数。
3. 迁移失败提供安全回退路径。

### 追问扩展

1. 什么时候应该拆 store？

当更新频率差异大、业务边界不一致或排障定位困难时。

2. Pinia 调试效率如何提升？

保持 action 语义清晰并记录关键状态快照变化。

## 面试补充

### 30 秒说明

Pinia 的价值是“让共享状态可治理”。重点不在能不能存，而在状态是否分层清晰、更新是否可追踪。

### 高频追问

1. 什么时候该上 Pinia？

当状态跨页面共享、生命周期长且需要统一治理时。

2. Pinia 和组件本地状态如何分工？

局部短生命周期放组件，本地外的跨域共享放 Pinia。

3. 持久化状态最容易踩什么坑？

数据结构升级后旧缓存不兼容。

4. 如何避免 store 膨胀？

按业务域拆 store，限制单 store 职责范围。
