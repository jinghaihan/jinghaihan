## 概念定位

Vue 渲染调度器的职责是：把大量状态变更合并为“有序、可控、低抖动”的更新执行序列。

核心问题不是“能不能更新”，而是“什么时候更新、按什么顺序更新、如何避免重复更新”。

## 调度核心结构

Vue 运行时可抽象为三类队列：

1. 组件更新队列（job queue）
2. 前置回调队列（pre flush callbacks）
3. 后置回调队列（post flush callbacks）

配合一个微任务触发器（通常 Promise microtask）完成一次 flush 周期。

## 工作机制

### 1) 触发更新

响应式 `trigger` 后，组件渲染 effect 不会立刻执行，而是通过 scheduler 进入 job queue。

### 2) 去重与排序

- 同一个 job 在同一轮 flush 中会去重。
- 通过组件创建顺序/优先关系保证父子更新顺序稳定。

### 3) flush 周期

一次刷新周期通常顺序为：

1. 执行 pre flush 回调
2. 执行组件更新 jobs
3. 执行 post flush 回调

### 4) nextTick

`nextTick` 等待当前 flush 完成后再执行回调，所以常用于“拿到更新后的 DOM”。

## 关键知识点

### 为什么批处理能提速

如果每次状态写入都立即重渲染，高频输入会导致大量重复计算。批处理把多个写入合并成一次渲染，减少无效开销。

### flush 时机选择

- `pre`：适合渲染前准备逻辑
- `post`：适合 DOM 依赖逻辑（如测量、滚动）
- `sync`：仅限少数必须同步的场景

### 与模板编译优化协同

SFC 编译产物中的静态提升、Patch Flags（补丁标记）会减少运行时 diff 成本。调度器决定“何时更新”，编译优化决定“更新多少”。

### 循环更新保护

Vue 内部有递归更新保护机制，防止 effect 自触发进入无限循环。

## 对页面性能的影响

- 调度稳定时：输入流畅、渲染抖动小、长任务更少。
- 调度失控时：同一交互触发多轮更新，INP（Interaction to Next Paint）恶化。
- 错误使用 `sync` 或过量 `nextTick`，容易把微任务链撑爆。

## 观测与排障

### 典型异常

1. “状态改一次，组件连渲染多次”
2. “nextTick 套 nextTick，逻辑越来越慢”
3. “watch flush 时机不对，拿到旧 DOM”

### 排障步骤

1. 确认更新来源：是响应式触发还是手动强刷。
2. 检查同一轮 flush 是否有重复 job。
3. 校验 watch flush 选项是否匹配业务需求。
4. 排查 `nextTick` 是否被滥用成流程控制工具。
5. 对高频交互链路做分帧或节流。

## 实战优化策略

1. 把“多次写入”合并到同一事件循环内。
2. 让派生状态走 `computed`，减少 watch 链式副作用。
3. DOM 依赖逻辑统一放 `post` 或 `nextTick` 后执行。
4. 避免在渲染 effect 中做昂贵同步计算。
5. 列表高频更新场景结合虚拟列表与局部更新策略。

## 常见误区

- “nextTick 越多越安全”：错误，过量 nextTick 会制造调度拥塞。
- “sync 更及时所以更好”：错误，sync 常带来连锁同步更新和卡顿。
- “批处理会导致状态不一致”：错误，问题通常来自副作用时机错误而非批处理本身。

## 题库深挖补充

### 高频对比与边界（基于题库）

- nextTick 解决的是“批处理更新后的读取时机”，不是性能万能键。
- pre flush、job queue、post flush 的先后顺序决定副作用执行时机。
- watch 的 flush: 'pre'|'post'|'sync' 选择会影响渲染一致性与性能。
- 过多同步 watch 会打断批处理，导致更新风暴。
- 调度器去重机制能避免重复 job，但不等于自动解决业务级竞态。
- 面试常问“为什么改了数据没立即看到 DOM”，核心就是调度批处理语义。

### 线上排障剧本

1. 先定位卡顿来自 job 过量还是副作用过重。
2. 检查是否有 sync flush 滥用。
3. 检查 nextTick 调用链是否嵌套过深。
4. 检查组件更新是否被不必要依赖放大。
5. 用组件级更新次数和耗时做回归对照。

### 高频追问补充（附答案）

1. nextTick 什么时候必须用？

当你需要在本轮响应式更新提交后读取最新 DOM 状态时。

2. flush 选错会带来什么问题？

副作用时机错位，可能出现读旧值或重复更新。

3. 调度器去重为什么还会卡？

去重只能减少重复任务，无法抵消单次任务过重。

## 进阶深挖（第二轮）

### flush 时机策略

- pre：适合在渲染前准备状态。
- post：适合在 DOM 更新后读取视图。
- sync：仅用于极少数必须同步的场景。

### 调度风暴防护

1. 降低同步 watcher 数量。
2. 合并同类状态更新。
3. 避免在 watcher 内再次触发高频状态写入。

### 追问扩展

1. 为什么 sync flush 容易引发卡顿？

它打破批处理节奏，放大主线程即时工作量。

2. nextTick 滥用会怎样？

会形成嵌套调度链，增加时序复杂度和维护成本。

## 面试补充

### 30 秒说明

Vue 调度器把响应式触发的更新放入队列，按 pre/job/post 顺序在微任务中批量执行。它解决的是更新去重和时机控制问题，核心收益是减少重复渲染。`nextTick` 用于拿到 flush 后结果，`watch` 的 flush 时机要和 DOM 依赖关系匹配。

### 高频追问

1. 为什么 Vue 不在 trigger 时同步重渲染？

因为同步重渲染会在高频写入下产生大量重复工作，批处理更稳定高效。

2. `watch` 的 `pre` 和 `post` 如何选？

是否依赖“更新后的 DOM”是分界线：依赖 DOM 用 `post`，否则优先 `pre`。

3. `nextTick` 的本质是什么？

等待当前刷新队列执行完成后的回调，不是“强制立刻刷新”。

4. 为什么会出现“父组件更新后子组件又更新一次”？

可能是状态扇出过大或副作用链路重复触发，需检查队列来源与依赖边界。

5. 如何判断调度器而非 diff 算法是瓶颈？

看同一交互是否触发多轮 flush 和重复 job，若是，优先治理调度链路。

6. 批处理和用户输入延迟之间如何平衡？

关键交互路径保持轻量同步反馈，重计算延后到批处理或分帧执行。
