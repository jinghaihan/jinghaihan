## 概念定义

- 本质定义：Webpack 热更新运行时是开发期在不整页刷新的前提下替换变更模块的机制。
- 解决问题：解决开发调试过程中频繁刷新页面导致状态丢失和反馈慢的问题。
- 边界说明：不用于生产环境静态资源发布，也不替代浏览器缓存策略。

## 核心机制拆解

### 1) 机制/流程

1. 文件变更后，增量编译产出新的模块和哈希。
2. 开发服务器通过长连接把变更通知发送到浏览器。
3. 客户端根据哈希请求更新清单和变更模块。
4. 运行时执行模块替换并触发接受回调。
5. 若模块不可热替换则回退到整页刷新。

### 2) 关键条件与触发点

- 条件 A：模块需要显式声明可接受更新，否则会向上冒泡直到触发全量刷新。
- 条件 B：当变更影响到无法局部替换的边界模块时，热更新会失败并回退。

### 3) 结果与影响

- 对性能的影响：开发期反馈速度显著提升，但运行时会增加热更新通信和补丁执行开销。
- 对可维护性的影响：组件状态保留利于调试，但需要清晰管理模块副作用。

## 相似概念对比

| 维度 | HMR | Live Reload | 结论 |
|---|---|---|---|
| 更新方式 | 局部模块替换 | 整页刷新 | 体验差异来自替换粒度 |
| 状态保留 | 大多可保留 | 通常丢失页面状态 | 调试效率差异明显 |
| 失败回退 | 可回退到刷新 | 无局部替换能力 | 两者常同时存在 |
| 适用场景 | 组件样式和逻辑迭代 | 静态资源或不可替换变更 | 不能混淆概念 |

## 配置示例

```ts
export default {
  mode: 'development',
  devServer: {
    hot: true,
    liveReload: true,
  },
}

if (import.meta.webpackHot) {
  import.meta.webpackHot.accept('./render', () => {
    // 仅替换变更模块，不刷新页面
  })
}
```

### 示例说明

- 这段配置展示了开启热更新并在模块侧声明接受更新的最小方式。
- `hot: true` 只开启能力，不代表所有模块都一定可热替换。
- 常见错误写法是只开 `devServer.hot`，但模块内没有 `accept`，最终仍频繁整页刷新。

## 真实场景示例

- 场景描述：表单页面开发时每次改样式都触发刷新，输入状态反复丢失。
- 问题表现：本地调试效率下降，联调周期被拉长。
- 处理方案：开启 HMR 并在表单入口组件接受局部更新，避免刷新整页。
- 验证方式：改动样式后页面不刷新且输入状态保留，控制台无热更新失败日志。

## 常见误区

- 误区：开启 `hot` 就一定不会刷新页面。
  - 正解：不可替换模块或冒泡到根模块时仍会刷新。
- 误区：HMR 只和前端代码有关，和服务器无关。
  - 正解：开发服务器负责推送变更信息和更新清单，是链路关键环节。

## 关联知识点

- 依赖：Loader 与 Plugin 扩展机制。
- 影响：构建提速策略和开发体验。
- 对比：浏览器刷新与缓存命中路径。

## 相关面试题

1. webpack 热更新的实现原理？
   - 标准答案：核心链路是“文件变更触发增量编译 -> 服务端推送新哈希 -> 客户端拉取更新清单与模块补丁 -> 运行时替换模块并执行回调 -> 失败回退整页刷新”。能否替换成功取决于模块接受边界和依赖关系。
   - 考察点：是否理解服务端通知、客户端拉取和运行时替换三段式机制。
