## 概念定位

实时通道是 Web 请求-渲染主链之外的“持续数据通道”。它的核心目标不是一次请求拿全量数据，而是在连接存续期间持续传递增量事件。

这一节点要解决三个问题：

- 选什么通道（短轮询、长轮询、SSE、WebSocket）
- 如何保证稳定性（重连、心跳、幂等、顺序）
- 如何让消息流不拖垮渲染（背压、批处理、调度）

## 工作机制

### WebSocket（Web 套接字）

WebSocket 通过 HTTP 升级握手建立全双工连接：

1. 客户端发 `Upgrade: websocket` 握手请求。
2. 服务端返回 `101 Switching Protocols`。
3. 双方进入帧级双向通信，直到关闭。

适合高频双向交互：协作编辑、在线游戏、语音控制、需要客户端主动上行事件的场景。

### SSE（Server-Sent Events, 服务端发送事件）

SSE 使用 `text/event-stream` 持续推送，客户端通常用 `EventSource` 订阅：

- 服务端持续发送 `data:` 事件块
- 可带 `id:`、`event:`、`retry:`
- 断线后浏览器可自动重连并带 `Last-Event-ID`

SSE 是单向（服务端 -> 客户端），更适合“前端主要消费增量流”的场景。

## 关键知识点

### 四种实时方案对比

| 方案 | 方向 | 建连开销 | 实时性 | 服务端复杂度 | 典型场景 |
| --- | --- | --- | --- | --- | --- |
| 短轮询 | 客户端拉取 | 高 | 低 | 低 | 简单状态刷新 |
| 长轮询 | 客户端拉取 | 中高 | 中 | 中 | 兼容优先实时通知 |
| SSE | 服务端单向推送 | 中 | 中高 | 中 | 通知流、日志流、AI 文本流 |
| WebSocket | 双向推送 | 一次握手后低 | 高 | 较高 | 协作、互动、高频双向指令 |

### 消息协议设计

无论 SSE 还是 WebSocket，建议统一事件模型：

- `eventType`：事件类型
- `messageId`：幂等键
- `timestamp`：服务端时间
- `payload`：业务内容
- `traceId`：链路追踪

这样可以统一重放、去重、监控和故障定位。

### 背压与渲染预算

高频消息最常见故障不是“断连”，而是“前端来不及渲染”：

- 消息入站速率 > 主线程处理速率
- 状态更新过于频繁导致重渲染风暴
- 大量 JSON 解析和 DOM 更新引发长任务

建议：

- 事件聚合后再渲染（批处理）
- 动画帧节奏更新（`requestAnimationFrame`）
- 高频流做降采样/节流
- 必要时把解析放到 Worker 线程

### 重连策略

重连必须有退避与抖动（Exponential Backoff + Jitter）：

- 避免固定间隔重连造成雪崩
- 服务恢复时避免瞬时洪峰
- 结合网络状态和页面可见性决定是否重连

## Agent 应用场景选型

### 结论先行

- 以模型输出流为主（token 连续下发、前端主要消费）优先用 SSE。
- 需要强双向实时控制（中断生成、多人协作、上行事件频繁）优先用 WebSocket。
- 大多数 Agent 聊天产品可采用“混合方案”：SSE 下行流 + HTTP 上行命令。

### 为什么 Agent 场景常先选 SSE

- 天然适配“服务端连续推送文本/事件”。
- 浏览器原生 `EventSource` 使用简单。
- 穿透代理和基础设施通常更平滑。
- 断线重连语义简单，改造成本低。

### 什么时候必须上 WebSocket

- 客户端需要持续上行（如语音片段、协作操作、频繁工具状态回传）。
- 需要低延迟双向控制（暂停、恢复、打断、追问并发控制）。
- 同一连接承载多类双向事件，且会话状态复杂。

### Agent 通道决策表

| 需求 | 推荐 |
| --- | --- |
| 纯文本流式回答 | SSE |
| 文本流 + 低频上行动作 | SSE + HTTP |
| 多人协作白板/编辑 | WebSocket |
| 语音对话/实时控制 | WebSocket |
| 企业内网代理限制严格 | 先验证 SSE，再评估 WebSocket |

## 对页面性能的影响

- 高频事件会抬高主线程占用，恶化交互到下一次绘制（Interaction to Next Paint, INP）。
- 重连风暴会拉高网络与服务端成本。
- 未做幂等和去重会导致重复渲染与状态污染。

## 观测与排障

### 最小可观测指标集

- 连接成功率
- 断线率与平均会话时长
- 重连次数分布
- 消息端到端延迟
- 消息丢失率/重复率
- 前端长任务占比与 INP

### 排障剧本

1. 先看连接层：握手是否成功，失败集中在哪类网络/设备。
2. 再看消息层：是否乱序、重复、丢失，是否有幂等键。
3. 再看渲染层：是否因高频 setState/响应式更新导致卡顿。
4. 检查重连策略是否无退避，是否引发雪崩。
5. 对 Agent 场景，单独观察“首 token 时间”和“完整回答时间”。

### 常见故障与处理

- 连接频繁断开：检查网关空闲超时、心跳间隔、代理策略。
- 页面越用越卡：检查消息是否逐条立即渲染，改为批处理。
- 重连后消息重复：引入 `messageId` 去重与断点续传语义。
- 只在移动网络下不稳定：提高超时容忍，优化重连与降级策略。

## 实战实现建议

### SSE 建议

- 响应头设置 `Content-Type: text/event-stream`、`Cache-Control: no-cache`。
- 事件设计尽量细粒度，前端可按 `event` 分路由处理。
- 对长流回复，服务端定期发送心跳事件避免中间链路超时。

### WebSocket 建议

- 明确连接生命周期：鉴权、续期、心跳、关闭码。
- 定义统一消息 envelope，避免多业务各自造格式。
- 连接数和频道订阅做上限控制，避免单用户过载。

### 前端调度建议

- 数据层和 UI 层解耦：先入缓冲区，再按帧刷新。
- 对高频更新区域做局部渲染，避免整页重绘。
- 结合可见性状态（Page Visibility）在后台标签降低更新频率。

## 常见误区

- “WebSocket 一定优于 SSE”：不对，单向流式输出场景 SSE 更轻。
- “连接不断就代表体验好”：不对，还要看延迟、抖动、丢包和渲染卡顿。
- “重连写个定时器就够了”：不对，没有退避和抖动会放大故障。
- “实时越高频越好”：不对，超出渲染预算会直接变成掉帧和卡顿。

## 面试补充

### 30 秒说明

实时通信方案要按方向性与交互强度选型。SSE 适合服务端单向流式推送，WebSocket 适合高频双向交互。工程上真正的难点不是建连，而是稳定性和渲染预算控制，包括重连退避、消息幂等、背压处理和指标闭环。

### 高频追问

1. WebSocket 和 SSE 的核心区别是什么？

WebSocket 是全双工双向通信；SSE 是服务端单向推送。SSE 实现更轻，WebSocket 交互能力更强。

2. Agent 场景下为什么很多产品先用 SSE？

因为大部分场景是“服务端连续下发 token，前端消费展示”，单向流更匹配，接入和运维成本更低。

3. 什么时候必须用 WebSocket？

当客户端上行频繁且低延迟要求高，例如语音实时控制、多人协作、复杂会话控制。

4. 如何避免重连风暴？

用指数退避和随机抖动，按网络状态控制重连，服务端也要做限流与熔断保护。

5. 实时消息为什么会导致页面卡顿？

因为消息处理和渲染都在主线程，高频逐条更新会产生长任务，必须做批处理和调度。

6. 如何保证重连后的消息一致性？

引入 `messageId` 幂等去重、断点续传标记（如 SSE 的 `Last-Event-ID`）和服务端可重放窗口。
