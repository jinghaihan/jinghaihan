## 概念定义

- 本质定义：实时通道是客户端与服务端在请求响应之外进行持续消息传输的通信方式，常见实现是 WebSocket 和 SSE（Server-Sent Events）。
- 解决问题：降低轮询开销，实现低延迟消息推送与状态同步。
- 边界说明：它不替代常规 REST 接口，数据一致性与幂等仍需业务层保证。

## 核心机制拆解

### 1) 机制/流程

1. 客户端建立长连接通道（WebSocket）或单向流（SSE）。
2. 服务端持续推送事件，客户端按事件类型分发处理。
3. 连接中断时执行重连和退避策略。
4. 关键消息通过确认机制保证投递可靠性。

### 2) 关键条件与触发点

- 条件 A：业务是否需要双向实时通信，决定选 WebSocket 还是 SSE。
- 条件 B：网络抖动与连接空闲时长，决定心跳和重连参数。

### 3) 结果与影响

- 对性能 / 稳定性 / 可维护性的影响：实时通道可降低轮询压力并提升交互实时性，但会增加连接管理复杂度和后端资源占用。

## 相似概念对比

| 维度 | WebSocket | SSE | 结论 |
|---|---|---|---|
| 定义 | 全双工长连接 | 服务端到客户端单向事件流 | 选型先看是否需要双向通信 |
| 触发时机 | 聊天、协同编辑、双向控制 | 通知流、行情流、日志流 | 简单推送优先 SSE，复杂交互优先 WebSocket |
| 性能影响 | 长连接常驻，需心跳维护 | 实现简单，重连成本较低 | 两者都优于高频轮询 |
| 适用场景 | 双向交互强 | 单向推送强 | 面试常问与轮询对比 |

## 代码

```js
const ws = new WebSocket('wss://api.example.com/realtime')
ws.onmessage = (event) => {
  console.log('message', event.data)
}
```

```js
const sse = new EventSource('https://api.example.com/events')
sse.onmessage = (event) => {
  console.log('event', event.data)
}
```

### 示例说明

- 这段代码 / 配置解决什么问题：分别展示 WebSocket 和 SSE 的最小接入方式。
- 关键参数含义：`wss://` 使用 TLS（Transport Layer Security）保护实时通道；`EventSource` 默认自动重连。
- 常见错误配置：只做无限重连不做退避，导致故障时雪崩放大。

## 真实场景示例

- 场景描述：交易看板原先用 1 秒轮询刷新报价。
- 问题表现：高峰期接口压力大，前端展示延迟明显。
- 处理方案：报价改为 SSE 推送，写操作仍走 HTTP 接口，补充断线重连和心跳监控。
- 验证方式：对比改造前后服务端 QPS、客户端延迟和丢包重连率。

## 常见误区

- 误区：实时业务必须使用 WebSocket。
- 正解：单向推送场景 SSE 更轻量，选型应按通信方向和维护成本决定。
- 误区：用了实时通道就不需要状态兜底。
- 正解：重连窗口内可能丢事件，需要补偿拉取或版本校验机制。

## 关联知识点

- 依赖：依赖 `tcp-tls` 的稳定连接和安全传输。
- 影响：影响 `request-lifecycle` 的连接策略与服务端并发治理。
- 对比：实时通道强调持续连接，常规 HTTP 强调请求响应。
- 组合：与 `http-evolution` 一起考“服务器推送 vs 实时通道”的边界。

## 相关面试题

1. HTTP 1.1 和 HTTP 2.0 的区别
- 标准答案：HTTP/2 的服务器推送是资源预推能力，不等于通用实时消息通道；实时双向通信通常使用 WebSocket，单向事件流可用 SSE。
- 考察点：是否能区分“资源推送”与“业务实时通信”。
- 追问 1：为什么很多场景不直接用轮询？
- 标准答案 1：高频轮询会产生大量无效请求和延迟抖动，长连接推送在实时性和成本上更可控。
- 追问 2：实时通道断开后怎么保证体验？
- 标准答案 2：使用心跳、指数退避重连和断线补偿拉取，保证状态最终一致。

2. 说一下 HTTP 3.0
- 标准答案：HTTP/3 基于 QUIC 改善弱网下传输恢复和建连时延，对实时场景有帮助，但不替代业务层的消息可靠性设计。
- 考察点：是否能把协议收益和业务可靠性边界说清楚。
