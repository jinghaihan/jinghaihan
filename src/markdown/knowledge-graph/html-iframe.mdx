## 概念定义

- 本质定义：`iframe` 用于在当前页面里嵌入一个独立文档上下文，并通过浏览器同源策略划清访问边界。
- 解决问题：在主页面中安全地加载第三方页面、独立系统或隔离运行的子页面。
- 边界说明：它不是通用布局容器，也不适合把核心业务界面无限制地拆进 `iframe`。

## 核心机制拆解

### 1) 机制/流程

1. 主页面创建 `iframe`，浏览器为其加载独立文档和浏览上下文。
2. 如果主页面与子页面同源，可以直接通过 `contentWindow` 和 DOM（Document Object Model）访问做深度交互。
3. 如果跨源，直接读写 DOM 和存储会被阻止，只能改用 `postMessage` 做消息通信。
4. 接入第三方内容时，再用 `sandbox`、`allow`、`loading`、`referrerpolicy` 收敛权限和性能开销。

### 2) 关键条件与触发点

- 条件 A：跨源场景直接访问 `contentWindow.document` 会触发同源策略拦截。
- 条件 B：消息通信不校验 `origin` 和 `source`，就有被伪造消息污染的风险。
- 条件 C：`iframe` 权限给得太宽，容易引出点击劫持、隐私泄露或多余能力暴露。

### 3) 结果与影响

- 对性能/稳定性/可维护性的影响：边界管理得当时，`iframe` 的隔离价值很高；边界管理失控时，通信、埋点、监控和排障都会变复杂。

## 相似概念对比

| 维度 | 同源 `iframe` | 跨源 `iframe` | 结论 |
|---|---|---|---|
| DOM 访问 | 可直接访问 | 不可直接访问 | 同源策略是硬边界 |
| 函数调用 | 可直接调用对方暴露的方法 | 不可直接调用 | 跨源要协议化 |
| 消息通信 | 可直接访问，也可 `postMessage` | 主要依赖 `postMessage` | 跨源以消息通道为主 |
| 存储能力 | 可共享同源上下文 | 不可直接读写彼此存储 | 边界必须说清 |

## 代码

```html
<iframe
  id="checkout-frame"
  src="https://pay.example.com/embed"
  sandbox="allow-scripts allow-forms"
  loading="lazy"
  referrerpolicy="strict-origin-when-cross-origin"
></iframe>

<script type="module">
  const frame = document.getElementById('checkout-frame')

  window.addEventListener('message', (event) => {
    if (event.origin !== 'https://pay.example.com')
      return

    if (event.data?.type === 'PAY_SUCCESS')
      console.log('支付完成')
  })

  frame?.addEventListener('load', () => {
    frame.contentWindow?.postMessage(
      { type: 'INIT_CHECKOUT', orderId: 'A1001' },
      'https://pay.example.com',
    )
  })
</script>
```

### 示例说明

- 这段代码解决什么问题：在跨源支付页场景下建立可控的 `iframe` 权限和消息通信链路。
- 关键语句或参数含义：`sandbox` 限制能力范围；`loading="lazy"` 控制延迟加载；`postMessage` 用于跨源通信；`event.origin` 用于来源校验。
- 常见错误写法：发送端 `targetOrigin` 用 `*`，接收端不校验消息来源，或者干脆不给 `sandbox`。

## 真实场景示例

- 场景描述：主站嵌入第三方支付页，支付完成后需要回写订单状态。
- 问题表现：偶发收不到支付成功消息，或者收到来源不明的消息事件。
- 处理方案：统一消息协议字段，发送端固定 `targetOrigin`，接收端校验 `origin` 和消息结构，并收紧 `iframe` 权限。
- 验证方式：支付链路成功率、消息超时率、非法消息拦截数和安全扫描结果。

## 常见误区

- 误区：跨源 `iframe` 不能通信。
- 正解：能通信，标准方案是 `postMessage`，只是不能直接访问 DOM 和存储。
- 误区：`iframe` 只要能显示出来就算接好了。
- 正解：权限、消息协议、加载策略、监控和异常回退都要一起设计。
- 误区：`iframe` 只能用在很老的系统里。
- 正解：支付、广告、外部文档、受控隔离场景今天仍然常用，只是不能滥用。

## 关联知识点

- 依赖：同源与跨域策略、加载与阻塞模型。
- 影响：安全边界、第三方接入稳定性、主子页面协作。
- 对比：同源直连访问与跨源消息通信。
- 组合：渐进增强与优雅降级、点击劫持防护。

## 相关面试题

1. HTML中的iframe标签是什么？如何使用？
- 标准答案：`iframe` 是在当前页面中嵌入另一个 HTML 页面或外部文档的标签。机制上它会创建独立的浏览上下文，可以通过 `src` 指定目标页面，通过 `width`、`height`、`loading`、`sandbox` 等属性控制展示和权限。场景上常见于支付页、广告位、文档预览和第三方系统嵌入。
- 考察点：是否知道 `iframe` 的基本职责和常用属性。

2. iframe 有那些优点和缺点？
- 标准答案：优点是内容隔离、部署边界清晰、接第三方系统方便；缺点是加载开销增加、通信更复杂、调试和埋点也更分裂。机制上它本质就是一个独立文档上下文，并受同源策略约束。场景上支付页、外部控制台适合用，主业务页面不要无边界地往里塞。
- 考察点：是否能从隔离收益和治理成本两边同时回答。

3. 什么是点击劫持攻击？如何防范？
- 标准答案：点击劫持是攻击者把受害页面放进透明或伪装的 `iframe`，诱导用户点击敏感按钮。防护上应在服务端设置 `X-Frame-Options` 或 CSP 里的 `frame-ancestors`，并对关键操作叠加二次确认。场景上支付、转账、权限变更页面要优先防护。
- 考察点：是否理解点击劫持依赖的是“可被嵌入 + 用户误点”双条件。

4. 什么是Tabnabbing攻击？如何防范？
- 标准答案：Tabnabbing 是恶意新标签页通过 `window.opener` 回改原页面地址，引导用户进入钓鱼站。防护上应在外链使用 `rel="noopener noreferrer"`，并避免给不可信页面保留 `opener` 引用。场景上 `target="_blank"` 的外部链接最容易被忽视。
- 考察点：是否知道外链安全不只涉及跳转地址，还涉及页面间引用关系。
