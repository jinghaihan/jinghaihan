## 代码

```ts
import { ref, watch } from 'vue'

const keyword = ref('')
const result = ref<string[]>([])

watch(keyword, async (next, _, onCleanup) => {
  const controller = new AbortController()
  onCleanup(() => controller.abort())

  const res = await fetch(`/api/search?q=${encodeURIComponent(next)}`, {
    signal: controller.signal,
  })
  result.value = await res.json()
})
```

### 示例说明

- 这段代码解决什么问题：处理输入联想请求的竞态和过期响应覆盖。
- 关键语句或参数含义：`onCleanup` 在下一次监听触发前取消上一次请求。
- 常见错误写法：不做取消，导致慢请求回包覆盖新结果。

## 相关面试题

1. Computed 和 Watch 的区别
- 标准答案：结论是 `computed` 适合派生状态，`watch` 适合副作用。机制上 `computed` 依赖缓存，依赖不变不会重复计算；`watch` 监听源变化后执行回调。场景上表单联动用 `computed`，请求、埋点、异步副作用用 `watch`。
- 考察点：是否能从定义、触发时机、性能影响和适用场景四个维度做对比。

2. Computed 和 Methods 的区别
- 标准答案：结论是 `computed` 有缓存，`methods` 每次渲染都会执行。机制上 `computed` 只有依赖变化才重算，`methods` 不参与依赖缓存。场景上模板里高频使用的派生值优先 `computed`，一次性动作函数放 `methods`。
- 考察点：是否能从定义、触发时机、性能影响和适用场景四个维度做对比。

3. 过滤器的作用，如何实现一个过滤器
- 标准答案：结论是过滤器本质是“展示层格式化函数”，Vue 3 已移除内置过滤器。机制上可用 `computed`、方法或全局工具函数替代。场景上金额、日期、状态文案格式化建议统一到工具层，避免模板内逻辑膨胀。

4. 如何保存页面的当前的状态

5. 常见的事件修饰符及其作用
- 标准答案：结论是常用修饰符包括 `.stop`、`.prevent`、`.capture`、`.self`、`.once`、`.passive`。机制上修饰符在编译期被转换为对应事件处理包装逻辑。场景上滚动监听优先 `.passive`，表单提交常配 `.prevent`，事件冒泡隔离用 `.stop`。
- 考察点：是否能说明结论背后的原因、成立条件和反例。

6. data 为什么是一个函数而不是对象
- 标准答案：结论是组件的 `data` 必须是函数，避免实例间共享引用。机制上每次创建组件实例都会执行 `data()` 返回全新对象。场景上如果写成对象，多实例会互相污染状态，这是线上高频 Bug。
- 考察点：是否能说明结论背后的原因、成立条件和反例。

7. v-if 和 v-for 哪个优先级更高？如果同时出现，应如何优化？
- 标准答案：结论是 `v-for` 优先级高于 `v-if`，两者同节点会先循环再判断。机制上这会导致每一项都执行条件判断，增加无效渲染。场景上应把 `v-if` 提前为外层包裹，或先用 `computed` 过滤数据再 `v-for`。

8. Vue 子组件和父组件执行顺序
- 标准答案：结论是初始化阶段一般为“父 beforeCreate/created -> 父 beforeMount -> 子 beforeCreate/created/beforeMount/mounted -> 父 mounted”。机制上挂载是深度优先。场景上父组件若依赖子 DOM，需在父 `mounted` 或 `nextTick` 后处理。

13. Composition API 与 React Hook 很像，区别是什么

14. 对虚拟 DOM 的理解？
- 标准答案：结论是虚拟 DOM 是对真实 DOM 的内存描述，用来做差异计算。机制上先生成新旧 vnode，再通过 Diff 找最小变更并提交真实 DOM。场景上它提升的是“跨平台抽象和可维护性”，不是任何场景都绝对更快。

- 标准答案：结论是过程分三步：模板/JSX 生成 vnode、Diff 计算变更、Patch 提交真实 DOM。机制上关键在同层比较和 key 命中，减少不必要节点移动。场景上列表更新性能主要看 key 是否稳定。

16. 为什么要用虚拟 DOM
- 标准答案：结论是为了把声明式状态更新转换成可控的 DOM 变更。机制上框架接管对比和批处理，开发者只描述状态。场景上中大型应用能显著降低手写 DOM 操作错误率和维护成本。

17. 虚拟 DOM 真的比真实 DOM 性能好吗
- 标准答案：结论是不是绝对。机制上在小规模、一次性更新中直接操作 DOM 可能更快；在复杂交互中，虚拟 DOM 通过批处理和差异计算更稳定。场景上要看更新频率、节点规模和重排成本，不应一刀切。

18. DIFF 算法的原理
- 标准答案：结论是 Diff 核心是“同层比较 + key 命中 + 最小化补丁”。机制上不同层级直接替换、同层按 key 复用和移动节点。场景上 key 不稳定会导致复用失败，引发状态错位和性能下降。
