## 概念定位

浏览器 DNS 缓存是 DNS 链路最前置命中点，负责复用近期解析结果，减少重复递归查询。

## 工作机制

1. 浏览器收到域名请求后，先查询本地 DNS 缓存。
2. 校验记录是否在 TTL（Time To Live） 有效期内。
3. 命中时直接返回 IP，进入建连阶段。
4. 未命中则继续查询系统 DNS 缓存。

## 关键知识点

- **首层命中价值高**：命中后可直接跳过后续 DNS 分层查询。
- **TTL 驱动复用**：过短 TTL 会显著增加重复解析。
- **域名粒度隔离**：子域通常独立缓存，跨域资源越多越容易失配。
- **资源提示协同**：`dns-prefetch` 可提升潜在命中概率。

## 对页面性能的影响

- 多域名首屏页面中，命中收益会被倍数放大。
- 首层未命中会把耗时传导到系统与递归链路。
- 记录波动会引发建连路径不稳定。

## 观测与排障

### 浏览器侧

- Network 观察 DNS Lookup 时长和重复域名请求行为。
- 对比首访/复访同域资源的 DNS 开销。

### 排障步骤

1. 统计首屏关键域名数量和复用情况。
2. 检查是否存在短 TTL 导致频繁重解析。
3. 优化域名收敛，减少非必要跨域首屏依赖。
4. 用 `dns-prefetch` 前置高概率域名。
5. 回归看 DNS 阶段耗时分布变化。

## 常见误区

- “浏览器缓存清了就等于 DNS 清了”：不准确，系统层可能仍命中。
- “同域资源一定只解析一次”：不准确，TTL 到期会重新解析。
- “DNS 阶段不重要”：不准确，它在建连前，慢一步会拖慢全链路。

## 题库深挖补充

### 高频对比与边界（基于题库）

- 浏览器 DNS 缓存命中后，通常不会继续向操作系统查询；操作系统缓存命中后，也不会进入递归解析。
- hosts 文件优先级高于外部 DNS 结果，常用于本地调试和应急切流。
- 负缓存（Negative Caching）会缓存“不存在域名”结果，排障时要注意缓存滞后。
- 本地缓存与公共 DNS 缓存并不是一层，清理浏览器缓存不等于清理系统解析缓存。
- 多网络环境切换（公司 VPN、移动热点）会改变本地解析路径，导致“同机不同时段结果不同”。
- 面试常问“为什么同一域名一会儿能访问一会儿不能访问”，核心就是多层缓存状态不一致。

### 线上排障剧本

1. 先确认问题是“解析失败”还是“解析到错误 IP”。
2. 在同一台机器分别检查浏览器缓存、系统缓存和 hosts 覆盖情况。
3. 对比本地解析结果和权威解析结果，判断是本地污染还是上游问题。
4. 检查 TTL 与负缓存窗口，确认是否处于缓存过渡期。
5. 若涉及紧急切流，优先用可控 DNS 策略，不建议长期依赖 hosts。

### 高频追问补充（附答案）

1. 为什么清了浏览器缓存问题还在？

因为解析可能命中系统缓存或网络侧缓存，浏览器缓存不是唯一层。

2. hosts 的风险点是什么？

它优先级很高，临时调试忘记清理会长期污染本机解析结果。

3. 什么是负缓存，为什么排障时容易忽略？

负缓存会记住“解析失败”结果，在 TTL 没过期前会持续失败，即使权威 DNS 已恢复。

### 术语速查与易错点

- A 记录：域名到 IPv4 地址映射。
- AAAA 记录：域名到 IPv6 地址映射。
- CNAME（Canonical Name）：别名记录，常用于接入 CDN。
- TTL（Time To Live）：缓存有效时长，不是“固定刷新时间”。
- NXDOMAIN：权威返回域名不存在。

常见易错点：

- 清浏览器缓存后立即下结论，忽略系统层缓存。
- 忘记 hosts 临时改动，导致排障方向偏离。
- 只在单网络环境验证解析结果。

## 深度延展（案例对照）

### 浏览器缓存偏差案例

- 场景：同机同网下，隐身模式正常、普通模式异常。
- 诊断：浏览器 DNS 缓存残留旧结果。
- 处理：清理浏览器网络栈缓存并回归。

### 实操提示

- 排障时固定使用普通窗口 + 隐身窗口双样本对照。
- 记录缓存清理前后解析结果，避免主观判断。

## 进阶排障清单（补充）

### 现场处置顺序

1. 先固定样本：同地域、同网络、同设备，避免样本漂移。
2. 抓三组结果：本地解析、公共 DNS 解析、权威解析。
3. 比对 TTL 与记录版本，确认是否处于传播窗口。
4. 标记是否命中本地特殊配置（hosts、代理、VPN）。
5. 形成“可复现路径 + 影响范围 + 临时止损”三件套再升级处理。

### 面试追问强化

1. 为什么 DNS 问题经常“不可稳定复现”？

因为多层缓存、地域路由和传播窗口共同影响，同一时间不同样本可能命中不同路径。

2. 如何用最少步骤判断是不是 DNS 问题？

先看不同解析器结果是否分叉，再看业务域名能否稳定映射到预期接入点。

3. 何时应该优先止损而不是继续追根因？

当影响范围在扩大且已有可控切流手段时，应先保可用性再做根因定位。

## 速查补充（第四轮）

### 快速判定问答

1. 什么时候先怀疑缓存层而不是权威记录本身？

当不同客户端结果分叉明显且权威查询稳定时，优先检查本地与递归缓存状态。

2. DNS 类问题为什么要强调样本固定？

因为网络、地域、时间窗口会影响解析路径，不固定样本容易得到互相矛盾结论。

## 面试补充

### 30 秒说明

浏览器 DNS 缓存是解析链路第一道加速层。命中即可跳过后续查询，直接进入建连。优化重点是控制域名数量、治理 TTL、用好资源提示策略。

### 高频追问

1. 为什么二次访问 DNS 仍可能慢？

TTL 到期、缓存淘汰、域名变化或网络环境切换都会导致再次解析。

2. DNS 优化第一优先级是什么？

先减少首屏关键域名数量，再做提示和缓存策略优化。

3. `dns-prefetch` 的真实收益场景？

适合“很可能访问但不确定立即请求”的域名，可把解析前置到浏览器空闲时段。
