## 概念定义

- 本质定义：HTML（HyperText Markup Language）解析与 DOM（Document Object Model）构建是浏览器把文本文档转换为可计算节点树的过程。
- 解决问题：为样式计算、布局和脚本执行提供结构化输入。
- 边界说明：它不负责最终像素绘制，绘制发生在后续阶段。

## 核心机制拆解

### 1) 机制/流程

1. 浏览器下载文档字节流并按字符编码解码。
2. 词法和语法分析生成节点并挂载到 DOM 树。
3. 解析过程中遇到外部资源会发起并行下载。
4. 遇到阻塞脚本时解析暂停，脚本执行后继续。
5. DOM 增量完成后进入样式计算阶段。

### 2) 关键条件与触发点

- 条件 A：文档结构是否规范，决定容错路径和修复成本。
- 条件 B：脚本加载策略是否阻塞解析，决定首屏构建速度。

### 3) 结果与影响

- 对性能 / 稳定性 / 可维护性的影响：解析阶段越平稳，后续样式和布局波动越小；结构混乱会放大后续渲染成本。

## 相似概念对比

| 维度 | DOM 树 | 渲染树 | 结论 |
|---|---|---|---|
| 定义 | 文档结构与节点关系 | 可见节点及样式计算结果 | 渲染树是 DOM 与样式计算的交集 |
| 触发时机 | HTML 解析阶段生成 | 样式计算后生成 | 先有 DOM 再有渲染树 |
| 性能影响 | 节点数量影响后续计算规模 | 直接影响布局与绘制成本 | 两者都需控制复杂度 |
| 适用场景 | 结构语义表达 | 视觉渲染执行 | 面试常考关系和差异 |

## 代码

```js
// 粗略观察文档节点规模
const nodeCount = document.getElementsByTagName('*').length
console.log('node count', nodeCount)
```

### 示例说明

- 这段代码解决什么问题：快速评估页面节点规模，辅助判断解析和后续渲染压力。
- 关键语句或参数含义：`getElementsByTagName('*')` 返回文档内全部元素集合。
- 常见错误写法：在超大页面高频执行全量遍历，反而增加主线程负担。

## 真实场景示例

- 场景描述：活动页模板叠加后首屏显著变慢。
- 问题表现：节点数量爆炸，解析和样式计算时间持续升高。
- 处理方案：精简嵌套层级，拆分非关键区域延迟渲染。
- 验证方式：对比优化前后 DOM 节点数和首屏阶段耗时。

## 常见误区

- 误区：只要接口快，首屏就一定快。
- 正解：解析和构树也会占据主线程预算，结构复杂同样会拖慢首屏。

## 关联知识点

- 依赖：依赖 `status-code-path` 返回文档响应。
- 影响：影响 `cssom-js`、`render-tree` 的输入规模。
- 对比：`html-parser` 负责结构构建，`cssom-js` 负责样式和脚本协同。
- 组合：常与 `parse-blocking` 一起定位首屏阻塞。

## 相关面试题

1. 浏览器的渲染过程
- 标准答案：渲染过程先解析 HTML 构建 DOM，再结合样式生成渲染树，随后执行布局、绘制和合成；HTML 解析是整个流程入口。
- 考察点：是否能按阶段顺序讲清职责链路。

2. 构建 DOM 与 CSSOM
- 标准答案：DOM 构建是增量解析，解析器会边读边挂节点；CSSOM 构建需要样式规则完整且可计算，因此在关键样式未就绪时会影响后续渲染阶段。两者汇合后才会进入渲染树构建。
- 考察点：是否能区分 DOM 增量构建和 CSSOM 对关键路径的约束差异。
