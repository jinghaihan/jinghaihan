## 概念定位

HTML 解析与 DOM（Document Object Model）节点描述的是：浏览器如何把网络响应的字节流，逐步转换为可参与渲染和脚本操作的结构化文档树。

这个阶段是渲染链的“地基层”：

- 结构不完整，后续样式、布局、绘制都无法稳定推进。
- 解析阶段被阻塞，首屏展示会整体后移。

## 工作机制

1. 浏览器读取 HTML 字节流并进行字符解码。
2. Tokenizer 扫描出开始标签、结束标签、属性、文本等 token。
3. Tree Builder 按规范插入节点，构建 DOM。
4. 预加载扫描器并行发现外部资源（CSS、JS、字体、图片）。
5. 遇到同步脚本时主解析可能暂停，等待脚本下载执行后继续。

## 关键知识点

### 增量解析而非一次性解析

DOM 是边收边建。网络慢、HTML 大、脚本阻塞都可能中断“可见结构”增长速度。

### 预加载扫描器的价值与边界

- 价值：提前发现关键资源，缩短下载启动时间。
- 边界：不改变主线程 DOM 构建顺序，不能消除脚本阻塞本质。

### 脚本与解析的交互

- 同步脚本可能打断解析与渲染节奏。
- `defer` 脚本并行下载，通常在解析完成后按顺序执行。
- `async` 脚本下载完成即执行，顺序不保证，可能插队打断。
- 模块脚本（`type="module"`）具备延迟执行语义，但依赖图仍需评估。

### defer / async / module 的适用场景

- `defer`：用于主业务逻辑脚本，尤其是有先后依赖关系、需要稳定顺序执行的场景。
- `async`：用于独立脚本（埋点、广告、统计），不依赖其他脚本、也不被其他脚本依赖。
- `module`：用于现代模块化工程入口，适合需要明确依赖图和按模块加载的场景。

简单原则：主链路脚本优先 `defer` 或 `module`；独立第三方脚本再考虑 `async`。

### 样式依赖导致的间接阻塞

当脚本读取布局/样式信息时，浏览器需保证 CSSOM（CSS Object Model）可用，可能出现“脚本等待样式”的阻塞链。

### DOM 规模与复杂度

- 节点数量、嵌套深度、属性复杂度都会影响解析和后续样式计算成本。
- 首屏结构越精简，越容易提前进入可见内容渲染。

## 对页面性能的影响

- HTML 解析慢会拉长 FCP（First Contentful Paint）与 LCP（Largest Contentful Paint）。
- 解析与脚本交错不当会导致主线程长任务，影响 INP（Interaction to Next Paint）。
- 关键结构出现过晚会让“页面有响应”明显推迟。

## 观测与排障

### 浏览器侧观测

1. Performance 面板查看 `Parse HTML` 与 `Evaluate Script` 交错关系。
2. Network 面板看关键 CSS/JS 下载是否前置。
3. Lighthouse 看阻塞渲染资源与关键请求链提示。

### 排障剧本

1. 列出首屏必需结构与必需资源。
2. 将非关键脚本改为 `defer`/懒加载。
3. 拆分关键 CSS，减轻首屏解析等待。
4. 优化模板输出，降低初始 DOM 规模与深度。
5. 回归验证 FCP/LCP/INP 是否同步改善。

## 实战优化策略

### 结构层

- 首屏 HTML 保留必要骨架，低优先内容延后注入。
- 避免模板层过深嵌套和冗余容器。
- 对重复大列表使用分页、虚拟化或分块渲染。

### 资源层

- 关键 CSS 内联或前置加载。
- 非关键脚本延后执行，第三方脚本异步隔离。
- 使用资源提示（`preload`/`preconnect`）缩短关键依赖启动时机。

### 运行时层

- 初始化逻辑分阶段执行，避免首帧前堆积。
- 长任务拆分为小任务，给渲染留出机会。
- 避免在首屏解析窗口做大量同步计算。

## 常见误区

- “HTML 解析只受网速影响”：错误，脚本阻塞和 DOM 复杂度同样关键。
- “脚本放底部就永远最优”：错误，应按依赖关系选 `defer`/`async`/模块策略。
- “预加载扫描器可以解决所有阻塞”：错误，它只能提前发现，不能改写主线程执行约束。

## 面试补充

### 30 秒说明

HTML 解析阶段负责把字节流转成 DOM，是渲染链第一步。浏览器会增量解析并并行探测资源，但同步脚本和样式依赖会打断流程。优化重点是减少首屏阻塞脚本、前置关键样式、控制 DOM 规模和初始化任务密度。

### 高频追问

1. `defer` 和 `async` 如何选择？

先看“是否有依赖和顺序要求”：

- 有依赖、要保证顺序：用 `defer`。
- 无依赖、独立脚本：用 `async`。
- 模块化主入口：优先 `type="module"`，再结合拆包策略。

2. 为什么脚本会“被 CSS 卡住”？

脚本若读取样式/布局，浏览器必须先保证 CSSOM 可用，因此脚本可能等待样式加载和解析。

3. DOM 过大为什么会影响性能？

会同时增加解析、样式计算、布局、绘制成本，且放大后续更新开销。

4. 预加载扫描器的局限是什么？

它可以提前发现资源，但不能跳过主解析顺序，也不能消除同步脚本阻塞。

5. 如何判断问题在解析阶段而非接口阶段？

若网络到首字节正常，但 `Parse HTML`/脚本评估占比高，通常瓶颈在解析和执行阶段。

6. HTML 优化如何与工程化协同？

通过模板瘦身、关键资源拆分、构建期注入资源提示与脚本策略，形成可持续优化链路。


## 进阶补充（深挖）

### 与上下游节点的因果关系

- 上游输入（常见）：响应流接收、解析阻塞治理、HTML 加载与语义策略。
- 本节点职责：将“HTML 解析与 DOM”的结果转化为可执行的后续分支决策。
- 下游影响（常见）：运行时与存储关联、构建渲染树。

### 线上故障案例

- 现象：首屏接口正常但 FCP/LCP 变慢，火焰图显示解析与脚本交错时间拉长。
- 根因：首屏同步脚本和非关键结构过多，打断 HTML 增量解析。
- 处置：下沉非关键脚本到 defer/懒执行，精简首屏 DOM 骨架并前置关键样式。
- 回归标准：关键指标恢复到基线区间，且连续观测窗口无二次抖动。

### 高频追问）

1. 为什么 HTML 结构也会影响交互性能？

结构复杂会放大样式和布局成本，影响后续每次更新。

2. 预加载扫描器能解决所有阻塞吗？

不能，它只能提前发现资源，无法改变同步执行约束。

3. 首屏优化中先改脚本还是改结构？

先定位占比，通常优先处理同步脚本与首屏冗余结构。


### 补充记忆框架

- 先判断该节点是“前置等待”问题还是“运行时抖动”问题。
- 再判断问题来自“资源策略”还是“执行调度”。
- 最后把定位结论转为可复用规则，进入门禁与动作库。

### 高频追问

1. 这个节点优化后如何做回归验收？

至少做两组对比：关键指标分位数对比 + 上下游链路稳定性对比，避免局部优化导致成本转移。

2. 为什么这个节点要单独做发布观察窗口？

因为它对主链路有放大效应，短时间异常会快速传导到首屏和交互体验。
