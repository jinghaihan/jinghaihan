## 概念定义

- 本质定义：枚举、类与装饰器关注 TypeScript 在接近运行时结构的一侧如何表达常量集合、面向对象抽象和元数据增强。
- 解决问题：为类式建模、权限控制和框架风格写法提供类型层支持。
- 边界说明：本节点不是推荐“所有前端都写成类”，而是解释这些能力何时有用、何时该克制。

## 核心机制拆解

### 1) 机制/流程

1. 枚举用于表达有限常量集合，必要时再考虑 `const enum` 以减少运行时代码。
2. 类通过访问修饰符和抽象类约束实例能力和继承边界。
3. 装饰器通过元数据或包装逻辑给类、方法、属性追加行为。
4. 只有当框架、SDK 或复杂建模真的需要时，才应该使用这些能力。

### 2) 关键条件与触发点

- 条件 A：简单常量集合若强行上 `enum`，有时不如字面量联合类型直观。
- 条件 B：前端业务逻辑如果到处滥用继承，代码会很快变重。
- 条件 C：装饰器牵涉编译配置和运行时行为，没必要时不应为了“高级”而上。

### 3) 结果与影响

- 对性能/稳定性/可维护性的影响：在需要类式抽象的场景下很有价值；在函数式和组合式风格更自然的场景里，过度使用只会增加复杂度。

## 相似概念对比

| 维度 | `enum` | `const enum` | 结论 |
|---|---|---|---|
| 运行时代码 | 会生成对象 | 常被内联展开 | 体积和调试体验不同 |
| 调试体验 | 更直观 | 编译后更轻 | 取舍要看场景 |
| 适用场景 | 需要运行时枚举对象 | 对性能和产物敏感的内部常量 | 团队要统一策略 |
| 风险 | 产物多一点 | 某些构建链有兼容顾虑 | 不是永远越轻越好 |

## 代码

```ts
enum Role {
  Admin = 'admin',
  User = 'user',
}

abstract class Animal {
  constructor(protected name: string) {}

  abstract speak(): string
}

class Dog extends Animal {
  speak() {
    return `${this.name}: wang`
  }
}
```

### 示例说明

- 这段代码解决什么问题：用枚举约束有限取值，用抽象类约束继承结构和子类实现。
- 关键语句或参数含义：`protected` 表示子类可访问、外部不可访问；`abstract` 表示必须由子类实现。
- 常见错误写法：明明一组字符串联合就够了，还强行引入重量级类层级。

## 真实场景示例

- 场景描述：设计系统 SDK 需要统一组件权限、状态和值对象表达，且这套结构会被多个项目复用。
- 问题表现：纯对象写法越来越散，继承关系和约束边界说不清。
- 处理方案：只在共享模型层使用类、抽象类和访问修饰符；简单常量优先评估字面量联合，必要时才上枚举。
- 验证方式：看公共模型是否更清晰、运行时代码是否可控，以及调用方使用体验是否改善。

## 常见误区

- 误区：用了 TypeScript 就应该多用类和继承。
- 正解：TypeScript 支持类，不代表类是默认最佳方案。
- 误区：装饰器只是语法糖，随便用都行。
- 正解：装饰器会改变声明方式和编译配置，要明确依赖链和运行时行为。
- 误区：`private`、`protected`、`public` 只是写法不同。
- 正解：它们直接决定外部、子类和类内部的可见边界。

## 相关面试题

1. 什么是 TypeScript 的装饰器（Decorators）？它有什么应用场景？
- 标准答案：装饰器是附加在类、方法、属性上的特殊声明，用来在声明阶段附加元数据或包装逻辑。机制上它依赖编译配置和运行时代码配合。场景上依赖注入、路由元数据、序列化和一些框架式写法里会用到，但普通业务代码不该滥用。
- 考察点：是否知道装饰器的价值和成本是一起出现的。

2. TypeScript 中的 `enum`（枚举）是什么？它和 `const enum` 的区别？
- 标准答案：`enum` 用来描述有限常量集合，并在编译后保留对应运行时对象；`const enum` 更偏编译期内联，生成代码更轻。机制上两者的主要区别是运行时代码保留与否。场景上需要运行时枚举对象时用 `enum`，内部常量、追求更轻产物时才考虑 `const enum`。
- 考察点：是否理解二者的运行时代码差异。

3. `public`, `private`, `protected` 修饰符有什么区别？
- 标准答案：`public` 任何地方都能访问，`private` 只能类内部访问，`protected` 则允许类内部和子类访问。机制上它们控制的是可见性边界。场景上公共 API 用 `public`，内部实现细节用 `private`，可复用的继承扩展点用 `protected`。
- 考察点：是否能把访问修饰符说成边界管理工具。

4. TypeScript 中的 `abstract` class（抽象类）是什么？
- 标准答案：抽象类是不能直接实例化、只能被继承的类，可以包含抽象方法和已实现方法。机制上它用来约束子类必须实现某些能力。场景上定义共享模型骨架、适配器或框架基类时有用，但别为了抽象而抽象。
- 考察点：是否知道抽象类的真正价值在于“约束继承结构”。

5. TypeScript 中的 Mixins 是什么？如何实现？
- 标准答案：Mixins 是把多个能力片段组合到同一个类上的模式，避免深层继承。机制上常见做法是高阶函数返回扩展后的类。场景上当多个类需要共享一组横切能力，但又不适合拉出公共父类时，Mixins 很有用。
- 考察点：是否能区分“组合能力”和“继承层级”。
