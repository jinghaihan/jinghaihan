## 概念定位

HTML 加载与阻塞模型决定了首屏可见时间和交互可用时间。核心问题是：哪些资源会阻塞解析、阻塞渲染、阻塞交互。

这个节点直接控制 First Contentful Paint（首次内容绘制）和 Largest Contentful Paint（最大内容绘制）的上限。

## 工作机制

1. 浏览器边下载 HTML 边解析，遇到同步脚本默认会暂停解析并执行脚本。
2. CSS 会阻塞渲染树构建；脚本如果依赖样式计算，间接会受到 CSS 下载与解析影响。
3. `defer` 脚本会在文档解析后、`DOMContentLoaded` 前按顺序执行。
4. `async` 脚本下载完成就立即执行，执行顺序不保证，适合独立脚本。
5. 通过 `preload/prefetch/preconnect/dns-prefetch` 提前建立关键链路，降低关键资源等待时间。

## 关键知识点

### 必须掌握

- `defer` 不是“比 `async` 慢”，而是“更可控”，适合主业务脚本。
- `async` 适合统计、广告等非关键且无顺序依赖脚本。
- 内联关键 CSS（Critical CSS）可降低首屏白屏时间，但要控制体积避免 HTML 过大。
- 模块脚本（`type="module"`）默认类似 `defer` 行为，且支持按需拆分。
- 资源提示（Resource Hints）要按关键路径使用，滥用会挤占带宽。
- 图片和 iframe 使用懒加载（`loading="lazy"`）可减少首屏竞争。

### 选型与取舍

- 全部改成 `async` 可能导致初始化时序错乱。
- 过多 `preload` 会抢占首屏关键资源带宽，反而拖慢。
- 把所有脚本都延后可能改善首屏却恶化首交互时机。

## 对页面性能与稳定性的影响

- 节点稳定时，前端基础（HTML/CSS/JavaScript） 的链路可预测性会明显提升，发布和排障成本更可控。
- 节点失稳时，问题会向上下游扩散，常表现为性能回退、交互异常或交付风险叠加。
- 该节点优化应和上下游协同验证，避免“局部优化把成本转移给后续节点”。

## 观测与排障

### 核心指标

- FCP（First Contentful Paint，首次内容绘制）
- LCP（Largest Contentful Paint，最大内容绘制）
- TTI（Time to Interactive，可交互时间）
- 阻塞资源数量与总阻塞时长

### 排障顺序

1. 先抓关键请求瀑布图，定位最慢的首屏阻塞资源。
2. 区分是“下载慢”还是“执行慢”：网络瓶颈和主线程瓶颈分开看。
3. 检查脚本属性和执行时机：同步、`defer`、`async` 是否匹配依赖关系。
4. 检查关键 CSS 体积和位置，避免关键样式晚到。
5. 回归验证：冷缓存与弱网场景都要看。

### 常见故障场景

- “首屏白屏长”：常见是主 CSS/主脚本阻塞并且体积过大。
- “偶发初始化报错”：常见是将有依赖关系的脚本误改成 `async`。
- “优化后反而变慢”：常见是滥用 `preload` 导致关键资源竞争。
- “弱网极慢而强网正常”：常见是关键资源链路过长且无预连接。

## 常见误区

- 把 `async` 当作默认优化策略，不看依赖关系。
- 只看实验室数据，不看真实用户监控（Real User Monitoring, RUM）。
- 忽略脚本执行成本，只优化下载体积。

## 深入拆解

### script 加载策略对比

| 方案 | 下载时机 | 执行时机 | 顺序保证 | 适用场景 |
| --- | --- | --- | --- | --- |
| 普通 script | 解析到标签时 | 立即执行并阻塞解析 | 有 | 极少量必须同步初始化逻辑 |
| script defer | 并行下载 | HTML 解析后执行 | 有 | 主业务初始化脚本 |
| script async | 并行下载 | 下载完成立刻执行 | 无 | 统计、广告、独立脚本 |
| type=module | 并行下载 | 类似 defer，支持模块依赖 | 有（依赖图内） | 现代前端主脚本 |

### 关键资源优先级治理

1. 首屏关键链只保留最小集合：关键 HTML、关键 CSS、关键字体和首屏必要脚本。
2. preload 只给马上要用的资源，避免把非关键资源抬到高优先级。
3. 非关键大资源延后：懒加载图片、延迟加载低频组件脚本。
4. 结合服务端压缩（Brotli/Gzip）和缓存，降低首访成本。

### 典型优化剧本

- 现象：LCP 高且首屏白屏明显。
- 分析：瀑布图发现主 CSS 和主脚本都在关键路径且体积偏大。
- 动作：主脚本改 defer，提取关键 CSS，次要组件改动态导入。
- 验证：对比实验组的 LCP P75 与首屏错误率。

## 题库深挖补充

### 高频对比与边界（基于题库）

- script 默认阻塞解析，defer 在解析后执行，async 下载完即执行且不保证顺序。
- preload 是“当前导航即将使用”的关键资源提示，prefetch 面向未来导航。
- module 脚本默认具备延后执行特性，但仍需关注依赖图大小。
- 关键 CSS 内联要控制体积，否则会拉长首包传输。
- 面试常问“async 一定优于 defer 吗”，正确答案是看依赖关系与时序要求。
- 解析阻塞问题要与脚本执行成本一起看，不能只看下载时间。

### 线上排障剧本

1. 抽取关键渲染路径瀑布图，锁定阻塞资源。
2. 校验脚本依赖关系，避免盲目 async 化。
3. 检查关键样式和关键字体到达顺序。
4. 对低频模块启用懒加载与分包。
5. 在冷缓存与弱网场景双回归。

### 高频追问补充（附答案）

1. defer 和 async 的核心区别？

defer 有序且解析后执行，async 无序且下载完立即执行。

2. 为什么有时优化下载后仍然慢？

因为执行和渲染阶段仍可能阻塞主线程。

3. preload 什么时候会“反优化”？

提示过多会抢占带宽，挤压真正关键资源。

## 进阶深挖（第二轮）

### 阻塞模型决策矩阵

| 资源类型 | 推荐策略 | 主要风险 | 典型修复 |
| --- | --- | --- | --- |
| 主业务脚本 | defer 或 module | 初始化顺序错乱 | 明确依赖顺序，拆分入口 |
| 统计与广告脚本 | async | 抢占关键带宽 | 延后加载或降优先级 |
| 关键样式 | 首屏关键 CSS 前置 | HTML 体积膨胀 | 只提取首屏必需规则 |
| 字体 | 预连接 + 子集化 | 字体阻塞布局稳定 | 结合 font-display 策略 |

### 事故复盘样例

- 现象：版本发布后首屏白屏从 1.1 秒升到 2.8 秒。
- 根因：把有依赖关系的初始化脚本改成 async，且新增多个 preload 抢占带宽。
- 止血：临时回滚脚本加载策略，移除低收益 preload。
- 根治：建立脚本策略基线，发布前跑关键路径检查。

### 追问扩展

1. module 脚本一定更优吗？

不一定。它能改善依赖组织，但若入口依赖图过大，首屏仍可能被执行阶段拖慢。

2. 为什么同样体积的脚本，慢机器更容易卡？

脚本下载只是前半段，解析与执行成本在低性能设备上会被显著放大。

## 面试补充

### 30 秒说明

HTML 阻塞治理的核心不是“全部延迟”，而是“让关键路径可控”：关键资源先到、脚本时序正确、主线程不过载。

### 高频追问

1. `async` 一定优于 `defer` 吗？

不一定。`async` 更快但无序，`defer` 可控且按顺序执行。主业务初始化通常更适合 `defer`。

2. 为什么 CSS 会影响脚本执行体验？

脚本若读取样式或布局信息，需要等待样式计算完成，CSS 迟到会放大脚本阻塞感知。

3. 如何判断一个资源该不该 `preload`？

只给关键路径且“即将使用”的资源加 `preload`，并通过瀑布图确认收益。

4. 冷缓存和热缓存评估为什么都要做？

冷缓存暴露首次访问瓶颈，热缓存反映回访体验，两者优化动作并不相同。
