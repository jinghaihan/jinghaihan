## 概念定位

State/Props 设计边界节点关注状态归属、数据传递与更新影响面，是 React 应用可维护性的核心。

状态放置错误会导致 props drilling、重复状态和性能回退。

## 工作机制

1. 先识别数据归属：局部状态、共享状态、远端状态分开处理。
2. 组件输入通过 props 显式声明，避免隐式依赖。
3. 状态提升只在确有共享需求时进行，避免无效上收。
4. 派生状态尽量由计算得到，减少冗余存储。
5. 复杂表单和交互场景明确受控/非受控策略。

## 关键知识点

### 必须掌握

- props 是只读输入，state 是组件内部可变数据。
- 重复存储同一事实会引发一致性问题。
- 局部状态优先可降低更新扇出。
- 状态提升后要关注渲染范围扩大。
- 异步更新场景要考虑闭包旧值问题。
- 状态初始化和重置策略要明确。

### 选型与取舍

- 状态上收一致性高，但性能与复杂度压力更大。
- 状态下放性能好，但跨组件协同变难。
- 受控组件可控性强，但代码量更大。

## 对页面性能与稳定性的影响

- 节点稳定时，React 工作流 的链路可预测性会明显提升，发布和排障成本更可控。
- 节点失稳时，问题会向上下游扩散，常表现为性能回退、交互异常或交付风险叠加。
- 该节点优化应和上下游协同验证，避免局部优化把成本转移给后续节点。

## 观测与排障

### 核心指标

- props drilling 深度
- 状态更新扇出范围
- 重复状态缺陷数
- 关键组件重渲染次数

### 排障顺序

1. 先定位数据是否多处维护。
2. 检查状态是否放在最小必要作用域。
3. 检查 props 链路是否过深。
4. 检查受控组件更新节奏是否合理。
5. 按边界重构后回归性能。

### 常见故障场景

- 父组件状态过重导致整页频繁刷新。
- 多个子组件维护同一状态副本。
- 表单输入延迟高。
- props 传递层级过深难维护。

## 常见误区

- 一上来就全局状态化。
- 把派生值也存入 state。
- 忽略状态更新带来的渲染连锁。

## 题库深挖补充

### 高频对比与边界（基于题库）

- React 核心是状态驱动渲染，不是命令式改 DOM。
- JSX 是语法糖，组件抽象质量取决于职责边界而非语法本身。
- state 与 props 分工不清会导致重复状态和渲染扩散。
- 测试与发布要围绕关键路径，而不是只追覆盖率数字。
- 团队治理要把 Hook 规则、副作用边界写进可执行规范。
- 面试常问“为什么 React 不断引入新范式”，核心是抽象与调度能力演进。

### 线上排障剧本

1. 先确认问题是状态边界错误还是组件抽象错误。
2. 检查是否存在重复状态来源。
3. 检查组件输入输出契约是否稳定。
4. 回归关键页面交互和异常路径。
5. 把结论沉淀到评审规范与测试用例。

### 高频追问补充（附答案）

1. 为什么单向数据流能提升可维护性？

更新路径可追踪，状态源更明确，排障成本更低。

2. 组件抽象越细越好吗？

不一定，过细会增加组合复杂度与通信成本。

3. React 项目治理最容易忽略什么？

副作用边界和状态生命周期规则。

## 进阶追问（补充）

1. React 优化中最容易造成反效果的动作是什么？

盲目加 memo/useCallback，增加复杂度却没有命中真正热点。

2. 为什么 effect 问题总在复杂页面里更明显？

组件层级深、状态来源多、异步并发高，会放大依赖与清理缺陷。

3. 如何判断状态管理选型是否需要升级？

当更新粒度无法收敛、排障不可追踪、跨页面一致性反复出问题时。

## 案例追问加深（第三轮）

### 真实故障片段

- 故障表现：交互偶发卡顿，且问题难以稳定复现。
- 排查结论：Hook 依赖、状态订阅粒度和组件边界共同导致重渲染扩散。
- 治理动作：先收敛状态边界，再治理 effect 依赖，最后做热点组件优化。

### 追问补充

1. 为什么 React 性能问题常和状态设计绑定？

状态边界决定更新扇出，扇出范围决定渲染成本上限。

2. 如何避免“优化 API 堆叠但收益很小”？

先定位热点和根因，再选择最小有效动作，而不是先上 memo 类技巧。

## 面试补充

### 30 秒说明

State/Props 设计本质是边界设计。边界清晰，更新范围和调试复杂度才可控。

### 高频追问

1. 什么时候该做状态提升？

当多个兄弟组件共享并共同驱动 UI 时再提升。

2. 为什么不建议存派生状态？

会造成冗余和一致性问题，优先通过计算得到。

3. 受控组件和非受控组件怎么选？

复杂校验和联动场景选受控，简单输入可考虑非受控。

4. props drilling 严重时怎么办？

引入上下文或状态管理，减少层层透传。
