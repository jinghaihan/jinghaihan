## 概念定义

- 本质定义：该节点聚焦 单向数据流可追踪。
- 解决问题：把高频面试问题转成可复述、可落地的机制说明。
- 边界说明：不替代其他节点的专属机制域。

## 核心机制拆解

### 1) 机制/流程

1. 明确输入状态和触发条件。
2. 进入框架调度或编译阶段。
3. 执行更新并产出结果。
4. 用回归验证结果是否稳定。

### 2) 关键条件与触发点

- 条件 A：组件边界和数据来源必须清晰。
- 条件 B：触发时机与副作用清理要成对出现。

### 3) 结果与影响

- 对性能/稳定性/可维护性的影响：机制清晰时可预测，机制混乱时会出现渲染回退、状态错乱或协作成本上升。

## 代码

```tsx
import { useState } from 'react'

export default function Counter() {
  const [count, setCount] = useState(0)

  function addTwice() {
    setCount(prev => prev + 1)
    setCount(prev => prev + 1)
  }

  return <button onClick={addTwice}>{count}</button>
}
```

### 示例说明

- 这段代码解决什么问题：演示 `setState` 批处理时如何避免读取旧值。
- 关键语句或参数含义：函数式更新 `prev => prev + 1` 能保证连续更新正确叠加。
- 常见错误写法：连续写 `setCount(count + 1)`，结果只加一次。

## 真实场景示例

- 场景描述：核心页面迭代中出现跨组件状态与渲染时序问题。
- 问题表现：功能可用但偶发抖动、错渲染或发布回退。
- 处理方案：先收敛边界，再做最小修复，最后补自动化验证。
- 验证方式：通过关键路径回归、性能指标和线上监控确认生效。

## 相关面试题

1. React 高阶组件、Render props、hooks 有什么区别，为什么要不断迭代
- 标准答案：结论是三者都是复用手段：高阶组件基于组件包装，Render Props 基于函数子组件，Hook 基于函数组合。机制上 Hook 在可读性和嵌套深度上通常更优。场景上现代 React 优先 Hook，历史代码常保留前两者。
- 考察点：是否能从定义、触发时机、性能影响和适用场景四个维度做对比。

2. React 中什么是受控组件和非控组件？
- 标准答案：结论是受控组件由 React state 驱动值，非受控组件由 DOM 自身维护值。机制上受控组件每次输入都会触发状态更新。场景上复杂校验与联动选受控，简单一次性表单可用非受控。
- 考察点：是否能给出明确结论，并补充机制和业务场景。

3. React setState 调用的原理
- 标准答案：结论是 `setState` 本质是向更新队列追加更新并触发调度。机制上 React 会按优先级批处理更新，再在提交阶段更新 DOM。场景上连续调用应使用函数式写法避免读取旧值。

4. React setState 调用之后发生了什么？是同步还是异步？
- 标准答案：结论是它不是“立即改 DOM”，而是先入队再调度。机制上是否同步取决于批处理上下文与 React 版本并发策略。场景上需要拿到更新后结果时应使用回调或 effect。
- 考察点：是否能给出明确结论，并补充机制和业务场景。

5. React 中的 setState 批量更新的过程是什么？
- 标准答案：结论是同一批次中的多个更新会被合并计算。机制上 React 先收集更新对象，再按队列顺序计算新 state，最后一次性提交。场景上这可减少重复渲染，但也要求避免直接依赖旧 state。

7. React 中 setState 的第二个参数作用是什么？
- 标准答案：结论是第二个参数回调在更新提交后执行。机制上它保证读取到的是更新后的状态与 DOM。场景上类组件里需要串联依赖更新结果的逻辑时会用到。
- 考察点：是否能说明结论背后的原因、成立条件和反例。

8. React 中的 setState 和 replaceState 的区别是什么？
- 标准答案：结论是 `setState` 合并更新，`replaceState`（历史 API）整体替换状态。机制上合并更新更安全，整体替换容易丢字段。场景上现代 React 基本不再使用 `replaceState`。
- 考察点：是否能从定义、触发时机、性能影响和适用场景四个维度做对比。

9. 在 React 中组件的 this.state 和 setState 有什么区别？
- 标准答案：结论是 `this.state` 是当前快照，`setState` 是触发更新的唯一官方入口。机制上直接改 `this.state` 不会触发重渲染。场景上任何需要驱动 UI 的状态变更都应走 `setState`。
- 考察点：是否能从定义、触发时机、性能影响和适用场景四个维度做对比。

10. state 是怎么注入到组件的，从 reducer 到组件经历了什么样的过程
- 标准答案：结论是从 reducer 到组件一般经过“store 更新 -> 订阅通知 -> 选择器映射 -> 组件接收新 props/state”。机制上 connect 或 Hook 负责订阅与比较。场景上排查状态不同步时要从这条链路逐段定位。
- 考察点：是否能按阶段讲清核心流程、关键数据结构和边界条件。

11. React 组件的 state 和 props 有什么区别？
- 标准答案：结论是 `props` 由父组件传入且只读，`state` 由组件内部维护可变。机制上二者都可触发渲染，但职责不同。场景上可复用输入用 props，本地交互过程状态用 state。
- 考察点：是否能从定义、触发时机、性能影响和适用场景四个维度做对比。

12. React 中的 props 为什么是只读的？
- 标准答案：结论是为了保持单向数据流和更新可追踪。机制上若子组件可直接改 props，会破坏父子状态一致性。场景上子组件需要变更时应通过回调通知父组件更新源状态。

13. 在 React 中组件的 props 改变时更新组件的有哪些方法？
- 标准答案：结论是常见处理点有 `componentDidUpdate`（类组件）和 `useEffect`（函数组件）。机制上应比较前后值后再执行副作用，避免循环更新。场景上联动请求、重置表单、缓存失效都在此处理。
