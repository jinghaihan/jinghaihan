## 概念定义

- 本质定义：TCP（Transmission Control Protocol）/TLS（Transport Layer Security）建连是请求发送前的连接建立和安全协商阶段。
- 解决问题：TCP 保证有序可靠传输，TLS 提供加密、身份认证和完整性校验。
- 边界说明：它不定义 HTTP（HyperText Transfer Protocol）方法和业务状态码语义。

## 核心机制拆解

### 1) 机制/流程

1. 客户端与服务端进行 TCP 三次握手建立连接。
2. 若是 HTTPS（HyperText Transfer Protocol Secure），请求会进入 TLS 握手协商加密套件与密钥。
3. 客户端校验证书链、域名和有效期。
4. 握手成功后，应用层开始发送 HTTP 报文。
5. 连接结束时通过四次挥手关闭或复用长连接。

### 2) 关键条件与触发点

- 条件 A：证书是否可信且与域名匹配，决定 TLS 是否可继续。
- 条件 B：网络质量和 RTT（Round-Trip Time），决定握手耗时和重传概率。

### 3) 结果与影响

- 对性能 / 稳定性 / 可维护性的影响：建连慢会直接抬高首字节时间；证书配置错误会导致连接失败；长连接复用可减少重复握手成本。

## 相似概念对比

| 维度 | TCP | TLS | 结论 |
|---|---|---|---|
| 定义 | 传输层可靠传输协议 | 在传输之上提供安全能力 | HTTPS 需要两者配合 |
| 触发时机 | 发起网络连接时 | TCP 建立后、发送 HTTP 前 | TLS 成功后才会发应用层数据 |
| 性能影响 | 握手和重传影响时延 | 握手与加解密有额外成本 | 复用连接和 TLS 1.3 可显著降成本 |
| 适用场景 | 所有 HTTP / HTTPS 请求 | 涉及安全传输的 HTTPS 请求 | 生产环境默认优先 HTTPS |

## 代码

```bash
openssl s_client -connect www.example.com:443 -servername www.example.com
```

```nginx
server {
  listen 443 ssl http2;
  server_name www.example.com;
  ssl_certificate /etc/nginx/certs/fullchain.pem;
  ssl_certificate_key /etc/nginx/certs/privkey.pem;
}
```

### 示例说明

- 这段代码 / 配置解决什么问题：`openssl s_client` 可验证证书链和握手细节；`nginx` 配置用于启用 TLS。
- 关键参数含义：`-servername` 触发 `SNI（Server Name Indication）`，用于多域名证书匹配。
- 常见错误配置：证书链不完整、证书域名不匹配、过期证书未轮换。

## 真实场景示例

- 场景描述：某次发布后，移动端 HTTPS 访问失败率飙升。
- 问题表现：接口日志无请求命中，但客户端报证书错误。
- 处理方案：排查证书链和 `SNI` 配置，补齐中间证书并回滚错误证书文件。
- 验证方式：全地区探测握手成功率、`TTFB（Time To First Byte）` 和错误码恢复。

## 常见误区

- 误区：使用 HTTPS 后就不会被中间人攻击。
- 正解：若证书校验被绕过或终端被劫持，仍有风险，必须启用严格证书校验和安全策略。
- 误区：TLS 只影响安全，不影响性能。
- 正解：握手次数和协议版本会显著影响首访延迟，连接复用和 TLS 1.3 是常见优化点。

## 关联知识点

## 相关面试题

1. HTTPS 通信（握手）过程
- 标准答案：先 TCP 建连，再由 TLS 协商版本和密钥、校验证书，握手成功后才开始加密传输 HTTP 数据。
- 考察点：是否理解“先连接后加密”的顺序和关键校验点。
- 追问 1：为什么握手失败时服务端可能没有业务日志？
- 标准答案 1：失败发生在应用层报文到达前，连接已在传输或安全层被中断，业务层自然无日志。
- 追问 2：TLS 1.3 的性能价值是什么？
- 标准答案 2：它减少握手往返次数，在高 RTT 场景能更快进入可传输状态。

2. TLS/SSL 的工作原理
- 标准答案：TLS 通过非对称机制完成身份验证和密钥交换，再用对称加密承载数据传输，兼顾安全性和传输效率。
- 考察点：是否能解释混合加密的动机与过程。

3. 描述一下 TCP 的三次握手过程？
- 标准答案：三次握手顺序是客户端发 `SYN`、服务端回 `SYN+ACK`、客户端再回 `ACK`，双方据此确认收发能力和初始序列号，连接才进入可传输状态。少于三次无法同时确认双方状态，多于三次会增加不必要往返。
- 考察点：是否能说清每次握手的目的，而不是只背顺序。

4. 描述一下 TCP 的四次挥手过程？
- 标准答案：四次挥手是因为 TCP 全双工需要双向独立关闭：主动方 `FIN`，被动方先 `ACK`，待发送完剩余数据后再发 `FIN`，最后主动方回 `ACK` 并进入 `TIME_WAIT`。核心条件是“确认收到关闭请求”和“确认数据已发完”不是同一个时刻。
- 考察点：是否理解四次挥手比三次握手多一步的根因。

5. 说说 TCP 和 UDP 的区别？
- 标准答案：TCP 面向连接、可靠有序、带流控与拥塞控制，适合可靠性优先场景；UDP 无连接、开销小、时延低但不保证到达顺序，适合实时性优先场景。选型要看业务容错能力，而不是只看谁“更快”。
- 考察点：是否能按连接性、可靠性、时延和场景四个维度对比。

6. HTTP 和 HTTPS 有什么区别？
- 标准答案：HTTP 明文传输，HTTPS 在 HTTP 之上增加 TLS 加密、身份认证和完整性校验，默认端口从 `80` 变为 `443`。HTTPS 首访会多握手成本，但通过连接复用和 TLS 版本升级可以降低开销，换来更高安全性。
- 考察点：是否能同时回答安全收益和性能成本。

7. 为什么 TCP 连接需要三次握手，而不是两次或四次？
- 标准答案：三次握手的关键价值是同时确认双方的发送与接收能力，并完成初始序列号同步。两次握手无法充分确认客户端接收能力，可能引入失效连接请求导致的资源浪费；四次握手在建连阶段没有新增有效信息，只会增加往返成本。
- 考察点：是否能说明“三次”背后的必要性，而不只是记结论。

8. 为什么 TCP 挥手需要四次？
- 标准答案：因为 TCP 是全双工连接，关闭时两个方向要分别终止。被动关闭方收到 `FIN` 后只能先回 `ACK`，等剩余数据发送完再发自己的 `FIN`，因此常见为四次。若把确认和结束强行合并，容易破坏“未发送完数据仍可继续发送”的语义。
- 考察点：是否理解“连接关闭分方向”这一根因。

9. 什么是 TIME_WAIT 状态？为什么要有这个状态？
- 标准答案：`TIME_WAIT` 是主动关闭方在最后一次 `ACK` 后保留一段时间的状态，用于两件事：确保对端重传 `FIN` 时还能重发 `ACK`，以及等待旧连接报文过期，避免影响后续同四元组新连接。它是可靠关闭与连接隔离的必要代价，不是纯粹的“无用等待”。
- 考察点：是否能把 `TIME_WAIT` 和可靠关闭、报文隔离关联起来。

10. 什么是 OSI 七层模型和 TCP/IP 四层模型？
- 标准答案：OSI 七层是教学模型，强调分层职责；TCP/IP 四层是工程事实标准，应用层、传输层、网际层、网络接口层覆盖了互联网协议栈。面试回答重点是把常见协议映射到层级，例如 HTTP 在应用层、TCP/UDP 在传输层、IP 在网际层。
- 考察点：是否能区分“理论分层”与“工程分层”，并给出协议映射。

11. ARP 协议是做什么的？
- 标准答案：ARP（Address Resolution Protocol）在局域网中把 IP 地址解析为 MAC 地址，让链路层能把数据帧发到正确网卡。它通常在同网段通信或网关转发前触发，结果会被本地 ARP 缓存复用；缓存异常或伪造会直接影响连通性和安全性。
- 考察点：是否理解 ARP 的定位是“IP 到 MAC 的本地解析”，而不是域名解析。
