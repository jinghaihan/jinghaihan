## 概念定位

TCP/TLS 建连节点回答的是：在 HTTP 请求真正发出前，浏览器要先付出多少连接成本。

它同时决定三件事：

- 传输可靠性（TCP）
- 安全性与身份验证（TLS）
- 后续协议能力（通过 ALPN 协商 `http/1.1`、`h2`、`h3`）

## 工作机制

1. **DNS 结束后**，浏览器拿到目标 IP，开始建连。
2. **TCP 三次握手**建立可靠连接。
3. **TLS 握手**完成证书校验、密钥协商与加密上下文建立。
4. **ALPN（Application-Layer Protocol Negotiation）** 协商应用层协议版本。
5. 握手成功后，才进入 HTTP 请求发送阶段。

## 关键知识点

### TCP 三次握手成本

- 连接建立至少需要一个 RTT（Round-Trip Time）级别的往返。
- 弱网和高延迟区域会放大这个成本。
- 若复用失败，多个关键资源会重复支付建连成本。

### TLS 握手关键步骤

- 客户端发起 `ClientHello`（能力、随机数、SNI）。
- 服务端返回证书链、协商参数。
- 客户端验证证书合法性并协商密钥。
- 双方进入加密通信。

### TLS 1.2 与 TLS 1.3 差异（性能视角）

- TLS 1.3 握手流程更精简，通常减少建连往返。
- 会话恢复（Session Resumption）可进一步降低重复握手成本。
- 0-RTT（Zero Round Trip Time）可提升恢复连接速度，但要注意重放风险与接口幂等性。

### SNI 与证书链

- SNI（Server Name Indication）用于多域共享 IP 的证书选择。
- 证书链不完整会导致握手失败或回退重试。
- OCSP Stapling 等机制可减少证书状态检查额外成本。

### 连接复用与预热

- `preconnect` 可前置 DNS/TCP/TLS 成本。
- 连接池复用对关键域尤为重要。
- 过多跨域关键资源会稀释复用收益。

## 对页面性能的影响

- 冷启动访问中，TCP/TLS 通常是首包前最显著成本之一。
- 若握手失败重试，会直接拉高 TTFB（Time To First Byte）。
- 多域名首屏 + 弱网会形成“多条连接链叠加延迟”。
- 连接复用不稳定会造成同站点“时快时慢”。

## 观测与排障

### 浏览器侧

- Network 查看 `DNS`、`Initial connection`、`SSL` 分段耗时。
- 对比首请求和后续请求是否仍重复出现完整握手。
- 分设备、地域、网络类型看分位数差异。

### 服务端与边缘侧

- 监控 TLS 握手失败率与错误类型。
- 检查证书链、过期时间、域名覆盖范围。
- 观察会话恢复命中率与握手重试率。

### 排障剧本

1. 若 `SSL` 阶段异常高，先查证书链与 TLS 版本协商。
2. 若同域请求反复冷连，查连接复用、代理层拆连、Keep-Alive 配置。
3. 若跨域关键资源多，评估域收敛与 `preconnect` 覆盖。
4. 若弱网尾延迟高，评估协议升级（h2/h3）和边缘调度策略。

## 实战优化策略

### 连接层优化

- 关键第三方域加 `preconnect`，低价值域避免盲目预连接。
- 收敛首屏关键域，降低并行握手数量。
- 保持连接复用与空闲回收策略平衡，避免抖动。

### TLS 层优化

- 优先 TLS 1.3，开启稳定会话恢复。
- 保证证书链完整、算法兼容、自动续期可靠。
- 关键域做握手失败监控和自动告警。

### 架构层优化

- CDN 就近接入，减少跨地域往返。
- 灰度评估 `h2/h3` 真实收益，按分位数决策。
- 把连接成本纳入发布门禁而非事后分析。

## 常见误区

- “HTTPS 只影响安全，不影响性能”：错误，握手成本真实存在。
- “上了 HTTP/2/3 就没有连接问题”：错误，首次连接仍有成本。
- “只要带宽高就不会慢”：错误，RTT 和握手与带宽是不同维度。
- “`preconnect` 越多越好”：错误，过度预连接会抢占资源和连接池。

## 面试补充

### 30 秒说明

TCP/TLS 建连是 HTTP 请求前的固定成本。TCP 提供可靠传输，TLS 提供加密和身份校验，ALPN 决定后续协议能力。优化关键是减少不必要建连、提升连接复用和会话恢复命中，并对握手失败率做持续监控。

### 高频追问

1. 为什么 `preconnect` 常能提升首屏？

因为把 DNS/TCP/TLS 成本提前到空闲窗口，关键请求发起时可直接发送。

2. 如何判断慢在建连还是慢在服务端处理？

若 `Initial connection`/`SSL` 明显高，偏连接问题；若这些正常而 TTFB 高，更多是服务端处理或回源慢。

3. TLS 1.3 的性能价值是什么？

握手流程更短，会话恢复更高效，弱网和移动网络下收益更明显。

4. 0-RTT 什么时候不建议开启？

对非幂等请求或重放风险敏感接口，需要谨慎使用或禁用。

5. 为什么同站点会出现“第一次慢、后面快”？

第一次走冷连，后续复用连接和会话恢复，建连成本被摊薄。

6. 跨域资源为什么会放大建连成本？

每个关键域都可能独立经历 DNS/TCP/TLS，首屏并行域越多，总成本越高。


## 进阶补充（深挖）

### 与上下游节点的因果关系

- 上游输入（常见）：浏览器 DNS 缓存、系统 DNS 缓存、Hosts 文件匹配。
- 本节点职责：将“TCP/TLS 建连”的结果转化为可执行的后续分支决策。
- 下游影响（常见）：发送 HTTP 请求。

### 线上故障案例

- 现象：某地区 SSL 建连耗时突增，TTFB 尾延迟明显抬升。
- 根因：证书链配置异常或会话恢复命中下降，导致重复完整握手。
- 处置：修复证书链、确认 TLS1.3 与会话恢复状态，关键域增加预连接策略。
- 回归标准：关键指标恢复到基线区间，且连续观测窗口无二次抖动。

### 高频追问）

1. 会话恢复命中率下降会带来什么影响？

后续请求重复支付握手成本，弱网下尾延迟放大明显。

2. preconnect 一定越多越好吗？

不是，过量 preconnect 会占用连接池并影响关键请求。

3. 如何分层确认是建连问题？

先看 DNS/Initial connection/SSL 分段，再看服务端处理耗时。
