## 概念定义

- 本质定义：泛型是在保持类型安全的前提下，让函数、接口和类可以适配多种类型参数的机制。
- 解决问题：避免为了复用而退回 `any`，也避免为不同类型重复写几乎一样的代码。
- 边界说明：本节点聚焦泛型和约束本身，不展开条件类型和工具类型实现细节。

## 核心机制拆解

### 1) 机制/流程

1. 用类型参数为函数、接口或类预留类型占位符。
2. 在调用时显式传入类型，或让编译器根据实参自动推断。
3. 当类型参数需要满足特定能力时，用 `extends` 加泛型约束。
4. 通过约束让“可复用”和“可校验”同时成立。

### 2) 关键条件与触发点

- 条件 A：只图省事用 `any` 时，复用是有了，类型安全没了。
- 条件 B：泛型过多、命名含糊时，可读性会急剧下降。
- 条件 C：没有约束的泛型，在访问属性或方法时很容易走到断言和逃逸。

### 3) 结果与影响

- 对性能/稳定性/可维护性的影响：泛型是大型组件库和工具库的核心复用手段；但滥用泛型也会让类型定义比业务还难读。

## 相似概念对比

| 维度 | `any` 复用 | 泛型复用 | 结论 |
|---|---|---|---|
| 类型信息 | 基本丢失 | 保留并可推断 | 泛型更安全 |
| 调用体验 | 提示弱 | 推断更准 | 公共 API 优势明显 |
| 约束能力 | 几乎没有 | 可配 `extends` 限制 | 可复用也可收窄 |
| 适用场景 | 仅迁移兜底 | 公共函数、组件库、数据容器 | 默认优先泛型 |

## 代码

```ts
function identity<T>(value: T): T {
  return value
}

function getLength<T extends { length: number }>(value: T): number {
  return value.length
}

const result = identity({ id: 1, name: 'Ada' })
const size = getLength(['a', 'b', 'c'])
```

### 示例说明

- 这段代码解决什么问题：同一个函数能适配不同类型，同时在需要时给类型参数加能力约束。
- 关键语句或参数含义：`<T>` 是类型参数；`T extends { length: number }` 表示只有带 `length` 的类型才能传入。
- 常见错误写法：泛型里大量使用单字母但没有语义，或者没有约束却直接访问属性。

## 真实场景示例

- 场景描述：组件库里的表格、表单和请求封装都要适配不同业务数据结构。
- 问题表现：要么到处复制模板代码，要么直接退回 `any`，调用方拿不到准确提示。
- 处理方案：为公共函数和数据容器引入泛型，在需要访问特定字段时加约束。
- 验证方式：看公共 API 的类型推断是否准确，调用点是否仍要手写大量断言。

## 常见误区

- 误区：泛型就是把 `any` 换个写法。
- 正解：泛型的关键价值是保留类型信息并支持推断。
- 误区：泛型参数越多越高级。
- 正解：能少就少，类型参数一多，理解成本会急剧上升。
- 误区：有泛型就不用约束。
- 正解：需要访问某些属性或方法时，约束是必须的。

- 依赖：基础类型与类型推断、对象建模与接口。
- 影响：工具函数、组件库、请求封装和数据容器设计。
- 对比：`any` 和泛型。
- 组合：类型守卫、工具类型、条件类型。

## 相关面试题

1. 什么是 TypeScript 中的泛型（Generics）？它有什么作用？
- 标准答案：泛型就是“把类型当参数传进去”的机制，让函数、接口和类能适配多种类型，同时保留类型信息。机制上它通过类型参数和推断工作，不需要把类型退回 `any`。场景上公共工具函数、组件库、请求返回值封装和数据容器最常见。
- 考察点：是否理解泛型解决的是“复用和类型安全同时成立”的问题。
