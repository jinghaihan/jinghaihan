## 概念定义

- 本质定义：Node.js 模块系统负责解析 `require()` 或 `import` 的路径、执行模块代码并缓存结果，让代码能按文件组织和复用。
- 解决问题：解释“为什么 `require()` 一次后模块会被缓存”，也解释“为什么改了导出方式会影响整个服务的加载行为”。
- 边界说明：本节点聚焦 Node.js 模块解析、执行和缓存，不展开打包器的模块图实现。

## 核心机制拆解

### 1) 机制/流程

1. 代码执行到 `require()` 时，Node.js 先解析模块标识符，判断是核心模块、相对路径还是第三方包。
2. 运行时定位文件后，检查模块缓存；命中则直接返回缓存导出。
3. 未命中时，Node.js 用函数包装器执行模块代码，生成 `module.exports`。
4. 模块执行完成后，导出对象被放入缓存，后续同一路径再次加载时复用同一份实例。

### 2) 关键条件与触发点

- 条件 A：同一进程内同一路径模块默认只执行一次，这是单例配置和连接池常见实现前提。
- 条件 B：循环依赖时，模块可能拿到“未完全初始化”的导出对象。
- 条件 C：`CommonJS` 和 `ESM（ECMAScript Modules）` 的加载时机和导出语义不同，混用时要明确边界。

### 3) 结果与影响

- 对性能/稳定性/可维护性的影响：模块缓存减少重复执行成本，但也会放大共享状态副作用；对缓存和导出边界不清楚，很容易引入隐藏状态污染。

## 相似概念对比

| 维度 | CommonJS | ESM | 结论 |
|---|---|---|---|
| 导出方式 | `module.exports` / `exports` | `export` / `export default` | 语法不同 |
| 导入方式 | `require()` | `import` | 调用方式不同 |
| 加载特征 | 运行时按需加载 | 静态分析更友好 | 构建与执行体验不同 |
| 常见场景 | 传统 Node.js 服务 | 新项目、前后端同构 | 需结合工程约束选型 |

## 代码

```js
// counter.js
let count = 0

function increment() {
  count += 1
  return count
}

module.exports = { increment }

// app.js
const counterA = require('./counter')
const counterB = require('./counter')

console.log(counterA.increment()) // 1
console.log(counterB.increment()) // 2
```

### 示例说明

- 这段代码解决什么问题：展示同一路径模块被缓存后，导出状态会在多次 `require()` 之间共享。
- 关键语句或参数含义：`module.exports` 定义导出对象；两次 `require()` 命中同一个缓存实例。
- 常见错误写法：把可变全局状态直接挂到模块导出上，却没有考虑并发请求之间的共享副作用。

## 真实场景示例

- 场景描述：服务启动时初始化数据库连接池和配置中心客户端，业务代码在多个文件里都会读取它们。
- 问题表现：如果不了解模块缓存，有人会在每个文件里重复初始化连接，导致资源浪费；如果共享状态设计不当，又会出现脏数据串用。
- 处理方案：把连接池和只读配置抽成独立模块，依赖缓存复用；有请求级状态时，改为函数工厂或上下文注入。
- 验证方式：观察进程启动日志里连接初始化次数、连接池数量和请求间状态隔离情况。

## 常见误区

- 误区：`require()` 每次都会重新执行文件。
- 正解：同一进程同一路径默认会走模块缓存，只在首次加载时执行。
- 误区：模块缓存一定是优点。
- 正解：缓存能提升复用，但共享可变状态也会带来副作用和排障难度。
- 误区：`exports` 和 `module.exports` 可以随意混用。
- 正解：`exports` 只是 `module.exports` 的引用，直接整体替换时要赋值给 `module.exports`。

## 相关面试题

1. Node.js 中的模块系统是如何工作的？
- 标准答案：Node.js 先解析模块路径，再检查缓存，未命中才定位文件并执行模块代码，执行结果通过 `module.exports` 暴露出去，最后把模块实例写入缓存。结论是模块系统同时解决了代码组织和复用问题，但缓存也意味着共享状态要谨慎设计，循环依赖也要特别小心。
- 考察点：是否理解模块解析、执行和缓存三件事是连在一起的。
