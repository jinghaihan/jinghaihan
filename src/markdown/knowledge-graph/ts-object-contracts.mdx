## 概念定义

- 本质定义：对象建模与接口关注如何用 `interface`、`type`、索引签名和只读约束把对象结构描述清楚。
- 解决问题：让“对象该长什么样、哪些字段必填、哪些字段只读、能不能扩展”在编译阶段说清楚。
- 边界说明：本节点聚焦对象契约，不展开泛型约束和条件类型等高级类型技巧。

## 核心机制拆解

### 1) 机制/流程

1. 先根据对象职责声明字段、可选字段和只读字段。
2. 需要复用和扩展时，用 `interface extends` 或 `type` 组合类型。
3. 需要描述动态键时，再引入索引签名。
4. TypeScript 会按结构化类型系统判断对象是否满足契约，而不是只看名义上的类型名。

### 2) 关键条件与触发点

- 条件 A：对象边界不清晰时，公共函数最容易出现“多传少传都没人拦”的问题。
- 条件 B：把所有东西都写成宽泛索引签名，会把精细字段约束一起抹掉。
- 条件 C：不知道结构化类型系统的判断方式时，赋值兼容性经常会“看着怪，其实合理”。

### 3) 结果与影响

- 对性能/稳定性/可维护性的影响：对象契约清楚，重构和协作都更稳；契约混乱，公共类型会迅速演变成大而空的“万能对象”。

## 相似概念对比

| 维度 | `type` | `interface` | 结论 |
|---|---|---|---|
| 表达能力 | 更适合联合、交叉、别名 | 更适合对象结构和扩展 | 两者各有侧重 |
| 扩展方式 | 交叉类型为主 | `extends` 和声明合并更自然 | 对象契约常偏 `interface` |
| 声明合并 | 不支持 | 支持 | 库类型扩展常用 `interface` |
| 适用场景 | 组合类型、工具类型结果 | 对象、类契约、公共 API | 不必教条统一 |

## 代码

```ts
interface User {
  readonly id: number
  name: string
  email?: string
}

type UserMap = {
  [userId: string]: User
}

const users: UserMap = {
  '1': { id: 1, name: 'Ada' },
}
```

### 示例说明

- 这段代码解决什么问题：同时展示对象字段约束、可选属性、只读属性和索引签名。
- 关键语句或参数含义：`readonly` 表示创建后不应被重新赋值；`?` 表示可选字段；索引签名描述动态键对象。
- 常见错误写法：公共对象全写成 `{ [key: string]: any }`，把真实结构全部丢掉。

## 真实场景示例

- 场景描述：组件库里的 `props` 类型一开始没约束，后来字段越来越多，调用方传错参数也没人知道。
- 问题表现：重构时删字段不敢删，新增字段到处搜，最终所有对象都像“黑盒字典”。
- 处理方案：为公共对象建立清晰接口，只在确实需要时补索引签名，并把关键标识字段设成只读。
- 验证方式：看组件调用报错是否前置、类型定义是否容易复用，以及重构时的类型连锁反应。

## 常见误区

- 误区：团队必须统一“只用 `type`”或“只用 `interface`”。
- 正解：两者都有价值，关键看是不是对象契约、是否需要联合类型或声明合并。
- 误区：结构化类型系统不安全，因为“长得像就能赋值”。
- 正解：它的目标本来就是按结构兼容，只要边界建模清晰，反而更灵活。
- 误区：索引签名越宽松越方便。
- 正解：索引签名过宽会把细粒度字段约束冲掉。

## 关联知识点

- 依赖：基础类型与类型推断。
- 影响：函数参数约束、泛型设计、组件 `props` 建模。
- 对比：`type` 和 `interface`、显式字段和索引签名。
- 组合：只读属性、可选属性、结构化类型系统。

## 相关面试题

1. `type` 和 `interface` 有什么区别？
- 标准答案：`type` 更像“类型别名”，适合联合类型、交叉类型和复杂类型组合；`interface` 更适合描述对象结构、类契约和可扩展接口。机制上 `interface` 支持声明合并，`type` 不支持。场景上对象 API、组件 `props` 常偏 `interface`，联合类型和工具类型结果常偏 `type`。
- 考察点：是否能从表达能力和扩展方式两方面回答。

2. 什么是 TypeScript 的结构化类型系统（Structural Typing）？
- 标准答案：结构化类型系统按“结构是否兼容”来判断能不能赋值，而不是只看类型名字。机制上如果一个对象拥有目标类型要求的字段和形状，它通常就兼容。场景上这让类型复用更灵活，但也要求你把对象边界建模得更清楚。
- 考察点：是否理解 TypeScript 不是名义类型系统。

3. 什么是索引签名（Index Signatures）？
- 标准答案：索引签名用来描述“键名事先不固定，但值类型有规则”的对象，比如字典、映射表和按 id 索引的数据集合。机制上它定义了某类键对应值的统一类型。场景上国际化词典、缓存映射很适合，但别因为省事就把所有对象都写成索引签名。
- 考察点：是否知道动态键对象该怎么约束。

4. 什么是类型别名（Type Aliases）？
- 标准答案：类型别名就是给一段类型表达式起名字。机制上它可以指向对象类型、联合类型、交叉类型、函数签名甚至工具类型结果。场景上重复出现的复杂联合或组合类型非常适合抽成别名，提高可读性。
- 考察点：是否理解它不是只能给对象取名。

5. `readonly` 修饰符有什么作用？
- 标准答案：`readonly` 用来限制属性在初始化后不能被重新赋值。机制上它作用在编译阶段，帮助你表达“这个字段创建后不应该再变”。场景上实体 id、配置对象和组件只读输入都适合用它，但要注意它不是深层不可变的运行时冻结。
- 考察点：是否知道 `readonly` 是类型约束，不是运行时冻结。
