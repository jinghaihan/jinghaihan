## 概念定义

- 本质定义：Git 协作与历史整合是围绕分支、提交和远程同步的版本治理机制。
- 解决问题：解决多人并行开发时的代码合并、冲突处理与历史可追溯问题。
- 边界说明：不解决构建工具配置、打包体积和运行时性能问题。

## 核心机制拆解

### 1) 机制/流程

1. 在本地分支完成独立开发，形成原子 commit。
2. 通过 `git fetch` 拉取远端更新并审阅差异。
3. 使用 `merge` 或 `rebase` 整合分支历史。
4. 解决冲突后执行回归验证。
5. 通过 `git push` 推送到远端并触发后续流水线。

### 2) 关键条件与触发点

- 条件 A：多人同时修改同一文件或同一逻辑块时，冲突概率显著上升。
- 条件 B：长期不与主干同步时，`rebase` 或 `merge` 的冲突成本会累积。

### 3) 结果与影响

- 对稳定性的影响：提交边界清晰且同步频繁时，冲突更早暴露，回滚更容易。
- 对可维护性的影响：线性历史有利于排障，分叉历史有利于保留协作过程。

## 相似概念对比

| 维度 | merge | rebase | 结论 |
|---|---|---|---|
| 定义 | 把两个分支历史汇合为一个新提交 | 把当前分支提交“平移”到目标分支之后 | 都可合并分支，但历史形态不同 |
| 触发时机 | 保留分支痕迹时 | 想要线性历史时 | 团队协作和审计目标决定选择 |
| 历史影响 | 保留分叉历史 | 改写当前分支历史 | 已共享分支谨慎 `rebase` |
| 适用场景 | 多人协作、强调可追溯 | 个人分支整理、提交前清理 | 推送前 `rebase` 更安全 |

## 代码

```bash
git init
git add .
git commit -m "feat: add build cache"
git fetch origin
git rebase origin/main
git push origin feature/build-cache
```

### 示例说明

- 这段命令展示了从本地提交到与主干对齐再推送的最小闭环。
- `git fetch` 只更新远端引用，不会直接改动当前分支内容。
- 常见错误写法是直接 `git pull` 后忽略自动合并结果，导致冲突被延后暴露。

## 真实场景示例

- 场景描述：两位同学同时修改构建脚本，一位先合并，一位晚一天同步。
- 问题表现：晚同步分支在 `git push` 时被拒绝，强推可能覆盖他人提交。
- 处理方案：先 `git fetch` + `git rebase origin/main`，逐个解决冲突并回归后再推送。
- 验证方式：检查远端分支提交顺序线性且功能回归通过。

## 常见误区

- 误区：`git pull` 和 `git fetch` 一样，只是写法不同。
  - 正解：`git pull` = `fetch + merge/rebase`，会直接改动当前分支状态。
- 误区：所有分支都应该使用 `rebase`。
  - 正解：公开协作分支优先 `merge`，个人未共享分支可用 `rebase` 整理历史。

## 关联知识点

- 依赖：仓库权限模型与分支保护策略。
- 影响：CI 执行触发频率、回滚定位效率。
- 对比：`fetch` / `pull`、`merge` / `rebase`。

## 相关面试题

1. git 和 svn 的区别
   - 标准答案：Git 是分布式版本控制，每个开发者本地都有完整仓库；svn 是集中式，离线能力弱且更依赖中心服务。Git 分支本质是提交指针，创建和切换成本低，适合高频分支协作；svn 分支更接近目录副本，管理成本更高。
   - 考察点：是否理解分布式协作、分支模型和容灾能力差异。

2. 经常使用的 git 命令？
   - 标准答案：高频命令至少包括初始化与暂存（`git init`、`git add`）、提交（`git commit`）、分支管理（`git branch`、`git checkout -b`）、同步（`git fetch`、`git pull`、`git push`）、状态查看（`git status`、`git log`）。其中同步命令要区分是否会改动工作分支。
   - 考察点：是否具备日常协作所需的命令分层和使用边界。

3. git pull 和 git fetch 的区别
   - 标准答案：`git fetch` 只更新远端跟踪分支，不自动合并到当前分支；`git pull` 会在拉取后直接合并或变基。多人协作中先 `fetch` 再手动整合更可控，能减少误合并风险。
   - 考察点：是否理解远端同步与本地历史改动的边界。

4. git rebase 和 git merge 的区别
   - 标准答案：`merge` 保留分叉历史并新增合并提交，适合团队审计；`rebase` 重写当前分支提交基线，历史更线性但会改写 commit 哈希。已推送到共享分支后再 `rebase` 容易造成他人同步冲突。
   - 考察点：是否能根据协作场景选择合适历史整合策略。
