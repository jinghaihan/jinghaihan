## 概念定义

- 本质定义：CDN 与边缘分发通过就近节点提供静态资源，减少用户到源站的网络距离。
- 解决问题：解决跨地域访问慢、源站压力高、流量突增易抖动的问题。
- 边界说明：不直接解决前端主线程阻塞和渲染卡顿问题。

## 核心机制拆解

### 1) 机制/流程

1. 用户访问资源域名，解析系统根据调度策略返回合适边缘节点。
2. 边缘节点判断本地是否命中资源。
3. 命中则直接返回；未命中则向上级节点或源站回源。
4. 回源成功后写入边缘缓存并返回用户。
5. 发布时通过预热与失效策略保证版本一致。

### 2) 关键条件与触发点

- 条件 A：跨地域访问、弱网访问、静态资源占比高时，边缘分发收益最明显。
- 条件 B：缓存键设计不合理或版本失效策略错误时，容易出现命中低和版本混用。

### 3) 结果与影响

- 对性能的影响：可显著降低首字节等待与下载耗时。
- 对稳定性的影响：源站压力下移后，可用性提升，但需要额外治理边缘一致性。

## 相似概念对比

| 维度 | CDN 与边缘分发 | 仅源站直出 | 结论 |
|---|---|---|---|
| 访问路径 | 用户优先访问就近边缘节点 | 用户直接访问源站 | 远距访问时边缘更稳定 |
| 源站压力 | 大量静态请求由边缘承接 | 请求集中打到源站 | 高峰期差异明显 |
| 发布一致性 | 依赖失效与预热策略 | 路径简单但扩展性差 | 边缘方案需配套治理 |
| 适用场景 | 全国或全球分发、高并发静态资源 | 小流量、单地域站点 | 按流量规模选择 |

## 配置示例

```nginx
location /assets/ {
  proxy_pass https://origin.example.com;
  add_header Cache-Control "public, max-age=31536000, immutable";
}

location = /index.html {
  proxy_pass https://origin.example.com;
  add_header Cache-Control "no-cache";
}
```

### 示例说明

- 这段配置展示了“静态资源长缓存 + 入口短缓存”的典型边缘策略。
- `index.html` 走短缓存可以避免页面长期引用旧资源。
- 常见错误写法是入口和静态资源都给长缓存，发布后出现新旧版本混用。

## 真实场景示例

- 场景描述：大促前首页静态资源访问量暴涨，源站带宽接近上限。
- 问题表现：部分地区首屏加载时间明显上升，源站响应抖动。
- 处理方案：将静态资源完全切到边缘分发，发布前做热点资源预热。
- 验证方式：观察边缘命中率、回源率和首字节分位数是否回落。

## 常见误区

- 误区：只要接入 CDN，性能就一定好。
- 正解：还要看缓存规则、回源链路和节点健康，配置错误会放大问题。
- 误区：命中率越高越好。
- 正解：还要同时看版本一致性，命中旧版本同样会造成故障。

## 关联知识点

- 依赖：缓存策略优化。
- 影响：资源预提示优化、图片优化策略。
- 组合：发布回归与性能门禁。

## 相关面试题

1. CDN 的概念
- 标准答案：CDN 是把内容分发到靠近用户的边缘节点，用户优先从边缘获取资源，未命中再回源。核心价值是降低访问时延并分担源站压力，特别适合跨地域高并发的静态资源场景。
- 考察点：是否理解边缘分发、缓存命中和回源三者关系。
- 追问 1：为什么接入后仍可能变慢？
- 标准答案 1：常见原因是缓存键碎片化、边缘节点异常或回源链路抖动，说明问题在策略和链路，而不在“是否接入”本身。

2. CDN 的使用场景
- 标准答案：高频场景包括静态资源分发、下载分发、直播流媒体分发和突发流量承接。共同条件是“资源可边缘复用 + 访问地域分散 + 对时延敏感”。
- 考察点：是否能把场景特征与方案收益对应起来。
