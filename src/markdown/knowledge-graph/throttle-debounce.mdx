## 概念定义

- 本质定义：节流与防抖是控制高频事件执行节奏的两种函数治理策略。
- 解决问题：解决滚动、输入、拖拽等高频事件导致的主线程拥塞与重复请求问题。
- 边界说明：不解决单次函数内部本身过慢的问题。

## 核心机制拆解

### 1) 机制/流程

1. 防抖：每次触发都重置计时器，只执行最后一次。
2. 节流：在固定窗口内最多执行一次。
3. 根据事件类型选择领先执行或尾随执行。
4. 在组件卸载或页面离开时清理定时器。
5. 通过采样验证交互延迟和请求频率是否下降。

### 2) 关键条件与触发点

- 条件 A：输入联想、按钮提交等“只关心最终结果”的场景更适合防抖。
- 条件 B：滚动监听、拖拽跟随等“需要持续反馈”的场景更适合节流。

### 3) 结果与影响

- 对性能的影响：可显著降低无效回调、重绘频次和请求风暴。
- 对可维护性的影响：统一封装后可减少业务层重复实现和参数误配。

## 相似概念对比

| 维度 | 防抖 | 节流 | 结论 |
|---|---|---|---|
| 执行策略 | 最后一次触发后执行 | 固定时间窗口执行一次 | 目标不同 |
| 典型场景 | 搜索输入、按钮防连点 | 滚动、拖拽、窗口缩放 | 按交互期望选 |
| 实时反馈 | 弱 | 强 | 节流更适合连续反馈 |
| 请求控制 | 抑制重复提交效果好 | 控制频率效果好 | 可组合使用 |

## 真实场景示例

- 场景描述：搜索框每输入一个字符都触发接口请求。
- 问题表现：请求量突增，返回乱序导致结果闪烁。
- 处理方案：输入事件改防抖，请求层增加取消前一请求逻辑；滚动埋点改节流。
- 验证方式：统计单位时间请求数、输入延迟和接口错误率变化。

## 常见误区

- 误区：节流和防抖可以互换。
  - 正解：一个控频率，一个控最终触发，交互语义不同。
- 误区：加了节流就不会卡。
  - 正解：如果单次处理逻辑太重，仍然会卡，需要同时优化函数体。

## 关联知识点

- 依赖：回流、重绘与渲染优化。
- 影响：发布回归与性能门禁中的交互指标。
- 组合：懒加载与按需加载。

## 相关面试题

1. 对节流与防抖的理解
   - 标准答案：防抖是在连续触发结束后执行一次，适合只关心最终结果的场景；节流是在固定窗口内最多执行一次，适合需要持续反馈的场景。两者本质是对高频事件的执行节奏治理。
   - 考察点：是否能根据场景做正确策略选择。

2. 实现节流函数和防抖函数
   - 标准答案：防抖实现关键是“重置定时器”，节流实现关键是“记录上次执行时间或定时窗口”。工程里还需要补充 `this` 绑定、参数透传和清理能力，避免内存泄漏。
   - 考察点：是否能写出可复用而非一次性实现。
```ts
function debounce<T extends (...args: any[]) => unknown>(fn: T, wait = 300) {
  let timer: ReturnType<typeof setTimeout> | null = null
  return function(this: unknown, ...args: Parameters<T>) {
    if (timer) clearTimeout(timer)
    timer = setTimeout(() => {
      fn.apply(this, args)
      timer = null
    }, wait)
  }
}

function throttle<T extends (...args: any[]) => unknown>(fn: T, wait = 300) {
  let last = 0
  let trailing: ReturnType<typeof setTimeout> | null = null

  return function(this: unknown, ...args: Parameters<T>) {
    const now = Date.now()
    const remaining = wait - (now - last)

    if (remaining <= 0) {
      if (trailing) {
        clearTimeout(trailing)
        trailing = null
      }
      last = now
      fn.apply(this, args)
      return
    }

    if (!trailing) {
      trailing = setTimeout(() => {
        last = Date.now()
        trailing = null
        fn.apply(this, args)
      }, remaining)
    }
  }
}
```
   - 追问 1：搜索联想为什么更常用防抖？
   - 标准答案 1：因为用户连续输入期间中间值大多无意义，防抖可把请求收敛到最后一次，减少服务压力和响应乱序。
