## 代码

```ts
// usePermission.ts
import { computed, ref } from 'vue'

const role = ref<'guest' | 'admin'>('guest')

export function usePermission() {
  const canEdit = computed(() => role.value === 'admin')
  function switchRole(next: 'guest' | 'admin') {
    role.value = next
  }
  return { role, canEdit, switchRole }
}
```

### 示例说明

- 这段代码解决什么问题：把权限规则沉淀为可复用 composable，降低页面重复逻辑。
- 关键语句或参数含义：`computed` 统一派生权限判断，避免散落条件分支。
- 常见错误写法：在多个组件复制权限判断，导致规则修改时漏改。

## 相关面试题

1. Vue 的基本原理
- 标准答案：结论是 Vue 的核心是响应式数据驱动视图。机制上通过依赖收集与触发更新，把状态变化映射到组件渲染。场景上它让业务代码聚焦状态而非手写 DOM 操作。
- 考察点：是否能按阶段讲清核心流程、关键数据结构和边界条件。

2. MVVM、MVC、MVP 的区别
- 标准答案：结论是 MVC 强调控制器，MVP 强调 Presenter，MVVM 强调 ViewModel 的双向绑定能力。机制上 Vue 更接近 MVVM，由响应式系统连接 View 与状态。场景上前端页面状态复杂时 MVVM 更易维护。
- 考察点：是否能从定义、触发时机、性能影响和适用场景四个维度做对比。

3. 简述 mixin、extends 的覆盖逻辑
- 标准答案：结论是两者都做逻辑复用，但来源和优先级不同。机制上 `mixin` 可多源合并，冲突时组件选项优先；`extends` 更像单继承扩展。场景上复杂复用优先 composable，老项目兼容才用 mixin/extends。
- 考察点：是否能给出明确结论，并补充机制和业务场景。

4. 对 React 和 Vue 的理解，它们的异同
- 标准答案：结论是两者都用组件化与声明式更新，但响应式模型不同。机制上 Vue 倾向细粒度依赖追踪，React 倾向函数重渲染与调度。场景上团队技术栈、工程生态和性能瓶颈类型决定选型。
- 考察点：是否能从定义、触发时机、性能影响和适用场景四个维度做对比。

5. 什么是 mixin ？
- 标准答案：结论是 mixin 是把可复用选项对象合并到组件。机制上数据、方法、生命周期按合并策略处理，命名冲突时组件自身优先。场景上它会带来隐式依赖，Vue 3 更推荐 composable。
- 考察点：是否能给出明确结论，并补充机制和业务场景。

6. extend 有什么作用
- 标准答案：结论是 `extend` 用于基于已有组件配置创建新构造器。机制上它复用父配置并允许局部覆写。场景上在动态挂载或旧版本代码中会见到，现代项目通常用 SFC + 组合式 API 替代。
- 考察点：是否能说明结论背后的原因、成立条件和反例。

7. MVVM 的优缺点？
- 标准答案：结论是优点是状态到视图同步、开发效率高；缺点是抽象层增加后调试链路更长。机制上响应式系统自动追踪依赖，但也会引入调度与缓存心智负担。场景上中大型交互项目收益明显，极轻量页面收益有限。
- 考察点：是否能从定义、触发时机、性能影响和适用场景四个维度做对比。

8. 对 Vue 组件化的理解
- 标准答案：结论是组件化是“按职责拆分 + 可组合复用”。机制上通过清晰输入输出契约（props/emits/slots）实现低耦合协作。场景上能降低多人协作冲突并提升测试与重构效率。
- 考察点：是否能给出明确结论，并补充机制和业务场景。

9. 对 Vue 设计原则的理解
- 标准答案：结论是要遵守单向数据流、最小暴露接口和可预测更新。机制上把状态源收敛、把副作用集中管理，避免跨层隐式修改。场景上这套原则能显著降低线上回归与重构成本。
- 考察点：是否能给出明确结论，并补充机制和业务场景。
