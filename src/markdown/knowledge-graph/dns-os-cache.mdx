## 概念定位

系统 DNS 缓存是浏览器 DNS 之后的第二层命中点，由操作系统 resolver 管理，可跨浏览器和应用复用。

## 工作机制

1. 浏览器缓存未命中后请求系统 resolver。
2. 系统 resolver 查询本地 DNS 缓存记录。
3. 命中时直接返回结果给浏览器。
4. 未命中再进入 hosts 或递归 DNS 查询流程。

## 关键知识点

- **跨应用复用**：同机多个应用共享系统缓存收益。
- **TTL 生效范围广**：系统层失效策略影响多个浏览器结果。
- **网络切换敏感**：切网或代理变化可能导致缓存与路径失配。
- **分层排障必要**：浏览器层与系统层问题常被混淆。

## 对页面性能的影响

- 命中系统层可进一步减少外部 DNS 查询。
- 对多浏览器并行场景收益明显。
- 系统层异常会造成“全浏览器同时变慢”的表象。

## 观测与排障

### 系统侧命令

```sh
# macOS
sudo dscacheutil -flushcache
# Windows
ipconfig /displaydns
ipconfig /flushdns
```

### 排障步骤

1. 先区分是单浏览器问题还是全系统问题。
2. 检查系统缓存是否过期策略异常或记录污染。
3. 切换网络环境验证是否路径相关。
4. 清理系统缓存后比对 DNS 阶段耗时。
5. 回归看关键域名解析稳定性。

## 常见误区

- “重启浏览器就能清系统 DNS”：不准确，系统缓存独立存在。
- “系统缓存命中收益不明显”：不准确，多应用场景收益很高。
- “只有浏览器 DevTools 才能排 DNS”：不准确，系统命令是必要手段。

## 题库深挖补充

### 高频对比与边界（基于题库）

- 浏览器 DNS 缓存命中后，通常不会继续向操作系统查询；操作系统缓存命中后，也不会进入递归解析。
- hosts 文件优先级高于外部 DNS 结果，常用于本地调试和应急切流。
- 负缓存（Negative Caching）会缓存“不存在域名”结果，排障时要注意缓存滞后。
- 本地缓存与公共 DNS 缓存并不是一层，清理浏览器缓存不等于清理系统解析缓存。
- 多网络环境切换（公司 VPN、移动热点）会改变本地解析路径，导致“同机不同时段结果不同”。
- 面试常问“为什么同一域名一会儿能访问一会儿不能访问”，核心就是多层缓存状态不一致。

### 线上排障剧本

1. 先确认问题是“解析失败”还是“解析到错误 IP”。
2. 在同一台机器分别检查浏览器缓存、系统缓存和 hosts 覆盖情况。
3. 对比本地解析结果和权威解析结果，判断是本地污染还是上游问题。
4. 检查 TTL 与负缓存窗口，确认是否处于缓存过渡期。
5. 若涉及紧急切流，优先用可控 DNS 策略，不建议长期依赖 hosts。

### 高频追问补充（附答案）

1. 为什么清了浏览器缓存问题还在？

因为解析可能命中系统缓存或网络侧缓存，浏览器缓存不是唯一层。

2. hosts 的风险点是什么？

它优先级很高，临时调试忘记清理会长期污染本机解析结果。

3. 什么是负缓存，为什么排障时容易忽略？

负缓存会记住“解析失败”结果，在 TTL 没过期前会持续失败，即使权威 DNS 已恢复。

### 术语速查与易错点

- A 记录：域名到 IPv4 地址映射。
- AAAA 记录：域名到 IPv6 地址映射。
- CNAME（Canonical Name）：别名记录，常用于接入 CDN。
- TTL（Time To Live）：缓存有效时长，不是“固定刷新时间”。
- NXDOMAIN：权威返回域名不存在。

常见易错点：

- 清浏览器缓存后立即下结论，忽略系统层缓存。
- 忘记 hosts 临时改动，导致排障方向偏离。
- 只在单网络环境验证解析结果。

## 深度延展（案例对照）

### 系统缓存偏差案例

- 场景：浏览器换了仍异常，但另一台机器正常。
- 诊断：操作系统 DNS 缓存保留旧记录。
- 处理：刷新系统 DNS 缓存并验证权威结果一致。

### 实操提示

- 浏览器级清理无效时，优先检查 OS 级缓存。
- 在运维手册中补充各系统清缓存步骤。

## 进阶排障清单（补充）

### 现场处置顺序

1. 先固定样本：同地域、同网络、同设备，避免样本漂移。
2. 抓三组结果：本地解析、公共 DNS 解析、权威解析。
3. 比对 TTL 与记录版本，确认是否处于传播窗口。
4. 标记是否命中本地特殊配置（hosts、代理、VPN）。
5. 形成“可复现路径 + 影响范围 + 临时止损”三件套再升级处理。

### 面试追问强化

1. 为什么 DNS 问题经常“不可稳定复现”？

因为多层缓存、地域路由和传播窗口共同影响，同一时间不同样本可能命中不同路径。

2. 如何用最少步骤判断是不是 DNS 问题？

先看不同解析器结果是否分叉，再看业务域名能否稳定映射到预期接入点。

3. 何时应该优先止损而不是继续追根因？

当影响范围在扩大且已有可控切流手段时，应先保可用性再做根因定位。

## 速查补充（第四轮）

### 快速判定问答

1. 什么时候先怀疑缓存层而不是权威记录本身？

当不同客户端结果分叉明显且权威查询稳定时，优先检查本地与递归缓存状态。

2. DNS 类问题为什么要强调样本固定？

因为网络、地域、时间窗口会影响解析路径，不固定样本容易得到互相矛盾结论。

## 面试补充

### 30 秒说明

系统 DNS 缓存是浏览器之后的复用层，能跨应用共享解析结果。排障时必须分层看浏览器缓存与系统缓存，否则容易误判问题归因。

### 高频追问

1. 为什么不同浏览器在同机上会同时变慢？

常见是系统 DNS 层出现问题，多个浏览器共享同一 resolver 路径。

2. 清缓存时为什么要分浏览器和系统两层？

两层缓存独立管理，单独清理其中一层可能仍命中另一层旧记录。

3. 系统缓存和 hosts 的关系是什么？

hosts 是静态映射优先覆盖，系统缓存是动态解析结果存储，两者作用层级不同。
