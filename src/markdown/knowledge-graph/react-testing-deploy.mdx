## 概念定位

测试、构建与发布节点用于把 React 改动稳定送达生产，核心是关键路径测试、发布门禁和回滚能力。

它直接决定上线风险和问题修复速度。

## 工作机制

1. 测试分层覆盖：单元测试 + 集成测试 + 关键路径 E2E。
2. 构建阶段校验类型、体积预算和静态规则。
3. 发布阶段采用灰度与观测联动。
4. 失败自动保留上下文，便于快速定位。
5. 复盘结果回流到测试用例和门禁规则。

## 关键知识点

### 必须掌握

- 测试重点是高价值流程，而非盲目追求覆盖率。
- 发布门禁应覆盖性能与稳定性双指标。
- 灰度放量和监控告警要打通。
- 回滚能力是发布流程的必选项。
- 测试数据和环境要可重复。
- flaky 用例要持续治理。

### 选型与取舍

- 全量测试最稳，但耗时高。
- 快测效率高，但风险覆盖不足。
- 高自动化建设成本高，但长期收益明显。

## 对页面性能与稳定性的影响

- 节点稳定时，React 工作流 的链路可预测性会明显提升，发布和排障成本更可控。
- 节点失稳时，问题会向上下游扩散，常表现为性能回退、交互异常或交付风险叠加。
- 该节点优化应和上下游协同验证，避免局部优化把成本转移给后续节点。

## 观测与排障

### 核心指标

- 回归通过率
- 发布成功率
- 线上缺陷逃逸率
- 回滚平均耗时

### 排障顺序

1. 先定位失败层级和环境差异。
2. 检查关键路径用例是否覆盖变更点。
3. 检查构建产物和配置一致性。
4. 执行回滚并验证恢复。
5. 补充回归用例并更新门禁。

### 常见故障场景

- 本地通过线上失败。
- 灰度正常全量异常。
- 回滚触发后恢复慢。
- 发布后性能明显回退。

## 常见误区

- 只看测试通过率，不看线上观测。
- 发布成功后不做观察窗口。
- 回滚预案长期不演练。

## 题库深挖补充

### 高频对比与边界（基于题库）

- React 核心是状态驱动渲染，不是命令式改 DOM。
- JSX 是语法糖，组件抽象质量取决于职责边界而非语法本身。
- state 与 props 分工不清会导致重复状态和渲染扩散。
- 测试与发布要围绕关键路径，而不是只追覆盖率数字。
- 团队治理要把 Hook 规则、副作用边界写进可执行规范。
- 面试常问“为什么 React 不断引入新范式”，核心是抽象与调度能力演进。

### 线上排障剧本

1. 先确认问题是状态边界错误还是组件抽象错误。
2. 检查是否存在重复状态来源。
3. 检查组件输入输出契约是否稳定。
4. 回归关键页面交互和异常路径。
5. 把结论沉淀到评审规范与测试用例。

### 高频追问补充（附答案）

1. 为什么单向数据流能提升可维护性？

更新路径可追踪，状态源更明确，排障成本更低。

2. 组件抽象越细越好吗？

不一定，过细会增加组合复杂度与通信成本。

3. React 项目治理最容易忽略什么？

副作用边界和状态生命周期规则。

## 进阶追问（补充）

1. React 优化中最容易造成反效果的动作是什么？

盲目加 memo/useCallback，增加复杂度却没有命中真正热点。

2. 为什么 effect 问题总在复杂页面里更明显？

组件层级深、状态来源多、异步并发高，会放大依赖与清理缺陷。

3. 如何判断状态管理选型是否需要升级？

当更新粒度无法收敛、排障不可追踪、跨页面一致性反复出问题时。

## 案例追问加深（第三轮）

### 真实故障片段

- 故障表现：交互偶发卡顿，且问题难以稳定复现。
- 排查结论：Hook 依赖、状态订阅粒度和组件边界共同导致重渲染扩散。
- 治理动作：先收敛状态边界，再治理 effect 依赖，最后做热点组件优化。

### 追问补充

1. 为什么 React 性能问题常和状态设计绑定？

状态边界决定更新扇出，扇出范围决定渲染成本上限。

2. 如何避免“优化 API 堆叠但收益很小”？

先定位热点和根因，再选择最小有效动作，而不是先上 memo 类技巧。

## 面试补充

### 30 秒说明

React 测试与发布节点的目标是把质量前置并把风险可控化，让交付速度和稳定性同时成立。

### 高频追问

1. 为什么要分层测试？

分层可在成本可控前提下覆盖不同风险面。

2. 发布后为什么还要观察？

很多问题只在真实流量下出现，需要观察窗口确认稳定。

3. 如何降低线上缺陷逃逸？

提升关键路径回归覆盖，并把故障复盘映射回测试。

4. 回滚能力应如何验证？

通过定期演练验证回滚链路和恢复时间。
