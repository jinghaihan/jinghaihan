## 概念定义

- 本质定义：从 URL（Uniform Resource Locator）输入到页面响应是浏览器导航请求从发起到拿到首个文档响应的端到端链路。
- 解决问题：把 DNS（Domain Name System）、建连、请求发送、响应返回串成可定位、可优化的统一流程。
- 边界说明：它在网络阶段收口于状态码处理，DOM（Document Object Model）/CSSOM（CSS Object Model）构建属于浏览器原理体系。

## 核心机制拆解

### 1) 机制/流程

1. 浏览器解析 URL，确定协议、域名、端口和路径。
2. 先做缓存判定，命中则走本地复用，未命中进入 DNS。
3. 对关键第三方域可通过 `dns-prefetch` 或 `preconnect` 做前置预热。
4. 拿到地址后完成 TCP（Transmission Control Protocol）/TLS（Transport Layer Security）建连。
5. 发送 HTTP（HyperText Transfer Protocol）请求并等待服务端返回响应。
6. 响应进入 `status-code-path` 分支，再决定是否进入 `html-parser`。

### 2) 关键条件与触发点

- 条件 A：刷新方式和缓存策略，决定请求是否回源。
- 条件 B：协议版本与连接复用能力，决定并发和时延表现。
- 条件 C：资源提示是否精准，决定 `dns-prefetch` 和 `preconnect` 的收益与副作用。

### 3) 结果与影响

- 对性能 / 稳定性 / 可维护性的影响：链路任何一段异常都会放大用户感知时延；分段观测是快速定位的前提。

## 相似概念对比

| 维度 | 首次访问 | 重复访问 | 结论 |
|---|---|---|---|
| 定义 | 冷缓存为主，需完整网络链路 | 可复用 DNS、连接或资源缓存 | 两者排障路径不同 |
| 触发时机 | 首次打开、缓存失效、强制刷新 | 正常刷新、资源未失效 | 需要区分样本再下结论 |
| 性能影响 | DNS 和握手成本更高 | 缓存命中时延更低 | 性能优化要分别做冷/热链路 |
| 适用场景 | 首屏体验优化 | 日常交互与回访体验优化 | 面试常追问差异原因 |

## 代码

```bash
curl -w 'dns=%{time_namelookup} connect=%{time_connect} tls=%{time_appconnect} ttfb=%{time_starttransfer}\n' -o /dev/null -s https://www.example.com
```

```bash
curl -I https://www.example.com
```

```html
<link rel="dns-prefetch" href="//api.example.com" />
<link rel="preconnect" href="https://api.example.com" crossorigin />
```

### 示例说明

- 这段代码 / 配置解决什么问题：拆分导航链路耗时，识别瓶颈在 DNS、连接、TLS 还是服务端处理。
- 关键参数含义：`time_namelookup`、`time_connect`、`time_appconnect`、`time_starttransfer` 对应关键阶段耗时；`dns-prefetch` 只做解析预热，`preconnect` 会额外前置连接握手。
- 常见错误配置：只盯总耗时，不拆阶段指标，导致优化动作方向错误。

## 真实场景示例

- 场景描述：用户反馈同一页面“有时很快，有时很慢”。
- 问题表现：总耗时波动大，但服务端耗时稳定。
- 处理方案：按链路拆指标后发现 DNS 与 TLS 波动大，优化解析策略并启用连接复用。
- 验证方式：比较优化前后链路阶段耗时分位数和用户侧首屏指标。

## 常见误区

- 误区：页面慢一定是服务端慢。
- 正解：DNS、建连、TLS、重定向都可能在服务端处理前消耗大量时间。
- 误区：缓存命中就不需要观测网络链路。
- 正解：缓存失效和强制刷新路径仍会触发完整网络链路，必须持续监控。

## 关联知识点

- 依赖：依赖 `dns-resolver`、`dns-result`、`tcp-tls`、`http-method-semantics`。
- 影响：结果进入 `status-code-path`，成功文档流向 `html-parser`。
- 对比：`request-lifecycle` 看端到端流程，单点节点看局部机制。
- 组合：常与 `refresh-modes`、`cache-policy` 做联合排障。

## 相关面试题

1. 当在浏览器中输入 Google.com 并且按下回车之后发生了什么？
- 标准答案：URL 解析后先做缓存判定，再做 DNS、TCP/TLS、HTTP 请求与响应，最后根据状态码进入后续处理并触发页面解析；对关键域名可用 `dns-prefetch` 或 `preconnect` 前置等待。
- 考察点：是否能按时间顺序讲清完整链路和每段职责。
- 追问 1：为什么同一页面强制刷新和普通刷新速度不同？
- 标准答案 1：两种刷新命中的缓存层不同，强制刷新更容易绕过本地缓存而回源，链路成本更高。
- 追问 2：定位“慢”时第一步做什么？
- 标准答案 2：先拆分阶段耗时并固定样本，再决定是网络、服务端还是渲染问题。

2. URL 有哪些组成部分
- 标准答案：协议决定传输栈，域名和端口决定连接目标，路径和参数决定服务端路由，锚点仅在客户端生效不参与请求发送。
- 考察点：是否能把 URL 结构映射到真实网络行为。
