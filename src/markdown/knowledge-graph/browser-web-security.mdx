## 概念定义

- 本质定义：浏览器安全机制是指浏览器在脚本执行、请求发送和资源访问阶段的一组隔离与校验规则。
- 解决问题：减少脚本注入、请求伪造、链路劫持和跨源滥用带来的数据泄露风险。
- 边界说明：浏览器机制只能做前端侧防线，最终安全仍依赖服务端鉴权和数据校验。

## 核心机制拆解

### 1) 机制/流程

1. 输入数据进入页面前进行编码和上下文转义。
2. 浏览器按同源规则限制脚本与资源访问边界。
3. 请求发送时携带凭证并触发跨源校验。
4. 服务端通过令牌、来源校验和权限模型做二次验证。
5. 传输链路通过 HTTPS（HyperText Transfer Protocol Secure）和证书体系保障完整性。

### 2) 关键条件与触发点

- 条件 A：页面是否存在不安全的字符串拼接点，决定注入风险是否可触发。
- 条件 B：跨源请求是否受控，决定浏览器是否放行或拦截。

### 3) 结果与影响

- 对性能 / 稳定性 / 可维护性的影响：安全策略越前置，漏洞修复成本越低；策略缺失会直接演化为数据泄露和账户风险。

## 相似概念对比

| 维度 | XSS（Cross-Site Scripting） | CSRF（Cross-Site Request Forgery） | 结论 |
|---|---|---|---|
| 定义 | 注入并执行恶意脚本 | 诱导用户在已登录状态发起非预期请求 | 一个偏脚本注入，一个偏请求伪造 |
| 触发时机 | 不可信内容进入可执行上下文时 | 凭证自动携带且缺少请求来源校验时 | 风险触发路径不同 |
| 性能影响 | 防护策略影响渲染和脚本注入点 | 防护策略影响请求校验链路 | 二者都属于安全开销 |
| 适用场景 | 评论区、富文本、模板拼接 | 转账、改密、敏感写操作接口 | 常被组合考察 |

## 代码

```http
Content-Security-Policy: default-src 'self'; script-src 'self' https://cdn.example.com; object-src 'none'
Set-Cookie: sid=xxx; HttpOnly; Secure; SameSite=Lax
```

```js
// 前端输出转义示意
const escapeHtml = (text) =>
  text
    .replaceAll('&', '&amp;')
    .replaceAll('<', '&lt;')
    .replaceAll('>', '&gt;')
```

### 示例说明

- 这段代码解决什么问题：第一段限制脚本来源并加固 Cookie 属性，第二段避免插入型脚本注入。
- 关键语句或参数含义：`HttpOnly` 禁止脚本读取 Cookie，`SameSite` 限制跨站携带。
- 常见错误写法：只做前端过滤，不做服务端校验和输出编码。

## 真实场景示例

- 场景描述：营销活动页接入了第三方评论组件。
- 问题表现：评论内容可注入脚本，用户会话被窃取。
- 处理方案：关闭危险富文本白名单，增加输出转义、CSP 和敏感接口令牌校验。
- 验证方式：用安全扫描和人工构造攻击向量验证注入路径是否被封堵。

## 常见误区

- 误区：全站 HTTPS 后就不会有前端安全问题。
- 正解：HTTPS 只解决链路安全，不会自动解决注入和伪造请求问题。
- 误区：同源策略能挡住所有跨站攻击。
- 正解：同源策略是浏览器限制，不等于服务端权限模型。

## 关联知识点

- 依赖：`security-cors` 提供跨源访问策略落地细节。
- 影响：直接影响 `browser-storage-security` 中 Cookie 与本地存储风险边界。
- 对比：`browser-web-security` 讲威胁模型，`security-cors` 讲跨域放行规则。
- 组合：常与 `tcp-tls` 一起考链路与应用层联合防护。

## 相关面试题

1. 什么是 XSS 攻击？有哪些类型？如何防范？
- 标准答案：XSS 是把恶意脚本注入页面并在用户浏览器执行的攻击，常见有存储型、反射型、DOM 型，根因是不可信输入进入可执行上下文。
- 考察点：是否能说清类型、根因和危害链路。

2. 如何防御 XSS 攻击？
- 标准答案：核心措施是输入校验与输出编码、限制脚本来源的 CSP（Content Security Policy）、禁用危险 API、敏感 Cookie 使用 `HttpOnly` 和 `Secure`。
- 考察点：是否能给出“编码 + 策略 + 运行时限制”组合方案。

3. 什么是 CSRF 攻击？如何防范？
- 标准答案：CSRF 是攻击者利用用户已登录状态，诱导浏览器向目标站点发起非预期请求，本质是凭证自动携带与来源校验缺失叠加。
- 考察点：是否理解攻击成立前提。

4. 如何防御 CSRF 攻击？
- 标准答案：应组合使用来源校验、CSRF Token、`SameSite` Cookie 和关键操作二次确认，避免只依赖单一手段。
- 考察点：是否能给出多层防护策略。

5. 什么是中间人攻击？如何防范？
- 标准答案：中间人攻击是在通信双方之间拦截并篡改消息；防护重点是 HTTPS、证书链校验、证书固定策略和敏感操作签名校验。
- 考察点：是否能区分链路安全与应用安全边界。

6. 有哪些可能引起前端安全的问题?
- 标准答案：常见问题包括 XSS、CSRF、恶意第三方脚本、不安全 `iframe`、明文传输劫持、权限边界缺失和敏感数据本地明文存储。
- 考察点：是否能完整列举主要风险面而不是只说单点漏洞。

7. 网络劫持有哪几种，如何防范？
- 标准答案：常见有 DNS 劫持和 HTTP 劫持；防护重点是全站 HTTPS、权威 DNS 管理、证书校验和关键请求完整性校验。
- 考察点：是否能给出“劫持类型 + 对应防护”一一映射。

8. 什么是同源策略
- 标准答案：同源策略要求协议、域名、端口同时一致，目的是限制不同源脚本直接读写彼此资源，避免跨站数据泄露。
- 考察点：是否清楚同源判定条件。

9. 如何解决跨域问题
- 标准答案：常见方案有 CORS 响应头、反向代理、`postMessage`、JSONP（仅 GET）和网关聚合转发，选型取决于请求类型和安全要求。
- 考察点：是否能显式列举方案并说明边界。

10. 正向代理和反向代理的区别
- 标准答案：正向代理面向客户端，替客户端访问目标资源；反向代理面向服务端，统一入口后再转发到后端实例。
- 考察点：是否能从“代理对象”和“部署位置”区分两者。

11. Nginx 的概念及其工作原理
- 标准答案：Nginx 是高性能 Web 服务器和反向代理，采用事件驱动和多路复用模型处理高并发请求，并可承担静态资源分发、负载均衡和网关转发。
- 考察点：是否理解 Nginx 在前端链路中的实际职责。

12. 什么是CSP？如何配置和使用？
- 标准答案：CSP 是通过响应头限制脚本、样式、图片等资源来源的安全策略，核心作用是降低脚本注入可执行面。机制上通常通过 `Content-Security-Policy` 头配置 `default-src`、`script-src`、`style-src` 等指令；场景上第三方脚本较多的页面更需要先收敛白名单再上线。
- 考察点：是否理解 CSP 的目标是“限制执行来源”，而不是替代输入校验。

13. 什么是SQL注入？前端如何防范？
- 标准答案：SQL 注入本质是把恶意语句拼进后端查询，真正防线在后端参数化查询。前端侧能做的是输入格式校验、长度限制和危险字符拦截，减少脏输入直达服务端的概率；但前端过滤不能当最终防护。
- 考察点：是否能说清“前端辅助防护”和“后端核心防护”的边界。

14. 什么是DOM Clobbering攻击？如何防范？
- 标准答案：DOM Clobbering 是攻击者通过注入标签或属性覆盖全局变量与对象属性，导致业务逻辑读取到被污染对象。防护上要减少全局变量依赖、避免把关键配置挂到 `window`、对注入点做严格过滤并在取值时做类型校验。
- 考察点：是否知道它不是传统脚本执行漏洞，而是对象解析语义被利用。

15. 什么是原型污染攻击？如何防范？
- 标准答案：原型污染是攻击者通过 `__proto__`、`constructor.prototype` 等路径污染对象原型，进而影响全局对象行为。防护要点是过滤危险键、避免无约束深合并、优先使用 `Object.create(null)` 或 `Map` 存储外部输入数据。
- 考察点：是否理解原型污染会通过共享原型扩散，而不是只影响单次请求。

16. 如何安全地处理用户输入？
- 标准答案：原则是永远不信任输入。机制上要做白名单校验、长度限制、上下文相关输出编码，并区分 HTML、URL、脚本字符串等不同上下文；场景上富文本、搜索词和评论区是最高频风险入口。
- 考察点：是否能把“输入校验 + 输出编码 + 上下文”作为组合策略回答。

17. 如何防范ReDoS（正则表达式拒绝服务）攻击？
- 标准答案：ReDoS 是利用灾难性回溯让正则匹配占满 CPU。防护要点是避免嵌套量词、限制输入长度、给复杂匹配设超时或改用线性时间算法。场景上对用户输入做复杂正则校验时最容易踩坑。
- 考察点：是否知道 ReDoS 的根因是回溯复杂度，而不是“正则慢一点”。

18. 如何安全地使用eval()和Function构造函数？
- 标准答案：默认策略是不用。若业务确实需要动态表达式执行，必须先做严格语法白名单和沙箱隔离，并限制可访问上下文。场景上 JSON 解析应使用 `JSON.parse`，模板渲染应使用模板引擎而不是 `eval()`。
- 考察点：是否能给出“禁用优先，隔离兜底”的工程策略。

19. 什么是子资源完整性（SRI）？如何使用？
- 标准答案：SRI 用于校验外部脚本和样式内容哈希，防止 CDN 资源被篡改后仍被浏览器执行。机制上在 `script` 或 `link` 标签设置 `integrity` 和 `crossorigin`；场景上公共 CDN 资源最常用这项能力。
- 考察点：是否理解 SRI 保护的是资源完整性，而不是业务授权。

20. 如何实现安全的前端日志记录？
- 标准答案：前端日志要最小化采集，禁止记录密码、令牌、身份证号等敏感信息，并对日志内容做换行与控制字符清理以防注入。机制上可按字段脱敏、按级别采样、按场景上报；场景上错误日志和行为日志要分级治理。
- 考察点：是否能在“可观测性”和“隐私安全”之间给出可执行平衡方案。

21. 前端安全之 XSS 与 CSRF
- 标准答案：XSS 和 CSRF 分别对应“脚本注入执行”和“凭证被动携带触发非预期请求”两类风险，防护必须组合输入输出治理、CSP、`HttpOnly`、`SameSite`、来源校验和 CSRF Token。只防其中一类无法覆盖完整攻击面，真实业务通常要并行建设两套防线。
- 考察点：是否能把两类攻击的触发条件和防护策略做并列对比，而不是混为一类问题。
