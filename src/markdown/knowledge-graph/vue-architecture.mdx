## 概念定位

Vue 架构认知与项目分层节点解决“项目如何按业务和职责组织”，核心是边界清晰、依赖可控、协作高效。

这个节点决定后续响应式、路由、状态管理是否会演变成耦合泥球。

## 工作机制

1. 先按业务域拆模块，再按技术层拆职责（页面、组件、服务、状态）。
2. 建立目录和依赖方向约束，避免页面层反向依赖基础设施层。
3. 公共能力（hooks、utils、design tokens）统一沉淀并设 owner。
4. 约定 API 层与视图层分离，避免组件直接拼装请求细节。
5. 把架构约束纳入 lint 和评审模板，防止规则失效。

## 关键知识点

### 必须掌握

- 架构目标是支持演进，而不是一次性完美设计。
- 业务域边界要先于技术边界，避免“技术分层看起来整齐但业务耦合高”。
- 组合式 API（Composition API）适合跨组件逻辑抽象，但要控制粒度。
- 模块间通信路径越短越好，避免层层透传。
- 组件库和业务组件要分层，避免双向污染。
- 架构文档要可执行，不能只停留在图示。

### 选型与取舍

- 强分层约束提高稳定性，但会增加初期开发心智成本。
- 弱约束上手快，后期改造成本高。
- 公共抽象可提升复用，但过度抽象会降低可读性。

## 对页面性能与稳定性的影响

- 节点稳定时，Vue 工作流 的链路可预测性会明显提升，发布和排障成本更可控。
- 节点失稳时，问题会向上下游扩散，常表现为性能回退、交互异常或交付风险叠加。
- 该节点优化应和上下游协同验证，避免局部优化把成本转移给后续节点。

## 观测与排障

### 核心指标

- 跨层依赖违规数
- 循环依赖数
- 核心模块变更影响面
- 新成员上手时长

### 排障顺序

1. 先定位问题是否由模块边界不清导致。
2. 检查依赖图是否存在反向依赖与循环。
3. 识别高频变更模块是否职责过大。
4. 收敛公共层接口并做向后兼容策略。
5. 修复后通过规则和评审固化。

### 常见故障场景

- 改一个公共组件影响多个业务线。
- 页面组件里混入大量请求和状态细节。
- 新需求总要跨多个目录改动。
- 多人并行开发冲突频发。

## 常见误区

- 把架构优化当目录重命名。
- 只定规范不做自动化检查。
- 公共层混入业务特例逻辑。

## 题库深挖补充

### 高频对比与边界（基于题库）

- Vue 架构与组件通信要优先保证单向数据流和职责边界。
- props/emits 与 provide/inject 的适用场景不同，不能混用代替状态管理。
- v-model 在自定义组件本质是 modelValue + update:modelValue 协议。
- 测试策略要覆盖组件契约、路由流程、发布回归三层。
- 团队治理要把高频问题沉淀成 lint/脚手架/评审清单。
- 面试常问“Vue 项目怎么落地规范”，重点是规则可执行而非文档数量。

### 线上排障剧本

1. 先确认问题属于通信边界、架构边界还是发布回归缺口。
2. 检查 props 透传深度和事件命名语义。
3. 检查是否存在隐式共享状态导致耦合。
4. 回归关键页面与关键交互流程。
5. 复盘后把问题映射到团队规则。

### 高频追问补充（附答案）

1. provide/inject 能替代 Pinia 吗？

不能完全替代。它更适合稳定上下文，不适合复杂共享状态治理。

2. 为什么通信协议要显式命名？

可追踪、可调试、可复盘，避免隐式副作用。

3. Vue 规范治理最有效抓手是什么？

自动化规则 + 评审清单 + 复盘闭环。

## 进阶追问（补充）

1. Vue 性能排障为什么不能只看组件层？

因为请求竞态、状态边界和调度批处理会共同影响组件表现。

2. 什么时候应该优先改架构而不是继续微优化？

当问题持续跨页面复发，且根因是边界设计不清时，应先做结构治理。

3. 如何防止 Vue 项目后期通信失控？

统一通信协议、状态分层和评审规则，并让规则可自动检查。

## 案例追问加深（第三轮）

### 真实故障片段

- 故障表现：功能正确但更新抖动明显，复杂页面偶发状态错乱。
- 排查结论：响应式更新时机、组件通信边界和状态层策略没有统一约束。
- 治理动作：统一调度时机规范、通信协议和状态分层规则，并纳入评审门禁。

### 追问补充

1. 为什么 Vue 项目后期常出现“能跑但难改”？

边界规则缺失导致耦合积累，修改一个点会牵动多个链路。

2. 如何把 Vue 优化变成可持续能力？

把高频问题转成自动化规则和发布检查，不依赖个人经验记忆。

## 面试补充

### 30 秒说明

Vue 架构节点的关键是把职责边界前置，确保后续响应式与渲染优化都能在清晰结构中落地。

### 高频追问

1. Vue 项目为什么要强调业务域分层？

因为业务域分层能降低跨模块耦合，减少改动扩散。

2. 组合式 API 会让架构更好吗？

会，但前提是抽象粒度和命名边界清晰。

3. 怎么判断架构开始劣化？

跨层依赖、循环依赖和回归影响面持续上升。

4. 架构规范如何避免流于形式？

把规则接入 lint/CI，并在评审中强制执行。
