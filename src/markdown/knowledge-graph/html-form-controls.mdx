## 概念定义

- 本质定义：表单与原生交互控件关注 HTML（HyperText Markup Language）如何声明输入项、校验规则、焦点顺序和交互语义。
- 解决问题：在不依赖重脚本的前提下，先把输入、提交、校验和键盘可达性做好。
- 边界说明：本节点聚焦表单和控件本身，不展开网络提交策略，也不展开框架层表单状态管理。

## 核心机制拆解

### 1) 机制/流程

1. 用 `form` 声明提交通道，用 `label`、`input`、`select`、`textarea`、`button` 组织输入结构。
2. 通过 `type`、`required`、`minlength`、`pattern`、`min`、`max` 等属性声明浏览器原生校验规则。
3. 用 `fieldset` 和 `legend` 给一组相关控件建立共同语义。
4. 通过 `tabindex`、`disabled`、`spellcheck` 等属性调整键盘路径和编辑体验。
5. 在“浏览器原生能力先兜底”的基础上，再补 JavaScript 自定义校验和提交流程。

### 2) 关键条件与触发点

- 条件 A：`label` 没有关联控件时，点击热区、读屏描述和焦点跳转都会变差。
- 条件 B：只写自定义校验，不写原生约束时，浏览器层的基础拦截和键盘语义会丢失。
- 条件 C：滥用正整数 `tabindex` 会让焦点顺序脱离 DOM（Document Object Model）结构，后续维护很痛苦。

### 3) 结果与影响

- 对性能/稳定性/可维护性的影响：原生控件和声明式校验用得对，代码更短、回归面更小；全部自己模拟，往往会把简单问题做复杂。

## 相似概念对比

| 维度 | `button` | `input[type="button"]` | 结论 |
|---|---|---|---|
| 内容能力 | 可包含文本、图标和其他内联内容 | 只能显示 `value` 文本 | 复杂按钮优先用 `button` |
| 默认行为 | 在表单内默认是 `submit` | 默认不提交表单 | 要显式声明 `type` |
| 样式能力 | 更容易做复杂样式和伪元素 | 样式能力更受限 | 组件化场景更适合 `button` |
| 适用场景 | 提交、弹窗确认、带图标按钮 | 极简纯文本按钮 | 现代项目通常优先 `button` |

## 代码

```html
<form action="/signup" method="post">
  <fieldset>
    <legend>账号信息</legend>
    <label for="email">邮箱</label>
    <input id="email" name="email" type="email" required />

    <label for="age">年龄</label>
    <input id="age" name="age" type="number" min="18" max="60" />
  </fieldset>

  <label for="city">城市</label>
  <select id="city" name="city">
    <option value="">请选择</option>
    <option value="shanghai">上海</option>
    <option value="shenzhen">深圳</option>
  </select>

  <button type="submit">提交</button>
</form>
```

### 示例说明

- 这段代码解决什么问题：用原生结构把“分组、关联、校验、选择、提交”一条链路交代清楚。
- 关键语句或参数含义：`fieldset + legend` 表示分组；`label for` 绑定控件；`type="email"` 和 `required` 开启浏览器内置校验。
- 常见错误写法：按钮不写 `type`、只放占位文字不写 `label`、所有校验都放到提交后才做。

## 真实场景示例

- 场景描述：注册页从设计稿看着没问题，但线上一直有“点标题聚焦不了输入框”“手机键盘类型不对”“提交后才知道格式错了”的反馈。
- 问题表现：转化率低，输入错误率高，键盘和读屏体验差。
- 处理方案：补 `label` 关联，给邮箱/电话/数字字段设置正确 `type`，优先使用浏览器原生校验，再补业务规则。
- 验证方式：看表单首提成功率、键盘路径回归、读屏抽检和错误提示触发率。

## 常见误区

- 误区：占位文字已经说明用途了，`label` 可以不要。
- 正解：占位文字会消失，`label` 才是稳定的控件语义入口。
- 误区：只要用了组件库，原生表单语义就不重要。
- 正解：组件库最终仍要落到原生元素和属性上，语义没处理好，问题只会被包起来。
- 误区：`tabindex` 数字越精细，焦点控制越专业。
- 正解：多数情况下应保持 DOM 顺序，优先用 `tabindex="0"` 或 `-1`，少用正整数重排焦点。

## 关联知识点

- 依赖：语义化与可访问性、文档声明与 head 配置。
- 影响：表单提交成功率、键盘可达性、移动端输入体验。
- 对比：原生校验与纯脚本校验。
- 组合：渐进增强与优雅降级、框架层表单状态管理。

## 相关面试题

1. HTML表单的基本结构是什么？
- 标准答案：核心结构是 `form` 作为容器，内部放 `label`、`input`、`textarea`、`select`、`button` 等控件。机制上 `form` 负责提交地址和方法，控件通过 `name` 参与序列化。场景上不管后续是不是 Ajax（Asynchronous JavaScript and XML）提交，原生结构都该先写对。
- 考察点：是否理解“表单 = 容器 + 控件 + 提交语义”。

2. HTML5中的新表单输入类型有哪些？
- 标准答案：常见新增类型包括 `email`、`url`、`tel`、`number`、`range`、`date`、`time`、`datetime-local`、`month`、`week`、`color`、`search`。机制上这些类型会影响浏览器校验规则和移动端键盘形态。场景上邮箱、电话、数字输入都应该优先用对应类型，而不是一律 `text`。
- 考察点：是否知道输入类型不仅影响样式，还影响校验和交互。

3. 如何在HTML中创建表单验证？
- 标准答案：可先用 `required`、`pattern`、`min`、`max`、`minlength`、`maxlength`、`type` 等原生属性声明规则，再按业务需要补自定义提示。机制上浏览器会在提交前做基础校验并阻止非法提交。场景上账号注册、预约、支付地址输入都适合先用原生能力兜底。
- 考察点：是否知道“声明式约束优先，脚本校验补充”。

4. label 的作用是什么？如何使用？
- 标准答案：`label` 的作用是给表单控件建立可读语义，并扩大可点击区域。机制上点击 `label` 会把焦点移到关联控件；常见写法是 `for + id`，也可以直接包裹控件。场景上登录、搜索、筛选面板这类高频输入区域都应该写 `label`。
- 考察点：是否把 `label` 理解成可访问性和可用性问题，而不只是文案。

5. HTML中的button和input[type="button"]有什么区别？
- 标准答案：`button` 可以放文本、图标和其他内联内容，在表单里默认是提交按钮；`input[type="button"]` 只能显示 `value` 文本，默认不提交表单。机制上两者的内容模型和默认行为不同。场景上现代项目里需要复杂样式或明确语义时通常优先 `button`。
- 考察点：是否能从内容能力、默认行为和语义三方面回答。

6. HTML中的fieldset和legend标签如何使用？
- 标准答案：`fieldset` 用于把一组相关控件包成一个语义分组，`legend` 是这组控件的标题。机制上读屏器会把这组输入当成同一块内容来理解。场景上注册信息、地址信息、支付信息这类分组表单很适合用它。
- 考察点：是否理解表单“分组语义”而不只会背标签名。

7. HTML中如何创建下拉菜单？
- 标准答案：标准做法是 `select` 包 `option`，分组场景可再配 `optgroup`。机制上浏览器会负责键盘选择、焦点和序列化提交。场景上固定候选项、数量不大时优先用原生下拉；复杂搜索型选择器再考虑脚本增强。
- 考察点：是否知道原生下拉的基本结构和适用边界。

8. HTML中的progress和meter标签有什么区别？
- 标准答案：`progress` 表示任务进度，强调“从 0 到完成”的过程；`meter` 表示某个区间内的当前测量值，比如电量、磁盘占用、评分。机制上两者语义不同，所以读屏和业务含义也不同。场景上上传进度用 `progress`，风险等级或库存占比更适合 `meter`。
- 考察点：是否能把“进度”和“测量值”分开。

9. HTML中的spellcheck属性有什么作用？
- 标准答案：`spellcheck` 控制浏览器是否对可编辑内容做拼写检查。机制上它主要影响 `input`、`textarea` 和 `contenteditable` 区域的输入提示。场景上文章编辑器可以开，代码输入、用户名、邮箱这类字段通常要关，避免错误标红。
- 考察点：是否能根据输入场景做开关取舍。

10. HTML中的tabindex属性如何控制焦点顺序？
- 标准答案：`tabindex="0"` 表示按自然顺序参与键盘导航，`tabindex="-1"` 表示不参与 Tab 导航但可被脚本聚焦，正整数会强行改写焦点顺序。机制上它直接影响键盘用户的访问路径。场景上自定义弹窗、菜单、组合控件会用到 `0` 和 `-1`，但通常不建议大量使用正整数。
- 考察点：是否理解焦点管理对可访问性和维护性的影响。
