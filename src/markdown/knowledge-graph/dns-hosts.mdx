## 概念定位

Hosts 文件匹配是 DNS 递归前的本地静态映射层，常用于开发联调、灰度验证和临时路由覆盖。

## 工作机制

1. 系统 resolver 在外部 DNS 前读取 hosts 映射。
2. 命中时直接返回指定 IP，不再继续 DNS 分层查询。
3. 未命中时才进入递归 DNS 路径。
4. hosts 生效通常不依赖 TTL，依赖本地文件配置与缓存刷新。

## 关键知识点

- **优先级高**：可覆盖线上权威 DNS 结果。
- **静态性强**：不会自动随线上变更更新。
- **风险可隐蔽**：本机“看似正常”但与线上真实环境不一致。
- **排障第一步**：同域行为分歧时优先排查 hosts。

## 对页面性能的影响

- 命中 hosts 可省去 DNS 查询链，解析阶段更快。
- 错误映射会导致建连失败、证书不匹配、地区路由异常。
- 遗留测试映射常导致“只有我机器有问题”。

## 观测与排障

### 检查命令

```sh
# macOS / Linux
cat /etc/hosts
# Windows
notepad C:\\Windows\\System32\\drivers\\etc\\hosts
```

### 排障步骤

1. 比对团队机器 hosts 是否一致。
2. 检查是否遗留历史测试域名映射。
3. 清浏览器与系统 DNS 缓存后重试。
4. 对比权威 DNS 结果和本机访问结果差异。
5. 建立 hosts 变更登记，避免长期漂移。

## 常见误区

- “nslookup 一定走 hosts”：不准确，很多实现直接查 DNS 服务器。
- “hosts 只影响浏览器”：不准确，依赖系统解析的应用都会受影响。
- “改完 hosts 立即全生效”：不一定，缓存层可能还在复用旧值。

## 题库深挖补充

### 高频对比与边界（基于题库）

- 浏览器 DNS 缓存命中后，通常不会继续向操作系统查询；操作系统缓存命中后，也不会进入递归解析。
- hosts 文件优先级高于外部 DNS 结果，常用于本地调试和应急切流。
- 负缓存（Negative Caching）会缓存“不存在域名”结果，排障时要注意缓存滞后。
- 本地缓存与公共 DNS 缓存并不是一层，清理浏览器缓存不等于清理系统解析缓存。
- 多网络环境切换（公司 VPN、移动热点）会改变本地解析路径，导致“同机不同时段结果不同”。
- 面试常问“为什么同一域名一会儿能访问一会儿不能访问”，核心就是多层缓存状态不一致。

### 线上排障剧本

1. 先确认问题是“解析失败”还是“解析到错误 IP”。
2. 在同一台机器分别检查浏览器缓存、系统缓存和 hosts 覆盖情况。
3. 对比本地解析结果和权威解析结果，判断是本地污染还是上游问题。
4. 检查 TTL 与负缓存窗口，确认是否处于缓存过渡期。
5. 若涉及紧急切流，优先用可控 DNS 策略，不建议长期依赖 hosts。

### 高频追问补充（附答案）

1. 为什么清了浏览器缓存问题还在？

因为解析可能命中系统缓存或网络侧缓存，浏览器缓存不是唯一层。

2. hosts 的风险点是什么？

它优先级很高，临时调试忘记清理会长期污染本机解析结果。

3. 什么是负缓存，为什么排障时容易忽略？

负缓存会记住“解析失败”结果，在 TTL 没过期前会持续失败，即使权威 DNS 已恢复。

### 术语速查与易错点

- A 记录：域名到 IPv4 地址映射。
- AAAA 记录：域名到 IPv6 地址映射。
- CNAME（Canonical Name）：别名记录，常用于接入 CDN。
- TTL（Time To Live）：缓存有效时长，不是“固定刷新时间”。
- NXDOMAIN：权威返回域名不存在。

常见易错点：

- 清浏览器缓存后立即下结论，忽略系统层缓存。
- 忘记 hosts 临时改动，导致排障方向偏离。
- 只在单网络环境验证解析结果。

## 深度延展（案例对照）

### hosts 污染案例

- 场景：仅开发机访问异常，线上用户无异常。
- 诊断：遗留 hosts 映射覆盖了正式域名。
- 处理：移除临时映射并加入开发环境启动自检。

### 实操提示

- 调试用 hosts 需加明确注释和过期时间。
- 团队内建立提交前 hosts 自检习惯。

## 进阶排障清单（补充）

### 现场处置顺序

1. 先固定样本：同地域、同网络、同设备，避免样本漂移。
2. 抓三组结果：本地解析、公共 DNS 解析、权威解析。
3. 比对 TTL 与记录版本，确认是否处于传播窗口。
4. 标记是否命中本地特殊配置（hosts、代理、VPN）。
5. 形成“可复现路径 + 影响范围 + 临时止损”三件套再升级处理。

### 面试追问强化

1. 为什么 DNS 问题经常“不可稳定复现”？

因为多层缓存、地域路由和传播窗口共同影响，同一时间不同样本可能命中不同路径。

2. 如何用最少步骤判断是不是 DNS 问题？

先看不同解析器结果是否分叉，再看业务域名能否稳定映射到预期接入点。

3. 何时应该优先止损而不是继续追根因？

当影响范围在扩大且已有可控切流手段时，应先保可用性再做根因定位。

## 速查补充（第四轮）

### 快速判定问答

1. 什么时候先怀疑缓存层而不是权威记录本身？

当不同客户端结果分叉明显且权威查询稳定时，优先检查本地与递归缓存状态。

2. DNS 类问题为什么要强调样本固定？

因为网络、地域、时间窗口会影响解析路径，不固定样本容易得到互相矛盾结论。

## 面试补充

### 30 秒说明

hosts 是本地静态覆盖层，优先级高于外部 DNS。它很适合调试，也很容易造成环境偏差。线上问题排障时，要把 hosts 检查作为第一步。

### 高频追问

1. 为什么我本机正常，其他人不正常？

常见是你本机 hosts 把域名指到了特殊 IP，而其他人走的是线上 DNS 结果。

2. hosts 覆盖会影响 HTTPS 吗？

会。若映射 IP 对应证书与域名不匹配，TLS 握手会失败。

3. 如何降低 hosts 误配置风险？

建立最小化使用原则、变更记录和定期清理流程。
