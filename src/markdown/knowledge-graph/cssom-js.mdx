## 概念定义

- 本质定义：CSSOM（CSS Object Model）与脚本执行阶段是浏览器在样式可用后完成样式计算并执行脚本逻辑的关键环节。
- 解决问题：保证脚本读取样式时拿到一致结果，避免样式与脚本执行顺序错乱。
- 边界说明：它不直接负责最终图层合成输出。

## 核心机制拆解

### 1) 机制/流程

1. 浏览器下载并解析 CSS（Cascading Style Sheets）生成 CSSOM。
2. 脚本执行若依赖样式信息，会等待样式可用。
3. 脚本运行可能修改 DOM 或样式，触发后续计算。
4. 变更结果流入渲染树和布局阶段。

### 2) 关键条件与触发点

- 条件 A：脚本是否需要读取计算样式，决定是否等待 CSSOM。
- 条件 B：样式资源是否及时到达，决定脚本执行延迟风险。

### 3) 结果与影响

- 对性能 / 稳定性 / 可维护性的影响：样式与脚本顺序管理不当会引发解析阻塞和主线程排队，直接拖慢首屏。

## 相似概念对比

| 维度 | CSSOM 构建 | 脚本执行 | 结论 |
|---|---|---|---|
| 定义 | 样式规则解析和对象化 | 代码运行与状态变更 | 两者互相影响执行时序 |
| 触发时机 | 样式资源可用时 | 解析器遇到脚本或任务调度触发 | 脚本读取样式会受 CSSOM 进度影响 |
| 性能影响 | 影响样式计算起点 | 影响主线程占用时长 | 顺序失控会形成阻塞链 |
| 适用场景 | 样式规则加载与合并 | 业务逻辑执行与视图更新 | 常一起考“为什么脚本被延迟” |

## 代码

```js
const style = window.getComputedStyle(document.body)
console.log(style.fontSize)
```

### 示例说明

- 这段代码解决什么问题：读取计算样式，验证样式是否已可用。
- 关键语句或参数含义：`getComputedStyle` 会读取最终计算结果。
- 常见错误写法：在高频回调中反复读写样式，触发不必要开销。

## 真实场景示例

- 场景描述：首屏脚本初始化依赖字体与尺寸计算。
- 问题表现：样式文件加载慢时，脚本启动延迟明显。
- 处理方案：关键样式内联、非关键样式延迟加载，脚本拆分为首屏必需与延后执行。
- 验证方式：对比样式下载时间、脚本开始时间和首屏渲染时间。

## 常见误区

- 误区：CSS 不会影响脚本执行时机。
- 正解：当脚本读取样式信息时，浏览器会等待样式可用以保证结果正确。

## 关联知识点

- 依赖：依赖 `html-parser` 生成结构节点。
- 影响：影响 `render-tree` 构建和 `layout` 触发频率。
- 对比：`cssom-js` 关注样式与脚本协作，`parse-blocking` 关注阻塞治理策略。
- 组合：常与 `js-event-loop` 联合分析主线程拥塞。

## 相关面试题

1. CSS 如何阻塞文档解析？
- 标准答案：样式本身不直接改变 DOM 解析逻辑，但脚本若依赖样式计算结果，浏览器会等待 CSSOM 可用再执行脚本，间接造成解析和执行延迟。
- 考察点：是否能解释“样式阻塞脚本，脚本再阻塞解析”的链路。
