## 概念定义

- 本质定义：浏览器事件派发模型描述事件从触发到处理完成的传播、排队和执行机制。
- 解决问题：解释事件冒泡、事件委托、任务排队和回调时序问题。
- 边界说明：它不替代业务状态管理，只定义浏览器侧调度规则。

## 核心机制拆解

### 1) 机制/流程

1. 用户输入或系统事件触发后进入事件系统。
2. 事件按捕获、目标、冒泡三个阶段传播。
3. 监听器回调进入任务调度，由主线程执行。
4. 回调可通过阻止冒泡控制传播范围。
5. 事件处理结束后再进入下一轮任务。

### 2) 关键条件与触发点

- 条件 A：监听器绑定在捕获还是冒泡阶段，决定回调顺序。
- 条件 B：主线程是否被长任务占用，决定事件处理延迟。

### 3) 结果与影响

- 对性能 / 稳定性 / 可维护性的影响：模型理解正确可降低交互 Bug 和卡顿，理解错误会导致事件失效与顺序错乱。

## 相似概念对比

| 维度 | 事件冒泡 | 事件委托 | 结论 |
|---|---|---|---|
| 定义 | 事件从目标节点向上层传播 | 在父节点统一监听并分发处理 | 委托建立在冒泡机制之上 |
| 触发时机 | 事件传播阶段自动触发 | 父节点监听到子节点事件时 | 二者是机制与策略关系 |
| 性能影响 | 与节点层级相关 | 减少监听器数量和绑定成本 | 列表场景委托收益明显 |
| 适用场景 | 一般交互传播 | 动态列表、批量节点交互 | 复杂隔离场景需谨慎使用 |

## 代码

```js
const list = document.querySelector('#todo-list')
list?.addEventListener('click', (event) => {
  const target = event.target
  if (target instanceof HTMLElement && target.matches('[data-remove]')) {
    target.closest('li')?.remove()
  }
})
```

### 示例说明

- 这段代码解决什么问题：通过事件委托处理动态列表删除，不需要给每个按钮单独绑监听。
- 关键语句或参数含义：`matches` 用于命中具体子节点，`closest` 回溯目标容器。
- 常见错误写法：不做目标过滤直接处理，导致误触发。

## 真实场景示例

- 场景描述：千级列表交互导致内存和响应时间上升。
- 问题表现：节点越多，点击回调越慢。
- 处理方案：改为父节点事件委托并减少高频同步逻辑。
- 验证方式：对比监听器数量、输入延迟和内存占用。

## 常见误区

- 误区：阻止冒泡可以解决所有事件冲突。
- 正解：滥用会破坏上层逻辑，应该优先做目标过滤和职责分层。

## 关联知识点

- 依赖：依赖 `js-event-loop` 执行任务调度。
- 影响：影响 `layout` 与 `paint` 阶段的触发密度。
- 对比：`browser-event-dispatch` 讲传播模型，`js-event-loop` 讲任务调度队列。
- 组合：常与 `throttle-debounce` 配合治理高频事件。

## 相关面试题

1. 事件是什么？事件模型？
- 标准答案：事件是浏览器对输入或状态变化的通知；事件模型包含捕获、目标、冒泡三个传播阶段，监听器在对应阶段执行。
- 考察点：是否能完整说出三阶段和执行顺序。

2. 如何阻止事件冒泡
- 标准答案：可在回调中调用 `event.stopPropagation()`，必要时配合 `stopImmediatePropagation()` 阻止同节点后续监听器。
- 考察点：是否理解两种阻止方式作用范围差异。

3. 对事件委托的理解
- 标准答案：事件委托是在父节点统一监听子节点事件，利用冒泡减少监听器数量并支持动态节点。
- 考察点：是否能说出原理和适用边界。

4. 事件委托的使用场景
- 标准答案：典型场景包括动态列表、表格批量操作、虚拟列表和重复结构组件。
- 考察点：是否能显式列举场景而不是只说“性能好”。

5. 同步和异步的区别
- 标准答案：同步任务按顺序阻塞执行，异步任务由外部事件触发后入队再执行；区别在执行时机、是否阻塞主线程和错误传播路径。
- 考察点：是否能从时序和阻塞角度比较。

6. 对事件循环的理解
- 标准答案：事件循环会不断取出任务执行，当前任务完成后再处理下一个，并在合适时机处理微任务队列。
- 考察点：是否理解“执行栈清空后再取任务”的核心规则。

7. 宏任务和微任务分别有哪些
- 标准答案：常见宏任务有 `setTimeout`、`setInterval`、I/O 回调，常见微任务有 `Promise.then`、`queueMicrotask`、`MutationObserver`。
- 考察点：是否能显式列举而非抽象描述。

8. 什么是执行栈
- 标准答案：执行栈是函数调用上下文的后进先出结构，用于管理同步代码执行顺序和返回路径。
- 考察点：是否理解栈结构与函数调用关系。

9. Node 中的 Event Loop 和浏览器中的有什么区别？process.nextTick 执行顺序？
- 标准答案：浏览器与 Node（Node.js）事件循环阶段划分不同，Node 有更细阶段队列；`process.nextTick` 在 Node 中优先级高于常规微任务队列。
- 考察点：是否能说出“平台差异 + 顺序差异”。

10. 事件触发的过程是怎样的
- 标准答案：事件触发后按捕获到目标再到冒泡传播，命中监听器后回调入栈执行，执行完成后继续后续传播或结束。
- 考察点：是否能按时间顺序描述完整过程。

11. 事件流：捕获与冒泡
- 标准答案：标准事件流是“捕获阶段 -> 目标阶段 -> 冒泡阶段”。默认 `addEventListener` 监听在冒泡阶段；若显式设置捕获参数，则会在捕获阶段先触发。排查顺序问题时要同时看绑定阶段和节点层级。
- 考察点：是否能准确说出三阶段和默认监听行为。

12. 事件委托：更高效的监听
- 标准答案：事件委托利用冒泡把多个子节点监听收敛到父节点，能降低监听器数量并支持动态节点。机制上依赖 `event.target` 做命中判断，必要时配合 `closest` 保证命中稳定。
- 考察点：是否理解委托的性能收益和目标过滤边界。

13. 阻止冒泡与默认行为
- 标准答案：`stopPropagation` 用于阻止事件继续传播，`preventDefault` 用于取消浏览器默认行为，两者作用面不同且可同时使用。场景上例如“按钮在链接内”常要先阻止默认跳转，再按需阻断冒泡。
- 考察点：是否能区分传播控制和默认行为控制。

14. 深入 addEventListener
- 标准答案：`addEventListener` 除了事件类型和回调，还要关注第三参数中的 `capture`、`passive`、`once`。例如滚动场景优先使用 `passive: true` 减少主线程等待，短生命周期事件可用 `once: true` 自动解绑。
- 考察点：是否能把监听选项和真实性能问题关联起来。

15. 事件循环揭秘：宏任务与微任务
- 标准答案：事件循环每轮会先执行一个宏任务，再在渲染前清空当前产生的微任务队列。结论是 `Promise` 回调这类微任务通常先于下一轮 `setTimeout` 执行；如果微任务持续堆积，也会阻塞渲染与输入响应。
- 考察点：是否能说清宏任务、微任务和渲染时机之间的顺序关系。
