## 概念定位

异步模式与错误处理关注“并发怎么控、错误怎么传、失败怎么恢复”。这是把 JS 语法能力变成稳定线上行为的关键节点。

该节点直接关联请求成功率、重试风暴风险和用户可感知稳定性。

## 工作机制

1. 统一异步风格：以 `async/await` 为主，保留 Promise 组合能力（`all/allSettled/race/any`）。
2. 定义并发策略：串行、固定并发、优先级队列，不同接口采用不同策略。
3. 定义超时、取消与重试协议：`AbortController` + 指数退避（Exponential Backoff）。
4. 错误分层：业务错误、网络错误、系统错误分开处理和上报。
5. 兜底策略：降级展示、缓存回退、局部失败可恢复。

## 关键知识点

### 必须掌握

- 并发不是越高越好，过高会触发服务端限流或客户端拥塞。
- `Promise.all` 一处失败全失败，聚合场景可用 `allSettled`。
- 重试必须幂等，否则可能造成重复写入。
- 超时不是失败终点，要和用户反馈与补偿动作绑定。
- 取消无效请求可减少竞态覆盖问题（后到请求覆盖先到新数据）。
- 错误上报需要上下文：接口、参数摘要、traceId、用户操作路径。

### 选型与取舍

- 强一致策略更安全，但可能牺牲响应速度。
- 激进重试能提高短时成功率，但会放大后端压力。
- 复杂并发控制提升稳定性，但实现和调试成本更高。

## 对页面性能与稳定性的影响

- 节点稳定时，前端基础（HTML/CSS/JavaScript） 的链路可预测性会明显提升，发布和排障成本更可控。
- 节点失稳时，问题会向上下游扩散，常表现为性能回退、交互异常或交付风险叠加。
- 该节点优化应和上下游协同验证，避免“局部优化把成本转移给后续节点”。

## 观测与排障

### 核心指标

- 请求成功率与重试成功率
- 超时率与取消率
- 同接口并发峰值
- 异步错误未捕获数量（Unhandled Rejection）

### 排障顺序

1. 先定位失败模式：超时、拒绝、业务校验失败还是竞态覆盖。
2. 检查是否存在未捕获 Promise 错误链。
3. 检查重试策略是否有退避与最大重试次数。
4. 检查取消机制是否覆盖页面切换和重复触发场景。
5. 对关键链路补充幂等键和上下文日志。

### 常见故障场景

- “列表闪烁且数据回退”：常见是并发请求返回顺序竞态。
- “接口雪崩后前端更慢”：常见是无退避重试导致放大故障。
- “线上偶发空白页”：常见是顶层异步错误未兜底。
- “切页后还在写状态”：常见是未取消过期请求。

## 常见误区

- 把错误统一 catch 后吞掉，不做可观测上报。
- 重试策略没有上限和退避。
- 忽视幂等约束，导致副作用重复执行。

## 深入拆解

### Promise 组合在实战中的选择

- Promise.all：强依赖全成功，任一失败立即失败。
- Promise.allSettled：聚合展示场景，允许部分失败。
- Promise.race：超时控制、首结果抢占场景。
- Promise.any：多源容灾，任一成功即成功。

### 重试策略设计清单

1. 判断是否可重试：是否幂等、是否暂时性错误。
2. 设置最大重试次数与指数退避。
3. 增加随机抖动（Jitter）防止同刻重试洪峰。
4. 每次重试带上 traceId 和 attempt 次数，便于追踪。

### 竞态与取消治理

- 页面切换时取消旧请求，避免旧数据覆盖新数据。
- 对列表检索类接口做请求防抖，减少无意义并发。
- 对关键写操作引入幂等键，防止重复提交。

## 题库深挖补充

### 高频对比与边界（基于题库）

- Promise.all 全成全败，allSettled 更适合聚合展示场景。
- race 常用于超时控制，any 常用于多源容灾。
- 重试策略必须结合幂等语义，写操作重试尤其要谨慎。
- AbortController 是取消过期请求的基础能力。
- 节流与防抖要按业务场景选：输入联想多用防抖，滚动监听多用节流。
- 面试常问“async/await 优缺点”，要能讲可读性与并发控制并存。

### 线上排障剧本

1. 先区分是失败重试问题还是请求竞态问题。
2. 检查是否存在未捕获 Promise rejection。
3. 检查重试是否有最大次数和退避。
4. 检查请求取消是否覆盖切页场景。
5. 增加 trace 字段做链路对齐。

### 高频追问补充（附答案）

1. Promise.all 什么时候不适合？

当部分失败可接受、页面仍需继续渲染时。

2. 为什么重试要加抖动？

避免大量请求同一时刻重试形成洪峰。

3. 防抖和节流怎么选？

看目标：减少触发次数选防抖，稳定触发频率选节流。

## 进阶深挖（第二轮）

### 重试策略精细化

| 错误类型 | 默认策略 | 风险 | 建议 |
| --- | --- | --- | --- |
| 网络超时 | 可重试 | 雪崩放大 | 指数退避 + 抖动 |
| 5xx 临时错误 | 可重试 | 后端压力升高 | 控制最大重试次数 |
| 4xx 参数错误 | 不重试 | 无意义流量 | 直接反馈并修正请求 |
| 写操作失败 | 谨慎重试 | 重复副作用 | 幂等键 + 去重日志 |

### 并发治理模板

1. 定义并发上限与队列策略。
2. 对同 key 请求做 in-flight 合并。
3. 页面切换统一取消过期请求。
4. 对关键写操作加幂等键。

### 追问扩展

1. 为什么很多项目“有重试”但线上还是抖？

因为重试没有按错误类型分层，且缺少退避与并发限制。

2. Promise.any 适合哪些实战场景？

多源容灾读取、镜像站点竞速、可接受任一成功结果的读取场景。

## 面试补充

### 30 秒说明

异步治理的核心是“并发可控、错误可见、失败可恢复”。只靠语法层面的 `await` 不足以支撑线上稳定性。

### 高频追问

1. `Promise.all` 和 `allSettled` 怎么选？

强依赖整体成功时用 `all`，部分失败可接受并需继续渲染时用 `allSettled`。

2. 重试为什么要指数退避？

避免所有请求在同一时刻重试，降低雪崩风险。

3. 如何处理请求竞态问题？

给请求打版本或序号，只接收最新请求结果，并在切换时取消旧请求。

4. 超时后的正确处理流程是什么？

记录上下文、给用户反馈、按幂等策略重试或降级，并上报观测系统。
