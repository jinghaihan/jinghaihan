## 概念定义

- 本质定义：该节点聚焦 数据变更驱动视图更新。
- 解决问题：把高频面试问题转成可复述、可落地的机制说明。
- 边界说明：不替代其他节点的专属机制域。

## 核心机制拆解

### 1) 机制/流程

1. 明确输入状态和触发条件。
2. 进入框架调度或编译阶段。
3. 执行更新并产出结果。
4. 用回归验证结果是否稳定。

### 2) 关键条件与触发点

- 条件 A：组件边界和数据来源必须清晰。
- 条件 B：触发时机与副作用清理要成对出现。

### 3) 结果与影响

- 对性能/稳定性/可维护性的影响：机制清晰时可预测，机制混乱时会出现渲染回退、状态错乱或协作成本上升。

## 相似概念对比

| 维度 | Vue 2 响应式 | Vue 3 响应式 | 结论 |
|---|---|---|---|
| 定义 | 关注单点能力 | 关注对照能力 | 职责边界不同 |
| 触发时机 | 偏局部触发 | 偏链路触发 | 需要按场景选择 |
| 性能影响 | 局部可见 | 全链路可见 | 不可只看单点 |
| 适用场景 | 快速落地 | 长期治理 | 建议组合使用 |

## 代码

```ts
import { computed, reactive, watch } from 'vue'

const state = reactive({ count: 0, price: 99 })
const total = computed(() => state.count * state.price)

watch(() => state.count, (next, prev) => {
  console.log('count changed:', prev, '->', next)
})
```

### 示例说明

- 这段代码解决什么问题：演示 `reactive + computed + watch` 的最小响应式闭环。
- 关键语句或参数含义：`computed` 自动缓存依赖结果，`watch` 用于处理副作用。
- 常见错误写法：把副作用写进 `computed`，导致语义混乱和调试困难。

## 真实场景示例

- 场景描述：核心页面迭代中出现跨组件状态与渲染时序问题。
- 问题表现：功能可用但偶发抖动、错渲染或发布回退。
- 处理方案：先收敛边界，再做最小修复，最后补自动化验证。
- 验证方式：通过关键路径回归、性能指标和线上监控确认生效。

## 相关面试题

1. 双向数据绑定的原理
- 标准答案：结论是双向绑定是“视图改动更新状态 + 状态改动驱动视图”。机制上输入事件触发状态更新，响应式系统再触发渲染。场景上表单场景效率高，但复杂表单要配合校验与状态分层防止联动失控。
- 考察点：是否能按阶段讲清核心流程、关键数据结构和边界条件。

2. 使用 Object.defineProperty() 来进行数据劫持有什么缺点？
- 标准答案：结论是 `Object.defineProperty` 无法直接监听新增/删除属性和数组索引变化。机制上它基于属性劫持而非对象级代理，初始化成本也更高。场景上这正是 Vue 3 转向 Proxy 的主要原因。
- 考察点：是否能给出可执行步骤、关键 API 和常见踩坑点。

3. Vue 中给 data 中的对象属性添加一个新的属性时会发生什么？如何解决？
- 标准答案：结论是 Vue 2 中直接新增属性默认不具备响应式。机制上初始化时未做 getter/setter 劫持，后加属性不会触发依赖。场景上 Vue 2 用 `Vue.set`，Vue 3 用 Proxy 可直接追踪新增属性。
- 考察点：是否能给出可执行步骤、关键 API 和常见踩坑点。

4. Vue 中封装的数组方法有哪些，其如何实现页面更新
- 标准答案：结论是 Vue 2 对 `push/pop/shift/unshift/splice/sort/reverse` 做了拦截。机制上这些方法内部会手动触发依赖通知以更新视图。场景上直接按索引赋值在 Vue 2 不可靠，通常改用 `splice`。
- 考察点：是否能给出可执行步骤、关键 API 和常见踩坑点。

5. Vue 是如何收集依赖的？
- 标准答案：结论是依赖收集发生在渲染读取响应式数据时。机制上当前活跃副作用函数（effect/watcher）会在 `track` 阶段记录到依赖表，数据写入时在 `trigger` 阶段统一调度。场景上这保证了只更新真正用到该状态的组件。
- 考察点：是否能给出可执行步骤、关键 API 和常见踩坑点。

6. delete 和 Vue.delete 删除数组的区别
- 标准答案：结论是 Vue 2 中 `delete` 不能触发响应式，`Vue.delete` 可以。机制上后者会补发依赖通知。场景上 Vue 3 使用 Proxy 后普通 `delete` 也能被追踪，但仍建议保持数据结构稳定。
- 考察点：是否能从定义、触发时机、性能影响和适用场景四个维度做对比。

7. vue 如何监听对象或者数组某个属性的变化
- 标准答案：结论是可以用 `watch` 精确监听路径或 getter。机制上通过深度监听 `deep` 或返回具体字段函数实现订阅。场景上深监听成本高，优先监听最小必要字段。
- 考察点：是否能给出可执行步骤、关键 API 和常见踩坑点。

8. defineProperty 和 proxy 的区别
- 标准答案：结论是 `defineProperty` 是属性级劫持，`Proxy` 是对象级拦截。机制上 Proxy 能覆盖新增/删除、数组索引和更多元操作。场景上 Vue 3 选 Proxy 以降低边界漏洞并提升一致性。
- 考察点：是否能从定义、触发时机、性能影响和适用场景四个维度做对比。

9. Vue3.0 为什么要用 proxy？
- 标准答案：结论是为了完整拦截对象操作并减少响应式边界问题。机制上 Proxy 支持 `get/set/delete/has` 等统一拦截，依赖收集更自然。场景上大型状态树、动态字段和数组操作更稳定。
- 考察点：是否能说明结论背后的原因、成立条件和反例。
