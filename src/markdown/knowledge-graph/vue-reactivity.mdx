## 概念定位

Vue 3 响应式系统的本质是：

- 在“读取数据”时收集依赖
- 在“写入数据”时触发依赖更新
- 用调度器把更新合并成可控的批处理

它不是语法糖，而是整个渲染调度链路的起点。理解这层，才能解释为什么某些状态更新不生效、为什么会出现更新风暴、为什么 `computed` 有缓存收益。

## 内部数据结构

### 依赖桶（核心）

Vue 内部可抽象为：

- `targetMap: WeakMap<object, Map<key, Dep>>`
- `Dep: Set<ReactiveEffect>`

含义：某个响应式对象（target）的某个属性（key）对应一组副作用函数（effect）。

### 当前活跃副作用

- `activeEffect`：当前正在执行并可被收集依赖的 effect
- effect stack：解决 effect 嵌套场景，确保收集上下文正确

### 响应式包装类型

- `reactive()`：对象级代理（Proxy）
- `ref()`：值容器，内部通过 `.value` 做依赖追踪
- `shallowReactive()` / `shallowRef()`：只追踪第一层
- `readonly()` / `shallowReadonly()`：只读代理
- `markRaw()`：显式跳过代理

## 工作机制

### 1) 读取阶段：track

当模板渲染或 effect 函数读取响应式属性时，触发 `track(target, key)`：

1. 从 `targetMap` 找到 `target` 对应的 Map
2. 从 Map 找到 `key` 对应的 Dep
3. 把 `activeEffect` 放入 Dep

### 2) 写入阶段：trigger

当属性变更时触发 `trigger(target, key, type)`：

1. 找到 `target+key` 对应的 Dep
2. 收集需要重新执行的 effects
3. 对普通 effect 走调度器（scheduler）合并执行
4. 对 `computed` 等特殊 effect 按优先级处理

### 3) 渲染更新联动

组件渲染函数本身也是 effect。状态变更后不会立刻同步重渲染，而是进入调度队列，批量刷新。

## 关键知识点

### ref 与 reactive 的边界

| 对比项 | `ref` | `reactive` |
| --- | --- | --- |
| 适用数据 | 基本类型和对象都可 | 仅对象/数组/Map/Set |
| 访问方式 | `.value` | 直接属性访问 |
| 替换整体值 | 方便（直接改 `.value`） | 通常保持同一代理对象 |
| 在模板中 | 自动解包 | 原样使用 |

### computed（计算属性）为何高效

- `computed` 内部是 lazy effect
- 通过 `dirty` 标记控制是否重新计算
- 依赖未变时直接返回缓存

这就是“同一渲染周期多次读取 computed 不会多次执行”的核心原因。

### watch 与 watchEffect

- `watchEffect`：自动收集依赖，适合快速建立响应链
- `watch`：显式指定依赖源，适合精细控制与比对新旧值

`watch` 的 `flush` 时机关键：

- `pre`：组件更新前（默认）
- `post`：组件更新后（拿到最新 DOM）
- `sync`：同步执行（慎用，易放大连锁更新）

### 响应式常见“失效”场景

1. 对 `reactive` 对象做结构赋值解构，丢失代理引用。
2. 把响应式对象转成普通对象后再写入。
3. 使用 `shallow*` 却期待深层自动追踪。
4. 在 class 实例或第三方对象上混用代理导致身份判断混乱。

### Map / Set 的追踪细节

Vue 对集合类型做了方法级劫持：`get/set/has/delete/forEach/iterate` 等都有对应依赖追踪逻辑。高频集合更新场景要特别关注迭代依赖触发成本。

## 对页面性能的影响

- 响应式粒度越粗，越容易触发无效更新。
- 依赖链越深，单次状态变更传播成本越高。
- `watch` 滥用和同步 flush 会抬高主线程负载，恶化 INP（Interaction to Next Paint）。

## 观测与排障

### 快速定位路径

1. 先确认“触发了没有”：状态写入是否走到代理。
2. 再确认“收集了没有”：依赖是否在正确上下文被 track。
3. 再确认“调度了没有”：effect 是否进入队列并执行。
4. 最后确认“渲染了没有”：组件更新是否被条件分支短路。

### 实战排障剧本

- 现象：状态改了，UI 不变
- 排查：
1. 是否解构丢失响应式引用
2. 是否用了 `markRaw` / `shallow*`
3. `watch` 是否监听错源
4. 组件是否被缓存/条件渲染拦截

### 监控建议

- 大页面关注：渲染次数、effect 执行次数、长任务占比
- 高频交互关注：单次交互触发的组件更新扇出数量

## 实战优化策略

1. 状态分层：热状态与冷状态分离，避免一个大对象驱动整页更新。
2. 限制深层响应式：大对象可使用 `shallowRef + replace` 策略。
3. `computed` 优先替代模板内重复表达式计算。
4. `watch` 只监听必要源，避免“全对象深监听”。
5. 高频场景结合节流与批处理，避免每次输入都触发重链路更新。

## 常见误区

- “Vue 响应式是脏检查”：错误，Vue 3 主要是 Proxy 依赖追踪模型。
- “ref 比 reactive 慢”：错误，性能取决于访问模式和更新扇出，不是 API 名称。
- “watch 越多越灵活”：错误，watch 过多通常意味着状态边界设计出了问题。

## 进阶深挖（第二轮）

### 响应式高级边界

- shallowRef 与 shallowReactive 适合大对象浅层追踪场景。
- markRaw 可避免对象被代理，常用于第三方实例或大型只读对象。
- customRef 可定制 track 与 trigger 节奏，适合防抖输入类场景。
- effectScope 有助于集中回收副作用，避免组件外泄漏。

### 高频故障样例

1. 大对象深层代理导致初始化开销异常。
2. 非必要深追踪引发频繁 trigger。
3. 副作用作用域未回收导致内存增长。

### 追问扩展

1. 什么时候该用 shallowRef 而不是 ref？

当你只关心引用层变化，不希望深层属性触发追踪时。

2. markRaw 的收益和风险？

收益是减少代理开销，风险是对象内部变化不再触发响应式更新。

## 面试补充

### 30 秒说明

Vue 3 响应式核心是 `track/trigger`：读取时收集依赖，写入时触发副作用。`ref` 适合值容器，`reactive` 适合对象代理；`computed` 通过 dirty+缓存减少重复计算；`watch` 负责副作用联动但要控制粒度和 flush 时机。线上问题大多来自依赖收集边界错误或调度策略失控。

### 高频追问

1. 为什么解构 `reactive` 后经常“不响应”？

因为解构得到的是普通值或普通引用，脱离了 Proxy 访问路径，后续写入不会触发原对象依赖。

2. `computed` 和 `watch` 的本质区别是什么？

`computed` 产出“可缓存值”，用于派生状态；`watch` 产出“副作用行为”，用于请求、日志、同步等。

3. 什么时候优先用 `shallowRef`？

大对象、外部库实例、频繁整体替换场景。它能减少深层追踪开销，但需手动控制更新边界。

4. `watch(..., { flush: 'post' })` 什么时候必须用？

当副作用依赖更新后的 DOM 状态时（如测量尺寸、滚动定位）。

5. 如何避免响应式更新风暴？

控制状态粒度、减少深监听、避免同步链式写入、把高频更新合并到批处理窗口。

6. Vue 2 和 Vue 3 响应式底层差异？

Vue 2 主要基于 `Object.defineProperty`，Vue 3 基于 Proxy，支持更完整的数据类型和更灵活的拦截语义。
