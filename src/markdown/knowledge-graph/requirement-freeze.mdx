## 概念定位

需求评审与方案冻结是交付稳定性的起点。它的目标不是“文档齐全”，而是提前锁定边界、风险和验收口径。

这个节点直接决定后续返工率、延期率和技术债规模。

## 工作机制

1. 需求评审时同时定义功能边界与非功能目标（性能、安全、可观测）。
2. 把“必须做 / 可以降级 / 明确不做”写入方案冻结单。
3. 为关键链路给出验收指标和回滚条件，避免上线后口径争议。
4. 拆分交付里程碑，标记外部依赖和阻塞风险。
5. 冻结后变更必须走变更评审，记录影响范围和资源成本。

## 关键知识点

### 必须掌握

- 方案冻结不是禁止变化，而是让变化有成本和流程。
- 非功能需求要前置，不应留到上线前“补票”。
- 验收标准要可量化，避免“感觉上线可以”。
- 跨团队依赖要有 SLA（Service Level Agreement）约束。
- 需求口径要和埋点、监控、告警对齐。
- 风险清单必须明确负责人和止损动作。

### 选型与取舍

- 冻结越严格，变更成本越高，但稳定性越好。
- 冻结太松看起来灵活，最终常演变为持续返工。
- 前置评审投入时间，会换来后期发布风险下降。

## 对页面性能与稳定性的影响

- 节点稳定时，前端工程化 的链路可预测性会明显提升，发布和排障成本更可控。
- 节点失稳时，问题会向上下游扩散，常表现为性能回退、交互异常或交付风险叠加。
- 该节点优化应和上下游协同验证，避免“局部优化把成本转移给后续节点”。

## 观测与排障

### 核心指标

- 需求变更率
- 返工工时占比
- 里程碑延期率
- 上线后需求缺口工单数

### 排障顺序

1. 先复盘问题是否源于边界不清。
2. 检查冻结单是否包含非功能目标。
3. 检查变更流程是否被绕过。
4. 补齐影响评估模板和审批节点。
5. 将高频问题沉淀到评审清单。

### 常见故障场景

- “每周改需求导致进度失控”：常见是冻结机制缺失。
- “上线后才发现性能不达标”：常见是非功能目标未前置。
- “业务说做了但验收不过”：常见是验收口径不一致。
- “跨团队互相等”：常见是依赖与 SLA 未定义。

## 常见误区

- 只冻结功能列表，不冻结质量目标。
- 把变更流程当形式，不记录影响。
- 风险识别后无负责人和动作。

## 题库深挖补充

### 高频对比与边界（基于题库）

- 需求冻结的核心是边界和验收口径，不是禁止变更。
- 仓库分层要定义依赖方向，避免“能 import 就 import”。
- 依赖治理要覆盖安全漏洞、体积增长、升级节奏三条线。
- 配置中心要支持审计与回滚，避免环境漂移。
- 复盘输出必须可执行：负责人、截止日期、验收标准缺一不可。
- 面试常问“工程化价值”，答案应覆盖效率、稳定性、可追踪性。

### 线上排障剧本

1. 先判断故障是流程缺失还是实现缺陷。
2. 检查需求/架构/依赖/配置是否存在边界越权。
3. 拉出变更时间线和责任链路。
4. 给出短期止血 + 中期治理动作。
5. 把结论沉淀为规则并跟踪复发率。

### 高频追问补充（附答案）

1. 为什么需求评审要带非功能指标？

因为性能、安全、可观测会影响架构路径，后置补救成本高。

2. 依赖治理最容易被忽视的点？

间接依赖和供应链风险，经常不在日常视野内。

3. 配置漂移为什么危险？

同代码在不同环境表现不同，事故定位会显著变慢。

## 进阶追问（补充）

1. 为什么工程化问题常在高峰期集中暴露？

因为流量放大后，缓存错配、发布窗口、告警噪声和回滚能力短板会被同时放大。

2. 门禁越严越好吗？

不是。要按风险分级阻断，既拦高风险问题，也要控制交付节奏。

3. 如何判断一次事故复盘是否有效？

看同类问题复发率是否下降，以及改进项是否转化为自动化规则。

## 案例追问加深（第三轮）

### 真实故障片段

- 故障表现：发布流水线显示通过，但上线后核心指标快速恶化。
- 排查结论：门禁覆盖了代码质量，却没有覆盖发布后观测与缓存一致性。
- 治理动作：把灰度阈值、缓存一致性检查和回滚演练纳入同一发布清单。

### 追问补充

1. 为什么工程化问题常在业务高峰暴露？

高峰流量会放大流程短板，平时隐性的风险会在短时间集中显现。

2. 如何衡量工程化治理是否真的有效？

看变更失败率、恢复时长和同类事故复发率是否持续下降。

## 面试补充

### 30 秒说明

方案冻结的本质是“边界共识 + 风险前置 + 验收可量化”。这是工程化链路稳定的第一步。

### 高频追问

1. 为什么需求冻结能提升交付质量？

它让边界和验收口径提前统一，减少后续返工和争议。

2. 冻结后还能改需求吗？

可以，但必须走变更评审并明确成本、影响和责任。

3. 非功能指标为什么要在评审阶段确定？

因为它们会影响架构与实现路径，后置补救成本极高。

4. 如何避免评审流于形式？

把风险清单、负责人、验收指标和回滚条件都写入可追踪文档。
