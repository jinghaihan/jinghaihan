## 概念定义

- 本质定义：文档声明与 head 配置用于定义浏览器如何解析文档，以及解析前需要建立的页面上下文。
- 解决问题：避免模式错误、编码错误、视口错误导致的兼容和展示异常。
- 边界说明：它不直接做渲染优化，但会决定后续加载链路是否在正确前提下运行。

## 核心机制拆解

### 1) 机制/流程

1. 浏览器读取 `<!doctype html>`，决定标准模式或混杂模式。
2. 解析 `head`，建立编码、标题、视口、索引策略等元信息。
3. 根据 `head` 中的 `link`、`script`、`meta` 发现关键资源和策略。
4. 进入 `body` 解析与渲染流程。

### 2) 关键条件与触发点

- 条件 A：Doctype 缺失或不规范，容易触发混杂模式。
- 条件 B：页面、接口、数据库编码不一致，容易出现乱码。

### 3) 结果与影响

- 对性能/稳定性/可维护性的影响：入口配置统一时跨浏览器行为更稳定；入口配置分裂时排障成本会持续增加。

## 相似概念对比

| 维度 | Doctype | `head` 元信息 | 结论 |
|---|---|---|---|
| 定义 | 声明解析模式 | 定义文档元数据与入口配置 | 二者共同决定起始上下文 |
| 触发时机 | 文档第一行 | 文档头部解析阶段 | 都早于正文渲染 |
| 性能影响 | 决定解析规则一致性 | 影响关键资源发现和显示正确性 | 基础配置错误会放大后续问题 |
| 适用场景 | 所有页面必须声明 | 所有页面必须配置 | 属于页面基础设施 |

## 代码

```html
<!doctype html>
<html lang="zh-CN">
  <head>
    <meta charset="UTF-8" />
    <meta name="viewport" content="width=device-width, initial-scale=1" />
    <meta name="description" content="商品详情页" />
    <title>商品详情</title>
  </head>
  <body>
    <h1>商品详情</h1>
  </body>
</html>
```

### 示例说明

- 这段代码解决什么问题：确保标准模式解析并补齐编码、视口、描述和标题等关键入口信息。
- 关键语句或参数含义：`<!doctype html>` 控制模式；`charset` 控制解码；`viewport` 控制移动端视口。
- 常见错误写法：省略 Doctype、错误编码声明、多个冲突 `viewport`。

## 真实场景示例

- 场景描述：同一页面在 Chrome 和 Safari 表现不一致，且出现部分中文乱码。
- 问题表现：布局偏移、字体异常、中文偶发乱码。
- 处理方案：统一 Doctype，统一页面/接口/数据库编码，清理冲突 `meta`。
- 验证方式：多浏览器回归 + 弱网/代理链路回归 + 编码一致性检查。

## 常见误区

- 误区：Doctype 是历史写法，删掉影响不大。
  - 正解：Doctype 决定解析模式，删掉会提升混杂模式风险。
- 误区：乱码只可能是后端问题。
  - 正解：前端页面声明编码错误同样会导致乱码。

## 关联知识点

- 依赖：HTML（HyperText Markup Language）体系。
- 影响：加载与阻塞模型、script 加载策略（defer / async）。
- 对比：标准模式与混杂模式。
- 组合：语义化与可访问性、跨浏览器兼容策略。

## 相关面试题

1. DOCTYPE(⽂档类型) 的作⽤
   - 标准答案：结论是 Doctype 用于告诉浏览器按哪套规则解析文档。机制上它决定标准模式还是混杂模式。场景上未声明或声明错误会导致跨浏览器差异。
   - 考察点：是否理解“解析模式 -> 展示一致性”的因果关系。

2. 常⽤的 meta 标签有哪些
   - 标准答案：常用 `meta` 包括 `charset`、`viewport`、`description`、`keywords`、`robots`、`refresh`。机制上分别控制编码、视口、摘要、检索策略和刷新跳转。场景上应按业务目标配置并避免冲突。
   - 考察点：是否能把标签功能与业务目标对应起来。

3. head 标签有什么作用，其中什么标签必不可少？
   - 标准答案：结论是 `head` 用于承载文档元信息和关键资源入口。机制上浏览器在此阶段建立页面基础上下文。场景上 `title` 是 `head` 中必需元素，`meta charset` 和 `meta viewport` 是高频基础配置。
   - 考察点：是否掌握文档声明与 head 配置的最小集合。

4. 文档声明（Doctype）和 `<!Doctype html>` 有何作用? 严格模式与混杂模式如何区分？它们有何意义?
   - 标准答案：结论是 `<!doctype html>` 用于触发标准模式。机制上标准模式遵循规范，混杂模式偏向历史兼容行为。场景上现代项目应固定标准模式以减少布局和脚本兼容风险。
   - 考察点：是否能解释模式差异和工程意义。
   - 追问 1：如何在代码里判断当前文档模式？
   - 标准答案 1：结论是可通过 `document.compatMode` 判断。机制上 `CSS1Compat` 代表标准模式，`BackCompat` 代表混杂模式。场景上它常用于线上兼容问题快速定位。

5. 浏览器乱码的原因是什么？如何解决？
   - 标准答案：结论是乱码本质是“同一字节流被错误编码规则解读”。机制上常见于页面声明编码、接口响应头编码、数据库编码三者不一致。场景上要统一存储和传输编码，并在页面显式声明 `UTF-8`。
   - 考察点：是否具备编码链路排障能力。
