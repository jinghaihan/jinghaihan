## 概念定义

- 本质定义：文档声明与 `head` 配置定义了 HTML（HyperText Markup Language）文档的解析模式、编码、标题、基础 URL（Uniform Resource Locator）和元信息入口。
- 解决问题：保证浏览器按正确模式解析页面，并在最早阶段拿到编码、视口、描述、资源基准路径等关键信息。
- 边界说明：本节点只管“文档入口怎么声明”，不展开脚本执行顺序和资源阻塞细节。

## 核心机制拆解

### 1) 机制/流程

1. 浏览器先读取 `<!doctype html>`，决定走标准模式还是混杂模式。
2. 解析 `html` 根节点和 `head`，建立语言、编码、标题、视口和索引信息。
3. 根据 `meta`、`link`、`base`、`script` 发现关键资源和基础策略。
4. 解析 `body`，再进入后续的资源加载、样式计算和脚本执行流程。

### 2) 关键条件与触发点

- 条件 A：Doctype 缺失或写错时，浏览器可能退回混杂模式，布局兼容问题会立刻放大。
- 条件 B：`charset`、响应头编码和数据源编码不一致时，会出现乱码。
- 条件 C：`base` 配错时，相对路径资源和链接会整体跑偏。

### 3) 结果与影响

- 对性能/稳定性/可维护性的影响：入口信息统一，线上行为更稳定；入口层配置混乱，后续所有“样式错位、资源 404、乱码、SEO 信息错误”都会变得难排查。

## 相似概念对比

| 维度 | `<!doctype html>` | `head` 元信息 | 结论 |
|---|---|---|---|
| 关注点 | 解析模式 | 文档元数据和资源入口 | 二者共同决定页面起跑线 |
| 生效时机 | 文档第一行 | HTML 头部解析阶段 | 都早于正文渲染 |
| 影响范围 | 盒模型、渲染兼容性 | 编码、标题、视口、链接、索引策略 | 都属于基础设施 |
| 适用场景 | 所有页面必须声明 | 所有页面都应有稳定最小集合 | 不该缺省 |

## 代码

```html
<!doctype html>
<html lang="zh-CN">
  <head>
    <meta charset="UTF-8" />
    <meta name="viewport" content="width=device-width, initial-scale=1" />
    <meta name="description" content="商品详情页" />
    <base href="https://example.com/shop/" />
    <title>商品详情</title>
  </head>
  <body>
    <h1>商品详情</h1>
  </body>
</html>
```

### 示例说明

- 这段代码解决什么问题：把解析模式、语言、编码、视口、描述和相对路径基准一次说明白。
- 关键语句或参数含义：`<!doctype html>` 触发标准模式；`charset` 决定页面解码；`viewport` 控制移动端视口；`base` 统一相对路径基准。
- 常见错误写法：Doctype 不在第一行、多个冲突 `viewport`、`base` 指向错误目录、只写 `h1` 不写 `title`。

## 真实场景示例

- 场景描述：活动页在 Chrome 和 Safari 表现不一致，图片和脚本路径偶发 404，部分中文还会乱码。
- 问题表现：布局偏移、资源地址拼错、标题和分享摘要不正确。
- 处理方案：统一 Doctype，校对 `head` 最小集合，梳理 `base` 是否必要，并统一页面与接口编码。
- 验证方式：多浏览器回归、抓包核对响应头编码、检查页面标题和相对路径解析结果。

## 常见误区

- 误区：Doctype 是历史包袱，删掉也没事。
- 正解：Doctype 决定解析模式，删掉会引出混杂模式问题。
- 误区：`title` 和 `h1` 选一个就够。
- 正解：`title` 面向标签页、搜索摘要和分享信息，`h1` 面向正文结构，两者职责不同。
- 误区：乱码只可能是后端问题。
- 正解：页面 `charset`、响应头、模板文件和数据库编码只要有一环不一致，前端照样会乱码。

## 关联知识点

- 依赖：HTML 体系。
- 影响：加载与阻塞模型、语义化与可访问性、兼容策略。
- 对比：标准模式与混杂模式。
- 组合：SEO 元信息、分享卡片信息、资源基准路径管理。

## 相关面试题

1. 什么是HTML？
- 标准答案：HTML 是用来描述网页结构和内容的标记语言。机制上浏览器通过标签、属性和层级关系理解页面结构，再交给样式和脚本继续处理。场景上它负责把“标题、段落、表单、图片、链接”这些内容组织成浏览器能识别的文档。
- 考察点：是否能把 HTML 说成“结构语言”，而不是把它和样式、脚本混在一起。

2. 解释HTML文档的基本结构
- 标准答案：标准结构至少包括 `<!doctype html>`、`html`、`head` 和 `body`。机制上 `head` 承载元信息和资源入口，`body` 承载可见内容。场景上一个最小可用页面至少要把解析模式、编码和标题交代清楚。
- 考察点：是否知道各层标签在文档中的职责。

3. 什么是HTML标签和元素？
- 标准答案：标签是写在尖括号里的标记，比如 `<p>`；元素是“开始标签 + 内容 + 结束标签”组成的完整结构，比如 `<p>文本</p>`。机制上浏览器真正构建的是元素节点而不是孤立标签。场景上理解这个区别后，才不容易把“空元素”“闭合规则”“属性作用域”说混。
- 考察点：是否具备基础概念准确度。

4. HTML属性是什么？如何使用？
- 标准答案：属性是写在开始标签里的键值对，用来补充元素信息，比如 `id`、`class`、`src`、`href`、`alt`。机制上属性会影响解析、可访问性、资源地址和行为。场景上图片要写 `alt`，链接要写 `href`，脚本和样式的外链标签也靠属性声明资源位置。
- 考察点：是否理解属性不是装饰，而是行为和语义声明。

5. HTML中的注释如何写？
- 标准答案：HTML 注释写法是 `<!-- 注释内容 -->`。机制上浏览器解析时会忽略注释，不会渲染到页面。场景上它适合标记结构段落或临时排查问题，但不该放敏感信息，因为查看源码仍然能看到。
- 考察点：是否掌握最基础的语法和安全边界。

6. DOCTYPE(⽂档类型) 的作⽤
- 标准答案：Doctype 的作用是告诉浏览器按什么模式解析文档。机制上现代页面通常用 `<!doctype html>` 触发标准模式，避免混杂模式下的历史兼容行为。场景上不写或写错时，最常见的问题就是盒模型和布局表现不一致。
- 考察点：是否理解 Doctype 和浏览器模式的关系。

7. 什么是HTML的DOCTYPE声明？
- 标准答案：HTML5 的 Doctype 声明就是 `<!doctype html>`，必须放在文档第一行。机制上它不再区分复杂的 DTD（Document Type Definition）版本写法，核心作用仍然是触发标准模式。场景上现代项目统一使用这一写法就够了。
- 考察点：是否知道现代 Doctype 的标准写法和位置要求。

8. 常⽤的 meta 标签有哪些
- 标准答案：常见的 `meta` 包括 `charset`、`viewport`、`description`、`keywords`、`robots`、`author`、`http-equiv="refresh"` 以及社交分享场景的 Open Graph（Open Graph Protocol）标签。机制上它们分别控制编码、视口、摘要、检索和部分浏览器行为。场景上至少要把 `charset`、`viewport`、`description` 这几项配置清楚。
- 考察点：是否能按作用分类，而不是只背标签名。

9. 什么是HTML的meta标签？
- 标准答案：`meta` 用来描述“关于文档的信息”，而不是正文内容。机制上浏览器和搜索系统会在解析 `head` 时读取这些元数据，用来决定编码、缩放、摘要和分享展示。场景上移动端视口、页面描述和社交卡片都依赖 `meta`。
- 考察点：是否理解元数据和正文内容的边界。

10. HTML中的head标签包含哪些内容？
- 标准答案：`head` 里通常包含 `title`、`meta`、`link`、`style`、`script`、`base` 等元素。机制上这里承载的不是正文，而是页面运行前要先知道的元信息和资源关系。场景上编码、视口、样式表、站点图标和页面标题都应该在这里配置。
- 考察点：是否知道 `head` 是资源和元信息入口。

11. head 标签有什么作用，其中什么标签必不可少？
- 标准答案：`head` 的作用是提供文档元信息、资源入口和浏览器策略声明。机制上浏览器在渲染正文前会先读取这里的配置。场景上 `title` 是规范要求的必备元素，`meta charset` 和 `meta viewport` 是现代前端的高频基础配置。
- 考察点：是否知道 `head` 的最小集合，而不只是罗列标签。

12. HTML中的base标签有什么作用？
- 标准答案：`base` 用来定义文档里所有相对 URL 的基准地址或默认跳转目标。机制上它会影响链接、图片、脚本、表单等相对路径解析。场景上子目录部署或静态资源前缀统一时会用到，但一旦配置错误，会导致整页相对路径一起出错。
- 考察点：是否理解 `base` 的全局影响范围。

13. 文档声明（Doctype）和 `<!Doctype html>` 有何作用? 严格模式与混杂模式如何区分？它们有何意义?
- 标准答案：`<!doctype html>` 的作用是让浏览器进入标准模式。机制上标准模式遵循现代规范，混杂模式会保留大量历史兼容行为；可通过 `document.compatMode` 区分，`CSS1Compat` 是标准模式，`BackCompat` 是混杂模式。场景上现代项目应固定标准模式，减少跨浏览器差异。
- 考察点：是否能把声明、模式判断和工程意义串起来。

14. 浏览器乱码的原因是什么？如何解决？
- 标准答案：乱码本质是“同一字节流被错误编码规则解读”。机制上常见原因是页面 `charset`、响应头 `Content-Type`、模板文件保存编码和数据库编码不一致。场景上排查时要沿着“页面声明 -> 接口响应 -> 存储源”一条链去核对，并统一为 `UTF-8`。
- 考察点：是否具备编码链路排障能力。
