## 概念定义

- 本质定义：发丝线问题是高像素密度屏下 1 CSS 像素边框视觉过粗或发虚的适配问题。
- 解决问题：实现多设备一致的细边框和小字号展示效果。
- 边界说明：整体响应式缩放策略在“响应式单位与媒体查询”节点。

## 核心机制拆解

### 1) 机制/流程

1. 识别设备像素比（DPR，Device Pixel Ratio）。
2. 选择发丝线方案：`transform: scale`、伪元素缩放、媒体查询分设备处理。
3. 字号小于 12px 场景结合缩放或设计替代策略。

### 2) 关键条件与触发点

- 条件 A：直接写 `border: 1px` 在高 DPR 设备上会偏粗。
- 条件 B：小字号强行压缩可能影响可读性和可访问性。

### 3) 结果与影响

- 对性能/稳定性/可维护性的影响：方案统一时多端边框一致；方案分裂时组件边界风格不统一。

## 相似概念对比

| 维度 | 直接 `1px` | 发丝线方案 | 结论 |
|---|---|---|---|
| 视觉一致性 | 中 | 高 | 高 DPR 场景优先发丝线 |
| 实现复杂度 | 低 | 中 | 需封装复用 |
| 兼容成本 | 低 | 中 | 要做设备回归 |
| 场景 | 普通边框 | 高精度分割线 | 按视觉要求选型 |

## 代码

```css
.hairline::after {
  content: '';
  position: absolute;
  left: 0;
  right: 0;
  bottom: 0;
  border-bottom: 1px solid #e5e7eb;
  transform: scaleY(.5);
  transform-origin: 0 100%;
}
```

### 示例说明

- 这段代码解决什么问题：用伪元素缩放生成更细的底部分割线。
- 关键语句或参数含义：`scaleY(.5)` 在高像素密度设备上可得到接近物理 1 像素效果。
- 常见错误写法：未设置 `transform-origin` 导致线条偏移。

## 真实场景示例

- 场景描述：移动端列表分割线在高端机上明显偏粗。
- 问题表现：视觉稿要求“细线”，实际显示为粗线。
- 处理方案：统一封装发丝线工具类，按设备像素比验证。
- 验证方式：真机截图比对、不同 DPR 机型回归。

## 常见误区

- 误区：`0.5px` 所有浏览器都完全一致。
  - 正解：支持和渲染细节仍有差异，需结合回归验证。
- 误区：小于 12px 字体只靠缩放就够。
  - 正解：还需考虑可读性、可访问性和设计规范。

## 关联知识点

- 依赖：响应式单位与媒体查询。
- 影响：布局视觉一致性和移动端细节质量。
- 对比：图像切图分割线方案。
- 组合：图形节点中的边框绘制技巧。

## 相关面试题

1. 画一条0.5px的线
   - 标准答案：可用伪元素 + `transform: scale`、媒体查询按 DPR 调整，或在支持环境直接写 `0.5px`。机制上核心是把 CSS 线宽映射到更细物理像素效果。场景上常用于移动端分割线。
   - 考察点：是否掌握至少一种稳定实现。

2. 设置小于12px的字体
   - 标准答案：常见方案是字体保持可读最小值，再通过 `transform: scale` 视觉缩小。机制上是视觉缩放而非真实排版字号变化。场景上需谨慎使用并做可读性验收。
   - 考察点：是否考虑可访问性边界。

3. 如何解决 1px 问题？
   - 标准答案：结论是本质在于 CSS 像素和物理像素映射差异。机制上可通过发丝线封装、DPR 媒体查询、矢量方案等处理。场景上推荐组件级统一方案，避免页面各写一套。
   - 考察点：是否能讲清根因与工程化解法。
