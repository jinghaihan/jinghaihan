## 概念定义

- 本质定义：该节点聚焦 高负载交互保持响应。
- 解决问题：把高频面试问题转成可复述、可落地的机制说明。
- 边界说明：不替代其他节点的专属机制域。

## 核心机制拆解

### 1) 机制/流程

1. 明确输入状态和触发条件。
2. 进入框架调度或编译阶段。
3. 执行更新并产出结果。
4. 用回归验证结果是否稳定。

### 2) 关键条件与触发点

- 条件 A：组件边界和数据来源必须清晰。
- 条件 B：触发时机与副作用清理要成对出现。

### 3) 结果与影响

- 对性能/稳定性/可维护性的影响：机制清晰时可预测，机制混乱时会出现渲染回退、状态错乱或协作成本上升。

## 相似概念对比

| 维度 | 同步渲染 | 并发渲染 | 结论 |
|---|---|---|---|
| 定义 | 关注单点能力 | 关注对照能力 | 职责边界不同 |
| 触发时机 | 偏局部触发 | 偏链路触发 | 需要按场景选择 |
| 性能影响 | 局部可见 | 全链路可见 | 不可只看单点 |
| 适用场景 | 快速落地 | 长期治理 | 建议组合使用 |

## 代码

```tsx
import { useMemo, useState, useTransition } from 'react'

export default function Search({ list }: { list: string[] }) {
  const [input, setInput] = useState('')
  const [keyword, setKeyword] = useState('')
  const [isPending, startTransition] = useTransition()

  const filtered = useMemo(
    () => list.filter(item => item.includes(keyword)),
    [list, keyword],
  )

  return (
    <>
      <input
        value={input}
        onChange={(e) => {
          const v = e.target.value
          setInput(v)
          startTransition(() => setKeyword(v))
        }}
      />
      {isPending && <span>calculating...</span>}
      <ul>{filtered.map(item => <li key={item}>{item}</li>)}</ul>
    </>
  )
}
```

### 示例说明

- 这段代码解决什么问题：把高成本过滤更新降级，保证输入框优先响应。
- 关键语句或参数含义：`startTransition` 标记低优先级更新。
- 常见错误写法：把所有状态都包进 transition，导致反馈延迟。

## 真实场景示例

- 场景描述：核心页面迭代中出现跨组件状态与渲染时序问题。
- 问题表现：功能可用但偶发抖动、错渲染或发布回退。
- 处理方案：先收敛边界，再做最小修复，最后补自动化验证。
- 验证方式：通过关键路径回归、性能指标和线上监控确认生效。

## 相关面试题

1. 对 React-Fiber 的理解，它解决了什么问题？
- 标准答案：结论是 Fiber 是 React 的可中断渲染架构。机制上把更新拆成可调度的小单元，允许高优任务插队，提升交互响应。场景上复杂页面输入、动画与大列表并存时收益明显。
- 考察点：是否能给出明确结论，并补充机制和业务场景。

2. 对虚拟 DOM 的理解？虚拟 DOM 主要做了什么？虚拟 DOM 本身是什么？
- 标准答案：结论是虚拟 DOM 是对真实 DOM 的内存描述，用来做差异计算。机制上先生成新旧 vnode，再通过 Diff 找最小变更并提交真实 DOM。场景上它提升的是“跨平台抽象和可维护性”，不是任何场景都绝对更快。
- 考察点：是否能给出明确结论，并补充机制和业务场景。

3. React diff 算法的原理是什么？
- 标准答案：结论是 React Diff 采用同层比较策略。机制上不同类型节点直接替换，同类型比较属性与子节点，列表依赖 key 定位复用。场景上稳定 key 是避免重建和状态错位的关键。
- 考察点：是否能按阶段讲清核心流程、关键数据结构和边界条件。

4. React key 是干嘛用的 为什么要加？key 主要是解决哪一类问题的
- 标准答案：结论是 key 用于标识同层节点身份。机制上 Diff 根据 key 判断复用、移动或删除。场景上使用可变 index 作为 key 会导致输入框状态错位、动画抖动等问题。
- 考察点：是否能说明结论背后的原因、成立条件和反例。

5. 虚拟 DOM 的引入与直接操作原生 DOM 相比，哪一个效率更高，为什么
- 标准答案：结论是不存在绝对谁更快。机制上小规模一次性更新直接 DOM 可能更快，复杂交互下虚拟 DOM 通过批处理和差异计算更稳定。场景上工程可维护性通常比单点基准更重要。
- 考察点：是否能说明结论背后的原因、成立条件和反例。

6. React 与 Vue 的 diff 算法有何不同？
- 标准答案：结论是两者都做同层比较，但优化细节不同。机制上 Vue 3 在编译阶段注入 patch flag 做静态提升，React 更依赖运行时 Fiber 调度与组件边界。场景上两者性能关键都在稳定 key 和合理组件拆分。
- 考察点：是否能给出明确结论，并补充机制和业务场景。
