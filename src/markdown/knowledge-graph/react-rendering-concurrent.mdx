## 概念定义

- 本质定义：该节点聚焦 高负载交互保持响应。
- 解决问题：把高频面试问题转成可复述、可落地的机制说明。
- 边界说明：不替代其他节点的专属机制域。

## 核心机制拆解

### 1) 机制/流程

1. 明确输入状态和触发条件。
2. 进入框架调度或编译阶段。
3. 执行更新并产出结果。
4. 用回归验证结果是否稳定。

### 2) 关键条件与触发点

- 条件 A：组件边界和数据来源必须清晰。
- 条件 B：触发时机与副作用清理要成对出现。

### 3) 结果与影响

- 对性能/稳定性/可维护性的影响：机制清晰时可预测，机制混乱时会出现渲染回退、状态错乱或协作成本上升。

## 相似概念对比

| 维度 | 同步渲染 | 并发渲染 | 结论 |
|---|---|---|---|
| 定义 | 关注单点能力 | 关注对照能力 | 职责边界不同 |
| 触发时机 | 偏局部触发 | 偏链路触发 | 需要按场景选择 |
| 性能影响 | 局部可见 | 全链路可见 | 不可只看单点 |
| 适用场景 | 快速落地 | 长期治理 | 建议组合使用 |

## 代码

```tsx
import { useMemo, useState, useTransition } from 'react'

export default function Search({ list }: { list: string[] }) {
  const [input, setInput] = useState('')
  const [keyword, setKeyword] = useState('')
  const [isPending, startTransition] = useTransition()

  const filtered = useMemo(
    () => list.filter(item => item.includes(keyword)),
    [list, keyword],
  )

  return (
    <>
      <input
        value={input}
        onChange={(e) => {
          const v = e.target.value
          setInput(v)
          startTransition(() => setKeyword(v))
        }}
      />
      {isPending && <span>calculating...</span>}
      <ul>{filtered.map(item => <li key={item}>{item}</li>)}</ul>
    </>
  )
}
```

### 示例说明

- 这段代码解决什么问题：把高成本过滤更新降级，保证输入框优先响应。
- 关键语句或参数含义：`startTransition` 标记低优先级更新。
- 常见错误写法：把所有状态都包进 transition，导致反馈延迟。

## 真实场景示例

- 场景描述：核心页面迭代中出现跨组件状态与渲染时序问题。
- 问题表现：功能可用但偶发抖动、错渲染或发布回退。
- 处理方案：先收敛边界，再做最小修复，最后补自动化验证。
- 验证方式：通过关键路径回归、性能指标和线上监控确认生效。

## 相关面试题

1. 对 React-Fiber 的理解，它解决了什么问题？
- 标准答案：结论是 Fiber 是 React 的可中断渲染架构。机制上把更新拆成可调度的小单元，允许高优任务插队，提升交互响应。场景上复杂页面输入、动画与大列表并存时收益明显。
- 考察点：是否能给出明确结论，并补充机制和业务场景。

2. 对虚拟 DOM 的理解？虚拟 DOM 主要做了什么？虚拟 DOM 本身是什么？
- 标准答案：结论是虚拟 DOM 是对真实 DOM 的内存描述，用来做差异计算。机制上先生成新旧 vnode，再通过 Diff 找最小变更并提交真实 DOM。场景上它提升的是“跨平台抽象和可维护性”，不是任何场景都绝对更快。
- 考察点：是否能给出明确结论，并补充机制和业务场景。

3. React diff 算法的原理是什么？
- 标准答案：结论是 React Diff 采用同层比较策略。机制上不同类型节点直接替换，同类型比较属性与子节点，列表依赖 key 定位复用。场景上稳定 key 是避免重建和状态错位的关键。
- 考察点：是否能按阶段讲清核心流程、关键数据结构和边界条件。

4. React key 是干嘛用的 为什么要加？key 主要是解决哪一类问题的
- 标准答案：结论是 key 用于标识同层节点身份。机制上 Diff 根据 key 判断复用、移动或删除。场景上使用可变 index 作为 key 会导致输入框状态错位、动画抖动等问题。
- 考察点：是否能说明结论背后的原因、成立条件和反例。

5. 虚拟 DOM 的引入与直接操作原生 DOM 相比，哪一个效率更高，为什么
- 标准答案：结论是不存在绝对谁更快。机制上小规模一次性更新直接 DOM 可能更快，复杂交互下虚拟 DOM 通过批处理和差异计算更稳定。场景上工程可维护性通常比单点基准更重要。
- 考察点：是否能说明结论背后的原因、成立条件和反例。

6. React 与 Vue 的 diff 算法有何不同？
- 标准答案：结论是两者都做同层比较，但优化细节不同。机制上 Vue 3 在编译阶段注入 patch flag 做静态提升，React 更依赖运行时 Fiber 调度与组件边界。场景上两者性能关键都在稳定 key 和合理组件拆分。
- 考察点：是否能给出明确结论，并补充机制和业务场景。

7. 虚拟DOM是什么？它解决了哪些实际问题？
- 标准答案：结论是虚拟 DOM 是对 UI 结构的内存描述层。机制上框架先在内存中做差异计算，再把最小变更提交到真实 DOM。场景上复杂组件化应用能获得更稳定的跨平台抽象与更新策略。
- 考察点：是否能区分“抽象收益”与“绝对性能收益”。

8. React 的 Fiber 架构主要是为了解决什么问题？
- 标准答案：结论是 Fiber 主要解决大更新任务阻塞主线程的问题。机制上把渲染拆成可中断、可恢复、可按优先级调度的单元。场景上输入交互与大列表并发存在时可显著改善响应性。
- 考察点：是否理解 Fiber 与调度优先级的关系。

9. React 中的 `key` 属性有什么作用？它的内部机制是怎样的，最佳实践是什么？
- 标准答案：结论是 key 用来标识同层节点身份，帮助 Diff 稳定复用。机制上 React 根据 key 判断节点复用、移动或重建。场景上应使用稳定业务主键，避免用索引导致状态错位和动画抖动。
- 考察点：是否能把 key 规则落到可执行的列表渲染实践。

10. useTransition 和 useDeferredValue 是如何优化用户体验的？它们之间有什么区别？
- 标准答案：结论是二者都用于降低高成本更新对交互流畅度的影响。机制上 `useTransition` 主动把更新降级，`useDeferredValue` 延后消费某个值。场景上输入筛选可用 transition，结果列表延后渲染可用 deferred value。
- 考察点：是否能按“更新发起方 vs 消费方”区分两者。

11. ** React 18 中的自动批处理 Automatic Batching 如何工作？
- 标准答案：结论是 React 18 会在更多异步边界自动合并多次状态更新。机制上同一批次内更新合并后统一提交，减少重复渲染。场景上 Promise、定时器和原生事件中的连续 setState 都能获得更稳定性能收益。
- 考察点：是否理解批处理范围在 React 18 的扩展点。

12. ** React 19 中的 Activity 组件与 Suspense 有什么区别？
- 标准答案：结论是两者都服务异步体验，但职责不同。机制上 Suspense 关注“等待中的占位与回退”，Activity 更偏向异步任务状态表达与交互反馈编排。场景上数据加载兜底用 Suspense，复杂任务流程反馈可引入 Activity。
- 考察点：是否能从职责边界而非名字相似性做区分。

13. **React 19 中的 use Hook 如何简化异步数据处理？
- 标准答案：结论是 `use` 允许直接消费 Promise 或上下文并与 Suspense 协同。机制上读取到未完成 Promise 时抛出等待信号，由边界统一处理回退 UI。场景上可减少手写加载态样板和重复状态同步逻辑。
- 考察点：是否理解 use 与 Suspense 的协同执行模型。

14. 说一下React 19 对表单处理的改进：useActionState 和 useFormStatus？
- 标准答案：结论是 React 19 把表单提交状态管理收敛到框架原生能力。机制上 `useActionState` 管理提交结果与错误态，`useFormStatus` 在表单层读取 pending 等状态。场景上可减少手写 loading/error 状态胶水代码。
- 考察点：是否能说明两者在表单链路中的分工。

15. React 19 中如何直接在组件中渲染 meta 标签和样式表？
- 标准答案：结论是 React 19 支持在组件树中声明文档元信息与样式资源。机制上框架会在渲染阶段统一管理这些 head 相关输出与去重。场景上路由级页面可就近声明 SEO 与样式依赖，减少跨层配置耦合。
- 考察点：是否理解“就近声明”对工程可维护性的价值。

16. React 19 的 ref 作为 prop 传递有什么优势？
- 标准答案：结论是它降低了 ref 透传的样板和心智负担。机制上组件可更自然地接收与转发引用能力，减少额外包装层。场景上基础组件库封装输入框、弹层、滚动容器时会更顺滑。
- 考察点：是否能把语法变化关联到组件库开发收益。
