## 概念定义

- 本质定义：`script` 加载策略决定脚本何时下载、何时执行、是否阻塞解析，以及多个脚本之间是否保序。
- 解决问题：在首屏可见速度和初始化稳定性之间做取舍。
- 边界说明：本节点只讨论脚本时序，不展开模块拆分和打包产物治理。

## 核心机制拆解

### 1) 机制/流程

1. 默认同步 `script`：解析到标签时开始下载并在下载后立即执行，期间会阻塞后续解析。
2. `defer`：脚本和 HTML（HyperText Markup Language）并行下载，等文档解析完成后按出现顺序执行。
3. `async`：脚本并行下载，谁先下载完谁先执行，不保证顺序。
4. 根据依赖关系决定脚本该走“保序执行”还是“尽快执行”。

### 2) 关键条件与触发点

- 条件 A：有依赖顺序的脚本误用 `async`，会导致初始化竞态。
- 条件 B：第三方统计、广告这类独立脚本如果仍然用默认同步策略，会平白增加首屏阻塞。
- 条件 C：把所有脚本统一成一种策略，看似简单，实际上最容易出线上偶发问题。

### 3) 结果与影响

- 对性能/稳定性/可维护性的影响：策略选对了，首屏更快且初始化稳定；策略选错了，线上通常表现为“偶发白屏”“偶发未定义”“本地复现不了”。

## 相似概念对比

| 维度 | 默认 `script` | `defer` | `async` |
|---|---|---|---|
| 下载时机 | 解析到标签后下载 | 与解析并行下载 | 与解析并行下载 |
| 执行时机 | 下载完立即执行 | DOM（Document Object Model） 解析完成后执行 | 下载完立即执行 |
| 顺序保证 | 按文档出现顺序 | 按文档出现顺序 | 不保证 |
| 适用场景 | 极少量必须同步逻辑 | 主链路初始化脚本 | 独立第三方脚本 |

## 代码

```html
<script defer src="/assets/runtime.js"></script>
<script defer src="/assets/main.js"></script>
<script async src="https://cdn.example.com/track.js"></script>
```

### 示例说明

- 这段代码解决什么问题：主链路脚本保序执行，第三方监控脚本独立异步执行。
- 关键语句或参数含义：`defer` 保证顺序和“等 DOM 解析完成再执行”；`async` 追求“尽快执行”，但不保序。
- 常见错误写法：把有依赖关系的一组脚本全部改成 `async`。

## 真实场景示例

- 场景描述：监控脚本接入后，线上偶发报“模块未定义”，本地又几乎复现不出来。
- 问题表现：运行时脚本还没准备好，业务入口已经先执行了。
- 处理方案：主链路脚本统一切到 `defer`，独立第三方脚本才用 `async`。
- 验证方式：回归初始化链路，观测错误率、首屏时间和脚本 waterfall（瀑布图）。

## 常见误区

- 误区：`async` 一定比 `defer` 更先进，所以应该全部用 `async`。
- 正解：`async` 更激进，但不保序；依赖链脚本通常更适合 `defer`。
- 误区：只要脚本体积小，同步执行就没影响。
- 正解：即便脚本很小，它依然会切断解析时序，只是影响大小不同。
- 误区：脚本顺序问题只会在开发环境暴露。
- 正解：真实网络环境更复杂，竞态问题往往只在生产环境放大。

## 关联知识点

- 依赖：加载与阻塞模型。
- 影响：首屏可见、初始化稳定性、第三方脚本治理。
- 对比：默认 `script`、`defer`、`async`。
- 组合：脚本位置、资源预加载、模块拆分。

## 相关面试题

1. script 标签中 defer 和 async 的区别
- 标准答案：两者都支持并行下载，但 `defer` 会等文档解析完成后按顺序执行，`async` 则在下载完成后立即执行且不保证顺序。机制差异就在执行时机和顺序约束。场景上依赖链脚本更适合 `defer`，独立统计脚本更适合 `async`。
- 考察点：是否能从“时序治理”角度解释差异。

2. script 标签位置的性能影响：为什么建议写在 body 末尾？
- 标准答案：默认同步脚本放在 `body` 末尾，是为了减少解析阻塞并避免 DOM 还没生成时脚本就提前执行。机制上浏览器会先把页面主要结构解析出来，再执行脚本。场景上如果不使用 `defer` / `async`，把脚本后移依然是简单有效的基础优化手段。
- 考察点：是否理解“标签位置”和“属性策略”是两层不同的优化手段。
