## 概念定义

- 本质定义：性能优化、存储与网络请求关注小程序如何在有限包体、有限通信成本和平台限制下，把数据获取和页面体验做稳。
- 解决问题：解释“为什么小程序容易因为 `setData`、请求并发和资源体积而卡”，也解释“不同存储方式到底该怎么选”。
- 边界说明：本节点聚焦小程序端的数据获取和性能治理，不展开后端接口设计。

## 核心机制拆解

### 1) 机制/流程

1. 请求侧通常通过 `wx.request` 或二次封装统一域名、鉴权头和错误处理。
2. 数据侧按作用范围选择页面 `data`、本地存储、全局数据或云侧存储。
3. 渲染侧通过控制 `setData` 粒度、分包、图片压缩、懒加载和列表分页降低通信与渲染成本。
4. 当体验要求更高时，再补骨架屏、缓存和错误兜底策略。

### 2) 关键条件与触发点

- 条件 A：请求域名必须提前配置为合法域名，并发请求数量也受平台约束。
- 条件 B：本地存储容量有限，适合轻量缓存，不适合无限堆积业务数据。
- 条件 C：性能问题通常不是单点造成的，而是包体、请求、图片和 `setData` 一起叠加。

### 3) 结果与影响

- 对性能/稳定性/可维护性的影响：数据获取链路收敛后，请求治理、缓存和性能优化才能统一落地；否则体验问题会散落在页面各处，很难系统治理。

## 相似概念对比

| 维度 | 本地存储 | 全局数据 | 云存储 | 结论 |
|---|---|---|---|
| 生命周期 | 持久化到本地 | 跟随小程序进程 | 依赖云侧服务 | 作用范围不同 |
| 适用场景 | Token、轻量缓存、草稿 | 登录态、全局配置 | 文件、共享数据 | 不能混用 |
| 容量边界 | 有容量限制 | 受内存限制 | 取决于服务端方案 | 成本不同 |
| 风险点 | 过期和清理 | 重启丢失 | 网络依赖和成本 | 要按数据性质选型 |

## 代码

```js
const request = (url, options = {}) => {
  return new Promise((resolve, reject) => {
    wx.request({
      url: `https://api.example.com${url}`,
      method: options.method || 'GET',
      data: options.data || {},
      header: {
        Authorization: wx.getStorageSync('token'),
      },
      success: resolve,
      fail: reject,
    })
  })
}

wx.setStorageSync('draft', { page: 1 })
```

### 示例说明

- 这段代码解决什么问题：统一网络请求入口，并演示本地轻量缓存的基本用法。
- 关键语句或参数含义：`wx.request` 是基础请求 API；请求头里可统一带 token；`setStorageSync` 适合存轻量持久化数据。
- 常见错误写法：各页面分散写请求逻辑、不做超时和错误收敛、把大对象长期堆进本地存储。

## 真实场景示例

- 场景描述：活动页首屏图片多、请求多，进入后经常白屏或掉帧。
- 问题表现：页面一次性并发拉太多接口、图片没有压缩，滚动时又频繁 `setData`，导致首屏慢和交互卡顿。
- 处理方案：合并请求、做接口预加载和缓存、压缩图片、长列表分页，并把 `setData` 控制在必要字段上。
- 验证方式：看首屏时间、请求数、包体积、图片下载体积和滚动时掉帧情况。

## 常见误区

- 误区：本地存储能当数据库用。
- 正解：本地存储容量有限，更适合轻量缓存和草稿，不适合长期堆大对象。
- 误区：请求封装只是代码风格问题。
- 正解：统一封装能收敛域名、鉴权、错误处理和重试策略，是稳定性的基础。
- 误区：性能优化只要少调 `setData` 就够了。
- 正解：包体、请求、图片、缓存和 `setData` 都可能是瓶颈，必须一起看。

## 关联知识点

- 依赖：双线程架构、数据绑定和页面生命周期。
- 影响：首屏速度、滚动流畅度、离线体验和请求稳定性。
- 对比：本地存储、全局数据、云存储。
- 组合：请求封装、缓存策略、图片优化、分包加载。

## 相关面试题

1. 小程序的性能优化策略有哪些？
- 标准答案：常见策略包括控制包体积、分包加载、压缩图片、减少 `setData` 频率和数据量、列表分页或虚拟化、请求合并和缓存、再配骨架屏与错误兜底。关键不是背清单，而是知道问题通常出在包体、请求、渲染和资源四条线上。
- 考察点：是否能把小程序性能问题拆成可执行的治理项。

2. 小程序的存储方式有哪些？
- 标准答案：常见方式有本地存储、全局数据、页面 `data` 和云侧存储。本地存储适合 token、轻量缓存和草稿，全局数据适合当前进程共享信息，页面 `data` 只服务当前视图，云存储适合需要跨设备或长期保存的数据。选型关键看数据作用范围和生命周期。
- 考察点：是否知道不同存储方式的范围和限制。

3. 小程序的网络请求如何处理？
- 标准答案：小程序主要通过 `wx.request()` 发请求，项目里通常会再封装一层统一域名、鉴权头、超时和错误处理。还要注意合法域名配置、并发限制和 HTTPS 要求。场景上如果每个页面各自为战，请求治理和异常处理会很快失控。
- 考察点：是否理解请求封装和平台约束同样重要。
