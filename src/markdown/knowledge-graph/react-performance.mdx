## 代码

```tsx
import { memo, useMemo, useState } from 'react'

const List = memo(function List({ items }: { items: string[] }) {
  return <ul>{items.map(item => <li key={item}>{item}</li>)}</ul>
})

export default function Page({ source }: { source: string[] }) {
  const [keyword, setKeyword] = useState('')
  const filtered = useMemo(
    () => source.filter(item => item.includes(keyword)),
    [source, keyword],
  )

  return (
    <>
      <input value={keyword} onChange={e => setKeyword(e.target.value)} />
      <List items={filtered} />
    </>
  )
}
```

### 示例说明

- 这段代码解决什么问题：减少输入时不必要的列表重复计算和渲染。
- 关键语句或参数含义：`useMemo` 缓存筛选结果，`memo` 避免 props 不变时重渲染。
- 常见错误写法：对所有组件盲目加 memo，却不先定位真实性能瓶颈。

## 真实场景示例

- 场景描述：核心页面迭代中出现跨组件状态与渲染时序问题。
- 问题表现：功能可用但偶发抖动、错渲染或发布回退。
- 处理方案：先收敛边界，再做最小修复，最后补自动化验证。
- 验证方式：通过关键路径回归、性能指标和线上监控确认生效。

## 相关面试题

1. React.Component 和 React.PureComponent 的区别
- 标准答案：结论是 `PureComponent` 默认做浅比较，`Component` 默认全量更新。机制上浅比较只比较引用，深层可变对象会导致误判。场景上配合不可变更新可减少渲染，否则可能产生显示不更新问题。
- 考察点：是否能从定义、触发时机、性能影响和适用场景四个维度做对比。

2. 哪些方法会触发 React 重新渲染？重新渲染 render 会做些什么？
- 标准答案：结论是触发源包括 `setState`、父组件重新渲染、`forceUpdate`、Context 值变化。机制上渲染阶段会重新执行组件函数/`render`，提交阶段才真正改 DOM。场景上应先确认是否真的提交 DOM，而不是把函数执行等同于页面变化。
- 考察点：是否能给出明确结论，并补充机制和业务场景。

3. React 如何判断什么时候重新渲染组件？
- 标准答案：结论是默认父更新会向下传播，是否跳过取决于比较策略。机制上可通过 `memo`、`PureComponent`、`shouldComponentUpdate` 做剪枝。场景上高频列表和大组件必须结合稳定引用做优化。
- 考察点：是否能给出可执行步骤、关键 API 和常见踩坑点。

4. 在 React 中如何避免不必要的 render？
- 标准答案：结论是核心手段有三类：稳定引用（`useMemo/useCallback`）、渲染剪枝（`memo/PureComponent`）、状态下沉与拆分。机制上目标是减少无关状态变更导致的渲染扇出。场景上先用 Profiler 找热点，再有针对性优化。
- 考察点：是否能给出可执行步骤、关键 API 和常见踩坑点。
