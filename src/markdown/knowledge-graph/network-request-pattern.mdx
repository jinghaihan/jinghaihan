## 概念定位

网络请求模式与重试治理节点关注“请求如何发、失败如何处理、并发如何控制、结果如何复用”。

这是把“接口调用”升级为“稳定数据链路”的核心节点，直接影响可用性与成本。

## 工作机制

1. 构建统一请求层：超时、重试、鉴权、日志、错误码映射集中治理。
2. 按业务语义区分读取与写入：读取可重试，写入必须幂等后再重试。
3. 并发控制：请求去重、并发上限、优先级队列，避免页面切换时洪峰。
4. 缓存策略：内存缓存 + 条件请求（ETag/If-None-Match）+ 失效规则。
5. 失败兜底：降级数据、离线回退、骨架屏与错误可恢复交互。

## 关键知识点

### 必须掌握

- 重试要区分错误类型：网络抖动可重试，参数错误不可重试。
- 同一资源并发请求可做 In-Flight 合并，减少重复请求。
- 取消过期请求可避免竞态覆盖和无效流量。
- SWR（Stale While Revalidate，陈旧可用同时后台刷新）模式适合列表类页面。
- 错误码分层要清晰：HTTP 状态、业务码、前端异常分开处理。
- 请求层日志需要 traceId 支持链路排障。

### 选型与取舍

- 激进缓存提升速度，但一致性风险更高。
- 高并发可缩短总时长，但会冲击后端和客户端资源。
- 强兜底可减少空白页，但错误感知可能被掩盖。

## 对页面性能与稳定性的影响

- 节点稳定时，前端基础（HTML/CSS/JavaScript） 的链路可预测性会明显提升，发布和排障成本更可控。
- 节点失稳时，问题会向上下游扩散，常表现为性能回退、交互异常或交付风险叠加。
- 该节点优化应和上下游协同验证，避免“局部优化把成本转移给后续节点”。

## 观测与排障

### 核心指标

- 请求成功率与失败类型分布
- 重试次数与重试成功率
- 请求重复率与取消率
- 接口平均延迟与 P95 延迟

### 排障顺序

1. 先按错误类型分层：DNS/TCP/超时/4xx/5xx/业务错误。
2. 检查是否存在重复请求与无效重试。
3. 检查并发峰值和排队情况，确认是否需限流。
4. 检查缓存命中和一致性问题是否共存。
5. 修复后在弱网和高并发场景复测。

### 常见故障场景

- “页面来回切换后数据乱跳”：常见是竞态覆盖。
- “后端恢复后前端仍失败”：常见是重试策略未退避。
- “流量异常升高”：常见是请求去重失效。
- “偶发旧数据”：常见是缓存失效策略不完整。

## 常见误区

- 把请求层写成简单 `fetch` 封装，不做策略治理。
- 所有失败都重试，导致副作用放大。
- 缺少统一 trace，上下游难以对齐定位。

## 深入拆解

### 请求治理分层

- 传输层：超时、重试、取消、代理与 DNS 异常处理。
- 协议层：状态码处理、重定向策略、压缩与缓存协商。
- 业务层：错误码映射、兜底文案、幂等控制与权限处理。

### 失败恢复优先级

1. 可恢复网络抖动：重试 + 退避。
2. 过期请求：取消并丢弃响应。
3. 业务失败：提示 + 可重试入口 + 上下文保留。
4. 服务故障：降级数据或只读模式，保护核心路径。

### Agent/流式场景补充

- 文本流场景可采用 SSE（Server-Sent Events）减少协议复杂度。
- 强双向交互采用 WebSocket，但要配套心跳和重连退避。
- 无论哪种通道都应定义统一消息 envelope（eventId、timestamp、traceId）。

## 题库深挖补充

### 高频对比与边界（基于题库）

- 请求治理要分传输层、协议层、业务层，不能只写一个 fetch 包装。
- 读取请求和写入请求的重试策略不同，写入必须保证幂等。
- 并发控制应包含去重、限流、取消、优先级，不是单点优化。
- 状态码处理要结合业务码，不可只看 HTTP 层。
- SSE/WebSocket 选型在流式场景非常高频，需能讲方向性和成本。
- 面试常问“如何做接口容错”，答案要覆盖超时、重试、降级、观测。

### 线上排障剧本

1. 先按错误类型分桶（超时/4xx/5xx/网络中断）。
2. 检查是否存在请求洪峰与重复请求。
3. 检查重试是否缺退避导致放大故障。
4. 检查缓存命中和数据一致性是否冲突。
5. 修复后做弱网和高并发回归。

### 高频追问补充（附答案）

1. 为什么请求去重很关键？

可减少无意义流量和竞态覆盖问题。

2. 写接口重试的风险是什么？

若无幂等控制，可能造成重复副作用。

3. 流式场景何时用 SSE？

主要是服务端单向持续推送的文本或事件流。

## 进阶深挖（第二轮）

### 请求治理落地清单

1. 建立统一请求层，收敛超时、重试、取消、鉴权和日志。
2. 读取与写入分策略，写入默认不自动重试。
3. 接口层引入 idempotency key 或版本号防重放。
4. 对关键接口建立降级与只读兜底路径。

### Agent 场景扩展

- 文本流主场景优先 SSE，降低双向协议复杂度。
- 需要双向低延迟控制时选 WebSocket。
- 无论选哪种通道，都要统一消息 envelope 和重连策略。

### 追问扩展

1. 为什么统一请求层会显著提升稳定性？

因为重试、超时、错误映射和可观测能在一个地方持续收敛，而不是散落在组件里。

2. 降级策略最常见误区是什么？

只有展示降级，没有数据一致性和恢复路径，最终会形成“表面可用”。

## 面试补充

### 30 秒说明

请求治理的目标不是“请求成功一次”，而是“在波动环境里持续可用且可观测”。

### 高频追问

1. 什么时候适合做请求重试？

短暂网络错误、超时、可恢复服务端错误可重试；参数错误和权限错误不应重试。

2. 如何避免并发请求把后端打爆？

做请求合并、并发上限和优先级队列，并在失败时退避重试。

3. SWR 方案的价值是什么？

先用缓存快速渲染，再后台刷新，平衡速度与新鲜度。

4. 为什么请求层要有 traceId？

便于前后端与网关日志串联，快速定位链路瓶颈。
