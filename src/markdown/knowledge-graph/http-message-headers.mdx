## 概念定义

- 本质定义：HTTP（HyperText Transfer Protocol）报文与头字段是客户端和服务端交换语义、能力和缓存意图的标准载体。
- 解决问题：用统一格式表达请求目标、传输内容、缓存条件、连接策略和安全约束。
- 边界说明：它不负责连接建立，不处理 TLS（Transport Layer Security）握手和 DNS（Domain Name System）解析。

## 核心机制拆解

### 1) 机制/流程

1. 客户端构造请求行、请求头、空行、请求体。
2. 服务端解析报文并执行业务逻辑。
3. 服务端返回状态行、响应头、空行、响应体。
4. 客户端根据头字段执行缓存复用、重定向或重试策略。

### 2) 关键条件与触发点

- 条件 A：`Content-Type` 与请求体格式匹配，决定服务端是否能正确解析。
- 条件 B：缓存相关头（`Cache-Control`、`ETag`、`If-None-Match`）是否成对使用，决定协商缓存是否生效。

### 3) 结果与影响

- 对性能 / 稳定性 / 可维护性的影响：头字段清晰可提升缓存命中和排障效率；字段混乱会导致缓存失效、兼容问题和调试成本上升。

## 相似概念对比

| 维度 | 请求头 | 响应头 | 结论 |
|---|---|---|---|
| 定义 | 客户端声明能力、意图和上下文 | 服务端声明结果、策略和元信息 | 两者是成对协作关系 |
| 触发时机 | 发送请求时 | 返回响应时 | 缓存协商需双向字段配合 |
| 性能影响 | 头过大增加传输开销 | 缓存头影响重复请求成本 | 头字段设计直接影响吞吐和时延 |
| 适用场景 | 鉴权、内容协商、条件请求 | 缓存控制、重定向、安全策略 | 面试常考字段配对关系 |

## 代码

```http
GET /users/1 HTTP/1.1
Host: api.example.com
Accept: application/json
If-None-Match: "v1-user-1"
```

```http
HTTP/1.1 304 Not Modified
Cache-Control: max-age=0, must-revalidate
ETag: "v1-user-1"
Content-Type: application/json
```

### 示例说明

- 这段代码 / 配置解决什么问题：示例展示协商缓存请求与 `304` 响应配合。
- 关键参数含义：`If-None-Match` 携带本地版本指纹，`ETag` 表示服务端版本标识。
- 常见错误配置：只返回 `ETag` 但客户端不带条件头，导致每次都回源传完整响应体。

## 真实场景示例

- 场景描述：静态资源版本未变但带宽成本持续升高。
- 问题表现：重复请求返回 `200` 和完整响应体，`304` 占比很低。
- 处理方案：补齐缓存协商头链路，统一 `Cache-Control` 与 `ETag` 策略。
- 验证方式：观察 `304` 占比、回源流量和首屏加载时间。

## 常见误区

- 误区：`304` 是错误响应。
  - 正解：`304` 是协商缓存命中信号，表示可复用本地副本。
- 误区：请求头越多越好，排障更方便。
  - 正解：头字段过大增加传输成本，应保留必要字段并做规范治理。

## 关联知识点

- 依赖：`http-method-semantics` 确定请求语义边界。
- 影响：报文头字段会影响 `status-code-path` 的分支处理。
- 对比：`http-message-headers` 关注报文结构，`http-evolution` 关注传输能力。
- 组合：常与 `cache-policy` 联合做缓存命中优化。

## 相关面试题

1. 常见的 HTTP 请求头和响应头
   - 标准答案：请求头用于声明客户端能力和上下文（如 `Accept`、`Host`、`Cookie`），响应头用于声明资源元信息和策略（如 `Content-Type`、`Cache-Control`）；关键在字段配对是否支撑缓存和内容协商。
   - 考察点：是否只会背字段，还是能说明字段在链路中的作用。
   - 追问 1：`Content-Type` 配错会造成什么问题？
   - 标准答案 1：服务端或客户端会按错误格式解析，轻则解析失败，重则触发兼容与安全风险。
   - 追问 2：为什么 `304` 能省带宽？
   - 标准答案 2：响应可不带实体，客户端直接复用本地缓存，减少传输体积和回源压力。

2. HTTP 请求报文的是什么样的？
   - 标准答案：请求报文由请求行、请求头、空行、请求体组成；请求行包含方法、URL 和协议版本，请求体仅在需要提交数据时出现。
   - 考察点：是否理解报文结构与字段职责。
