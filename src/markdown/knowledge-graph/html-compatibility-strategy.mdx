## 概念定义

- 本质定义：渐进增强与优雅降级是前端在不同浏览器能力下控制功能可用性的两种兼容设计思路。
- 解决问题：在能力不一致环境下保证核心功能可达，并为高能力环境提供更好体验。
- 边界说明：本节点聚焦策略方法，不替代具体 API 的兼容实现细节。

## 核心机制拆解

### 1) 机制/流程

1. 先识别业务核心功能和可选增强功能。
2. 渐进增强路径：先交付可用基础，再叠加高级体验。
3. 优雅降级路径：先实现完整高级方案，再为低能力环境回退。
4. 建立能力检测和回退分支，避免直接按浏览器名硬编码。

### 2) 关键条件与触发点

- 条件 A：用户环境差异大（旧设备、弱网、老浏览器）时，更需要基础可用优先。
- 条件 B：高交互项目若只做单一路径，容易在低能力环境失效。

### 3) 结果与影响

- 对性能/稳定性/可维护性的影响：策略清晰可减少兼容返工，提升发布稳定性。

## 相似概念对比

| 维度 | 渐进增强 | 优雅降级 | 结论 |
|---|---|---|---|
| 起点 | 基础可用版本 | 完整高级版本 | 起点不同 |
| 目标 | 向高能力环境增强 | 向低能力环境回退 | 方向不同 |
| 风险 | 高级体验实现成本上移 | 低能力兼容成本上移 | 需结合团队能力选择 |
| 适用场景 | 用户环境复杂、覆盖广 | 产品能力集中、版本可控 | 可组合使用 |

## 代码

```js
if ('serviceWorker' in navigator) {
  navigator.serviceWorker.register('/sw.js')
}
```

### 示例说明

- 这段代码解决什么问题：在支持环境启用离线增强，不支持环境保持基础可用。
- 关键语句或参数含义：`in` 检测用于能力判断，不依赖浏览器 UA（User Agent）字符串。
- 常见错误写法：不做能力检测直接调用高级 API。

## 真实场景示例

- 场景描述：后台系统需要兼容企业内旧浏览器，同时满足现代浏览器交互体验。
- 问题表现：新功能上线后旧环境出现白屏或功能不可用。
- 处理方案：核心流程采用渐进增强，复杂特性提供降级替代路径。
- 验证方式：按浏览器能力分层回归，验证核心流程可达率。

## 常见误区

- 误区：二选一，只能采用其中一种。
- 正解：可按模块组合使用，核心链路渐进增强，局部特性优雅降级。
- 误区：做了降级就不用关注基础可用。
- 正解：基础可用始终是底线。

## 关联知识点

- 依赖：HTML5 能力总览、离线存储与 Service Worker。
- 影响：发布稳定性、跨端体验一致性。
- 对比：能力检测与浏览器嗅探。
- 组合：文档声明与 head 配置、媒体与图形降级策略。

## 相关面试题

1. 渐进增强和优雅降级之间的区别
- 标准答案：结论是渐进增强从基础可用出发，逐步增加高级能力；优雅降级从完整能力出发，向低能力环境回退。机制上二者区别在设计起点和兼容方向。场景上用户环境差异大时更适合渐进增强，强控环境可结合优雅降级。
- 考察点：是否能把策略区别落到工程决策。
