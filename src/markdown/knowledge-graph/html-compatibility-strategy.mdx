## 概念定义

- 本质定义：渐进增强与优雅降级是两种处理浏览器能力差异的设计思路，核心都在于“高能力环境更好用，低能力环境至少能完成核心任务”。
- 解决问题：避免页面在老设备、弱网、脚本关闭或部分能力缺失时直接不可用。
- 边界说明：本节点讲方法和判断原则，不展开某个具体 API（Application Programming Interface） 的兼容补丁。

## 核心机制拆解

### 1) 机制/流程

1. 先分清什么是核心功能，什么是增强体验。
2. 渐进增强是先把基础可用链路搭起来，再给高能力环境叠加体验。
3. 优雅降级是先按完整能力实现，再为低能力环境补回退路径。
4. 通过能力检测、回退内容和 `noscript` 兜底保证关键路径不失效。

### 2) 关键条件与触发点

- 条件 A：用户环境差异大时，基础可用比高级效果更重要。
- 条件 B：页面严重依赖脚本但没有回退内容时，一旦脚本失败就可能直接白屏。
- 条件 C：只按浏览器名称做嗅探，随着版本变化很快会失效。

### 3) 结果与影响

- 对性能/稳定性/可维护性的影响：策略清晰时，兼容问题能被提前收口；策略混乱时，每次发版都像开盲盒。

## 相似概念对比

| 维度 | 渐进增强 | 优雅降级 | 结论 |
|---|---|---|---|
| 起点 | 基础可用版本 | 完整高级版本 | 起点不同 |
| 方向 | 向高能力环境增强 | 向低能力环境回退 | 方法不同 |
| 风险 | 高级特性要逐层加 | 低能力回退容易补不全 | 可以组合使用 |
| 适用场景 | 用户环境复杂、覆盖面广 | 技术栈可控、能力集中 | 要按业务选 |

## 代码

```html
<div id="chart-root"></div>

<script>
  if ('serviceWorker' in navigator) {
    navigator.serviceWorker.register('/sw.js')
  }
</script>

<noscript>
  <p>当前页面的高级交互依赖 JavaScript。如无法启用脚本，请查看静态说明页。</p>
  <a href="/report-static.html">打开静态报告</a>
</noscript>
```

### 示例说明

- 这段代码解决什么问题：支持环境启用离线增强，不支持脚本时仍给出静态回退入口。
- 关键语句或参数含义：`'serviceWorker' in navigator` 是能力检测；`noscript` 是脚本关闭时的兜底内容。
- 常见错误写法：高级能力直接写死，没有任何回退内容或替代路径。

## 真实场景示例

- 场景描述：企业后台既要兼容旧浏览器，又要在现代浏览器里支持更好的图表和离线体验。
- 问题表现：新能力上线后，老环境直接白屏或图表区域空白。
- 处理方案：核心流程先保留基础表格和静态报告，现代环境再增强图表、拖拽和离线能力。
- 验证方式：按浏览器能力分层回归，核对核心流程可达率和关键页面失败率。

## 常见误区

- 误区：渐进增强和优雅降级只能二选一。
- 正解：一个系统完全可以核心路径走渐进增强，局部特性走优雅降级。
- 误区：做了回退就等于兼容完成了。
- 正解：回退内容如果不能完成关键任务，仍然不算真正可用。
- 误区：`noscript` 已经过时，不需要了。
- 正解：只要页面核心信息高度依赖脚本，它就仍然是很实用的最后一道兜底。

## 关联知识点

- 依赖：HTML5 能力总览、离线存储与 Service Worker。
- 影响：发布稳定性、跨端体验和容灾能力。
- 对比：能力检测和浏览器嗅探。
- 组合：`noscript`、静态兜底页、功能开关。

## 相关面试题

1. 渐进增强和优雅降级之间的区别
- 标准答案：渐进增强是先保证基础版本可用，再逐步增强；优雅降级是先实现完整能力，再为低能力环境补回退。机制上它们的区别是设计起点不同。场景上用户环境复杂时更适合渐进增强，受控后台系统则经常两者一起用。
- 考察点：是否能把两种策略讲到工程决策层面。

2. HTML中的noscript标签有什么用途？
- 标准答案：`noscript` 用于在浏览器禁用脚本或脚本不可用时提供替代内容。机制上只有在脚本无法执行时，它内部内容才会显示。场景上地图、图表、富交互表单这类强依赖脚本的页面，最好至少给一个静态说明或回退入口。
- 考察点：是否知道脚本失败时页面也要有基本可用方案。
