## 概念定义

- 本质定义：Webpack 构建提速策略是通过并行、缓存、范围收敛和预编译降低全量与增量构建耗时。
- 解决问题：解决构建链路慢、开发反馈慢和 CI 队列拥堵问题。
- 边界说明：不直接解决线上首屏渲染慢或业务逻辑性能问题。

## 核心机制拆解

### 1) 机制/流程

1. 识别耗时阶段：转译、压缩、依赖解析、产物写盘。
2. 对 CPU 密集步骤启用并行和增量缓存。
3. 对稳定依赖做预构建或外部化，减少重复编译。
4. 收敛 Loader 作用范围，避免扫描无关目录。
5. 通过构建分析持续回归，防止提速策略失效。

### 2) 关键条件与触发点

- 条件 A：CI 构建时长持续上涨且变更规模未增加时，应优先排查缓存与插件链。
- 条件 B：本地二次构建仍接近全量耗时时，通常是缓存未命中或范围配置过宽。

### 3) 结果与影响

- 对性能的影响：构建提速提升研发反馈速度，间接提升优化迭代频率。
- 对稳定性的影响：过度并行或激进缓存配置不当会引入偶发构建不一致。

## 相似概念对比

| 维度 | 打包速度优化 | 构建速度优化 | 结论 |
|---|---|---|---|
| 关注点 | 产物处理阶段耗时 | 全链路总耗时 | 两者范围不同 |
| 典型手段 | 压缩并行、插件精简 | 缓存、预编译、范围收敛 | 常常需要组合 |
| 指标 | 压缩阶段耗时 | 全量与增量构建耗时 | 评估口径要分开 |
| 适用场景 | 生产构建慢 | 本地和 CI 整体慢 | 先定位瓶颈再选手段 |

## 配置示例

```ts
import path from 'node:path'

export default {
  cache: {
    type: 'filesystem',
  },
  module: {
    rules: [
      {
        test: /\.tsx?$/,
        include: path.resolve(process.cwd(), 'src'),
        use: [
          'thread-loader',
          {
            loader: 'babel-loader',
            options: { cacheDirectory: true },
          },
        ],
      },
    ],
  },
  externals: {
    jquery: 'jQuery',
  },
}
```

### 示例说明

- 这段配置覆盖了文件系统缓存、并行转译、`include` 收敛和 `externals` 外部化。
- `thread-loader` 适合 CPU 密集转换，不适合 I/O 受限步骤。
- 常见错误写法是对所有规则都加并行，反而增加线程调度和通信开销。

## 真实场景示例

- 场景描述：中台项目 CI 构建从 7 分钟涨到 19 分钟，研发等待时间过长。
- 问题表现：压缩和转译阶段耗时激增，提交峰值时队列阻塞严重。
- 处理方案：启用持久化缓存、收敛 Babel 扫描范围、下线低收益分析插件、外部化稳定大依赖。
- 验证方式：一周内全量构建降到 9 分钟，增量构建稳定在 2 分钟以内。

## 常见误区

- 误区：提速只能靠升级机器。
- 正解：构建瓶颈更多来自配置和链路设计，硬件只是放大器。
- 误区：所有依赖都适合 `externals`。
- 正解：`externals` 会把加载责任转给运行环境，不适合强一致业务依赖。

## 关联知识点

- 依赖：Webpack 模块图与构建流程、Loader 与 Plugin 扩展机制。

- 组合：CI 构建门禁和发布流程。

## 相关面试题

1. 如何提高 webpack 的打包速度？
- 标准答案：可从并行处理、缓存命中、插件精简和第三方依赖外部化入手。典型手段包括 `thread-loader`、`babel-loader` 缓存、限制 Loader 处理范围、对稳定大依赖使用 `externals` 或预编译。关键条件是先定位耗时阶段再选策略。
- 考察点：是否能把提速手段和瓶颈类型对应起来。

2. 如何提高 webpack 的构建速度？
- 标准答案：构建提速要覆盖全量与增量两类场景，可结合公共代码抽离、依赖预编译、持久化缓存、并行压缩和摇树优化。若项目是多入口，还需优先控制入口依赖重复率，否则优化收益会被重复编译抵消。
- 考察点：是否理解“局部快”与“全链路快”的区别。
