## 概念定义

- 本质定义：该节点聚焦 Hook 调用顺序稳定。
- 解决问题：把高频面试问题转成可复述、可落地的机制说明。
- 边界说明：不替代其他节点的专属机制域。

## 核心机制拆解

### 1) 机制/流程

1. 明确输入状态和触发条件。
2. 进入框架调度或编译阶段。
3. 执行更新并产出结果。
4. 用回归验证结果是否稳定。

### 2) 关键条件与触发点

- 条件 A：组件边界和数据来源必须清晰。
- 条件 B：触发时机与副作用清理要成对出现。

### 3) 结果与影响

- 对性能/稳定性/可维护性的影响：机制清晰时可预测，机制混乱时会出现渲染回退、状态错乱或协作成本上升。

## 相似概念对比

| 维度 | 自定义 Hook | 工具函数 | 结论 |
|---|---|---|---|
| 定义 | 关注单点能力 | 关注对照能力 | 职责边界不同 |
| 触发时机 | 偏局部触发 | 偏链路触发 | 需要按场景选择 |
| 性能影响 | 局部可见 | 全链路可见 | 不可只看单点 |
| 适用场景 | 快速落地 | 长期治理 | 建议组合使用 |

## 代码

```tsx
import { useEffect, useState } from 'react'

function useWindowWidth() {
  const [width, setWidth] = useState(window.innerWidth)

  useEffect(() => {
    const onResize = () => setWidth(window.innerWidth)
    window.addEventListener('resize', onResize)
    return () => window.removeEventListener('resize', onResize)
  }, [])

  return width
}
```

### 示例说明

- 这段代码解决什么问题：示范自定义 Hook 的标准写法和清理时机。
- 关键语句或参数含义：Hook 顶层调用、依赖数组稳定、返回清理函数。
- 常见错误写法：在条件分支里调用 Hook，导致调用顺序变化。

## 真实场景示例

- 场景描述：核心页面迭代中出现跨组件状态与渲染时序问题。
- 问题表现：功能可用但偶发抖动、错渲染或发布回退。
- 处理方案：先收敛边界，再做最小修复，最后补自动化验证。
- 验证方式：通过关键路径回归、性能指标和线上监控确认生效。

## 相关面试题

1. 对 React Hook 的理解，它的实现原理是什么
- 标准答案：结论是 Hook 让函数组件拥有状态与副作用能力。机制上 React 依赖固定调用顺序把 Hook 状态与 Fiber 节点关联。场景上它把逻辑按“功能”组织，便于跨组件复用。
- 考察点：是否能按阶段讲清核心流程、关键数据结构和边界条件。

2. 为什么 useState 要使用数组而不是对象
- 标准答案：结论是返回数组是为了 API 语义稳定和可重命名解构。机制上固定位置的 `[state, setState]` 易于压缩和推断。场景上对象返回会引入命名冲突和额外属性约定，不利于简单一致的调用体验。
- 考察点：是否能给出可执行步骤、关键 API 和常见踩坑点。

3. React Hooks 解决了哪些问题？
- 标准答案：结论是主要解决类组件逻辑分散、复用困难和 this 绑定心智负担。机制上通过自定义 Hook 抽离状态与副作用逻辑，实现组合复用。场景上复杂页面的请求、订阅、表单逻辑可明显降耦。
- 考察点：是否能给出明确结论，并补充机制和业务场景。

4. React Hook 的使用限制有哪些？
- 标准答案：结论是 Hook 只能在函数组件或自定义 Hook 顶层调用，不能放在条件、循环、嵌套函数里。机制上这是为了保证每次渲染 Hook 调用顺序一致。场景上一旦破坏顺序会出现状态错位和难排查 Bug。
- 考察点：是否能给出可执行步骤、关键 API 和常见踩坑点。

5. React Hooks 在平时开发中需要注意的问题和原因
- 标准答案：结论是重点关注依赖数组、闭包捕获和副作用清理。机制上依赖不完整会读到旧值，清理缺失会产生内存泄漏。场景上网络请求、事件监听、定时器都要在 effect 返回函数中清理。
- 考察点：是否能给出明确结论，并补充机制和业务场景。

6. 请解释装饰器模式（Decorator Pattern），它在现代前端框架中有什么应用？
- 标准答案：装饰器模式是在不改原对象代码的前提下，通过包装扩展能力。React 里高阶组件就是典型实现，输入一个组件返回增强组件；TypeScript 和 Angular 的装饰器语法则用于给类和属性附加元数据。场景上埋点注入、权限包装和路由增强都常用这一思路。
- 考察点：是否能把“包装增强”与“继承改造”区分开。

7. 在React中，自定义Hooks（Custom Hooks）可以看作是哪种设计模式的应用？请说明理由。
- 标准答案：自定义 Hook 可看作策略模式和组合模式的结合。每个 Hook 封装一段可替换逻辑策略，组件通过调用多个 Hook 进行能力组合。场景上数据请求、事件订阅、表单状态管理都能拆成独立 Hook 复用。
- 考察点：是否理解 Hook 的价值在“逻辑复用与组合”，而不是语法糖。

8. useEffect 的执行时机具体是什么时候？它和 useLayoutEffect 有什么核心区别？
- 标准答案：结论是 `useEffect` 在浏览器绘制后异步执行，`useLayoutEffect` 在绘制前同步执行。机制上后者会阻塞绘制，适合必须先读写布局再展示的场景。场景上网络请求和订阅优先 `useEffect`，布局测量用 `useLayoutEffect`。
- 考察点：是否能把执行时机与性能影响关联说明。

9. 你能深入剖析一下 useEffect 的依赖项吗？它背后的原理是什么？
- 标准答案：结论是依赖数组描述“副作用重新执行的输入集合”。机制上 React 用浅比较判断依赖是否变化，变化才重新执行并先清理上一次 effect。场景上依赖缺失会出现闭包旧值，依赖冗余会造成无效执行。
- 考察点：是否理解依赖声明与闭包行为的关系。

10. 在 useEffect 中，如何正确地处理异步请求并避免竞态条件（Race Condition）？
- 标准答案：结论是要在 effect 清理阶段取消或失效旧请求。机制上可用 `AbortController` 或本地标记位阻断过期回包写入状态。场景上搜索联想和快速切换详情页最容易出现竞态覆盖问题。
- 考察点：是否能给出可执行的竞态治理方案。

11. 什么时候应该使用 useCallback 和 useMemo？如果滥用它们会带来什么后果？
- 标准答案：结论是只在“引用稳定性会带来实际收益”时使用。机制上二者用缓存换计算和比较成本，滥用会增加依赖管理复杂度并可能更慢。场景上作为 `memo` 子组件 props、昂贵计算结果缓存时价值最大。
- 考察点：是否能从收益与成本两侧解释取舍。

12. useRef 有哪些常见的应用场景？它和 useState 的根本区别是什么？
- 标准答案：结论是 `useRef` 用于保存可变引用且不触发渲染，`useState` 用于驱动 UI 更新。机制上 ref 写入不会进入调度更新链。场景上 DOM 引用、定时器句柄、上次值缓存适合 ref。
- 考察点：是否能区分“数据存储”与“界面驱动”职责。

13. forwardRef 和 useImperativeHandle 这两个 Hooks 是为了解决什么特定问题而设计的？
- 标准答案：结论是它们用于在组件封装后仍可受控暴露引用能力。机制上 `forwardRef` 透传 ref，`useImperativeHandle` 限定对外暴露的方法集合。场景上表单组件聚焦、滚动定位、受控弹层常用这组组合。
- 考察点：是否能说明“暴露最小能力面”的设计价值。

14. React 18 新增的 useId 解决了什么问题（ SSR 场景下的 ID 冲突、可访问性问题）？
- 标准答案：结论是 `useId` 用于稳定生成跨服务端和客户端一致的 ID。机制上它避免 hydration 期间随机 ID 不一致导致的匹配错误。场景上表单标签关联、无障碍属性绑定在 SSR 场景最受益。
- 考察点：是否理解 useId 与 hydration 一致性的关系。

15. 你能详细解释一下 React Hooks 的执行顺序和依赖规则吗？
- 标准答案：结论是 Hook 必须在组件顶层按固定顺序调用。机制上 React 依赖调用顺序把 Hook 状态槽位与组件实例关联，顺序变化会造成状态错位。场景上条件分支或循环内调用 Hook 是高危错误。
- 考察点：是否能从底层槽位模型解释规则来源。
