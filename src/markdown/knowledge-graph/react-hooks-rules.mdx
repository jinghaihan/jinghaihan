## 概念定位

Hooks 规则不是语法洁癖，而是 Fiber 状态机能够正确工作的必要条件。

一句话：React 依赖“调用顺序稳定”来把每次 render 映射到正确的 Hook 状态槽位。

## Fiber 里的 Hook 存储方式

在函数组件对应的 Fiber 上，Hooks 以链表形式挂在 `memoizedState` 上。

渲染时 React 维护两个指针：

- 当前处理的 Hook（current hook）
- 工作中的 Hook（workInProgress hook）

每调用一次 Hook，指针向后移动一次。只要调用顺序稳定，状态就能正确对齐。

## 为什么“只能在顶层调用 Hook”

如果把 Hook 放进条件分支/循环：

- 某次 render 调用了 5 个 Hook
- 下次 render 因条件变化只调用了 4 个

链表槽位错位，后续 Hook 全部读取到错误状态。这就是规则来源的底层原因。

## 工作机制

### 1) mount 阶段

首次渲染，Hooks Dispatcher 走 mount 分支：

- 创建 Hook 节点
- 初始化 `memoizedState`
- 建立更新队列（如 `useState`）

### 2) update 阶段

更新渲染，Dispatcher 走 update 分支：

- 按顺序读取已有 Hook 节点
- 处理队列里的 pending updates
- 计算新状态并继续向后推进

### 3) useState 队列模型

`useState` 通常维护更新队列（常见实现为环形链表队列），更新对象包含 action、lane 等信息。调度时按优先级决定何时消费这些更新。

## 关键知识点

### 常见 Hook 的职责边界

- `useState`：本地可变状态
- `useReducer`：复杂状态转换逻辑
- `useMemo`：缓存计算结果
- `useCallback`：缓存函数引用
- `useEffect`：提交后副作用
- `useLayoutEffect`：布局阶段副作用（慎用）

### 自定义 Hook 设计原则

1. 只暴露业务语义，不暴露内部细节。
2. 把副作用边界收口，避免组件层重复拼装。
3. 返回值稳定（必要时用 memo/callback 保持引用稳定）。

### 闭包陷阱

Hook 场景下高频问题是 stale closure（陈旧闭包）：

- 回调里读到旧 state
- effect 依赖漏写导致逻辑漂移

解决要么补齐依赖，要么改用函数式更新或 ref 持有最新值。

## 对页面性能的影响

- Hook 规则正确可保证更新链路稳定。
- Hook 误用会导致重复 effect、无效重渲染和内存泄漏。
- 过度使用 `useMemo/useCallback` 也会增加维护成本和心智负担。

## 观测与排障

### 典型故障模式

1. 条件调用 Hook 导致状态错位
2. effect 依赖不全导致数据不同步
3. 自定义 Hook 内部副作用重复触发

### 排障步骤

1. 先看 ESLint Hooks 规则是否有违规。
2. 再看组件每次 render 的 Hook 调用路径是否一致。
3. 检查 effect 依赖是否完整并验证闭包值。
4. 用 Profiler 看是否存在无效重复渲染。

## 实战优化策略

1. 强制启用 `eslint-plugin-react-hooks`。
2. 自定义 Hook 只做一类职责，避免“大一统 Hook”。
3. 对性能敏感组件先做数据边界优化，再考虑 memo 化。
4. effect 逻辑遵循“可重入、可清理、可重试”原则。

## 常见误区

- “Hooks 规则只是官方建议”：错误，违反会破坏状态映射。
- “useMemo/useCallback 用得越多越快”：错误，滥用可能适得其反。
- “依赖数组为空最省事”：错误，常引入陈旧数据问题。

## 进阶深挖（第二轮）

### Hook 规则工程化治理

- 禁止条件分支内调用 Hook。
- 自定义 Hook 统一命名和输入输出契约。
- 复杂 Hook 逻辑拆分为状态层与副作用层。

### stale closure 高发场景

1. 定时器回调读取旧状态。
2. 异步回调持有旧 props。
3. effect 依赖不完整导致旧引用持续生效。

### 追问扩展

1. 为什么 Hook 规则违反后 bug 常很隐蔽？

状态槽位错位后表现依赖运行路径，通常是偶发且难复现。

2. 自定义 Hook 何时该拆？

当单个 Hook 同时承担状态管理、请求和副作用时应拆分职责。

## 面试补充

### 30 秒说明

Hooks 依赖调用顺序稳定来映射 Fiber 上的状态链表，所以必须在函数组件顶层调用，不能放在条件/循环中。`useState` 等 Hook 在 mount/update 阶段走不同 dispatcher 分支，更新通过队列和优先级调度。工程上核心是遵守规则、控制副作用边界、避免陈旧闭包。

### 高频追问

1. 为什么条件调用 Hook 会出问题？

因为会改变 Hook 调用数量和顺序，导致 Fiber 上后续状态槽位错位。

2. Hooks 状态存在哪里？

存于函数组件对应 Fiber 的 `memoizedState` Hook 链表中。

3. `useEffect` 和 `useLayoutEffect` 怎么选？

默认优先 `useEffect`；只有确实需要布局同步读取/写入时才用 `useLayoutEffect`。

4. 如何解决陈旧闭包问题？

补齐依赖、使用函数式更新、或用 ref 保存最新值。

5. 自定义 Hook 的“坏味道”有哪些？

职责过多、内部副作用不可控、返回引用不稳定、难以测试。

6. Hooks 性能优化第一优先级是什么？

先减少无效状态与渲染扇出，再考虑局部 memo 化；不要本末倒置。
