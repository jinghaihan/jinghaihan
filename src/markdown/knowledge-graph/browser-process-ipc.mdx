## 概念定义

- 本质定义：进程通信与标签页协同是指浏览器内不同执行单元之间交换数据和同步状态的机制。
- 解决问题：让多进程架构下的页面、网络、存储和标签页能够可靠协作。
- 边界说明：它不负责渲染管线的绘制细节，也不替代业务协议设计。

## 核心机制拆解

### 1) 机制/流程

1. 进程通过通道传递消息或共享受控数据。
2. 浏览器标签页可通过存储事件、共享线程或消息通道做状态广播。
3. 消息到达后由接收端做幂等校验与状态合并。
4. 协作失败时进入重试或降级路径。

### 2) 关键条件与触发点

- 条件 A：通信双方是否同源，决定可用 API 范围。
- 条件 B：消息是否有唯一标识，决定重复消息能否去重。

### 3) 结果与影响

- 对性能 / 稳定性 / 可维护性的影响：通信机制选择不当会导致状态漂移、消息丢失或资源竞争放大。

## 相似概念对比

| 维度 | `BroadcastChannel` | `localStorage` 事件 | 结论 |
|---|---|---|---|
| 定义 | 同源上下文实时广播通道 | 通过键值变化触发跨页通知 | 二者都可做多标签协同 |
| 触发时机 | 主动发送消息 | 写入存储触发事件 | 前者语义更直接 |
| 性能影响 | 轻量消息分发 | 需序列化并写入存储 | 高频同步优先通道模型 |
| 适用场景 | 登录态同步、协同操作 | 简单状态同步和兜底 | 常做组合降级方案 |

## 代码

```js
const channel = new BroadcastChannel('tab-sync')
channel.postMessage({ type: 'logout', ts: Date.now() })
channel.onmessage = (event) => {
  if (event.data.type === 'logout') {
    location.reload()
  }
}
```

### 示例说明

- 这段代码解决什么问题：在多个同源标签页间同步退出登录事件。
- 关键语句或参数含义：`BroadcastChannel` 通过频道名称建立同源广播域。
- 常见错误写法：不做消息去重和版本判断，导致重复消费。

## 真实场景示例

- 场景描述：用户在一个标签页退出登录，其他标签页仍保留旧状态。
- 问题表现：跨标签页状态不一致，引发权限异常。
- 处理方案：建立消息广播 + 本地存储兜底双通道同步。
- 验证方式：并发打开多标签页，验证状态变更是否一致收敛。

## 常见误区

- 误区：只要能通信就不会有一致性问题。
  - 正解：通信只解决“能传”，还需要幂等、顺序和去重机制保证“传对”。

## 关联知识点

- 依赖：`browser-process-model` 提供多进程隔离基础。
- 影响：影响 `browser-storage-security` 的同源数据同步策略。
- 对比：`browser-process-ipc` 关注通道与同步，`browser-event-dispatch` 关注事件传播。
- 组合：常与 `sw-check` 一起用于离线状态同步。

## 相关面试题

1. 进程之前的通信方式
   - 标准答案：常见方式包括管道、消息队列、信号量、信号、共享内存和套接字；浏览器场景里多用消息传递与受控共享来保持隔离。
   - 考察点：是否能把操作系统通信方式映射到浏览器实际协作。

2. 如何实现浏览器内多个标签页之间的通信?
   - 标准答案：可用 `BroadcastChannel`、`localStorage` 事件、`SharedWorker`、`postMessage` 或服务端转发通道；选型要看同源约束、实时性和兼容要求。
   - 考察点：是否能列举方案并给出选型依据。
```ts
const channel = new BroadcastChannel('tab-sync')
const KEY = 'tab-sync-event'

function publish(event: { type: string, payload?: unknown }) {
  const data = { ...event, ts: Date.now() }
  channel.postMessage(data)
  localStorage.setItem(KEY, JSON.stringify(data))
}

channel.onmessage = ({ data }) => {
  if (data.type === 'logout') location.reload()
}

window.addEventListener('storage', (e) => {
  if (e.key !== KEY || !e.newValue) return
  const data = JSON.parse(e.newValue)
  if (data.type === 'logout') location.reload()
})
```
   - 追问 1：为什么常做双通道兜底？
   - 标准答案 1：单一通道在兼容或异常场景可能失效，双通道可提升一致性和可恢复性。

3. 僵尸进程和孤儿进程是什么？
   - 标准答案：孤儿进程是父进程退出后仍在运行的子进程，僵尸进程是子进程退出后资源描述未被父进程回收；两者都属于进程生命周期管理问题。
   - 考察点：是否理解进程退出与资源回收的关系。

4. 死锁产生的原因？ 如果解决死锁的问题？
   - 标准答案：死锁通常由互斥、请求保持、不可剥夺、循环等待共同触发；可通过资源有序分配、一次性申请、超时回滚等策略打破必要条件。
   - 考察点：是否能给出“原因 + 条件 + 解除策略”的完整回答。
