## 概念定义

- 本质定义：HTML5 能力总览关注 HTML（HyperText Markup Language）从“只描述结构”扩展到“原生支持媒体、表单、拖拽、模板、离线和自定义数据”的能力升级。
- 解决问题：建立一张“新能力有哪些、该放在哪用、什么时候不该硬上”的选型地图。
- 边界说明：本节点做能力总览，具体的媒体、Worker、离线缓存和 `iframe` 细节会在各自节点展开。

## 核心机制拆解

### 1) 机制/流程

1. 先按能力分类：语义标签、表单增强、媒体图形、交互能力、离线与存储、模板与组件协作。
2. 再按使用方式分类：纯声明式能力、声明 + 脚本协作能力、必须依赖浏览器支持的增强能力。
3. 最后按兼容风险决定是直接使用，还是走渐进增强和回退路径。

### 2) 关键条件与触发点

- 条件 A：只记 API 名字不记用途，面试答题就会变成零散罗列。
- 条件 B：看到 HTML5 原生能力就全都自己实现一遍，往往会忽略更成熟的浏览器默认行为。
- 条件 C：把“浏览器支持”误解成“业务一定适合”，会导致交互体验和维护成本失衡。

### 3) 结果与影响

- 对性能/稳定性/可维护性的影响：用原生能力能少写很多胶水代码；但能力边界没搞清楚时，也会把浏览器差异和兼容回退问题一起引进来。

## 相似概念对比

| 维度 | HTML5 新增能力 | 历史方案或旧写法 | 结论 |
|---|---|---|---|
| 结构表达 | `header`、`main`、`article` 等 | 大量无语义容器 | 语义表达更直接 |
| 交互增强 | drag API、`details`、`contenteditable` | 纯脚本模拟 | 原生能力更省代码 |
| 模板能力 | `template`、`slot` | 字符串拼接、手写占位结构 | 结构复用更清晰 |
| 数据承载 | `data-*` | 自定义非标准属性或额外映射对象 | 数据挂载更规范 |

## 代码

```html
<div
  draggable="true"
  data-card-id="todo-1"
>
  待办卡片
</div>

<details>
  <summary>查看说明</summary>
  <p>拖拽时通过 data-card-id 识别卡片身份。</p>
</details>

<template id="card-template">
  <article class="card">
    <h3 class="title"></h3>
  </article>
</template>
```

```js
const source = document.querySelector('[draggable="true"]')

source?.addEventListener('dragstart', (event) => {
  event.dataTransfer?.setData('text/plain', source.dataset.cardId ?? '')
})
```

### 示例说明

- 这段代码解决什么问题：把拖拽、折叠说明、模板复用和自定义数据属性放到一个最小 HTML5 场景里。
- 关键语句或参数含义：`draggable` 开启拖拽；`data-card-id` 存放自定义数据；`details` / `summary` 提供原生展开收起；`template` 保存待克隆结构。
- 常见错误写法：自定义数据不用 `data-*`，拖拽只写 `dragstart` 不写落点处理，模板内容直接写成字符串拼接。

## 真实场景示例

- 场景描述：后台看板需要卡片拖拽、可折叠说明、局部模板复用和组件初始化数据挂载。
- 问题表现：全部用脚本模拟，代码量很大，回归时容易在边角交互翻车。
- 处理方案：能用原生能力的地方优先用原生能力，复杂交互只在原生能力之上做增强。
- 验证方式：看代码复杂度、交互缺陷数、键盘回归和低网速下的初始化稳定性。

## 常见误区

- 误区：HTML5 的更新只是一批新语义标签。
- 正解：它还带来了表单、拖拽、模板、数据属性、媒体和离线相关能力。
- 误区：`data-*` 只是给测试同学留标记。
- 正解：它本来就是标准的自定义数据承载方式，业务配置和行为挂载都能用。
- 误区：原生拖拽只要元素能拖起来就算完成。
- 正解：拖拽源、落点、`dataTransfer` 和回退体验要一起设计。

## 关联知识点

- 依赖：语义化与可访问性、渐进增强与优雅降级。
- 影响：表单与原生交互控件、媒体与图形、离线存储与 Service Worker。
- 对比：原生增强能力和纯脚本模拟方案。
- 组合：自定义数据、模板复用、Web Components 协作。

## 相关面试题

1. HTML5 相比 HTML4 有哪些主要改进？
- 标准答案：可按五类回答。第一类是语义标签，如 `header`、`nav`、`article`、`section`；第二类是媒体和图形，如 `audio`、`video`、Canvas；第三类是表单增强，如新的输入类型和原生校验属性；第四类是存储和离线相关能力，如 Web Storage（Web Storage）、离线缓存；第五类是交互能力，如 Web Worker（Web Worker）、drag API。场景上这些更新的核心价值是减少插件依赖和脚本模拟成本。
- 考察点：是否能按类别组织答案，而不是散着背特性名。

2. HTML5 有哪些更新
- 标准答案：如果面试官这样问，通常还是想听“分类 + 代表项”。至少要覆盖语义标签、媒体标签、表单增强、存储与离线、拖拽和图形能力。机制上这些能力让浏览器原生职责更完整。场景上真正答题时，最好顺手补一两个项目里用过的例子，比如 `srcset`、`required` 或 drag API。
- 考察点：是否具备分类表达和场景化回答能力。

3. 什么是HTML的data属性？
- 标准答案：`data-*` 是标准的自定义数据属性，用来把轻量配置或业务标识挂在元素上。机制上浏览器会把它们暴露到 `dataset` 上，脚本可以直接读写。场景上组件初始化参数、埋点标识、行为配置都很适合放这里。
- 考察点：是否知道 `data-*` 是标准能力，而不是随便造属性。

4. HTML中的details和summary标签是什么？
- 标准答案：`details` 和 `summary` 用来创建原生的展开/收起区域，`summary` 是标题，`details` 是容器。机制上浏览器会维护 `open` 状态，并处理基本交互。场景上 FAQ、说明文档和非关键附加信息很适合直接用它们。
- 考察点：是否知道浏览器已经提供了声明式折叠能力。

5. HTML中的template标签是什么？
- 标准答案：`template` 用来存放不会直接渲染到页面上的结构模板，需要脚本克隆后再插入文档。机制上它的内容保存在 `template.content` 里。场景上列表项模板、卡片模板和轻量组件初始化都很常见。
- 考察点：是否理解“声明模板”和“立即渲染”的区别。

6. HTML中的slot标签在Web Components中如何使用？
- 标准答案：`slot` 是 Web Components 的内容分发入口，允许组件外部内容插进组件内部指定位置。机制上可以有默认插槽，也可以有命名插槽。场景上做卡片、弹窗、面板这类可定制容器组件时，`slot` 非常关键。
- 考察点：是否知道插槽不是框架专属概念，浏览器原生就有。

7. HTML中的contenteditable属性如何使用？
- 标准答案：`contenteditable` 让普通元素变成可编辑区域。机制上浏览器会把该元素当作输入上下文，用户可以直接修改其内容。场景上富文本编辑器、便签、内联编辑很常见，但要注意粘贴内容清洗和输出安全。
- 考察点：是否理解它适合“轻编辑”，但不等于完整编辑器方案。

8. HTML中的draggable属性如何实现拖拽功能？
- 标准答案：把元素设为 `draggable="true"` 后，再配合 `dragstart`、`dragover`、`drop`、`dragend` 等事件和 `dataTransfer` 数据传递，就能完成拖拽链路。机制上拖拽源负责放数据，目标区域必须在 `dragover` 里调用 `preventDefault()` 才能接收落点。场景上列表排序、文件拖入上传和看板编排都很常见。
- 考察点：是否掌握拖拽链路的关键事件和触发条件。

9. HTML中的hidden属性如何使用？
- 标准答案：`hidden` 是声明式隐藏属性，表示元素当前不应该展示。机制上浏览器会把它当作不可见内容处理。场景上高级筛选、模态框、折叠区域和加载完成前的占位内容都可以用它做基础状态控制，但复杂动画切换通常还要配合 CSS 或脚本。
- 考察点：是否知道它是语义上的“隐藏状态”，而不只是一个样式技巧。

10. HTML5中的自定义数据属性（data-*）在实际开发中有哪些应用场景？
- 标准答案：常见场景有组件初始化配置、埋点标识、行为开关、商品卡片元信息、列表项主键和测试钩子。机制上这些数据会跟随元素存在，脚本读取成本低，也不会污染标准属性空间。场景上它适合轻量数据，不适合承载大对象或真正的业务状态中心。
- 考察点：是否知道 `data-*` 的典型应用边界。
