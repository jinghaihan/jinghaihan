## 概念定义

- 本质定义：Web Worker 是浏览器提供的后台线程执行环境，用来把可独立计算的任务从主线程拆出去。
- 解决问题：减少主线程被大计算任务占满时的卡顿和交互延迟。
- 边界说明：它不能直接操作 DOM（Document Object Model），也不负责网络代理和离线缓存，这些属于 Service Worker 的职责。

## 核心机制拆解

### 1) 机制/流程

1. 主线程通过 `new Worker()` 创建 Worker 并加载独立脚本。
2. 主线程用 `postMessage` 把数据或任务描述发给 Worker。
3. Worker 在隔离线程里完成计算，再把结果回传给主线程。
4. 主线程接收结果，最后再做 UI（User Interface）更新。

### 2) 关键条件与触发点

- 条件 A：任务本身计算量足够大，而且能和页面渲染解耦，才值得迁移到 Worker。
- 条件 B：频繁传输大对象或复制大数组时，通信成本可能抵消收益。
- 条件 C：如果任务最终还是要频繁操作 DOM，Worker 带来的价值会明显下降。

### 3) 结果与影响

- 对性能/稳定性/可维护性的影响：划分清楚时，长任务明显减少；划分随意时，通信协议和调试成本会迅速上升。

## 相似概念对比

| 维度 | Web Worker | Service Worker | 结论 |
|---|---|---|---|
| 核心职责 | 后台计算 | 网络代理和离线能力 | 两者解决的问题不同 |
| 生命周期 | 通常跟页面生命周期绑定 | 独立于页面，可后台存活 | 不可互相替代 |
| DOM 操作 | 都不能直接操作 DOM | 都不能直接操作 DOM | 结果仍需主线程更新界面 |
| 适用场景 | 大数据计算、解析、压缩 | 缓存、离线、推送、后台同步 | 一个算数据，一个管请求 |

## 代码

```js
// main.js
const worker = new Worker('/workers/calc.js')

worker.postMessage({ numbers: [1, 2, 3, 4, 5] })

worker.onmessage = (event) => {
  console.log('sum:', event.data.total)
}
```

```js
// /workers/calc.js
self.onmessage = (event) => {
  const total = event.data.numbers.reduce((sum, item) => sum + item, 0)
  self.postMessage({ total })
}
```

### 示例说明

- 这段代码解决什么问题：把求和计算挪到后台线程，主线程只保留交互和展示。
- 关键语句或参数含义：`Worker` 创建后台线程；`postMessage` 负责线程间消息传递。
- 常见错误写法：在 Worker 脚本里直接访问 `document` 或频繁来回传大对象。

## 真实场景示例

- 场景描述：报表页需要解析大体量数据并生成图表，用户一滚动页面就发卡。
- 问题表现：输入延迟变高，主线程长任务堆积，图表渲染前页面几乎没法操作。
- 处理方案：把数据解析和聚合挪进 Worker，主线程只接收处理后的结果并负责绘制。
- 验证方式：看长任务数量、输入延迟、图表首次可见时间和主线程占用。

## 常见误区

- 误区：只要用了 Worker，页面一定更快。
- 正解：任务太小或通信太频繁时，收益可能还不如直接在主线程做。
- 误区：Worker 是更高级的异步任务。
- 正解：它不是 `Promise` 的替代品，而是线程层面的任务隔离。
- 误区：Web Worker 和 Service Worker 只是名字不同。
- 正解：前者管计算，后者管请求和缓存，生命周期也完全不同。

## 关联知识点

- 依赖：JavaScript 体系中的事件循环与消息传递。
- 影响：主线程可交互性、图形渲染稳定性。
- 对比：主线程计算和后台线程计算。
- 组合：媒体处理、加密解密、离线存储与 Service Worker。

## 相关面试题

1. 对 web worker 的理解
- 标准答案：Web Worker 的核心价值是把计算密集任务放到后台线程，避免主线程被长任务阻塞。机制上主线程和 Worker 通过 `postMessage` 通信，Worker 不能直接操作 DOM。场景上大数据解析、复杂计算、图像处理和压缩任务很适合放进 Worker。
- 考察点：是否理解 Worker 的职责边界和通信方式。

2. 多线程之 Web Worker 与 Service Worker
- 标准答案：Web Worker 解决“算得慢”的问题，把 CPU 密集任务从主线程剥离；Service Worker 解决“请求慢和离线可用”的问题，位于网络请求链路中管理缓存与回源。两者都不能直接操作 DOM，但生命周期和职责边界完全不同，不能互相替代。
- 考察点：是否能准确对比两类 Worker 的目标、位置和生命周期。
