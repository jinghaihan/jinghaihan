## 概念定义

- 本质定义：Web Worker 是浏览器提供的后台线程执行模型，用于把计算密集任务从主线程拆出去。
- 解决问题：降低主线程阻塞，提升页面交互响应性。
- 边界说明：Worker 不能直接操作 DOM，只能通过消息通信与主线程协同。

## 核心机制拆解

### 1) 机制/流程

1. 主线程创建 Worker 实例并加载 Worker 脚本。
2. 主线程通过 `postMessage` 发送任务数据。
3. Worker 在线程内执行计算并回传结果。
4. 主线程接收结果后更新 UI（User Interface）。

### 2) 关键条件与触发点

- 条件 A：任务计算量大且与 UI 解耦，适合迁移到 Worker。
- 条件 B：频繁大对象消息传输会抵消部分收益。

### 3) 结果与影响

- 对性能/稳定性/可维护性的影响：迁移得当可明显减少卡顿；边界设计不清会引入通信复杂度。

## 相似概念对比

| 维度 | 主线程执行 | Web Worker 执行 | 结论 |
|---|---|---|---|
| 执行位置 | UI 主线程 | 后台线程 | 并行模型不同 |
| DOM 操作 | 可直接操作 | 不可直接操作 | 职责要分层 |
| 对交互影响 | 易造成卡顿 | 可减轻卡顿 | 计算任务优先下沉 |
| 适用场景 | 轻量逻辑和 UI 更新 | 大数据计算、解析、压缩 | 按任务类型分配 |

## 代码

```js
// main.js
const worker = new Worker('/workers/sum.js')
worker.postMessage({ numbers: [1, 2, 3, 4, 5] })
worker.onmessage = (event) => {
  console.log('sum:', event.data)
}
```

```js
// /workers/sum.js
self.onmessage = (event) => {
  const total = event.data.numbers.reduce((acc, n) => acc + n, 0)
  self.postMessage(total)
}
```

### 示例说明

- 这段代码解决什么问题：把求和计算放到后台线程，避免主线程被同步计算占满。
- 关键语句或参数含义：`postMessage` 负责主线程和 Worker 双向通信。
- 常见错误写法：在 Worker 中尝试直接访问 DOM。

## 真实场景示例

- 场景描述：报表页面需要解析大体量数据，滚动和点击明显卡顿。
- 问题表现：主线程长任务增多，输入延迟升高。
- 处理方案：把数据解析迁移到 Worker，主线程只负责渲染和交互。
- 验证方式：观察长任务数量、交互延迟、页面卡顿投诉。

## 常见误区

- 误区：Worker 可以替代所有异步任务。
- 正解：Worker 更适合计算密集且可独立的数据处理任务。
- 误区：只要用了 Worker，性能一定提升。
- 正解：消息序列化和通信开销也需要评估。

## 关联知识点

- 依赖：JavaScript 体系的事件循环与任务调度。
- 影响：交互稳定性、渲染流畅度。
- 对比：主线程执行与后台线程执行。
- 组合：离线能力、图形渲染任务分层。

## 相关面试题

1. 对 web worker 的理解
- 标准答案：结论是 Web Worker 用于把计算任务放到后台线程，减少主线程阻塞。机制上主线程与 Worker 通过 `postMessage` 通信，Worker 不能直接操作 DOM。场景上适合大数据解析、加密压缩和复杂计算。
- 考察点：是否理解 Worker 的价值边界和通信机制。
- 追问 1：什么场景不建议用 Worker？
- 标准答案 1：结论是轻量计算或高频小任务不一定适合。机制上频繁通信和序列化会引入额外成本。反例是每次输入都把小计算发到 Worker，整体开销可能更高。
