## 概念定义

- 本质定义：基础类型与类型推断关注 TypeScript 如何在编译阶段描述值的形状、空值约束和不可达分支，并在大多数场景下自动推导类型。
- 解决问题：减少“参数传错了还没人拦”“变量越写越像 `any`”“空值处理靠运气”这类问题。
- 边界说明：本节点聚焦基础类型系统，不展开对象契约、泛型和高级类型编程。

## 核心机制拆解

### 1) 机制/流程

1. 开发者为变量、参数和返回值显式标注类型，或让编译器基于初始值做类型推断。
2. `strictNullChecks`、字面量类型和元组会进一步收紧可赋值范围。
3. 当分支不可能到达时，类型系统会推断出 `never`，帮助做穷尽检查。
4. 当外部数据不可信时，优先用 `unknown` 接住，再逐步收窄。

### 2) 关键条件与触发点

- 条件 A：一旦引入 `any`，相关变量会迅速失去后续类型保护。
- 条件 B：关闭严格空检查后，很多空值错误会从编译阶段溜到运行时。
- 条件 C：把推断当成“完全不用写类型”，常见结果是公共边界处的类型越来越模糊。

### 3) 结果与影响

- 对性能/稳定性/可维护性的影响：类型越早被约束，重构越稳；基础类型放任失控，后面的高级类型再多也补不回来。

## 相似概念对比

| 维度 | `any` | `unknown` | 结论 |
|---|---|---|---|
| 类型检查 | 基本关闭 | 保留，使用前必须收窄 | 外部不可信数据优先 `unknown` |
| 可赋值范围 | 几乎可赋给任何类型 | 赋值前需要判断或断言 | 安全边界不同 |
| 适用场景 | 仅历史迁移或极少数兜底 | `JSON.parse`、第三方输入、边界数据 | 默认优先后者 |
| 风险 | 类型污染扩散很快 | 使用成本更高但更安全 | 宁可多一步收窄 |

## 代码

```ts
type Status = 'idle' | 'loading' | 'done'

const pair: [number, string] = [1, 'Ada']
const raw: unknown = JSON.parse('{"name":"Ada"}')

function getMessage(status: Status): string {
  switch (status) {
    case 'idle':
      return 'waiting'
    case 'loading':
      return 'loading'
    case 'done':
      return 'done'
    default: {
      const exhaustive: never = status
      return exhaustive
    }
  }
}

if (typeof raw === 'object' && raw !== null && 'name' in raw)
  console.log((raw as { name: string }).name)
```

### 示例说明

- 这段代码解决什么问题：同时演示字面量类型、元组、`unknown` 接外部数据和 `never` 穷尽检查。
- 关键语句或参数含义：`Status` 把状态收成有限集合；`[number, string]` 固定元组位置；`never` 用来兜底不可能分支。
- 常见错误写法：`JSON.parse` 直接给 `any`，或者把可枚举状态写成宽泛字符串。

## 真实场景示例

- 场景描述：活动页接口字段经常变动，前端又喜欢直接从响应里拿值，结果一改字段就线上报错。
- 问题表现：空值、字段缺失和状态拼写错误只能靠测试阶段碰运气。
- 处理方案：外部数据先接成 `unknown`，关键状态用字面量联合类型，打开严格空检查。
- 验证方式：看编译期报错数量、空值相关线上错误和接口变更后的重构成本。

## 常见误区

- 误区：`any` 最省事，后面需要时再慢慢补类型。
- 正解：`any` 一旦扩散，后面补类型的成本会更高。
- 误区：类型推断很强，所以所有地方都不用写显式类型。
- 正解：局部变量可以多用推断，公共函数边界和返回值仍然建议明确声明。
- 误区：`never` 没什么用，平时见不到。
- 正解：它在穷尽检查、抛错函数和不可能分支里很关键。

## 关联知识点

- 依赖：JavaScript 的基础类型和控制流。
- 影响：对象建模、类型守卫、泛型和工具类型的可用性。
- 对比：`any`、`unknown`、`never`。
- 组合：严格模式、空值处理、状态建模。

## 相关面试题

1. TypeScript 中的 `any` 和 `unknown` 有什么区别？
- 标准答案：`any` 会基本关闭类型检查，后续对它做什么操作编译器都很难拦；`unknown` 则保留检查，使用前必须先收窄或断言。机制上两者都能接住未知输入，但安全边界完全不同。场景上接口原始数据、`JSON.parse` 结果优先用 `unknown`，不要默认上来就是 `any`。
- 考察点：是否知道两者不是“语义差不多的两个兜底类型”。

2. 什么是 TypeScript 中的 `never` 类型？它有什么用？
- 标准答案：`never` 表示“不应该出现任何值”的类型，常出现在永远抛错的函数、死循环和穷尽检查里。机制上它帮助编译器确认某个分支已经不可能继续往下走。场景上写 `switch` 穷尽检查时最有价值，能防止新增状态漏处理。
- 考察点：是否真正理解 `never` 的语义，而不是只会背定义。

3. 什么是元组（Tuple）类型？
- 标准答案：元组是“长度和每个位置类型都已知”的数组类型。机制上它按下标保存类型信息，不同位置可以是不同类型。场景上返回 `[状态码, 消息]`、`[id, name]` 这类固定结构时很合适，但如果位置语义不清晰，最终还是建议改成对象。
- 考察点：是否知道元组和普通数组的边界。

4. TypeScript 是如何进行类型推断的？
- 标准答案：TypeScript 会根据初始值、上下文和返回表达式自动推导类型。机制上局部变量、函数返回值和泛型参数都能参与推断。场景上推断能减少样板代码，但在公共 API 和复杂返回值场景里，显式声明往往更稳。
- 考察点：是否知道推断很强，但不是“处处都不用写类型”。

5. `void` 类型有什么用？
- 标准答案：`void` 表示函数没有可用返回值。机制上它通常出现在副作用函数或回调签名里，提醒调用方不要依赖返回结果。场景上事件处理、日志函数和纯执行型流程经常会用到 `void`。
- 考察点：是否知道 `void` 是返回值约束，而不是“什么都能放”的空类型。

6. TypeScript 的严格空检查 (`strictNullChecks`) 是什么？
- 标准答案：开启后，`null` 和 `undefined` 不再默认能赋给所有类型，必须显式处理。机制上它让空值成为类型系统的一部分。场景上这能大幅减少“运行时读取了空对象属性”的错误，是中大型项目几乎必开的选项。
- 考察点：是否理解严格空检查的实际价值。

7. `unknown` 和 `any` 在 JSON 解析场景下如何选择？
- 标准答案：优先选 `unknown`。机制上 `JSON.parse` 得到的数据天然不可信，用 `unknown` 可以强制你先做校验再使用；用 `any` 则会把风险直接放进业务逻辑。场景上接口响应、LocalStorage 反序列化结果都适用这个原则。
- 考察点：是否会把类型系统用在真实边界数据上。

8. 什么是字面量类型（Literal Types）？
- 标准答案：字面量类型就是把类型收窄到某个具体值，比如 `'success'`、`404`、`true`。机制上它通常和联合类型配合，形成一组有限状态集合。场景上状态机、按钮尺寸、主题模式和角色值很适合用字面量类型约束。
- 考察点：是否知道字面量类型是状态建模工具，而不只是语法细节。

9. 如何优雅地处理可选链 (`?.`) 和空值合并 (`??`)？
- 标准答案：`?.` 用来安全访问可能为空的链路，`??` 用来只在值为 `null` 或 `undefined` 时给默认值。机制上它们避免了层层空值判断和误用 `||` 带来的假值问题。场景上接口回显、可选配置和深层对象访问时非常常见。
- 考察点：是否理解它和 `||` 的语义差异。

10. `satisfies` 操作符是做什么的？
- 标准答案：`satisfies` 用来校验一个值是否满足目标类型，但尽量保留原始值更精细的推断结果。机制上它不像显式类型注解那样直接把值“压扁”成目标类型。场景上配置对象、常量映射表和字典结构特别适合它。
- 考察点：是否理解“校验契约”和“保留字面量信息”这两个点。

11. `never` 类型在 TypeScript 中有什么用？
- 标准答案：结论是 `never` 用于表达“不应该出现任何值”的分支。机制上它常用于穷尽检查、永不返回函数和不可能到达路径。场景上状态机 `switch` 里用 `never` 可防止新增分支漏处理。
- 考察点：是否能把 `never` 从定义落到真实排错场景。

12. `unknown` 和`any` 在 JSON 解析场景下如何选择？
- 标准答案：结论是优先选 `unknown`。机制上 `unknown` 强制先校验再使用，`any` 会让类型风险直接扩散。场景上接口响应和本地缓存反序列化都应先用 `unknown` 承接边界数据。
- 考察点：是否理解边界数据类型策略。
