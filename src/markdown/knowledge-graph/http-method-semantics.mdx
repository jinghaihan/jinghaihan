## 概念定义

- 本质定义：HTTP（HyperText Transfer Protocol）方法语义定义了请求对资源的意图和副作用边界。
- 解决问题：为缓存策略、重试策略、幂等保障和接口治理提供统一约束。
- 边界说明：它不决定传输性能上限，性能更多由协议版本和连接策略决定。

## 核心机制拆解

### 1) 机制/流程

1. 客户端按业务意图选择方法（GET、POST、PUT、PATCH、DELETE 等）。
2. 服务端按方法语义匹配路由与处理逻辑。
3. 网关和中间层根据方法语义执行缓存、限流和重试策略。
4. 响应返回后，客户端按方法与状态码决定后续行为。

### 2) 关键条件与触发点

- 条件 A：方法是否幂等，决定是否可以自动重试。
- 条件 B：是否触发预检（如自定义头和跨域复杂请求），决定额外往返成本。

### 3) 结果与影响

- 对性能 / 稳定性 / 可维护性的影响：语义清晰可避免错误重试和脏写；方法滥用会导致缓存失效、审计困难和线上副作用。

## 相似概念对比

| 维度 | POST | PUT | 结论 |
|---|---|---|---|
| 定义 | 常用于创建资源或触发动作 | 常用于更新或替换指定资源 | 两者都可带请求体，但语义不同 |
| 触发时机 | 新增、提交、触发非幂等操作 | 已知资源地址的覆盖式更新 | 是否幂等是核心区分点 |
| 性能影响 | 默认不宜自动重试 | 可在幂等前提下做重试 | 重试策略必须跟语义绑定 |
| 适用场景 | 创建订单、提交表单 | 配置更新、资源覆盖 | 面试常追问“为什么 PUT 更适合重试” |

## 代码

```bash
curl -X GET 'https://api.example.com/users/1'
curl -X POST 'https://api.example.com/users' -H 'Content-Type: application/json' -d '{"name":"A"}'
curl -X PUT 'https://api.example.com/users/1' -H 'Content-Type: application/json' -d '{"name":"B"}'
```

```http
OPTIONS /orders HTTP/1.1
Origin: https://app.example.com
Access-Control-Request-Method: POST
Access-Control-Request-Headers: Content-Type, Authorization
```

### 示例说明

- 这段代码 / 配置解决什么问题：第一段展示常见方法语义，第二段展示跨域复杂请求触发预检。
- 关键参数含义：`Access-Control-Request-Method` 声明后续真实请求的方法。
- 常见错误配置：把非幂等接口接入自动重试，导致重复下单或重复扣款。

## 真实场景示例

- 场景描述：支付接口偶发超时后客户端自动重试。
- 问题表现：同一笔订单被重复创建。
- 处理方案：将创建接口保持 POST 且引入幂等键，重试仅在具备幂等保护时放行。
- 验证方式：压测超时重试场景，校验订单去重率与错误率。

## 常见误区

- 误区：GET 和 POST 只有参数位置不同。
  - 正解：两者在语义、缓存行为、幂等性和安全暴露面上都不同。
- 误区：PUT 和 POST 可以随便互换。
  - 正解：PUT 强调对指定资源的幂等更新，POST 更偏向创建或动作触发。

## 关联知识点

- 依赖：`tcp-tls` 提供可用连接通道。
- 影响：方法语义会影响 `status-code-path` 和缓存策略。
- 对比：`http-method-semantics` 关注“做什么”，`http-message-headers` 关注“怎么描述”。
- 组合：与 `security-cors` 结合高频考预检触发条件。

## 相关面试题

1. GET 和 POST 的请求的区别
   - 标准答案：GET 主要用于读取且通常可缓存，POST 常用于创建或提交操作；两者不仅是参数位置差异，还涉及幂等性、缓存策略和安全暴露面差异。
   - 考察点：是否能从语义和工程后果两个层面回答。
   - 追问 1：GET 一定比 POST 安全吗？
   - 标准答案 1：不是，HTTP 本身都可被抓包；安全核心在 HTTPS 和鉴权设计，GET 只是更容易暴露在 URL 与日志里。
   - 追问 2：GET 长度为什么会受限？
   - 标准答案 2：限制来自浏览器和服务器实现，不是 HTTP 规范硬限制。

2. POST 和 PUT 请求的区别
   - 标准答案：POST 常用于创建或触发非幂等动作，PUT 常用于对已知资源做幂等更新，因此 PUT 更适合在确定幂等时做自动重试。
   - 考察点：是否理解幂等性对重试和稳定性的影响。
