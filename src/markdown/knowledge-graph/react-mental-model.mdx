## 概念定位

React 心智模型节点关注声明式 UI、单向数据流和状态驱动渲染原则，它是理解后续 Hook、并发渲染和性能优化的前提。

心智模型错误会导致过度副作用、状态错位和更新不可预测。

## 工作机制

1. 把 UI 看作状态到视图的映射，而不是命令式 DOM 操作。
2. 数据流从父到子单向流动，变更通过事件向上反馈。
3. 组件每次渲染都基于当前 props/state 重新计算输出。
4. 副作用与渲染分离，渲染阶段保持纯函数思维。
5. 把共享状态上收或外置，避免多源真相。

## 关键知识点

### 必须掌握

- React 不会自动帮你管理状态边界，需主动设计。
- 渲染不是 DOM 立即更新，而是先计算再提交。
- 组件函数会重复执行，闭包值需特别关注。
- 单向数据流是可预测性的基础。
- 状态最小化有助于减少无效更新。
- 把副作用放在渲染里会导致时序问题。

### 选型与取舍

- 状态集中便于一致性，但会增加全局复杂度。
- 状态分散灵活，但同步成本高。
- 抽象过深可复用高，但调试难度上升。

## 对页面性能与稳定性的影响

- 节点稳定时，React 工作流 的链路可预测性会明显提升，发布和排障成本更可控。
- 节点失稳时，问题会向上下游扩散，常表现为性能回退、交互异常或交付风险叠加。
- 该节点优化应和上下游协同验证，避免局部优化把成本转移给后续节点。

## 观测与排障

### 核心指标

- 组件重渲染分布
- 状态更新扇出范围
- 副作用重复执行次数
- 关键交互响应时延

### 排障顺序

1. 先确认问题是状态边界还是副作用时序。
2. 检查是否存在多源状态。
3. 检查渲染函数是否包含副作用。
4. 识别高频重渲染组件。
5. 通过状态收敛与组件拆分回归验证。

### 常见故障场景

- 状态在多个组件重复维护导致不一致。
- 渲染阶段触发请求导致重复调用。
- 组件改动引发大范围重渲染。
- 闭包值过期导致逻辑错误。

## 常见误区

- 把 React 当成模板引擎而非状态机。
- 忽略单向数据流，直接跨层修改状态。
- 把副作用逻辑塞进渲染函数。

## 题库深挖补充

### 高频对比与边界（基于题库）

- React 核心是状态驱动渲染，不是命令式改 DOM。
- JSX 是语法糖，组件抽象质量取决于职责边界而非语法本身。
- state 与 props 分工不清会导致重复状态和渲染扩散。
- 测试与发布要围绕关键路径，而不是只追覆盖率数字。
- 团队治理要把 Hook 规则、副作用边界写进可执行规范。
- 面试常问“为什么 React 不断引入新范式”，核心是抽象与调度能力演进。

### 线上排障剧本

1. 先确认问题是状态边界错误还是组件抽象错误。
2. 检查是否存在重复状态来源。
3. 检查组件输入输出契约是否稳定。
4. 回归关键页面交互和异常路径。
5. 把结论沉淀到评审规范与测试用例。

### 高频追问补充（附答案）

1. 为什么单向数据流能提升可维护性？

更新路径可追踪，状态源更明确，排障成本更低。

2. 组件抽象越细越好吗？

不一定，过细会增加组合复杂度与通信成本。

3. React 项目治理最容易忽略什么？

副作用边界和状态生命周期规则。

## 进阶追问（补充）

1. React 优化中最容易造成反效果的动作是什么？

盲目加 memo/useCallback，增加复杂度却没有命中真正热点。

2. 为什么 effect 问题总在复杂页面里更明显？

组件层级深、状态来源多、异步并发高，会放大依赖与清理缺陷。

3. 如何判断状态管理选型是否需要升级？

当更新粒度无法收敛、排障不可追踪、跨页面一致性反复出问题时。

## 案例追问加深（第三轮）

### 真实故障片段

- 故障表现：交互偶发卡顿，且问题难以稳定复现。
- 排查结论：Hook 依赖、状态订阅粒度和组件边界共同导致重渲染扩散。
- 治理动作：先收敛状态边界，再治理 effect 依赖，最后做热点组件优化。

### 追问补充

1. 为什么 React 性能问题常和状态设计绑定？

状态边界决定更新扇出，扇出范围决定渲染成本上限。

2. 如何避免“优化 API 堆叠但收益很小”？

先定位热点和根因，再选择最小有效动作，而不是先上 memo 类技巧。

## 面试补充

### 30 秒说明

React 心智模型的关键是“状态驱动 + 单向数据流 + 渲染纯函数”，这是所有高级特性的共同前提。

### 高频追问

1. 为什么单向数据流这么重要？

它让数据来源可追踪，降低状态错乱和调试成本。

2. React 渲染函数能做副作用吗？

不建议，副作用应放在 effect 或事件回调中。

3. 多源状态为什么危险？

同一事实在多个地方维护会导致一致性崩溃。

4. 怎么判断心智模型是否正确？

看状态边界是否清晰、更新路径是否可预测。
