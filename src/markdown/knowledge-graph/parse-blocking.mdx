## 概念定义

- 本质定义：解析阻塞治理是控制文档解析、脚本执行、样式加载之间互相等待关系的优化方法。
- 解决问题：缩短关键渲染路径，降低首屏等待时间。
- 边界说明：它不直接替代后端接口优化和资源体积压缩。

## 核心机制拆解

### 1) 机制/流程

1. 文档解析遇到阻塞脚本会暂停，等待脚本下载和执行。
2. 脚本若依赖样式信息，可能等待样式可用后再执行。
3. 预解析线程可提前发现后续资源并并行下载。
4. 通过 `defer`、`async`、资源优先级和内联关键样式减少阻塞。
5. 优化后关键路径缩短，首屏更早可见。

### 2) 关键条件与触发点

- 条件 A：脚本是否同步执行，决定是否阻塞文档解析。
- 条件 B：关键样式是否及时到达，决定脚本和渲染是否排队。

### 3) 结果与影响

- 对性能 / 稳定性 / 可维护性的影响：阻塞治理到位可显著降低首屏时延，治理缺失会引发“网络空闲但页面未显示”的体感卡顿。

## 相似概念对比

| 维度 | `defer` | `async` | 结论 |
|---|---|---|---|
| 定义 | 下载并行，文档解析后按顺序执行 | 下载并行，下载完成立即执行 | 二者都可减少同步阻塞 |
| 触发时机 | DOM（Document Object Model）构建完成后 | 资源下载完成后 | `async` 执行时机不保证顺序 |
| 性能影响 | 对首屏稳定性更友好 | 对独立脚本更灵活 | 需按依赖关系选型 |
| 适用场景 | 有顺序依赖脚本 | 无依赖统计脚本 | 面试高频对比点 |

## 代码

```html
<script defer src="/app.js"></script>
<script async src="https://example.com/analytics.js"></script>
<link rel="preload" href="/critical.css" as="style" />
```

### 示例说明

- 这段代码解决什么问题：把关键业务脚本与独立统计脚本分策略加载，减少主解析阻塞。
- 关键语句或参数含义：`defer` 保序执行，`async` 不保序，`preload` 提前拉取关键资源。
- 常见错误写法：所有脚本都设为 `async`，导致依赖脚本执行顺序错乱。

## 真实场景示例

- 场景描述：首页脚本拆分后，首屏白屏时间仍长。
- 问题表现：网络请求返回快，但内容渲染晚。
- 处理方案：关键样式前置、同步脚本改为 `defer`、非关键脚本延迟加载。
- 验证方式：对比优化前后首屏渲染时点和主线程阻塞时间。

## 常见误区

- 误区：把脚本放到 `body` 末尾就等于完全不阻塞。
- 正解：仍可能阻塞后续任务执行，需结合加载属性和依赖关系设计。

## 关联知识点

- 依赖：依赖 `html-parser` 和 `cssom-js` 的执行时序。
- 影响：直接影响 `render-tree` 首次构建时点。
- 对比：`parse-blocking` 关注阻塞治理策略，`layout` 关注计算成本。
- 组合：常与 `resource-hints`、`cache-policy` 联动优化。

## 相关面试题

1. 渲染过程中遇到 JS 文件如何处理？
- 标准答案：解析器遇到同步脚本会暂停文档解析，先下载并执行脚本，再继续解析；可通过 `defer` 或 `async` 改造阻塞路径。
- 考察点：是否能说清暂停点和恢复点。

2. 什么是文档的预解析？
- 标准答案：预解析是在主解析执行脚本时，由辅助线程提前扫描后续资源引用并并行下载，以减少空等待时间。
- 考察点：是否理解“预解析只预取资源，不直接改 DOM”。

3. 如何优化关键渲染路径？
- 标准答案：关键动作包括减少关键资源数量、缩短依赖链、降低关键字节量、前置关键样式和延后非关键脚本。
- 考察点：是否能显式列举优化动作，而不是只讲原则。

4. 什么情况会阻塞渲染？
- 标准答案：常见阻塞来源包括同步脚本执行、关键样式未就绪、主线程长任务、频繁同步布局读取和高成本绘制任务。
- 考察点：是否能完整列举阻塞类型并说明触发条件。

5. JS 的加载与阻塞
- 标准答案：默认同步 `script` 会同时阻塞 HTML 解析和后续渲染关键路径。`defer` 会并行下载并在 DOM 完成后按顺序执行，`async` 会并行下载且下载完成即执行但不保序。工程上应把有依赖的业务脚本放在 `defer`，把独立统计脚本放在 `async`。
- 考察点：是否能把阻塞行为、执行时机和脚本依赖关系一起讲清。
