## 概念定位

解析阻塞治理节点用于回答：为什么浏览器已经收到 HTML，页面却迟迟不出首屏。

所谓“解析阻塞”，本质是主线程在构建 DOM/CSSOM 过程中被同步依赖打断，导致渲染关键路径被拉长。

## 工作机制

1. HTML 解析推进时遇到外部资源与脚本。
2. 若资源被判定为阻塞型，主线程暂停解析推进。
3. 资源下载、解析、执行完成后，主线程继续构建。
4. 阻塞链越长，首屏关键节点出现越晚。

## 关键知识点

### 常见阻塞源

- 同步脚本（无 `defer/async`）
- 关键 CSS 下载与解析
- 脚本读取样式导致的“样式就绪等待”
- `document.write` 注入脚本或资源
- 第三方脚本早期执行

### script 策略差异

| 策略 | 下载时机 | 执行时机 | 对解析影响 |
| --- | --- | --- | --- |
| 普通 script | 发现即下载 | 下载后立即执行 | 常阻塞 |
| `defer` | 并行下载 | 解析完成后按顺序执行 | 通常不阻塞解析主流程 |
| `async` | 并行下载 | 下载完成即执行 | 可能插队打断 |
| `module` | 并行拉依赖 | 模块图就绪后执行 | 相对可控，但仍需关注依赖 |

### async / defer / module 选型规则

- `defer`：适合主业务脚本、存在依赖顺序的脚本、需要 DOM 基本就绪后再执行的脚本。
- `async`：适合彼此独立、与页面主流程弱耦合的脚本，如统计、埋点、广告、A/B 实验。
- `module`：适合现代 ESM（ECMAScript Modules）工程入口，依赖关系清晰、可分模块管理。

不建议把“有执行顺序要求的业务脚本”用 `async`，否则脚本先后顺序不稳定，容易出现偶发问题。

### CSS 阻塞的真实边界

- CSS 本身主要阻塞“首次有样式内容的呈现”。
- 当脚本依赖样式计算结果时，CSS 会间接阻塞脚本执行。
- `@import` 可能形成串行样式链，放大首屏等待。

### 第三方脚本的隐形阻塞

统计、广告、A/B、埋点 SDK 若在首屏窗口同步初始化，常造成不可见却昂贵的阻塞。

## 对页面性能的影响

- 解析阻塞直接推迟 FCP（First Contentful Paint）和 LCP（Largest Contentful Paint）。
- 阻塞期间主线程忙碌，交互输入得不到及时响应，影响 INP（Interaction to Next Paint）。
- 阻塞链不稳定会导致“同页面偶发慢”。

## 观测与排障

### 观测方法

1. Performance 观察主线程是否在首屏窗口被脚本长任务占满。
2. Network 检查关键 CSS/JS 是否串行加载。
3. 统计关键请求链长度与阻塞资源类型。

### 排障剧本

1. 标出首屏关键路径资源：哪些必须阻塞，哪些不必阻塞。
2. 普通脚本优先改 `defer`，独立脚本再评估 `async`。
3. 清理 `@import` 串行链，合并并前置关键样式。
4. 第三方脚本延后或在空闲时机初始化。
5. 回归对比：阻塞资源数、FCP/LCP、首屏主线程长任务。

## 实战优化策略

### 资源组织策略

- 首屏关键 CSS 保证尽早可用，非关键 CSS 延后加载。
- 首屏非必要 JS 统一延后到 `DOMContentLoaded` 后。
- 拆分大 bundle，避免首屏一次性下载和执行过重。

### 调度策略

- 把初始化任务分片，避免单个任务长时间占用主线程。
- 对非关键逻辑使用空闲调度（如 `requestIdleCallback` 兜底方案）。
- 降低首屏前同步计算与同步存储访问。

### 第三方治理策略

- 明确第三方脚本分级（阻塞级/可延后级/可懒加载级）。
- 建立上线门禁，防止新脚本回归阻塞首屏。
- 对第三方性能开销做独立指标上报。

## 常见误区

- “只要用了 `defer` 就万事大吉”：错误，样式依赖和大脚本执行仍可卡首屏。
- “CSS 不会阻塞脚本”：错误，脚本读取样式信息时会等待 CSSOM。
- “解析阻塞只能从网络层解决”：错误，执行时机和任务拆分同样关键。

## 面试补充

### 30 秒说明

解析阻塞是首屏变慢的高频根因。核心治理思路是把“必须阻塞”和“可延后”分开：脚本用 `defer/async` 合理调度，样式做关键路径拆分，第三方脚本分级治理，再用指标验证阻塞链是否缩短。

### 高频追问

1. 为什么页面资源都不大，首屏仍慢？

常见不是下载慢，而是脚本执行与样式依赖导致解析阻塞。

2. `defer` 与 `module` 在阻塞治理中如何选？

都能降低解析阻塞，但模块有依赖图管理特性；选择取决于工程体系与依赖复杂度。

3. 为什么 `@import` 可能拖慢首屏？

它容易形成样式串行下载链，关键样式到达时间被后移。

4. 第三方脚本治理第一步做什么？

先量化每个脚本对首屏和主线程的开销，再决定是否降级或延后。

5. 如何验证阻塞治理有效？

看阻塞资源数量、首屏关键链长度、FCP/LCP、首屏长任务是否同步改善。

6. 解析阻塞和强制布局有什么关系？

解析阻塞偏首屏前期资源与执行顺序问题，强制布局偏运行时读写顺序问题，两者都占主线程但阶段不同。


## 进阶补充（深挖）

### 与上下游节点的因果关系

- 上游输入（常见）：HTML 加载与语义策略。
- 本节点职责：将“解析阻塞治理”的结果转化为可执行的后续分支决策。
- 下游影响（常见）：响应流接收、HTML 解析与 DOM、CSSOM 构建与脚本执行。

### 线上故障案例

- 现象：接入第三方脚本后，首页首屏时间抖动明显增大。
- 根因：第三方脚本在解析关键路径同步执行，阻塞解析推进。
- 处置：将第三方脚本分级延后，保留关键必需脚本，建立阻塞资源门禁。
- 回归标准：关键指标恢复到基线区间，且连续观测窗口无二次抖动。

### 高频追问）

1. async 一定优于 defer 吗？

不一定。两者没有绝对优劣，关键看依赖关系和执行时机：

- 需要顺序执行、依赖 DOM 或其他脚本：优先 `defer`。
- 独立且越早执行越好、无需顺序保证：可用 `async`。
- 若是模块化主入口：优先 `type="module"`，再配合模块拆分。

2. 如何识别阻塞链是否被第三方放大？

比较引入前后关键请求链长度与首屏窗口长任务。

3. 为什么解析阻塞会影响 INP？

首屏主线程被占满会延迟后续输入响应链路。


### 补充记忆框架

- 先判断该节点是“前置等待”问题还是“运行时抖动”问题。
- 再判断问题来自“资源策略”还是“执行调度”。
- 最后把定位结论转为可复用规则，进入门禁与动作库。

### 高频追问

1. 这个节点优化后如何做回归验收？

至少做两组对比：关键指标分位数对比 + 上下游链路稳定性对比，避免局部优化导致成本转移。

2. 为什么这个节点要单独做发布观察窗口？

因为它对主链路有放大效应，短时间异常会快速传导到首屏和交互体验。
