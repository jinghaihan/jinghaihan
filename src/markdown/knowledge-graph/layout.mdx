## 概念定义

- 本质定义：布局计算是浏览器根据渲染树计算每个渲染对象位置和尺寸的过程。
- 解决问题：把“该显示哪些元素”转化为“这些元素具体画在什么位置”。
- 边界说明：它不负责像素填充，像素填充属于绘制阶段。

## 核心机制拆解

### 1) 机制/流程

1. 浏览器从根节点开始计算盒模型尺寸。
2. 按文档流和定位规则递归计算子节点几何信息。
3. 输出每个节点的坐标和尺寸结果。
4. 将结果传给绘制阶段。

### 2) 关键条件与触发点

- 条件 A：DOM 结构或几何样式变化会触发布局重算。
- 条件 B：读写布局信息交错执行会放大同步布局成本。

### 3) 结果与影响

- 对性能 / 稳定性 / 可维护性的影响：布局是高成本阶段，频繁触发会导致掉帧和交互延迟。

## 相似概念对比

| 维度 | 回流 | 重绘 | 结论 |
|---|---|---|---|
| 定义 | 几何信息重算 | 外观像素重画 | 回流通常包含重绘 |
| 触发时机 | 尺寸、位置、结构变化 | 颜色、阴影等视觉变化 | 二者触发条件不同 |
| 性能影响 | 成本高，影响范围大 | 成本相对低 | 优化优先减少回流 |
| 适用场景 | 布局变更场景 | 纯视觉样式更新 | 面试高频考点 |

## 代码

```js
// 先读后写，避免读写交错触发同步布局
const width = element.offsetWidth
requestAnimationFrame(() => {
  element.style.width = `${width + 20}px`
})
```

### 示例说明

- 这段代码解决什么问题：把布局读取和写入分阶段执行，降低同步布局抖动概率。
- 关键语句或参数含义：`requestAnimationFrame` 将写操作推迟到下一帧。
- 常见错误写法：循环里读写交错 `offsetWidth` 与 `style`。

## 真实场景示例

- 场景描述：搜索建议面板输入时明显卡顿。
- 问题表现：每次键入都触发大量回流。
- 处理方案：合并样式更新、批量 DOM 操作、减少同步读取。
- 验证方式：对比输入延迟和布局耗时。

## 常见误区

- 误区：只要帧率掉了就是绘制慢。
  - 正解：很多掉帧来自布局重算过于频繁。

## 关联知识点

- 依赖：依赖 `render-tree` 输出渲染对象。
- 影响：影响 `paint` 与 `composite` 任务量。
- 对比：`layout` 处理几何信息，`paint` 处理像素绘制。
- 组合：常与 `forced-layout` 联合做回流治理。

## 相关面试题

1. 浏览器渲染优化
   - 标准答案：布局优化重点是减少回流触发次数，常见手段包括批量 DOM 更新、读写分离、避免频繁强制同步布局和控制节点层级。
   - 考察点：是否能显式列举布局层可执行优化项。
