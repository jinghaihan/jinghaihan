## 概念定义

- 本质定义：类型守卫与类型编程关注如何利用控制流、`keyof`、`typeof`、`infer`、映射类型和条件类型，让类型系统跟着业务规则一起运转。
- 解决问题：减少断言、提高类型复用度，并把“输入不同、输出不同”的规则表达进类型层。
- 边界说明：本节点聚焦高级类型能力，不展开工程配置和模块系统。

## 核心机制拆解

### 1) 机制/流程

1. 先用类型守卫和 `is` 把宽类型逐步收窄到可安全访问的范围。
2. 再用 `keyof`、`typeof` 获取已有类型信息，减少手写重复类型。
3. 需要批量转换类型时，使用映射类型；需要条件分支时，使用条件类型。
4. 当要从复杂类型里“抽出一部分”时，再用 `infer` 做类型提取。

### 2) 关键条件与触发点

- 条件 A：频繁用断言跳过检查时，类型系统的价值会快速下降。
- 条件 B：类型转换规则重复出现时，应该考虑映射类型或工具类型抽象。
- 条件 C：类型表达已经比业务本身还难懂时，说明高级类型可能用过头了。

### 3) 结果与影响

- 对性能/稳定性/可维护性的影响：高级类型用得好，公共类型能非常稳；用得过头，类型定义会变成新的维护负担。

## 相似概念对比

| 维度 | 类型断言 | 类型守卫 | 结论 |
|---|---|---|---|
| 本质 | 告诉编译器“按这个类型看” | 通过判断收窄类型 | 守卫更安全 |
| 安全性 | 依赖开发者自觉 | 依赖运行时判断结果 | 默认优先守卫 |
| 适用场景 | 已知外部约束成立时 | 联合类型、边界数据 | 能判断就别硬断言 |
| 风险 | 容易把错带到运行时 | 成本高一点但更稳 | 工程上更推荐后者 |

## 代码

```ts
type ApiResult<T> = T extends Promise<infer U> ? U : T

function isStringArray(value: unknown): value is string[] {
  return Array.isArray(value) && value.every(item => typeof item === 'string')
}

type PartialRecord<T> = {
  [K in keyof T]?: T[K]
}
```

### 示例说明

- 这段代码解决什么问题：同时展示用户自定义类型守卫、`infer` 提取和映射类型。
- 关键语句或参数含义：`value is string[]` 是类型谓词；`infer U` 用于从 `Promise` 里提取内部类型；`[K in keyof T]` 是映射类型语法。
- 常见错误写法：大量断言替代守卫，或者把简单问题也写成多层条件类型。

## 真实场景示例

- 场景描述：请求层既要处理同步值，也要处理异步包装值，还要对不可信入参做收窄。
- 问题表现：同样的类型转换在多个模块里重复实现，业务里充满 `as` 断言。
- 处理方案：把复用规则抽成条件类型和工具类型，边界数据统一走类型守卫。
- 验证方式：看断言数量是否下降、公共类型是否能复用，以及调用方提示是否更准确。

## 常见误区

- 误区：高级类型越多越能体现水平。
- 正解：能把业务规则表达清楚才有价值，炫技没有意义。
- 误区：类型守卫和断言效果差不多。
- 正解：断言是跳过检查，守卫是建立检查。
- 误区：工具类型全靠背名字。
- 正解：真正关键的是理解它们背后的映射和条件分发机制。

## 关联知识点

- 依赖：基础类型与类型推断、对象建模与接口、泛型。
- 影响：工具类型封装、框架类型推断、组件 API 设计。
- 对比：断言和守卫、映射类型和条件类型。
- 组合：`keyof`、`typeof`、`infer`、工具类型。

## 相关面试题

1. 什么是 TypeScript 的类型断言（Type Assertion）？
- 标准答案：类型断言是告诉编译器“把这个值按某个类型看待”，但它不会做运行时检查。机制上它只影响编译阶段，不会真的验证值结构。场景上在你明确知道外部约束成立时可以用，但不能拿它替代正常的类型判断。
- 考察点：是否知道断言的风险在于“不会帮你兜底”。

2. `keyof` 和 `typeof` 在 TypeScript 中有什么作用？
- 标准答案：`keyof` 用来拿到某个对象类型的键集合，`typeof` 用来从一个已有值推导出它的类型。机制上二者都是“从已有信息反推类型”，避免重复手写。场景上配置对象、常量映射和工具函数封装里经常组合使用。
- 考察点：是否知道这两个关键字服务的是类型复用。

3. 什么是 TypeScript 中的类型守卫（Type Guards）？
- 标准答案：类型守卫是通过运行时判断，让编译器在某个分支里把类型收窄。机制上常见形式有 `typeof`、`instanceof`、`in` 以及用户自定义的 `value is X`。场景上处理 `unknown`、联合类型和接口响应时特别常见。
- 考察点：是否理解“判断结果会影响后续类型”。

4. 解释一下 TypeScript 的 `infer` 关键字。
- 标准答案：`infer` 只能出现在条件类型里，用来在匹配类型结构时临时声明一个待推断的类型变量。机制上它常被用来从函数返回值、数组元素或 `Promise` 内部提取类型。场景上很多内置工具类型底层都依赖它。
- 考察点：是否知道 `infer` 是“提取类型”的工具，而不是普通泛型替代品。

5. 什么是映射类型（Mapped Types）？
- 标准答案：映射类型就是遍历一个已有类型的键集合，然后批量生成新类型。机制上常见写法是 `[K in keyof T]`。场景上 `Partial`、`Readonly`、`Pick` 这类工具类型本质都是映射类型。
- 考察点：是否知道工具类型背后的原理。

6. 什么是条件类型（Conditional Types）？
- 标准答案：条件类型就是在类型层写 `A extends B ? X : Y` 这样的分支逻辑。机制上它会根据类型关系决定最终结果，联合类型下还可能发生分发。场景上“如果传入 Promise 就拿内部值类型，否则保持原类型”这类需求很典型。
- 考察点：是否理解条件类型表达的是类型层分支。

7. `is` 关键字的作用是什么？
- 标准答案：`is` 用于用户自定义类型守卫的返回值里，告诉编译器“如果函数返回 true，这个值就是某个更窄的类型”。机制上它把运行时判断结果反馈给类型系统。场景上校验接口返回、表单值和动态对象结构时很常用。
- 考察点：是否知道 `is` 的核心是让守卫生效。

8. 什么是“工具类型”（Utility Types）？请举例。
- 标准答案：工具类型是 TypeScript 提供或团队封装的可复用类型工具，用来对已有类型做批量变换。常见例子有 `Partial`、`Required`、`Readonly`、`Pick`、`Omit`、`Record`、`Exclude`、`Awaited`。场景上它们能大幅减少重复类型定义，但前提是你理解它们的输入输出。
- 考察点：是否能给出清单，并说明它们不是“魔法”。

9. 什么是模板字面量类型（Template Literal Types）？
- 标准答案：模板字面量类型可以把字符串字面量类型按模板组合成新的字符串类型。机制上它和联合类型结合后能生成一组受控字符串。场景上事件名、路由名、样式变量名和权限键名生成很常见。
- 考察点：是否知道它主要服务字符串规则建模。

10. `Awaited<T>` 工具类型是做什么的？
- 标准答案：`Awaited<T>` 用来拿到 `Promise` 最终解析后的值类型。机制上它内部会递归处理嵌套的 `Promise`。场景上封装请求函数、异步工具和服务层返回值时非常实用。
- 考察点：是否知道它解决的是“异步结果类型提取”。

11. `Omit<T, K>` 和 `Exclude<T, U>` 有什么区别？
- 标准答案：`Omit<T, K>` 用于对象类型，表示从对象里去掉某些键；`Exclude<T, U>` 用于联合类型，表示从联合集合中排除某些成员。机制上它们的输入对象不同，一个处理键，一个处理联合成员。场景上接口裁剪用前者，联合状态过滤用后者。
- 考察点：是否知道两者不是简单的“一个更高级版本”。

12. 如何理解协变（Covariance）和逆变（Contravariance）？
- 标准答案：结论是协变表示子类型可替代父类型位置，逆变表示父类型可替代子类型参数位置。机制上 TypeScript 在函数参数位置通常按逆变或双向兼容规则处理。场景上回调函数类型设计若忽略变体规则容易引发类型不安全。
- 考察点：是否能结合函数参数和返回值位置解释变体。

13. 解释一下 TypeScript 的“名义化类型”模拟。
- 标准答案：结论是 TypeScript 默认结构化类型，但可通过品牌字段等手段模拟名义化约束。机制上给类型附加不可见标记，避免结构相同但语义不同的值互相赋值。场景上金额、用户 ID、订单 ID 等语义强区分字段常用该模式。
- 考察点：是否理解“结构相同不代表语义相同”。

14. `infer` 与泛型参数有什么不同？
- 标准答案：结论是泛型参数由调用方显式或隐式传入，`infer` 在条件类型匹配时临时推断。机制上 `infer` 只能在条件类型的 `extends` 分支中声明。场景上提取函数返回值、Promise 内层类型时 `infer` 更直接。
- 考察点：是否能从“输入来源”角度区分二者。

15. 什么是 Variance（变体）？它对函数类型有什么影响？
- 标准答案：结论是 Variance 描述类型参数在子类型关系中的变化方向。机制上函数参数位置和返回值位置变体不同，直接影响回调兼容性判断。场景上事件处理器和高阶函数类型设计需要显式考虑变体约束。
- 考察点：是否理解变体对可赋值性的实际影响。

16. `Omit<T, K>` 和`Exclude<T, U>` 有什么区别？
- 标准答案：结论是 `Omit<T, K>` 用于对象键裁剪，`Exclude<T, U>` 用于联合成员过滤。机制上两者处理对象不同，一个操作键集合，一个操作联合集合。场景上接口字段裁剪用 Omit，状态联合剔除用 Exclude。
- 考察点：是否能准确按数据结构选择工具类型。
