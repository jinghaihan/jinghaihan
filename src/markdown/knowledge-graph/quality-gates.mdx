## 概念定位

Lint/Test/Typecheck 门禁节点用于把质量风险前置拦截，避免低级错误直接进入生产。

它是“开发效率”和“线上稳定性”之间的平衡器。

## 工作机制

1. 静态规则门禁：ESLint/Stylelint/格式检查，保障代码一致性与常见 bug 预防。
2. 类型门禁：TypeScript 或 `vue-tsc` 保证接口与状态边界正确。
3. 测试门禁：单元、集成、端到端按风险分层执行。
4. 质量阈值门禁：覆盖率、变更影响面、快照稳定性。
5. 发布前聚合门禁结果并给出放行/阻断结论。

## 关键知识点

### 必须掌握

- 门禁目标是“拦高风险”，不是“追求 0 warning 形式主义”。
- 类型检查能提前发现跨模块协议破坏。
- 测试策略应以关键路径优先，而不是平均用力。
- 门禁失败要输出可操作修复信息。
- 门禁速度要可控，避免拖垮研发节奏。
- 门禁规则要随故障复盘持续更新。

### 选型与取舍

- 门禁越严格，稳定性越高，但构建时间与修复成本会上升。
- 门禁过松上线快，但缺陷后移成本更高。
- 全量测试最稳妥，但可通过分层策略控制耗时。

## 对页面性能与稳定性的影响

- 节点稳定时，前端工程化 的链路可预测性会明显提升，发布和排障成本更可控。
- 节点失稳时，问题会向上下游扩散，常表现为性能回退、交互异常或交付风险叠加。
- 该节点优化应和上下游协同验证，避免“局部优化把成本转移给后续节点”。

## 观测与排障

### 核心指标

- 门禁失败率与失败类型分布
- 缺陷前移率（上线前拦截占比）
- 门禁平均耗时
- 同类缺陷复发率

### 排障顺序

1. 先分类失败来源：规则、类型、测试、环境。
2. 检查是否是误报（flaky）还是真实缺陷。
3. 对高频失败项补充模板和自动修复。
4. 优化门禁并行度和缓存以降低耗时。
5. 把真实故障映射回门禁规则闭环。

### 常见故障场景

- “线上出现类型不匹配崩溃”：常见是类型门禁缺失。
- “测试经常红但没人信”：常见是 flaky 测试未治理。
- “门禁太慢被绕过”：常见是规则设计与执行策略不合理。
- “同类 bug 重复出现”：常见是复盘未沉淀到规则。

## 常见误区

- 把门禁当流程负担而不是风险控制工具。
- 只设规则，不治理执行时间和误报。
- 手工放行缺乏审计记录。

## 深入拆解

### 门禁分层建议

- P0：类型错误、单测核心路径失败、安全高危依赖，直接阻断。
- P1：覆盖率下降、性能预算轻微超限，需评审放行。
- P2：风格与低风险规范问题，可在后续版本修复。

### 速度治理

1. 任务并行：lint/test/typecheck 并发执行。
2. 结果缓存：只校验变更影响范围。
3. 测试分层：PR 跑快测，全量夜间回归。

### 门禁质量评估

- 看缺陷前移率而不只看拦截总数。
- 对误报规则定期清退，保持团队信任度。

## 题库深挖补充

### 高频对比与边界（基于题库）

- 构建优化看三件事：速度、体积、可追踪，不是只看包大小。
- 质量门禁应分级阻断：P0 直接阻断，P1 评审放行，P2 持续治理。
- 缓存发布要做入口与静态资源差异化策略，避免版本错配。
- CI/CD 需要幂等和审计，失败重试不能污染环境状态。
- 灰度与监控联动是发布安全关键，不可分离。
- 回滚预案要可演练，不是“有命令就算有能力”。

### 线上排障剧本

1. 先定位问题发生在 build、verify、deploy、release 哪一段。
2. 检查门禁日志与放行记录，确认是否越权放行。
3. 检查缓存策略与发布顺序是否一致。
4. 灰度异常先止损，再做根因定位。
5. 回滚后验证核心链路并进入复盘。

### 高频追问补充（附答案）

1. 为什么说发布成功不等于上线成功？

真实流量下仍可能暴露性能或稳定性问题。

2. 灰度阶段最关键看什么？

错误率、核心性能分位数和关键业务转化。

3. 回滚后还要做什么？

确认恢复、记录时间线、沉淀治理规则。

## 进阶深挖（第二轮）

### 门禁策略升级

- 静态门禁：类型、规范、危险用法拦截。
- 动态门禁：关键路径测试与性能预算。
- 发布门禁：灰度阈值和告警联动。

### 误报治理流程

1. 标记高频误报规则。
2. 在不降低风险前提下收敛规则范围。
3. 记录放行依据并保留审计。
4. 定期复盘误报与漏报平衡。

### 追问扩展

1. 为什么门禁会被团队绕过？

常见是耗时过长、误报过多或失败信息不可操作。

2. 如何提高门禁信任度？

减少误报、给出可执行修复建议，并保持执行时间可控。

## 面试补充

### 30 秒说明

门禁体系的价值在于“缺陷前移 + 规则沉淀 + 可追踪放行”。它是稳定交付的基础设施。

### 高频追问

1. 门禁越多越好吗？

不是。要优先拦高风险项，同时控制耗时和误报。

2. 为什么类型检查很重要？

它能在编译阶段发现接口契约破坏，避免运行时崩溃。

3. 如何处理 flaky 测试？

先识别并隔离，再修根因，必要时临时降级但要有修复时限。

4. 门禁失败可以强行放行吗？

可以但必须记录风险说明、审批人和回滚预案。
