## 概念定义

- 本质定义：资源引入与图像策略关注样式资源加载顺序和图片资源体积质量平衡。
- 解决问题：降低阻塞、缩短加载时间、提升图像清晰度与传输效率。
- 边界说明：像素级适配细节在“响应式单位与媒体查询”节点。

## 核心机制拆解

### 1) 机制/流程

1. 样式引入优先使用 `link`，减少加载链路不确定性。
2. 图片格式按场景选型：照片、图标、透明、动画分别处理。
3. 小图标按体积和维护成本选择 Sprites 或现代方案。

### 2) 关键条件与触发点

- 条件 A：滥用大图会拉高首屏传输成本。
- 条件 B：格式选型错误会造成体积浪费或清晰度下降。

### 3) 结果与影响

- 对性能/稳定性/可维护性的影响：策略合理时请求和体积可控；策略失衡时首屏慢和资源管理复杂度上升。

## 相似概念对比

| 维度 | `link` | `@import` | 结论 |
|---|---|---|---|
| 加载时机 | HTML 解析阶段可发现 | 依赖 CSS 解析后发现 | `link` 更可控 |
| 兼容性 | 高 | 历史浏览器存在差异 | 工程化优先 `link` |
| 可维护性 | 结构清晰 | 层级深时追踪困难 | 大项目建议禁用 `@import` |
| 典型场景 | 主样式加载 | 局部历史样式复用 | 现代项目谨慎使用 |

## 代码

```html
<link rel="stylesheet" href="/assets/app.css" />
```

### 示例说明

- 这段代码解决什么问题：把主样式放在可预期加载路径中。
- 关键语句或参数含义：`rel="stylesheet"` 明确样式关系。
- 常见错误写法：主样式用 `@import` 多层嵌套，导致加载路径变长。

## 真实场景示例

- 场景描述：活动页图片多，低网速用户加载时间过长。
- 问题表现：首屏卡住，图像逐步抖动出现。
- 处理方案：重选图片格式（照片用 WebP、图标用 SVG），压缩体积并梳理样式加载顺序。
- 验证方式：首屏资源体积、请求数、LCP（Largest Contentful Paint）和图像清晰度对比。

## 常见误区

- 误区：图片只看清晰度，不看体积。
- 正解：Web 场景要平衡清晰度、体积和解码成本。
- 误区：Sprites 一定过时。
- 正解：在特定场景仍可用，但要综合 HTTP/2（HyperText Transfer Protocol 2）和维护成本评估。

## 关联知识点

- 依赖：响应式单位与媒体查询。
- 影响：性能优化与工程化产物策略。
- 对比：新旧图像格式与资源组织方式。
- 组合：构建压缩、CDN（Content Delivery Network）缓存策略。

## 相关面试题

1. link和@import的区别
- 标准答案：结论是 `link` 是 HTML 标签，`@import` 是 CSS 语法。机制上 `link` 在 HTML 解析阶段可被更早发现，`@import` 往往在 CSS 解析后继续请求。场景上主链路样式优先 `link`。
- 考察点：是否能说明加载链路差异。

2. 常见的图片格式及使用场景
- 标准答案：照片常用 JPEG/WebP，透明图常用 PNG/WebP，图标常用 SVG，动画短图常见 GIF/WebP 动图。机制上格式差异在压缩方式、透明支持和缩放失真。场景上按内容类型做格式分层。
- 考察点：是否能按“内容类型 -> 格式”做选型。

3. 对 CSSSprites 的理解
- 标准答案：结论是 Sprites 通过合并小图减少请求次数。机制上依靠 `background-position` 定位显示子区域。场景上在请求开销敏感且图标稳定的场景仍可用，但维护成本要评估。
- 考察点：是否能同时说明优点和维护代价。
