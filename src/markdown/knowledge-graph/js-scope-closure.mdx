## 概念定位

作用域、闭包与模块化是 JavaScript 运行时行为的基础。它决定变量可见性、状态封装能力和内存生命周期。

该节点做不好，会出现变量污染、闭包泄漏、模块边界混乱等长期维护问题。

## 工作机制

1. JavaScript 采用词法作用域（Lexical Scope），变量可见性在定义阶段确定。
2. 闭包让函数在离开定义上下文后仍可访问其词法环境。
3. `var` 存在函数作用域和变量提升，`let/const` 存在块级作用域和暂时性死区。
4. 模块系统（ESM, ECMAScript Modules）提供显式导入导出和静态分析能力。
5. 通过工厂函数和模块封装共享状态，避免全局对象污染。

## 关键知识点

### 必须掌握

- 闭包不是问题本身，失控引用链才是内存泄漏根因。
- `this` 绑定规则和词法作用域不同，面试常混淆。
- 循环里的闭包要关注变量捕获行为。
- ESM 的静态结构有利于 Tree Shaking（摇树优化）。
- 不要在模块顶层放重逻辑副作用，影响首屏执行时机。
- 模块边界要和业务域边界一致，避免“巨型 util”。

### 选型与取舍

- 过度封装会提升抽象成本，过少封装会导致全局耦合。
- 闭包可保持私有状态，但长期持有大对象引用要谨慎。
- 动态导入可降低初始包体，但会增加运行时请求复杂度。

## 对页面性能与稳定性的影响

- 节点稳定时，前端基础（HTML/CSS/JavaScript） 的链路可预测性会明显提升，发布和排障成本更可控。
- 节点失稳时，问题会向上下游扩散，常表现为性能回退、交互异常或交付风险叠加。
- 该节点优化应和上下游协同验证，避免“局部优化把成本转移给后续节点”。

## 观测与排障

### 核心指标

- 模块循环依赖数
- 全局变量暴露数
- 内存快照中闭包保留对象体积
- 首屏脚本执行时长

### 排障顺序

1. 先判断是作用域访问错误还是 `this` 绑定错误。
2. 再看是否因闭包引用导致对象无法释放。
3. 检查模块依赖图是否存在跨层循环依赖。
4. 对大型模块拆分并引入明确导出边界。
5. 回归验证：功能正确性 + 内存基线。

### 常见故障场景

- “回调里拿不到最新值”：常见是闭包捕获旧状态。
- “页面停留越久越卡”：常见是闭包持有大对象未释放。
- “构建优化不生效”：常见是模块导出结构不利于摇树。
- “命名冲突频发”：常见是全局作用域污染。

## 常见误区

- 把闭包当成“高级技巧”而不关注生命周期。
- 混淆 `this` 绑定和词法作用域导致排障方向错误。
- 模块划分按技术层而非业务域，导致职责不清。

## 题库深挖补充

### 高频对比与边界（基于题库）

- 词法作用域在定义时确定，this 绑定在调用时决定，二者常被混淆。
- var 存在函数作用域与提升行为，let/const 具备块级作用域与暂时性死区。
- 闭包本身不是问题，长生命周期引用链才是泄漏来源。
- 原型链与继承模型是 JS 面试高频，需能解释查找路径。
- ESM（ECMAScript Modules）静态结构有利于 Tree Shaking。
- 面试常问“闭包应用场景”，要同时能说收益和风险。

### 线上排障剧本

1. 先区分变量可见性错误还是 this 绑定错误。
2. 检查闭包是否长期持有大对象或 DOM 引用。
3. 检查模块依赖是否存在循环引用。
4. 对共享状态做边界收敛，避免全局泄漏。
5. 用内存快照验证修复效果。

### 高频追问补充（附答案）

1. 闭包为什么会导致内存问题？

因为闭包可能持有本可释放的对象引用。

2. var 与 let 在循环中的核心差别？

let 每轮迭代形成新绑定，var 共享同一绑定。

3. ESM 为什么更利于构建优化？

导入导出可静态分析，便于摇树与按需打包。

## 进阶追问（补充）

1. 为什么很多前端性能问题最终会回到“基础知识点”？

因为事件循环、布局模型、缓存与请求语义是上层框架和业务逻辑的底层约束。

2. 如何区分“语法会用”和“机制掌握”？

能解释边界、失效条件和排障顺序，才算机制掌握，而不只是 API 会写。

3. 性能优化里最容易被忽略的一步是什么？

对照验证和防回退机制，经常只做了修复动作没做持续守护。

## 案例追问加深（第三轮）

### 真实故障片段

- 故障表现：页面逻辑正确但体感卡顿，且在低端设备明显更差。
- 排查结论：基础层问题叠加，包括事件循环拥塞、样式重算成本和请求并发失控。
- 治理动作：按主线程任务、样式布局、请求治理三条线拆解并建立基线。

### 追问补充

1. 为什么基础知识点决定上层框架表现？

框架最终都运行在浏览器基础机制之上，底层约束没处理好，上层优化很难生效。

2. 如何避免“会写代码但不会排障”？

建立机制到现象的映射：看到症状能快速关联到可能的底层阶段。

## 面试补充

### 30 秒说明

JS 作用域与闭包的工程价值在于“状态边界可控”。只有边界清晰，后续异步调度和性能治理才有稳定前提。

### 高频追问

1. 闭包会导致内存泄漏吗？

闭包本身不会，问题在于闭包持续引用不再需要的大对象或 DOM。

2. `var` 和 `let` 在循环里最常见的差异是什么？

`var` 共享同一绑定，`let` 每次迭代创建新绑定，闭包行为不同。

3. ESM 为什么更利于构建优化？

导入导出是静态可分析的，打包器可以更准确做摇树和代码分割。

4. 如何避免“过度全局状态”？

通过模块边界、工厂函数和依赖注入收敛共享状态。
