## 概念定义

- 本质定义：作用域链决定变量查找路径，闭包让函数在定义环境外继续访问其词法环境。
- 解决问题：解决变量可见性、状态封装和回调持久化问题。
- 边界说明：不直接解决 `this` 绑定和原型继承问题。

## 核心机制拆解

### 1) 机制/流程

1. 函数定义时确定词法作用域。
2. 函数执行时创建执行上下文并链接外层作用域。
3. 内层函数引用外层变量时形成闭包。
4. 引用链存在则相关变量不会被提前回收。

### 2) 关键条件与触发点

- 条件 A：循环中的异步回调容易捕获同一变量绑定。
- 条件 B：闭包长期持有大对象会放大内存泄漏风险。

### 3) 结果与影响

- 对稳定性的影响：作用域边界清晰可减少变量污染。
- 对性能的影响：不当闭包引用会增加内存占用和回收压力。

## 相似概念对比

| 维度 | 词法作用域 | this 绑定 | 结论 |
|---|---|---|---|
| 决定时机 | 定义时 | 调用时 | 两者机制不同 |
| 影响对象 | 变量可见性 | 函数执行上下文 | 不应混淆 |
| 常见误用 | 以为调用位置可改作用域 | 以为箭头函数可动态改 this | 面试高频混淆点 |
| 典型排障 | 变量取值异常 | 方法调用对象错误 | 诊断路径不同 |

## 代码

```ts
function makeCounter() {
  let count = 0
  return () => ++count
}

const c1 = makeCounter()
console.log(c1(), c1())

for (let i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 0)
}
```

### 示例说明

- `makeCounter` 展示了闭包状态持久化。
- `let` 在循环中形成每轮新绑定，避免 `var` 常见捕获问题。
- 常见错误写法是用 `var` 造成异步回调同值输出。

## 真实场景示例

- 场景描述：表单页面离开后仍有回调引用页面大对象。
- 问题表现：内存快照显示对象长期未释放。
- 处理方案：解绑监听、清理闭包引用、避免全局缓存持有页面上下文。
- 验证方式：重复进出页面后内存曲线回落。

## 相关面试题

1. 对闭包的理解
- 标准答案：闭包是函数与其词法环境的组合，可在函数外持续访问定义时作用域变量。它适合私有状态封装，但不当引用会导致内存泄漏。
- 考察点：是否能同时说收益和风险。

2. 对作用域、作用域链的理解
- 标准答案：作用域规定变量可见范围，作用域链是查找变量时从当前到外层逐级查找的路径。词法作用域在定义阶段确定，不受调用位置影响。
- 考察点：是否理解“定义时决定”。

3. 对执行上下文的理解
- 标准答案：执行上下文包含变量环境、词法环境和 `this` 绑定。函数调用会入栈创建上下文，执行完出栈，决定了代码执行顺序和变量解析语义。
- 考察点：是否理解调用栈与上下文关系。

4. JavaScript 为什么要进行变量提升，它导致了什么问题？
- 标准答案：提升让声明在编译阶段提前可见，便于引擎统一处理标识符；但会带来“先使用后声明”的可读性问题。`let/const` 用暂时性死区缓解该类风险。
- 考察点：是否能讲“设计原因 + 问题 + 解决”。

### 代码输出题

1. 代码输出结果
```js
(function(){
   var x = y = 1;
})();
var z;

console.log(y); // 1
console.log(z); // undefined
console.log(x); // Uncaught ReferenceError: x is not defined
```
- 输出结果：
```text
1
undefined
Uncaught ReferenceError: x is not defined
```
- 关键机制：变量查找遵循词法作用域链，`var` 存在提升与函数作用域；闭包会持有外层变量引用，导致值在后续调用中继续可见。
- 边界条件：`let/const` 与 `var` 的提升和作用域规则不同；同名变量在不同词法环境会产生遮蔽。
