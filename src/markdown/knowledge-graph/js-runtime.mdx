## 概念定位

JavaScript 执行与调度节点关注的是：主线程如何分配执行时间，以及为什么“代码没报错但页面很卡”。

在 Knowledge Graph 中，它是连接“网络/渲染”与“交互体验”的核心桥梁。

## 工作机制

1. 浏览器主线程执行脚本任务（宏任务）。
2. 每轮任务结束后处理微任务队列。
3. 浏览器在可用时机执行渲染更新。
4. 若任务持续时间过长，渲染和交互会被延后。

## 关键知识点

### 事件循环（Event Loop）与任务队列

- **宏任务（Macrotask）**：脚本、定时器回调、I/O 回调等。
- **微任务（Microtask）**：Promise 回调、`queueMicrotask` 等。
- 微任务会在当前宏任务末尾尽快清空，过多微任务会饿死渲染机会。

### 渲染机会与主线程竞争

浏览器渲染不是随时发生，而是在任务间隙。若长任务连续占用主线程，用户输入和绘制都会被延迟。

### 长任务（Long Task）

单个任务超过 50ms 常被视为长任务，会直接拉高 TBT（Total Blocking Time）并恶化 INP（Interaction to Next Paint）。

### 框架更新与调度

现代框架会批量更新，但若业务层在一次事件中做了大量同步计算，框架仍无力避免主线程拥塞。

## 对页面性能的影响

- 首屏阶段：初始化脚本过重会推迟 FCP/LCP。
- 交互阶段：长任务会让点击/输入反馈明显滞后。
- 动画阶段：主线程繁忙导致掉帧和滚动不流畅。

## 观测与排障

### 关键观测指标

- Long Task 数量与总时长
- TBT 分位数
- INP 分位数
- JS 执行耗时与样式/布局耗时占比

### 排障剧本

1. 用 Performance 找出最长任务及调用栈。
2. 区分任务类型：初始化、数据处理、渲染提交、第三方脚本。
3. 将大任务拆分成小任务，释放渲染机会。
4. 对非关键逻辑延后执行或放入 Worker。
5. 回归看 INP/TBT 与长任务数量变化。

## 实战优化策略

### 任务拆分

- 把大计算拆成分片任务，避免一次执行过长。
- 利用调度 API 或定时切片，在帧间隙推进非关键工作。
- 高频事件处理加入节流/防抖和批处理。

### 线程分工

- 重计算、数据转换、序列化可迁移到 Web Worker。
- 主线程只做必要的 UI 状态提交与交互反馈。

### 初始化优化

- 首屏只初始化关键模块，次要模块懒加载。
- 路由级与组件级按需加载，降低启动脚本体积。
- 第三方 SDK 延后注入，避免抢占首帧预算。

## 常见误区

- “代码量小就不会卡”：错误，执行时机与同步密度更关键。
- “Promise 都是异步，所以不会阻塞”：错误，微任务清空阶段仍占主线程。
- “只优化网络体积就够了”：错误，运行时调度常是交互瓶颈主因。

## 面试补充

### 30 秒说明

JavaScript 运行时性能的核心是主线程调度。事件循环按宏任务和微任务推进，渲染只能在任务间隙发生。长任务会直接造成交互延迟和掉帧。优化重点是拆任务、控同步、分线程、按优先级调度执行时机。

### 高频追问

1. 微任务为什么可能让页面更卡？

微任务会在当前宏任务后尽快清空，若大量堆积，会占满渲染前窗口。

2. 如何定位 INP 变差是 JS 导致？

看输入事件后的主线程火焰图，若长任务或连续微任务占用明显，多半是 JS 调度问题。

3. 什么时候要用 Web Worker？

当计算任务重、与 UI 渲染耦合低且可序列化传输时，应优先迁移。

4. TBT 与 INP 的关系是什么？

TBT 反映整体阻塞总量，INP 反映真实交互延迟；TBT 高常会推高 INP，但需结合交互链路具体分析。

5. 防抖和节流在调度上的区别？

防抖偏“只执行最后一次”，节流偏“限制执行频率”，适用于不同事件模式。

6. 框架批量更新能否解决所有卡顿？

不能。业务同步计算和第三方脚本同样会占主线程，仍需整体调度治理。


## 进阶补充（深挖）

### 与上下游节点的因果关系

- 上游输入（常见）：实时通道分支（WebSocket/SSE）。
- 本节点职责：将“JavaScript 执行与调度”的结果转化为可执行的后续分支决策。
- 下游影响（常见）：运行时与存储关联、CSSOM 构建与脚本执行、强制布局与回流治理。

### 线上故障案例

- 现象：INP 偶发飙高，定位到交互后存在连续微任务清空。
- 根因：微任务队列堆积导致渲染机会被持续推迟。
- 处置：拆分长逻辑、降低同步链长度、将非关键计算迁移到 Worker。
- 回归标准：关键指标恢复到基线区间，且连续观测窗口无二次抖动。

### 高频追问）

1. 微任务为什么会饿死渲染？

当前宏任务结束后会优先清空微任务，可能持续占用主线程。

2. 如何快速找出最贵任务？

用性能火焰图按耗时排序并定位调用栈。

3. 框架批处理能解决所有长任务吗？

不能，业务同步计算和第三方脚本仍可阻塞主线程。


### 补充记忆框架

- 先判断该节点是“前置等待”问题还是“运行时抖动”问题。
- 再判断问题来自“资源策略”还是“执行调度”。
- 最后把定位结论转为可复用规则，进入门禁与动作库。

### 高频追问

1. 这个节点优化后如何做回归验收？

至少做两组对比：关键指标分位数对比 + 上下游链路稳定性对比，避免局部优化导致成本转移。

2. 为什么这个节点要单独做发布观察窗口？

因为它对主链路有放大效应，短时间异常会快速传导到首屏和交互体验。

### 补充策略

- 交互关键路径优先“快响应、慢补全”，把非关键逻辑后移。
- 将高频事件的状态更新做批处理，避免逐事件渲染。
