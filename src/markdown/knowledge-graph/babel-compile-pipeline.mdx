## 概念定义

- 本质定义：Babel 编译三阶段是把源代码解析为语法树、执行转换、再生成目标代码的过程。
- 解决问题：解决语法兼容、语法降级和代码改写自动化问题。
- 边界说明：不负责模块打包、资源拆分和运行时缓存策略。

## 核心机制拆解

### 1) 机制/流程

1. Parse：把源码解析为抽象语法树。
2. Transform：遍历语法树并执行插件转换。
3. Generate：把转换后的语法树输出为新代码。
4. 结合 `preset` 和插件链控制目标语法和兼容范围。

### 2) 关键条件与触发点

- 条件 A：目标环境不支持新语法时，需要对应转换插件。
- 条件 B：转换链过多或顺序不当会导致构建变慢或语义偏差。

### 3) 结果与影响

- 对性能的影响：合理配置可减少无效转换，缩短转译耗时。
- 对可维护性的影响：稳定的 Babel 策略可避免“本地能跑、线上报错”的兼容事故。

## 相似概念对比

| 维度 | Parse | Transform / Generate | 结论 |
|---|---|---|---|
| 作用 | 构建语法树 | 改写并输出代码 | 三阶段缺一不可 |
| 输入 | 源代码文本 | 语法树 | 输入对象不同 |
| 输出 | 语法树 | 新代码文本 | 从结构到文本 |
| 适用场景 | 语法分析 | 语法降级与改写 | 主要成本在转换阶段 |

## 配置示例

```ts
export default {
  presets: [
    ['@babel/preset-env', { targets: '> 0.5%, not dead' }],
    '@babel/preset-typescript',
  ],
  plugins: [
    '@babel/plugin-transform-runtime',
  ],
}
```

### 示例说明

- 这段配置展示了语法降级和运行时代码复用的最小组合。
- `targets` 决定转换范围，范围越宽通常转译成本越高。
- 常见错误写法是无目标环境配置，导致转换过度或不足。

## 真实场景示例

- 场景描述：项目引入新语法后，旧版浏览器用户出现白屏。
- 问题表现：线上日志提示语法错误，但本地和现代浏览器正常。
- 处理方案：补齐 `preset-env` 目标配置并校验 Babel 插件顺序。
- 验证方式：旧版浏览器回归通过，构建产物不再出现未降级语法。

## 常见误区

- 误区：Babel 会自动处理所有兼容问题。
- 正解：兼容范围取决于目标环境和插件配置。
- 误区：只要能编译通过，转换链越多越安全。
- 正解：冗余转换会拖慢构建，且可能引入语义偏差。

## 关联知识点

- 依赖：Loader 与 Plugin 扩展机制。
- 影响：Webpack 构建速度和产物可兼容性。
- 组合：Git 协作中的版本升级策略。

## 相关面试题

1. Babel 的原理是什么？
- 标准答案：Babel 按 Parse、Transform、Generate 三阶段工作，先把源码解析为语法树，再由插件改写语法树，最后生成目标代码。是否兼容目标环境取决于 `preset` 和插件配置，而不是 Babel 自动“全兜底”。
- 考察点：是否理解 Babel 的抽象语法树驱动机制与配置边界。
