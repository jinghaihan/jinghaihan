## 概念定位

磁盘缓存检查是浏览器的持久化缓存层，负责在跨页面、跨会话场景中复用历史资源副本，是“请求降本”的核心节点。

## 工作机制

1. 浏览器根据缓存键查找磁盘索引中的历史响应。
2. 若资源仍新鲜，直接命中强缓存并返回副本。
3. 若过期但可验证，发起条件请求（`If-None-Match`/`If-Modified-Since`）。
4. 服务端返回 `304` 时复用本地副本，`200` 时更新缓存。

## 关键知识点

- **强缓存**：有效期内不发请求，收益最高。
- **协商缓存**：过期后验证，节省带宽但仍有 RTT（Round-Trip Time） 成本。
- **ETag（Entity Tag） 优先**：通常比时间戳校验更精细。
- **缓存污染**：错误键策略和个性化响应易导致误命中。

## 对页面性能的影响

- 可显著降低重复下载和服务器压力。
- 高 304 比例并不等于最优，往返成本仍存在。
- 策略错配会导致“每次都回源”或“长期旧资源”。

## 观测与排障

### 浏览器侧

- Network 看 `200`/`304` 比例、缓存来源和响应头。
- 检查 `Cache-Control`、`ETag`、`Last-Modified` 是否一致。

### 服务端与 CDN 侧

- 监控回源率、命中率、带宽变化。
- 关注发布后缓存收敛速度是否符合预期。

### 排障步骤

1. 先确认资源是否指纹化，能否安全长缓存。
2. 再确认动态资源是否错误走了强缓存。
3. 再看协商字段是否缺失或被代理层改写。
4. 用发布前后回源率和响应分布验证策略。

## 常见误区

- “304 是免费命中”：不准确，仍有网络往返。
- “静态资源都该短缓存”：不准确，指纹化资源应长缓存。
- “缓存失效只能靠手动清理”：不准确，版本化和 TTL 管理更关键。

## 题库深挖补充

### 高频对比与边界（基于题库）

- 磁盘缓存（Disk Cache）比内存缓存慢，但可跨会话保留，适合静态资源复用。
- 同一资源是否落盘由缓存策略、资源类型和浏览器实现共同决定。
- 协商缓存（ETag/Last-Modified）常与磁盘缓存联动，返回 304 时重用本地副本。
- “缓存命中但体验仍慢”常见原因是入口 HTML 不可缓存或协商开销偏高。
- 强制刷新会绕过部分缓存路径，排障时要明确用户操作方式。
- 面试常问 200/304 比例是否越高越好，正确答案是看链路总体成本。

### 线上排障剧本

1. 先确认资源是 200 from disk cache 还是 304 协商命中。
2. 检查静态资源是否带内容指纹，避免旧资源复用风险。
3. 检查入口 HTML 与静态资源是否采用差异化缓存策略。
4. 检查发布后 CDN 与本地缓存失效是否协同。
5. 对异常用户做“普通刷新/强刷/新会话”三种路径对照。

### 高频追问补充（附答案）

1. 304 多是不是好事？

不绝对。304 说明可复用，但仍有网络往返，关键看整体收益。

2. 为什么磁盘缓存命中还会卡？

命中资源不代表主链路完成，入口、脚本执行和渲染仍可能是瓶颈。

3. 指纹资源为什么适合长缓存？

内容变化会导致 URL 变化，可安全长期缓存且避免版本错配。

### 术语速查与易错点

- Disk Cache：可跨会话复用的本地缓存层。
- ETag：实体标签，协商缓存核心字段。
- Last-Modified：上次修改时间校验字段。
- 304 Not Modified：复用本地副本。
- Cache Busting：通过指纹 URL 强制版本切换。

常见易错点：

- 入口 HTML 与静态资源使用同一缓存策略。
- 发布顺序错误导致资源错配。
- 把 304 当“零成本请求”。

## 深度延展（案例对照）

### 磁盘缓存错配案例

- 场景：发布后少量用户出现旧样式与新脚本混用。
- 诊断：入口缓存与静态资源缓存策略不匹配，导致版本错配。
- 处理：入口短缓存 + 资源长缓存（带指纹）并加发布原子性校验。

### 排查补充

1. 抓取异常用户请求头/响应头样本。
2. 核对入口引用的资源 URL 是否完整带 hash。
3. 验证 CDN 失效与本地缓存协商是否一致。

## 进阶排障清单（补充）

### 现场处置顺序

1. 核对入口文档缓存策略和静态资源缓存策略是否分离。
2. 抓取异常用户响应头样本，确认协商缓存路径。
3. 核对资源 URL 是否稳定带 hash。
4. 检查 CDN 边缘状态与本地缓存状态是否冲突。
5. 修复后用普通刷新、强刷、新会话三路径回归。

### 面试追问强化

1. 为什么磁盘缓存问题常在发布后爆发？

发布时版本切换和缓存失效策略若不同步，最容易出现错配。

2. 304 命中后仍慢的常见原因？

关键路径仍受入口、脚本执行或渲染阶段限制。

3. 如何衡量缓存策略是否健康？

同时看命中率、版本一致性、回源率和发布后错误率。

## 速查补充（第四轮）

### 快速判定问答

1. 什么时候先怀疑缓存层而不是权威记录本身？

当不同客户端结果分叉明显且权威查询稳定时，优先检查本地与递归缓存状态。

2. DNS 类问题为什么要强调样本固定？

因为网络、地域、时间窗口会影响解析路径，不固定样本容易得到互相矛盾结论。

## 面试补充

### 30 秒说明

磁盘缓存是持久化复用层。先走强缓存，再走协商缓存。优化重点是资源分层、版本指纹、校验头完整性，以及发布链路下的缓存收敛控制。

### 高频追问

1. 为什么要“长缓存 + 文件指纹”？

指纹让内容变化映射到 URL 变化，既能安全长缓存，又能确保版本及时更新。

2. 什么时候 304 会很多但体验仍差？

弱网或高 RTT 场景下，条件请求往返本身成本高，304 比例高不代表首屏快。

3. 如何判断缓存策略是否真的成功？

看复访链路总耗时是否下降，并结合命中率、回源率、关键页面 LCP 共同判断。
