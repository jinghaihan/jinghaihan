## 概念定位

仓库结构与分层约束定义了代码组织方式和依赖方向。它直接影响协作效率、复用能力和变更风险。

这个节点是“开发速度”和“长期维护成本”的平衡点。

## 工作机制

1. 按业务域与技术层双维度划分目录，避免“按页面堆文件”。
2. 定义依赖方向：上层可依赖下层，下层禁止反向依赖。
3. 公共能力抽到 package/模块层，禁止复制粘贴式复用。
4. 通过 lint/依赖图工具自动检查跨层违规和循环依赖。
5. 为关键模块维护 owner 与变更评审责任。

## 关键知识点

### 必须掌握

- 架构目标是“可演进”，不是一次性“最优目录”。
- Monorepo（单体仓多包）适合共享组件与统一工具链治理。
- 跨层调用一旦放开，后续重构成本会指数上升。
- 公共层必须稳定，避免把业务波动传递给全仓。
- 依赖图可视化有助于识别风险耦合点。
- 架构规范需要配套自动化校验，否则执行会走样。

### 选型与取舍

- 强分层约束会增加初期心智成本，但后续维护收益高。
- 弱约束开发快，长期会形成“全局耦合泥球”。
- 集中公共层复用高，但需治理版本兼容风险。

## 对页面性能与稳定性的影响

- 节点稳定时，前端工程化 的链路可预测性会明显提升，发布和排障成本更可控。
- 节点失稳时，问题会向上下游扩散，常表现为性能回退、交互异常或交付风险叠加。
- 该节点优化应和上下游协同验证，避免“局部优化把成本转移给后续节点”。

## 观测与排障

### 核心指标

- 跨层依赖违规数
- 循环依赖数量
- 跨模块变更影响面
- 公共包版本冲突率

### 排障顺序

1. 先看问题是否源于依赖方向错误。
2. 检查循环依赖与隐式全局耦合。
3. 识别高频变更热点模块并评估是否应拆分。
4. 补齐边界文档和自动化依赖检查。
5. 回归观察变更失败率与评审效率。

### 常见故障场景

- “改一个组件多个业务一起炸”：常见是公共层职责过宽。
- “新同学上手慢”：常见是目录和边界缺少一致规则。
- “重构无法推进”：常见是循环依赖太多。
- “多人并行冲突高”：常见是模块划分和 owner 不清晰。

## 常见误区

- 把架构当目录命名问题，不管依赖方向。
- 规范写了但没有工具校验。
- 公共层沉积业务逻辑导致污染。

## 题库深挖补充

### 高频对比与边界（基于题库）

- 需求冻结的核心是边界和验收口径，不是禁止变更。
- 仓库分层要定义依赖方向，避免“能 import 就 import”。
- 依赖治理要覆盖安全漏洞、体积增长、升级节奏三条线。
- 配置中心要支持审计与回滚，避免环境漂移。
- 复盘输出必须可执行：负责人、截止日期、验收标准缺一不可。
- 面试常问“工程化价值”，答案应覆盖效率、稳定性、可追踪性。

### 线上排障剧本

1. 先判断故障是流程缺失还是实现缺陷。
2. 检查需求/架构/依赖/配置是否存在边界越权。
3. 拉出变更时间线和责任链路。
4. 给出短期止血 + 中期治理动作。
5. 把结论沉淀为规则并跟踪复发率。

### 高频追问补充（附答案）

1. 为什么需求评审要带非功能指标？

因为性能、安全、可观测会影响架构路径，后置补救成本高。

2. 依赖治理最容易被忽视的点？

间接依赖和供应链风险，经常不在日常视野内。

3. 配置漂移为什么危险？

同代码在不同环境表现不同，事故定位会显著变慢。

## 进阶追问（补充）

1. 为什么工程化问题常在高峰期集中暴露？

因为流量放大后，缓存错配、发布窗口、告警噪声和回滚能力短板会被同时放大。

2. 门禁越严越好吗？

不是。要按风险分级阻断，既拦高风险问题，也要控制交付节奏。

3. 如何判断一次事故复盘是否有效？

看同类问题复发率是否下降，以及改进项是否转化为自动化规则。

## 案例追问加深（第三轮）

### 真实故障片段

- 故障表现：发布流水线显示通过，但上线后核心指标快速恶化。
- 排查结论：门禁覆盖了代码质量，却没有覆盖发布后观测与缓存一致性。
- 治理动作：把灰度阈值、缓存一致性检查和回滚演练纳入同一发布清单。

### 追问补充

1. 为什么工程化问题常在业务高峰暴露？

高峰流量会放大流程短板，平时隐性的风险会在短时间集中显现。

2. 如何衡量工程化治理是否真的有效？

看变更失败率、恢复时长和同类事故复发率是否持续下降。

## 面试补充

### 30 秒说明

仓库架构的核心是“分层边界 + 依赖方向 + 自动化守护”。这样才能在规模增长时保持可维护。

### 高频追问

1. Monorepo 的主要价值是什么？

统一依赖和工具链、提升跨项目复用、便于全局质量治理。

2. 为什么要限制跨层依赖？

防止耦合扩散，降低局部变更引发的全局回归风险。

3. 如何发现架构劣化？

看循环依赖、跨层违规和变更影响面是否持续上升。

4. 架构规范怎么落地最有效？

规则文档 + lint/CI 自动校验 + 代码评审联合执行。
