## 概念定义

- 本质定义：浏览器进程模型是指浏览器以多个进程分工协作来完成页面加载与执行的运行方式。
- 解决问题：通过进程隔离提升稳定性和安全性，避免单点崩溃扩散。
- 边界说明：它不描述渲染进程内部线程调度细节。

## 核心机制拆解

### 1) 机制/流程

1. 浏览器进程负责窗口管理、用户交互和子进程调度。
2. 网络进程负责请求发送、响应接收和协议栈处理。
3. 渲染进程负责文档解析、脚本执行和页面渲染。
4. 图形进程负责图层合成和图形加速。
5. 进程通过进程间通道交换数据并维持同步。

### 2) 关键条件与触发点

- 条件 A：标签页是否独立分配渲染进程，决定故障隔离粒度。
- 条件 B：站点隔离策略是否开启，决定跨站脚本和数据隔离强度。

### 3) 结果与影响

- 对性能 / 稳定性 / 可维护性的影响：多进程提升隔离和安全，但会增加内存占用和进程通信成本。

## 相似概念对比

| 维度 | 进程 | 线程 | 结论 |
|---|---|---|---|
| 定义 | 资源分配与隔离单位 | 进程内执行调度单位 | 进程管资源，线程管执行 |
| 触发时机 | 程序启动或站点隔离策略触发 | 进程内部并发任务处理 | 同一进程可包含多个线程 |
| 性能影响 | 切换成本高但隔离强 | 切换成本低但共享风险高 | 浏览器用“多进程 + 多线程”平衡隔离与效率 |
| 适用场景 | 标签页隔离、安全边界隔离 | 渲染、脚本、定时器并发协作 | 面试常考两者职责差异 |

## 代码

```bash
# Chrome 查看进程信息
open -a "Google Chrome" "chrome://process-internals"
```

### 示例说明

- 这段代码解决什么问题：快速查看标签页与进程映射，定位“哪个页面占用高”问题。
- 关键语句或参数含义：`chrome://process-internals` 显示浏览器内部进程诊断信息。
- 常见错误写法：只看操作系统总内存，不看进程粒度和页面对应关系。

## 真实场景示例

- 场景描述：线上某页面偶发崩溃，但浏览器未整体退出。
- 问题表现：当前标签白屏，其他标签可继续使用。
- 处理方案：确认渲染进程崩溃并抓取崩溃前日志，定位脚本异常与内存峰值。
- 验证方式：复现后观察进程崩溃是否仍局限在单标签。

## 常见误区

- 误区：浏览器只有一个进程。
  - 正解：现代浏览器通常至少包含浏览器进程、网络进程、渲染进程和图形进程。
- 误区：进程越少越快。
  - 正解：进程少会降低隔离能力，故障和安全风险会扩大。

## 相关面试题

1. 进程与线程的概念
   - 标准答案：进程是资源分配和隔离单位，线程是进程内部的执行调度单位；浏览器通过多进程隔离风险，通过多线程提升并发执行效率。
   - 考察点：是否能把概念和浏览器真实架构对应起来。
   - 追问 1：为什么浏览器不只用多线程？
   - 标准答案 1：仅多线程缺少进程级隔离，一旦线程级故障或安全问题扩散，影响范围会更大。

2. 进程和线程的区别
   - 标准答案：区别主要在资源隔离、切换成本和通信方式：进程隔离强但切换重，线程共享快但需要同步控制；浏览器综合使用两者做平衡。
   - 考察点：是否能给出“差异 + 取舍 + 浏览器实践”完整回答。
