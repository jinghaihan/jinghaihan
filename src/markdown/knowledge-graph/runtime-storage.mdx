## 概念定义

- 本质定义：垃圾回收用于自动释放不可达对象，内存泄漏是本应可回收对象被错误持有。
- 解决问题：解决页面长期运行后内存上涨、卡顿和崩溃问题。
- 边界说明：不负责网络超时和渲染链路瓶颈。

## 核心机制拆解

### 1) 机制/流程

1. 引擎标记可达对象。
2. 清理不可达对象并回收内存。
3. 长期引用链会阻止回收。
4. 业务层需主动释放无用引用和监听器。

### 2) 关键条件与触发点

- 条件 A：闭包、全局变量、缓存容器容易形成长生命周期引用。
- 条件 B：事件监听、定时器、DOM 引用未清理会放大泄漏。

### 3) 结果与影响

- 对性能的影响：内存持续增长会导致 GC 频繁，交互抖动明显。
- 对稳定性的影响：极端情况下触发页面崩溃或标签页被回收。

## 相似概念对比

| 维度 | 正常缓存 | 内存泄漏 | 结论 |
|---|---|---|---|
| 生命周期 | 有明确淘汰策略 | 无边界持续增长 | 关键在可控性 |
| 回收行为 | 可按策略释放 | 对象一直可达 | 泄漏不可自动恢复 |
| 指标表现 | 曲线有波动回落 | 曲线持续上升 | 可通过快照识别 |
| 处理方式 | 调整容量与策略 | 断开引用链 | 手段不同 |

## 代码

```ts
function bindResize(handler: () => void) {
  window.addEventListener('resize', handler)
  return () => window.removeEventListener('resize', handler)
}

const dispose = bindResize(() => {
  // do something
})

// 页面卸载或组件销毁时调用
dispose()
```

### 示例说明

- 示例体现了“注册即返回卸载函数”的泄漏防护习惯。
- 若不移除监听，闭包可能持续持有页面上下文。
- 常见错误写法是只注册不清理，页面多次进入后监听器倍增。

## 真实场景示例

- 场景描述：监控页长时间打开后内存不断上涨。
- 问题表现：页面越用越卡，切换标签恢复慢。
- 处理方案：排查未清理定时器和事件监听，修正缓存淘汰策略。
- 验证方式：内存快照对比，确认对象数量不再持续增长。

## 相关面试题

1. 浏览器的垃圾回收机制
   - 标准答案：主流引擎基于可达性标记和分代回收策略，按对象存活特征分层处理。核心结论是“不可达对象会被回收，可达对象不会”。
   - 考察点：是否理解可达性而非只背算法名。

2. 哪些情况会导致内存泄漏
   - 标准答案：常见包括全局变量滥用、未清理监听器、未清理定时器、闭包持有大对象、DOM 脱离文档但仍被引用。定位要结合堆快照和引用路径。
   - 考察点：是否能给出可落地排查方向。
