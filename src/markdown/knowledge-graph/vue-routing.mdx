## 概念定位

路由设计与页面状态节点关注路由结构、权限守卫、缓存策略和数据预取，它决定页面切换体验与状态一致性。

这个节点是“页面组织”和“数据加载”之间的桥梁。

## 工作机制

1. 定义路由层级和命名规则，避免路径与业务域错位。
2. 在路由守卫里处理鉴权、重定向和前置校验。
3. 结合按路由拆包减少首包体积。
4. 页面缓存（keep-alive）按业务规则控制激活与失活。
5. 路由切换时配合请求取消和预取策略。

## 关键知识点

### 必须掌握

- 路由不是菜单映射，而是应用信息架构。
- 守卫逻辑要避免重复请求和死循环跳转。
- 路由元信息（meta）可承载权限、标题、埋点策略。
- 长列表页面要设计返回态恢复策略。
- 分包策略需结合访问频率而非静态目录。
- 错误路由和异常页面要有统一兜底。

### 选型与取舍

- 预取可提速，但会增加后台流量。
- keep-alive 提升切换体验，但会增加内存占用。
- 守卫前置校验越多越安全，但切换时延可能上升。

## 对页面性能与稳定性的影响

- 节点稳定时，Vue 工作流 的链路可预测性会明显提升，发布和排障成本更可控。
- 节点失稳时，问题会向上下游扩散，常表现为性能回退、交互异常或交付风险叠加。
- 该节点优化应和上下游协同验证，避免局部优化把成本转移给后续节点。

## 观测与排障

### 核心指标

- 路由切换耗时
- 切换失败率（守卫拒绝/异常）
- 页面恢复正确率
- 路由相关白屏率

### 排障顺序

1. 先确定问题出在守卫逻辑还是页面加载。
2. 检查路由懒加载与依赖资源是否命中。
3. 检查 keep-alive 缓存命中和失效条件。
4. 检查切换时是否存在未取消请求。
5. 修复后验证前进/后退/刷新三种路径。

### 常见故障场景

- 登录态失效后出现跳转死循环。
- 返回列表页状态丢失。
- 弱网下路由切换白屏时间长。
- 路由预取导致非必要流量激增。

## 常见误区

- 把守卫逻辑分散到多个入口。
- keep-alive 全开不做策略。
- 忽略浏览器前进后退路径回归。

## 深入拆解

### 守卫执行顺序治理

- 全局前置守卫：鉴权、全局拦截。
- 路由独享守卫：模块级规则。
- 组件内守卫：页面级细粒度保护。

要避免在多个层级重复做同一校验。

### 路由缓存策略

1. 列表页返回保留滚动位置。
2. 详情页按业务是否复用决定缓存。
3. 表单页离开保护与草稿恢复分离设计。

### 分包与预取协同

- 首屏主路径保留在主包。
- 次级路径按用户行为概率预取。

## 题库深挖补充

### 高频对比与边界（基于题库）

- 路由驱动请求要处理竞态：后发先至会覆盖新状态。
- Pinia 状态分层要区分持久状态、会话状态、派生状态。
- SFC 模板编译需理解 Patch Flag、静态提升和 key 稳定性。
- Vue 性能优化先看热点组件和更新频率，再决定是否缓存或拆分。
- 异步请求治理需覆盖超时、重试、取消和错误兜底。
- 面试常问“v-if/v-show、computed/watch、nextTick”，要能结合性能语义说明。

### 线上排障剧本

1. 先区分卡顿来自请求链路还是渲染链路。
2. 检查请求取消与去重是否生效。
3. 检查列表 key 和模板复杂表达式。
4. 检查 store 更新粒度是否过粗。
5. 通过灰度对照验证优化收益。

### 高频追问补充（附答案）

1. 为什么 v-for 一定要稳定 key？

否则节点复用错位，性能和状态正确性都会受影响。

2. computed 和 watch 的边界？

computed 用于派生值缓存，watch 用于副作用响应。

3. nextTick 解决的核心问题是什么？

在 DOM 更新批处理后读取最新视图状态。

## 速查补充（第四轮）

### 快速判定问答

1. 路由切换体验差先看什么？

先分离代码加载耗时和数据加载耗时，再看守卫与缓存策略是否引入额外阻塞。

2. 路由问题如何避免“修一处坏一处”？

把守卫、缓存、预取策略放到统一规则里，不在页面层各自实现特例。

## 面试补充

### 30 秒说明

Vue 路由治理的重点是“切换可预测 + 状态可恢复 + 异常可兜底”。路由层设计好，页面体验会稳定很多。

### 高频追问

1. 路由守卫最容易出什么问题？

鉴权逻辑重复和重定向循环。

2. 什么时候该用 keep-alive？

列表返回场景和高频切换页面，且有明确失效策略时。

3. 路由懒加载的收益是什么？

降低首包体积，提升首次进入速度。

4. 如何保证返回态恢复？

结合路由参数、缓存策略和滚动位置管理。
