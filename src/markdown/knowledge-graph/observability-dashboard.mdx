## 概念定位

监控看板与告警联动节点把日志、指标、追踪串成可操作的观测体系，让故障“可发现、可定位、可追责”。

这是线上质量治理的中枢，决定 MTTA（Mean Time to Acknowledge）与 MTTR（Mean Time to Recovery）。

## 工作机制

1. 采集三类信号：Metrics（指标）、Logs（日志）、Traces（链路追踪）。
2. 按业务域和技术域构建统一看板：体验、错误、接口、资源、发布版本。
3. 告警分级：通知、预警、紧急，分别绑定响应策略。
4. 告警去噪与合并，避免告警风暴淹没真实事故。
5. 告警事件自动关联版本、变更单、责任人。

## 关键知识点

### 必须掌握

- 没有上下文的告警价值很低，必须带版本与链路信息。
- 看板要可钻取，从总览直达样本级详情。
- 告警阈值要按时段与基线动态校准。
- 前端观测要覆盖资源加载失败和用户交互失败。
- 监控指标应与 SLO（Service Level Objective）对齐。
- 观测体系需要定期演练，验证链路可用。

### 选型与取舍

- 采集越全定位越快，但成本和噪声也会上升。
- 阈值越敏感越早发现问题，但误报率可能升高。
- 单一平台集中管理方便，但迁移成本较高。

## 对页面性能与稳定性的影响

- 节点稳定时，前端工程化 的链路可预测性会明显提升，发布和排障成本更可控。
- 节点失稳时，问题会向上下游扩散，常表现为性能回退、交互异常或交付风险叠加。
- 该节点优化应和上下游协同验证，避免“局部优化把成本转移给后续节点”。

## 观测与排障

### 核心指标

- MTTA/MTTR
- 告警准确率与误报率
- 关键指标覆盖率
- 告警闭环时长

### 排障顺序

1. 先确认告警是否真实反映用户影响。
2. 通过版本和 trace 定位变更关联。
3. 从宏观指标下钻到错误样本与请求样本。
4. 执行止损并观察恢复趋势。
5. 更新阈值和告警策略避免重复噪声。

### 常见故障场景

- “问题发生了但没人知道”：常见是监控覆盖不足。
- “告警太多没人看”：常见是阈值与分级策略失衡。
- “知道报错却找不到根因”：常见是缺少链路上下文。
- “修复后仍频繁报警”：常见是告警规则未回收。

## 常见误区

- 把看板做成“数据展示墙”，不支持行动。
- 只监控错误率，不监控体验指标。
- 告警规则长期不维护。

## 深入拆解

### 看板最小闭环

- 发现：异常指标自动告警。
- 定位：下钻到版本、页面、接口、用户样本。
- 修复：关联发布、回滚、开关动作。
- 复盘：沉淀到规则与阈值。

### 告警治理建议

1. 按严重级别设置不同通知通道。
2. 对同源告警做聚合，减少噪声。
3. 引入静默窗口避免重复轰炸。
4. 告警必须带建议动作和责任人。

### 前端侧关键观测补充

- 资源加载错误（JS/CSS/font）
- JS 未捕获异常与 Promise 拒绝
- 首屏与交互核心指标分位数

## 题库深挖补充

### 高频对比与边界（基于题库）

- 构建优化看三件事：速度、体积、可追踪，不是只看包大小。
- 质量门禁应分级阻断：P0 直接阻断，P1 评审放行，P2 持续治理。
- 缓存发布要做入口与静态资源差异化策略，避免版本错配。
- CI/CD 需要幂等和审计，失败重试不能污染环境状态。
- 灰度与监控联动是发布安全关键，不可分离。
- 回滚预案要可演练，不是“有命令就算有能力”。

### 线上排障剧本

1. 先定位问题发生在 build、verify、deploy、release 哪一段。
2. 检查门禁日志与放行记录，确认是否越权放行。
3. 检查缓存策略与发布顺序是否一致。
4. 灰度异常先止损，再做根因定位。
5. 回滚后验证核心链路并进入复盘。

### 高频追问补充（附答案）

1. 为什么说发布成功不等于上线成功？

真实流量下仍可能暴露性能或稳定性问题。

2. 灰度阶段最关键看什么？

错误率、核心性能分位数和关键业务转化。

3. 回滚后还要做什么？

确认恢复、记录时间线、沉淀治理规则。

## 进阶深挖（第二轮）

### SLO 驱动看板设计

- 定义 SLI：可用性、错误率、交互性能分位数。
- 定义 SLO：目标阈值和观察窗口。
- 定义告警策略：误报可控、升级路径清晰。

### 告警风暴治理

1. 同源事件聚合与去重。
2. 告警静默窗口与抑制规则。
3. 告警附带建议动作和负责人。

### 追问扩展

1. 为什么有看板却定位慢？

缺少版本、链路、用户样本三者关联，数据可见但不可行动。

2. 前端可观测最应该补哪类信号？

交互失败与资源加载失败，这两类最贴近用户真实感知。

## 面试补充

### 30 秒说明

可观测体系的核心不是“有多少图表”，而是“是否能让团队快速做正确决策”。

### 高频追问

1. 为什么前端也要做 Trace？

能把用户报错与后端链路关联，显著缩短定位路径。

2. 如何降低告警噪声？

做分级、去重、聚合和动态阈值校准。

3. MTTA 和 MTTR 分别反映什么？

MTTA 反映发现速度，MTTR 反映恢复速度。

4. 看板设计最容易忽略什么？

缺少从总览到样本的下钻能力，导致可见不可用。
