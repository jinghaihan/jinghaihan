## 概念定位

依赖治理与版本策略关注“引什么、升不升、怎么升、出问题怎么回”。它是供应链安全与构建稳定性的核心。

该节点决定漏洞风险、构建可重复性和产物体积可控性。

## 工作机制

1. 建立依赖分层：运行时依赖、构建时依赖、开发辅助依赖分开治理。
2. 版本策略统一：关键依赖锁定版本，普通依赖按策略升级。
3. 引入漏洞扫描与许可证检查，阻断高危依赖进入主干。
4. 对大型依赖做体积评估和替代方案评审。
5. 定期升级窗口 + 回归套件，避免“长期积压一次性爆雷”。

## 关键知识点

### 必须掌握

- 锁文件（lockfile）是可重复构建的基础，不能随意漂移。
- 直接依赖与间接依赖都要监控漏洞风险。
- 升级要分批，避免一次性跨多个 major 版本。
- 引入新依赖前先确认是否已有内部能力可复用。
- 体积敏感依赖应做按需引入或替换评估。
- 供应链风险需结合发布回滚预案。

### 选型与取舍

- 严格锁版本稳定性高，但升级频率需要流程保障。
- 激进升级可追新特性，但回归风险上升。
- 依赖越少越轻，但自研成本可能变高。

## 对页面性能与稳定性的影响

- 节点稳定时，前端工程化 的链路可预测性会明显提升，发布和排障成本更可控。
- 节点失稳时，问题会向上下游扩散，常表现为性能回退、交互异常或交付风险叠加。
- 该节点优化应和上下游协同验证，避免“局部优化把成本转移给后续节点”。

## 观测与排障

### 核心指标

- 高危漏洞数量
- 依赖总量与体积增长
- 升级失败率
- 构建可重复率

### 排障顺序

1. 先定位问题来自哪一层依赖（直接/间接）。
2. 检查锁文件和安装环境是否一致。
3. 复现最小依赖集合并做二分定位。
4. 制定临时回退和长期替代方案。
5. 补齐依赖引入评审与升级节奏机制。

### 常见故障场景

- “同代码不同机器构建结果不同”：常见是锁文件或 Node 版本不一致。
- “上线后才发现漏洞”：常见是缺少门禁扫描。
- “包体突然暴涨”：常见是引入大依赖无体积评估。
- “升级一次崩一片”：常见是长期不升级导致跨版本鸿沟。

## 常见误区

- 把依赖升级当“有空再做”的事务。
- 只看功能可用，不看安全与体积成本。
- 出问题只回退，不追根因与替代方案。

## 题库深挖补充

### 高频对比与边界（基于题库）

- 需求冻结的核心是边界和验收口径，不是禁止变更。
- 仓库分层要定义依赖方向，避免“能 import 就 import”。
- 依赖治理要覆盖安全漏洞、体积增长、升级节奏三条线。
- 配置中心要支持审计与回滚，避免环境漂移。
- 复盘输出必须可执行：负责人、截止日期、验收标准缺一不可。
- 面试常问“工程化价值”，答案应覆盖效率、稳定性、可追踪性。

### 线上排障剧本

1. 先判断故障是流程缺失还是实现缺陷。
2. 检查需求/架构/依赖/配置是否存在边界越权。
3. 拉出变更时间线和责任链路。
4. 给出短期止血 + 中期治理动作。
5. 把结论沉淀为规则并跟踪复发率。

### 高频追问补充（附答案）

1. 为什么需求评审要带非功能指标？

因为性能、安全、可观测会影响架构路径，后置补救成本高。

2. 依赖治理最容易被忽视的点？

间接依赖和供应链风险，经常不在日常视野内。

3. 配置漂移为什么危险？

同代码在不同环境表现不同，事故定位会显著变慢。

## 进阶追问（补充）

1. 为什么工程化问题常在高峰期集中暴露？

因为流量放大后，缓存错配、发布窗口、告警噪声和回滚能力短板会被同时放大。

2. 门禁越严越好吗？

不是。要按风险分级阻断，既拦高风险问题，也要控制交付节奏。

3. 如何判断一次事故复盘是否有效？

看同类问题复发率是否下降，以及改进项是否转化为自动化规则。

## 案例追问加深（第三轮）

### 真实故障片段

- 故障表现：发布流水线显示通过，但上线后核心指标快速恶化。
- 排查结论：门禁覆盖了代码质量，却没有覆盖发布后观测与缓存一致性。
- 治理动作：把灰度阈值、缓存一致性检查和回滚演练纳入同一发布清单。

### 追问补充

1. 为什么工程化问题常在业务高峰暴露？

高峰流量会放大流程短板，平时隐性的风险会在短时间集中显现。

2. 如何衡量工程化治理是否真的有效？

看变更失败率、恢复时长和同类事故复发率是否持续下降。

## 面试补充

### 30 秒说明

依赖治理的目标是“可控引入、可预测升级、可追踪风险”。这是前端工程稳定性的底层保障。

### 高频追问

1. 为什么必须维护 lockfile？

它保证团队和 CI 安装结果一致，是可重复构建的关键。

2. 依赖升级为什么要小步快跑？

降低单次变更风险，便于快速定位与回滚。

3. 漏洞扫描应该放在哪个环节？

开发阶段预警 + CI 门禁阻断，双层保障。

4. 如何控制依赖体积膨胀？

做引入评审、按需加载、定期清理和替换重依赖。
