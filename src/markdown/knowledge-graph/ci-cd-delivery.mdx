## 概念定位

CI/CD 发布流水线是把代码变更自动、安全地送到生产的执行系统，核心是可重复、可审计、可回滚。

它决定交付节奏上限与故障止损速度。

## 工作机制

1. 持续集成阶段完成编译、测试、门禁与产物生成。
2. 持续交付阶段按环境执行部署、验证、灰度与放量。
3. 流水线中每一步都保留审计记录：版本、执行人、结果、耗时。
4. 引入环境审批与变更窗口控制，降低高峰时段风险。
5. 失败自动回滚或暂停放量，并通知责任人响应。

## 关键知识点

### 必须掌握

- 流水线要支持幂等执行，避免“重试一次状态更乱”。
- 环境配置注入必须和产物解耦，避免重新打包。
- 发布前健康检查要覆盖关键依赖服务可用性。
- 流水线模板化可降低人为操作差异。
- 权限最小化，避免高权限发布滥用。
- 发布日志与告警系统要联动。

### 选型与取舍

- 自动化程度越高，效率越高，但前期建设成本更大。
- 审批越严格，风险更低，但发布灵活性下降。
- 发布窗口越保守，事故概率低，但业务响应速度降低。

## 对页面性能与稳定性的影响

- 节点稳定时，前端工程化 的链路可预测性会明显提升，发布和排障成本更可控。
- 节点失稳时，问题会向上下游扩散，常表现为性能回退、交互异常或交付风险叠加。
- 该节点优化应和上下游协同验证，避免“局部优化把成本转移给后续节点”。

## 观测与排障

### 核心指标

- 发布成功率
- 平均发布时间
- 发布失败恢复时长
- 变更失败率（Change Failure Rate）

### 排障顺序

1. 先定位失败阶段：构建、部署、验证还是放量。
2. 检查流水线上下文和凭据是否有效。
3. 复核环境差异和配置注入是否正确。
4. 根据预案执行回滚或重试。
5. 复盘并更新流水线模板。

### 常见故障场景

- “同版本多次发布结果不一致”：常见是流水线非幂等。
- “发布成功但服务异常”：常见是健康检查覆盖不足。
- “夜间事故响应慢”：常见是告警链路未打通。
- “人工步骤导致漏操作”：常见是自动化不足。

## 常见误区

- 把 CI/CD 当“自动执行脚本”，忽略治理属性。
- 发布失败后只重试，不做状态清理。
- 没有演练回滚流程。

## 深入拆解

### 流水线关键阶段

1. Build：构建与产物归档。
2. Verify：测试、类型、质量门禁。
3. Deploy：部署到目标环境。
4. Verify Runtime：烟测、关键路径检查。
5. Release：灰度放量与全量切换。

### 可审计设计

- 每次发布记录 commit、构建号、配置版本和执行人。
- 发布日志可关联告警与回滚动作。
- 失败步骤保留上下文，支持自动重试与人工接管。

### 提升稳定性的实践

- 发布前冻结高风险变更。
- 生产变更采用双人审批。
- 建立可回退即发布的硬性规则。

## 题库深挖补充

### 高频对比与边界（基于题库）

- 构建优化看三件事：速度、体积、可追踪，不是只看包大小。
- 质量门禁应分级阻断：P0 直接阻断，P1 评审放行，P2 持续治理。
- 缓存发布要做入口与静态资源差异化策略，避免版本错配。
- CI/CD 需要幂等和审计，失败重试不能污染环境状态。
- 灰度与监控联动是发布安全关键，不可分离。
- 回滚预案要可演练，不是“有命令就算有能力”。

### 线上排障剧本

1. 先定位问题发生在 build、verify、deploy、release 哪一段。
2. 检查门禁日志与放行记录，确认是否越权放行。
3. 检查缓存策略与发布顺序是否一致。
4. 灰度异常先止损，再做根因定位。
5. 回滚后验证核心链路并进入复盘。

### 高频追问补充（附答案）

1. 为什么说发布成功不等于上线成功？

真实流量下仍可能暴露性能或稳定性问题。

2. 灰度阶段最关键看什么？

错误率、核心性能分位数和关键业务转化。

3. 回滚后还要做什么？

确认恢复、记录时间线、沉淀治理规则。

## 进阶深挖（第二轮）

### 流水线稳定性要点

- 幂等：失败重试不破坏环境状态。
- 可审计：每次发布有完整版本与配置轨迹。
- 可回退：任意失败阶段可快速止损。
- 可观测：发布后指标自动关联变更版本。

### 发布风险控制模板

1. 高风险变更前置冻结窗口。
2. 发布中强制健康检查与门禁。
3. 发布后固定观察窗口和阈值回滚。

### 追问扩展

1. 为什么说 CI/CD 的核心不是自动化脚本？

核心是把交付流程变成可预测、可追踪、可回滚的系统。

2. 幂等发布最容易遗漏什么？

环境外部状态清理与配置回滚一致性。

## 面试补充

### 30 秒说明

CI/CD 的关键价值是把发布变成“可预测流程”，而不是“依赖经验的手工操作”。

### 高频追问

1. 为什么流水线要强调幂等？

因为失败重试必须安全，否则会造成环境状态不一致。

2. 发布前健康检查要检查什么？

关键依赖可用性、配置正确性、基础接口连通性和核心页面可达性。

3. 自动化发布还需要人工审批吗？

关键环境通常需要，目的是在高风险变更前加入业务与技术双确认。

4. 如何评估发布体系成熟度？

看成功率、恢复时长、可审计性和是否支持自动回滚。
