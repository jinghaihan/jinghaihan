## 概念定义

- 本质定义：内存缓存是浏览器进程在当前会话中保存资源副本的高速缓存层。
- 解决问题：减少重复下载和磁盘读取，降低复访延迟。
- 边界说明：它随进程生命周期变化，不保证长期持久化。

## 核心机制拆解

### 1) 机制/流程

1. 首次请求成功后资源按策略进入缓存系统。
2. 后续请求先检查内存缓存是否命中。
3. 命中且有效时直接返回副本。
4. 未命中再回退到磁盘缓存或网络请求。

### 2) 关键条件与触发点

- 条件 A：资源是否仍在有效期，决定是否可直接复用。
- 条件 B：进程是否重启或内存回收，决定缓存是否还存在。

### 3) 结果与影响

- 对性能 / 稳定性 / 可维护性的影响：命中内存缓存可显著降低请求时延，但进程回收后命中不稳定，需要与磁盘和网络层联合设计。

## 相似概念对比

| 维度 | 内存缓存 | 磁盘缓存 | 结论 |
|---|---|---|---|
| 定义 | 进程内存中的快速副本 | 本地磁盘中的持久副本 | 前者快，后者稳 |
| 触发时机 | 同会话高频复用 | 进程重启后复访 | 两层常按顺序级联检查 |
| 性能影响 | 访问速度快 | 读取速度慢于内存 | 多层协作可平衡性能与命中率 |
| 适用场景 | 短周期重复请求 | 中长期资源复用 | 面试常考优先级链路 |

## 代码

```js
performance
  .getEntriesByType('resource')
  .filter((entry) => entry.transferSize === 0)
```

### 示例说明

- 这段代码解决什么问题：通过资源性能条目识别缓存命中请求。
- 关键语句或参数含义：`transferSize === 0` 常见于缓存复用场景。
- 常见错误写法：只看单次请求，不做冷缓存和热缓存对照。

## 真实场景示例

- 场景描述：用户在同一会话内频繁切换页面。
- 问题表现：重复资源请求数量仍高，切页耗时波动大。
- 处理方案：调整静态资源缓存头并收敛重复请求路径。
- 验证方式：对比优化前后内存命中率与资源请求次数。

## 常见误区

- 误区：缓存命中就一定来自内存。
  - 正解：命中可能来自内存、磁盘或 Service Worker，需要结合网络面板判断。

## 关联知识点

- 依赖：依赖 `refresh-modes` 和 `cache-policy` 策略。
- 影响：未命中时会流向 `disk-cache` 和网络层。
- 对比：`memory-cache` 偏短期命中，`disk-cache` 偏跨会话复用。
- 组合：常与 `status-code-path` 联合判断 `304` 行为。

## 相关面试题

1. 对浏览器的缓存机制的理解
   - 标准答案：浏览器缓存通常按“内存缓存 -> 磁盘缓存 -> 协商请求 -> 回源下载”逐层判定，内存缓存位于靠前层，负责高频快速复用。
   - 考察点：是否能说明缓存链路顺序和判定关系。

2. 为什么需要浏览器缓存？
   - 标准答案：浏览器缓存可以减少重复传输、降低服务器压力、提升页面响应速度；内存缓存在会话内收益最明显。
   - 考察点：是否能显式列举缓存收益，而非抽象描述。
