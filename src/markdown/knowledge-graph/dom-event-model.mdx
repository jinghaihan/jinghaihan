## 概念定位

DOM（Document Object Model）/BOM（Browser Object Model）与事件模型决定了页面如何响应用户输入。它是“交互正确性”和“交互性能”的共同基础。

事件传播路径和监听策略会直接影响输入延迟、内存占用以及组件耦合度。

## 工作机制

1. 事件分三阶段传播：捕获（capture）-> 目标（target）-> 冒泡（bubble）。
2. 事件委托通过在父节点集中监听，减少监听器数量并降低动态节点管理成本。
3. 浏览器输入事件（click、pointer、keydown）会进入任务队列，与渲染调度竞争主线程。
4. `passive` 监听可告诉浏览器不阻塞滚动，显著降低滚动卡顿风险。
5. 复杂交互需要统一事件协议：触发源、业务意图、埋点字段保持一致。

## 关键知识点

### 必须掌握

- `target` 是事件源，`currentTarget` 是当前监听节点，混淆会导致逻辑错误。
- 频繁绑定/解绑监听器会产生额外 GC（Garbage Collection）压力。
- 移动端手势优先用 Pointer Events（指针事件）统一鼠标和触摸处理。
- 避免在高频事件（scroll、mousemove）中直接做昂贵计算。
- 事件委托适合列表类动态节点，但不适合需要强隔离的敏感交互。
- 自定义事件总线要限制作用域，避免全局广播失控。

### 选型与取舍

- 大量独立监听逻辑直观但难维护，委托更集中但需要清晰事件命名。
- `stopPropagation` 能快速止血，但滥用会导致上层逻辑失效。
- 统一代理层能提升治理能力，但会增加调试路径长度。

## 对页面性能与稳定性的影响

- 节点稳定时，前端基础（HTML/CSS/JavaScript） 的链路可预测性会明显提升，发布和排障成本更可控。
- 节点失稳时，问题会向上下游扩散，常表现为性能回退、交互异常或交付风险叠加。
- 该节点优化应和上下游协同验证，避免“局部优化把成本转移给后续节点”。

## 观测与排障

### 核心指标

- 输入响应延迟（INP, Interaction to Next Paint）
- 长任务数量（Long Tasks）
- 单页监听器总量与增长趋势
- 高频事件回调平均耗时

### 排障顺序

1. 先确定问题属于传播错误还是执行过慢。
2. 用 DevTools 事件监听面板检查监听器绑定位置与数量。
3. 对高频事件回调做性能剖析，定位热点函数。
4. 检查是否误用 `preventDefault`/`stopPropagation` 破坏交互链路。
5. 把修复动作固化到事件规范与 ESLint 规则中。

### 常见故障场景

- “点击无响应”：常见是事件被上游阻断或命中元素层级异常。
- “滚动卡顿”：常见是非 `passive` 的 touch/scroll 监听阻塞主线程。
- “内存持续上涨”：常见是组件卸载后监听未清理。
- “列表交互偶发异常”：常见是委托节点选择过高导致误匹配。

## 常见误区

- 把所有问题都归因于框架，忽略 DOM 事件本身的传播规则。
- 高频事件中直接 setState/响应式更新，不做节流。
- 只修业务逻辑，不治理监听器生命周期。

## 题库深挖补充

### 高频对比与边界（基于题库）

- 事件传播分捕获、目标、冒泡三阶段，target 与 currentTarget 语义不同。
- 事件委托能降低监听器数量，但命中逻辑要有明确约束。
- passive 监听可避免滚动阻塞，是移动端性能高频考点。
- 高频事件（scroll/mousemove）应节流或批处理，避免主线程拥塞。
- BOM（Browser Object Model）API 使用要关注兼容和副作用边界。
- 面试常问“事件代理原理”，关键在冒泡路径与命中筛选。

### 线上排障剧本

1. 先确认问题是事件没触发还是触发后执行慢。
2. 检查监听器绑定位置与解绑时机。
3. 检查是否被 stopPropagation 误阻断。
4. 分析高频回调中的重计算和同步布局读取。
5. 收敛到统一事件规范并加 lint 约束。

### 高频追问补充（附答案）

1. passive 为什么能提升滚动体验？

浏览器可提前执行默认滚动，不必等待回调结果。

2. 事件委托的边界是什么？

动态列表场景收益高，强隔离交互场景不一定适用。

3. target 和 currentTarget 常见误用？

把两者混用会导致命中节点判断错误。

## 进阶追问（补充）

1. 为什么很多前端性能问题最终会回到“基础知识点”？

因为事件循环、布局模型、缓存与请求语义是上层框架和业务逻辑的底层约束。

2. 如何区分“语法会用”和“机制掌握”？

能解释边界、失效条件和排障顺序，才算机制掌握，而不只是 API 会写。

3. 性能优化里最容易被忽略的一步是什么？

对照验证和防回退机制，经常只做了修复动作没做持续守护。

## 案例追问加深（第三轮）

### 真实故障片段

- 故障表现：页面逻辑正确但体感卡顿，且在低端设备明显更差。
- 排查结论：基础层问题叠加，包括事件循环拥塞、样式重算成本和请求并发失控。
- 治理动作：按主线程任务、样式布局、请求治理三条线拆解并建立基线。

### 追问补充

1. 为什么基础知识点决定上层框架表现？

框架最终都运行在浏览器基础机制之上，底层约束没处理好，上层优化很难生效。

2. 如何避免“会写代码但不会排障”？

建立机制到现象的映射：看到症状能快速关联到可能的底层阶段。

## 面试补充

### 30 秒说明

事件模型的本质是“传播规则 + 调度预算 + 生命周期治理”。把这三点做对，交互稳定性和性能会同时提升。

### 高频追问

1. 事件委托为什么能提升性能？

它减少监听器数量和动态节点管理成本，尤其在长列表和频繁增删节点场景收益明显。

2. `passive` 的核心价值是什么？

它让浏览器知道监听器不会阻止默认滚动，从而提前执行滚动，降低卡顿。

3. 什么时候不适合事件委托？

当节点隔离要求高、权限边界强或事件命中逻辑复杂时，单独监听更清晰。

4. INP 高一定是事件模型问题吗？

不一定。也可能是脚本执行、布局抖动、网络回调集中触发等主线程拥塞问题。
