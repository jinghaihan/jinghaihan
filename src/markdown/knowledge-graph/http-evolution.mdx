## 概念定义

- 本质定义：HTTP（HyperText Transfer Protocol）版本演进是应用层协议为降低延迟、提升并发和改善弱网体验而进行的能力升级。
- 解决问题：减少连接开销、缓解队头阻塞、降低头部冗余、提升多资源并发传输效率。
- 边界说明：版本升级不能替代业务层缓存策略和资源体积治理。

## 核心机制拆解

### 1) 机制/流程

1. HTTP/1.0 默认短连接，请求串行且连接成本高。
2. HTTP/1.1 引入持久连接和更完整缓存协商字段。
3. HTTP/2 使用二进制分帧、多路复用和头部压缩。
4. HTTP/3 基于 QUIC（Quick UDP Internet Connections），减少握手往返并改善弱网下传输恢复。
5. 版本能力与服务端配置、客户端支持共同决定实际收益。

### 2) 关键条件与触发点

- 条件 A：服务端、网关、CDN（Content Delivery Network）是否已启用对应版本协议。
- 条件 B：资源类型与并发模式，决定多路复用和头压缩收益大小。

### 3) 结果与影响

- 对性能 / 稳定性 / 可维护性的影响：协议升级能降低首包和并发请求开销，但若资源治理和缓存策略缺失，收益会被抵消。

## 相似概念对比

| 维度 | HTTP/1.1 | HTTP/2 | 结论 |
|---|---|---|---|
| 定义 | 文本报文，持久连接 | 二进制分帧，多路复用 | HTTP/2 更适合多资源并发场景 |
| 触发时机 | 默认长连接但仍存在队头影响 | 同连接内并发传输多个流 | 不是“升级后无需任何优化” |
| 性能影响 | 连接数和排队成本较高 | 头压缩和并发能力更强 | 对图片、脚本密集页收益更明显 |
| 适用场景 | 兼容性广、基础场景 | 资源并发高、延迟敏感场景 | 生产常见是 HTTP/2 为主、HTTP/3 逐步补充 |

### 示例说明

- 这段代码 / 配置解决什么问题：启用并验证 HTTP/2 协议能力。
- 关键参数含义：`--http2` 强制客户端以 HTTP/2 发起请求验证协商结果。
- 常见错误配置：服务端启用了 HTTP/2，但中间代理降级到 HTTP/1.1。

## 真实场景示例

- 场景描述：首页有大量小图与拆分脚本，HTTP/1.1 下排队严重。
- 问题表现：同域资源并发请求耗时长，首屏延迟高。
- 处理方案：开启 HTTP/2，多路复用同连接传输，并同步治理头字段体积和缓存策略。
- 验证方式：对比升级前后连接数、`TTFB（Time To First Byte）`、首屏时间和错误率。

## 常见误区

- 误区：升级到 HTTP/2 后域名分片一定继续有效。
- 正解：HTTP/2 下过度分片可能降低连接复用收益，需按真实链路评估。
- 误区：HTTP/3 一定全面优于 HTTP/2。
- 正解：收益受网络环境、终端支持和链路配置影响，需要灰度验证。

## 关联知识点

- 依赖：`tcp-tls` 提供连接和安全基础。
- 影响：版本能力改变 `request-lifecycle` 的请求并发表现。

## 相关面试题

1. HTTP 1.0 和 HTTP 1.1 之间有哪些区别？
- 标准答案：HTTP/1.1 默认持久连接并补充更多缓存控制与方法能力，减少了 HTTP/1.0 频繁建连和能力不足的问题。
- 考察点：是否能从连接、缓存、方法与主机标识多维度回答。
- 追问 1：`keep-alive` 的核心价值是什么？
- 标准答案 1：减少重复握手和连接释放成本，在同域多请求场景降低整体延迟。
- 追问 2：为什么同域连接数限制在 HTTP/1.1 下仍重要？
- 标准答案 2：请求仍会受连接数和排队影响，资源多时仍可能出现队头等待。

2. HTTP 1.1 和 HTTP 2.0 的区别
- 标准答案：HTTP/2 通过二进制分帧、多路复用和头压缩减少排队与冗余，提升并发资源加载效率；但仍需配合资源治理才能稳定收益。
- 考察点：是否理解“协议能力提升”和“工程治理”是两件事。

3. HTTP 的演进之路
- 标准答案：HTTP 演进主线是从“减少重复建连成本”到“提升并发传输效率”再到“改善弱网恢复能力”：HTTP/1.1 解决基础连接复用，HTTP/2 强化同连接并发，HTTP/3 通过 QUIC 减少握手和队头阻塞影响。是否有实际收益取决于客户端支持、网关配置和资源治理是否同步到位。
- 考察点：是否能按版本回答“解决了什么问题、还剩什么边界”。
