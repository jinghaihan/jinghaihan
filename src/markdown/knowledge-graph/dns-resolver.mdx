## 概念定义

- 本质定义：DNS（Domain Name System）解析流程是把域名转换为可连接地址的分层查询过程。
- 解决问题：让浏览器在发起连接前拿到目标主机地址，并在可复用时走缓存减少查询成本。
- 边界说明：它不负责建立 TCP（Transmission Control Protocol）/TLS（Transport Layer Security）连接，也不负责 HTTP（HyperText Transfer Protocol）报文传输。

## 核心机制拆解

### 1) 机制/流程

1. 浏览器发起导航请求，先查浏览器缓存和系统缓存。
2. 本地缓存未命中时，本地解析器向递归解析器发起查询。
3. 递归解析器按根域、顶级域、权威域逐级获取下一跳信息。
4. 解析器拿到最终记录后返回给客户端。
5. 客户端用结果进入 `tcp-tls` 节点继续建连。

### 2) 关键条件与触发点

- 条件 A：缓存是否命中且未过期，决定是否需要递归查询。
- 条件 B：是否存在 CNAME（Canonical Name）链和多条记录，决定查询跳数和后续连接选择。
- 条件 C：是否使用 `dns-prefetch` 和 `preconnect`，决定能否把解析或建连前置到浏览器空闲阶段。

### 3) 结果与影响

- 对性能 / 稳定性 / 可维护性的影响：冷启动时 DNS 延迟会直接抬高首包时间；CNAME 链过长会增加失败点；TTL（Time To Live）配置不当会导致切流慢或查询压力高。

## 相似概念对比

| 维度 | 递归查询 | 迭代查询 | 结论 |
|---|---|---|---|
| 定义 | 客户端把“查到底”交给递归解析器 | 解析器逐级问不同层级服务器 | 浏览器通常只做递归请求，迭代发生在 DNS 体系内部 |
| 触发时机 | 本地缓存未命中时 | 递归解析器向上游查询时 | 两者常同时出现在一次完整解析中 |
| 性能影响 | 受本地到递归解析器质量影响 | 受根、顶级域、权威链路质量影响 | 任何一层抖动都会放大首访波动 |
| 适用场景 | 浏览器 / 应用侧常规解析 | DNS 服务侧分层路由 | 面试需能同时说清客户端视角和服务端视角 |

### 资源提示对比

| 维度 | `dns-prefetch` | `preconnect` | 结论 |
|---|---|---|---|
| 定义 | 仅提前做域名解析 | 提前做 DNS + TCP/TLS 建连 | `preconnect` 前置动作更多 |
| 触发时机 | 预计后续会访问某域名 | 预计很快就会发起关键请求 | 关键资源优先用 `preconnect` |
| 性能影响 | 降低 DNS 等待 | 同时降低解析和握手等待 | 过量使用会占用连接资源 |
| 适用场景 | 次级资源域名预热 | 首屏关键接口 / 关键静态域 | 先识别关键域名再投放 |

## 代码

```bash
dig +trace www.example.com
```

```bash
nslookup www.example.com 8.8.8.8
```

```html
<link rel="dns-prefetch" href="//api.example.com" />
<link rel="preconnect" href="https://api.example.com" crossorigin />
```

### 示例说明

- 这段代码 / 配置解决什么问题：`dig +trace` 用于逐跳验证解析链路，`nslookup` 用于对比不同解析器结果。
- 关键参数含义：`+trace` 表示显示根域到权威域的完整路径；`dns-prefetch` 只预解析，`preconnect` 会预建连接并可配合 `crossorigin`。
- 常见错误配置：只看单个解析器结果就下结论，忽略运营商和地域差异。

## 真实场景示例

- 场景描述：新版本发布后，华东地区首屏明显变慢。
- 问题表现：同域名在不同网络环境下解析耗时和目标地址差异很大。
- 处理方案：先用 `dig +trace` 确认委派链正确，再按运营商对比递归解析结果，最后收敛 CNAME 链和 TTL。
- 验证方式：对比优化前后 DNS 耗时分位数和首屏 `TTFB（Time To First Byte）`。

## 常见误区

- 误区：DNS 只影响首次访问一次，后续可忽略。
  - 正解：缓存失效、网络切换、域名变更都会重新触发解析。
- 误区：TTL 越短越好，切流更灵活。
  - 正解：TTL 过短会放大查询压力和抖动，需按业务变更频率分阶段设置。

## 关联知识点

- 依赖：`disk-cache` 未命中后进入 DNS 解析流程。
- 影响：解析结果直接影响 `tcp-tls` 建连目标。
- 对比：`dns-resolver` 关注查询过程，`dns-result` 关注结果复用与地址选择。
- 组合：常与 `request-lifecycle`、`status-code-path` 一起考全链路定位。

## 相关面试题

1. 当在浏览器中输入 Google.com 并且按下回车之后发生了什么？
   - 标准答案：URL（Uniform Resource Locator）解析和缓存判断后会进入 DNS 查询，客户端对本地解析器是递归请求，解析器向根域、顶级域、权威域是迭代查询，拿到地址后才会进入 TCP/TLS 建连。
   - 考察点：是否能说清浏览器视角和 DNS 分层视角。
   - 追问 1：为什么根域服务器不直接返回业务 IP？
   - 标准答案 1：根域只负责把请求指向正确顶级域，分层设计保证全局可扩展和治理边界清晰。
   - 追问 2：CNAME 链为什么会让首访变慢？
   - 标准答案 2：每一层别名都可能引入一次额外查询与缓存依赖，链越长，额外往返和失败点越多。

2. URL 有哪些组成部分
   - 标准答案：URL 中域名部分决定 DNS 查询对象，端口会影响连接目标，路径和参数不参与 DNS 查询但会影响服务端路由处理。
   - 考察点：是否能把 URL 结构和 DNS 职责边界对应起来。
