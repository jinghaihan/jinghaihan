## 概念定义

- 本质定义：同源与跨域策略是浏览器对跨源请求实施的 CORS（Cross-Origin Resource Sharing）安全约束与放行机制。
- 解决问题：在保证安全边界的前提下，让前后端在不同源下可控通信。
- 边界说明：它只约束浏览器环境下的跨源访问，不等于服务端权限控制。

## 核心机制拆解

### 1) 机制/流程

1. 浏览器先判断请求是否同源。
2. 同源请求直接发送，跨源请求进入 CORS 规则判断。
3. 简单请求直接带 `Origin`，复杂请求先发 `OPTIONS` 预检。
4. 服务端返回允许的源、方法、头字段后，浏览器才放行真实请求。
5. 响应若不满足 CORS 规则，浏览器会拦截结果。

### 2) 关键条件与触发点

- 条件 A：是否使用自定义头、非简单方法或特殊 `Content-Type`，决定是否触发预检。
- 条件 B：服务端是否返回正确的 `Access-Control-*` 头，决定浏览器是否放行。

### 3) 结果与影响

- 对性能 / 稳定性 / 可维护性的影响：预检会增加一次往返，配置混乱会导致线上“接口正常但前端报跨域错误”。

## 相似概念对比

| 维度 | 同源请求 | 跨源请求（CORS） | 结论 |
|---|---|---|---|
| 定义 | 协议、域名、端口都一致 | 任一维度不同 | 跨源默认受限，需要显式放行 |
| 触发时机 | 页面访问同源资源 | 访问不同源接口或资源 | 复杂请求常伴随预检 |
| 性能影响 | 无预检额外成本 | 可能增加 `OPTIONS` 往返 | 跨源接口需控制预检比例 |
| 适用场景 | 单域部署 | 前后端分离、多子域服务 | 安全策略和性能要同时设计 |

## 代码

```http
OPTIONS /api/orders HTTP/1.1
Origin: https://app.example.com
Access-Control-Request-Method: POST
Access-Control-Request-Headers: Content-Type, Authorization
```

```nginx
location /api/ {
  add_header Access-Control-Allow-Origin https://app.example.com always;
  add_header Access-Control-Allow-Methods 'GET,POST,PUT,DELETE,OPTIONS' always;
  add_header Access-Control-Allow-Headers 'Content-Type,Authorization' always;
  if ($request_method = OPTIONS) {
    return 204;
  }
}
```

### 示例说明

- 这段代码 / 配置解决什么问题：第一段展示预检请求格式，第二段给出服务端跨域放行配置。
- 关键参数含义：`Access-Control-Allow-Origin` 决定允许的源，`Access-Control-Allow-Methods` 决定允许的方法。
- 常见错误配置：把 `Allow-Origin` 配成 `*` 且同时携带凭证，导致浏览器拒绝。

## 真实场景示例

- 场景描述：前端改成多域部署后，登录态接口频繁跨域失败。
- 问题表现：服务端日志显示请求成功，但浏览器控制台报 CORS 错误。
- 处理方案：校正 `Origin` 白名单、补齐允许头字段，并对预检结果设置合理缓存时间。
- 验证方式：抓包确认预检与真实请求都返回允许字段，前端错误率下降。

## 常见误区

- 误区：只要服务端返回 `200`，前端就一定能拿到数据。
  - 正解：浏览器会在 CORS 校验失败时拦截响应，前端仍会报错。
- 误区：跨域问题只能前端解决。
  - 正解：核心在服务端响应头和网关策略，前端只能调整触发条件。

## 关联知识点

- 依赖：依赖 `http-method-semantics` 的方法语义和预检触发条件。
- 影响：跨域校验失败会直接中断 `request-lifecycle`。
- 对比：同源策略是浏览器安全边界，鉴权是服务端业务边界。
- 组合：与 `html-iframe` 节点常一起考跨源通信方案。

## 相关面试题

1. OPTIONS 请求方法及使用场景
   - 标准答案：`OPTIONS` 用于查询资源支持的方法与权限，在 CORS 复杂请求中充当预检请求，浏览器据此决定是否发送真实请求。
   - 考察点：是否能说清预检的触发条件和作用。
   - 追问 1：为什么预检会拖慢接口？
   - 标准答案 1：它引入额外往返请求，弱网或高延迟场景下成本会被放大。
   - 追问 2：如何减少预检成本？
   - 标准答案 2：收敛自定义头和复杂方法，合理设置预检缓存并合并跨域接口策略。

2. 常见的 HTTP 请求方法
   - 标准答案：`OPTIONS` 是方法集合中的探测方法，常用于跨域能力确认；与业务读写方法不同，它通常不承载业务实体处理。
   - 考察点：是否理解方法的职责分工。
