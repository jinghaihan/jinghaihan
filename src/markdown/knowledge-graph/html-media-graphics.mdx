## 概念定义

- 本质定义：媒体与图形节点关注图片、音视频、图形绘制和嵌入式媒体的原生表达方式，以及它们和带宽、清晰度、交互之间的关系。
- 解决问题：在“看得清、下得快、交互稳”之间找到合适的媒体策略。
- 边界说明：`iframe` 的跨文档通信在独立节点展开，离线缓存和资源预加载不在本节点细讲。

## 核心机制拆解

### 1) 机制/流程

1. 图片场景先确定是否只需静态展示，还是需要响应式切图和格式协商。
2. 音视频场景优先使用原生 `audio`、`video` 和 `source`，减少插件或过度定制。
3. 图形场景再判断用 Canvas 还是 SVG（Scalable Vector Graphics）：高频重绘偏前者，结构化矢量偏后者。
4. 如果是图像映射、外部媒体嵌入或旧格式兼容，再评估 `map` / `area`、`object`、`embed` 等补充方案。

### 2) 关键条件与触发点

- 条件 A：所有设备都下同一张大图时，低端机带宽和高分屏清晰度都会出问题。
- 条件 B：高频动画和复杂图形若选错渲染模型，后面会一直被性能和维护成本追着打。
- 条件 C：旧嵌入标签或外部媒体方案没有回退内容时，兼容和可访问性会很差。

### 3) 结果与影响

- 对性能/稳定性/可维护性的影响：媒体策略写对了，页面更轻也更清晰；写错了，最直接的结果就是糊图、卡顿和资源浪费。

## 相似概念对比

| 维度 | Canvas | SVG | 结论 |
|---|---|---|---|
| 渲染模型 | 像素绘制 | 矢量节点 | 关注点不同 |
| 更新方式 | 常靠重绘 | 常靠更新节点属性 | 高频动画更偏 Canvas |
| 交互支持 | 命中检测多靠脚本 | 可直接绑定节点事件 | 结构化图形更偏 SVG |
| 适用场景 | 游戏、粒子效果、频繁刷新图表 | 图标、流程图、可缩放图表 | 按更新模式选型 |

## 代码

```html
<picture>
  <source media="(min-width: 900px)" srcset="/images/hero-lg.webp" type="image/webp" />
  <source media="(min-width: 480px)" srcset="/images/hero-md.webp" type="image/webp" />
  <img
    src="/images/hero-sm.jpg"
    srcset="/images/hero-sm.jpg 480w, /images/hero-md.jpg 800w, /images/hero-lg.jpg 1200w"
    sizes="100vw"
    alt="活动主视觉"
  />
</picture>
```

### 示例说明

- 这段代码解决什么问题：让浏览器根据视口宽度和格式支持能力选择更合适的图片资源。
- 关键语句或参数含义：`picture` 负责媒体条件和格式协商；`source` 提供候选资源；`img` 是兜底显示元素。
- 常见错误写法：只写 `srcset` 不写 `sizes`，或者 `picture` 里没有稳定的 `img` 兜底。

## 真实场景示例

- 场景描述：商品详情页在高分屏上主图发糊，但移动端首屏流量又很高。
- 问题表现：桌面端清晰度不够，移动端下载体积过大；图表切到大数据量后还开始掉帧。
- 处理方案：核心图片接入 `srcset + sizes` 或 `picture`，图表按更新频率在 Canvas 和 SVG 之间重选。
- 验证方式：看图片平均下载体积、清晰度反馈、滚动流畅度和图表帧率。

## 常见误区

- 误区：`srcset` 只是给高分屏准备的。
- 正解：它的核心是让浏览器按展示尺寸和设备条件自动选更合适的资源。
- 误区：Canvas 一定比 SVG 快。
- 正解：场景才决定性能，高频重绘偏 Canvas，结构化矢量图偏 SVG。
- 误区：`object`、`embed` 已经过时，所以完全不用知道。
- 正解：现代项目确实少用，但面试仍然会问它们和原生媒体标签的区别。

## 关联知识点

- 依赖：HTML5 能力总览、加载与阻塞模型。
- 影响：图片优化、渲染性能、可访问性。
- 对比：响应式图片和固定图片、Canvas 和 SVG。
- 组合：懒加载、资源预加载、`iframe` 嵌入。

## 相关面试题

1. 如何在HTML中插入图片？
- 标准答案：使用 `img` 标签，并至少写好 `src` 和 `alt`。机制上 `src` 决定资源地址，`alt` 负责替代文本和可访问性。场景上现代项目除了基础写法，还常配 `srcset`、`sizes`、`loading="lazy"` 做资源优化。
- 考察点：是否知道图片标签不是只写一个地址就结束。

2. HTML中如何插入音频和视频？
- 标准答案：优先使用 `audio`、`video` 配合多个 `source` 标签提供不同格式。机制上浏览器会选择自己支持的第一种格式。场景上常见做法是准备 `mp4`、`webm` 或常用音频格式，同时给出不支持时的兜底文案。
- 考察点：是否知道原生媒体标签和多格式回退写法。

3. HTML5中的Canvas元素是什么？
- 标准答案：Canvas 是 HTML5 提供的像素绘图画布，真正绘制动作要通过脚本完成。机制上它本身只是一个位图容器，拿到上下文后才能画线、矩形、文字和图像。场景上图表、游戏、图像处理和复杂动画很常见。
- 考察点：是否理解 Canvas 不是“声明图形”，而是“提供画布”。

4. Canvas 和 SVG 的区别
- 标准答案：Canvas 是像素绘制，更新通常依赖整块重绘；SVG 是矢量节点，更新通常是改节点属性。机制上两者的渲染模型不同。场景上高频动画、粒子效果偏 Canvas，图标、流程图、结构化图表偏 SVG。
- 考察点：是否能把选型结论和渲染机制绑定起来。

5. HTML中如何实现响应式图片？
- 标准答案：常见做法有 `srcset + sizes`、`picture + source`、再配合 CSS 的 `max-width: 100%`。机制上浏览器会根据媒体条件、像素密度和展示宽度选择资源。场景上轮播图、商品图和 Banner（Banner）图都适合这样做。
- 考察点：是否知道响应式图片是浏览器原生能力，不必全靠脚本判断。

6. img 的 srcset 属性的作⽤？
- 标准答案：`srcset` 用于提供多份候选图片资源，让浏览器按设备条件和展示宽度自动选择。机制上它通常配合 `sizes` 一起使用，避免盲目下载大图。场景上同一位置在手机、平板、桌面上展示尺寸不同，就该考虑它。
- 考察点：是否理解 `srcset` 的核心价值是“清晰度和带宽平衡”。

7. HTML中的map和area标签如何创建图像映射？
- 标准答案：做法是给 `img` 写 `usemap`，再用 `map` 包多个 `area` 定义可点击区域。机制上 `area` 通过坐标定义矩形、圆形或多边形热点。场景上园区图、座位图、老式导航图仍然会用到，但现代项目要额外注意响应式适配和可维护性。
- 考察点：是否知道图像映射的基本结构和局限。

8. HTML中的object和embed标签有什么区别？
- 标准答案：`object` 更标准，支持回退内容和更丰富的参数配置；`embed` 更直接，但标准化程度和回退能力弱一些。机制上两者都用于嵌入外部资源，但现代项目更常用原生 `audio`、`video`、`iframe` 或直接下载跳转。场景上面试回答时要顺手补一句“现代项目通常优先原生媒体标签”。
- 考察点：是否知道旧嵌入方案和现代替代方案的关系。
