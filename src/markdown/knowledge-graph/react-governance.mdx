- 场景描述：核心页面迭代中出现跨组件状态与渲染时序问题。

2. Component, Element, Instance 之间有什么区别和联系？
- 标准答案：结论是 Element 是描述 UI 的普通对象，Component 是定义 UI 的函数/类，Instance 是运行时组件实例（类组件明显）。机制上 React 根据 Element 创建/更新 Instance。场景上理解三者有助于定位渲染和状态问题。
- 考察点：是否能从定义、触发时机、性能影响和适用场景四个维度做对比。

3. React.createClass 和 extends Component 的区别有哪些？
- 标准答案：结论是 `createClass` 是旧 API，`extends Component` 是 ES6 类写法。机制上前者自动绑定 `this` 且依赖 mixin，后者更符合现代 JavaScript 标准。场景上新项目统一使用函数组件或 ES6 类组件。
- 考察点：是否能从定义、触发时机、性能影响和适用场景四个维度做对比。

7. 对 React 和 Vue 的理解，它们的异同
- 标准答案：结论是二者都走组件化与声明式更新，但响应式模型和生态侧重点不同。机制上 React 更偏函数重渲染与调度，Vue 更偏依赖追踪与编译优化。场景上选型应看团队经验、项目复杂度和工程生态。
- 考察点：是否能从定义、触发时机、性能影响和适用场景四个维度做对比。

12. 在 React 中页面重新加载时怎样保留数据？
- 标准答案：结论是要把关键状态持久化到 URL、localStorage、sessionStorage 或服务端。机制上刷新会重建运行时，只有外部存储或可重放状态能恢复。场景上筛选条件用 URL，用户草稿和会话状态用本地存储加版本控制。
- 考察点：是否能给出明确结论，并补充机制和业务场景。

13. 如何理解 UI = f(state) 这个公式？
- 标准答案：结论是界面是状态的纯映射结果。机制上同一输入状态应得到同一渲染输出，状态变化触发重渲染。场景上这要求把可变数据集中管理，避免直接改 DOM 造成状态与视图不一致。
- 考察点：是否能把“声明式渲染”讲成可执行的状态建模规则。

14. React 的事件机制和合成事件是如何工作的？
- 标准答案：结论是 React 用合成事件统一浏览器差异。机制上事件在根容器委托监听，再按组件树分发回调并参与批处理。场景上这能降低跨浏览器差异，但与原生事件混用时要注意传播和时序。
- 考察点：是否理解事件委托、统一封装和批处理三者关系。

15. React 的严格模式 (StrictMode) 有什么作用？
- 标准答案：结论是它用于开发期发现不安全副作用和过时 API。机制上会对部分逻辑做额外检查与重复调用探测。场景上重构阶段长期开启可提前暴露潜在线上问题。
- 考察点：是否能区分“开发期检查”与“生产行为”边界。

16. 为什么在 React 开发中，我们总是提倡“组合优于继承”？
- 标准答案：结论是组合更符合组件化复用模型。机制上通过 props、children、Hook 组合能力，避免继承层级带来的强耦合。场景上权限、布局、数据加载等横切能力都更适合组合实现。
- 考察点：是否能给出组合替代继承的具体落地方式。
