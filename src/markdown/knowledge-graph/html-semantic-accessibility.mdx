## 概念定义

- 本质定义：语义化与可访问性是用合适的标签表达合适的结构、含义和交互角色，让浏览器、搜索系统和辅助技术都能稳定理解页面。
- 解决问题：提升结构可读性、可维护性、检索友好度和键盘/读屏体验。
- 边界说明：本节点聚焦标签语义和结构表达，不展开表单校验逻辑和脚本交互细节。

## 核心机制拆解

### 1) 机制/流程

1. 先按页面结构选标签，比如 `header`、`main`、`article`、`section`、`footer`。
2. 再按文本含义选标签，比如标题、引用、强调、注音、缩写、预格式化文本。
3. 遇到导航、列表、表格等结构时，优先使用原生标签而不是 `div` 平铺。
4. 最后用键盘导航、读屏和页面大纲回头验证语义是否真的成立。

### 2) 关键条件与触发点

- 条件 A：页面一律使用 `div + span` 拼结构时，后续可访问性和维护成本会一起升高。
- 条件 B：标题层级、强调标签和结构标签混用时，搜索系统和读屏器对页面的理解会偏掉。
- 条件 C：把视觉样式标签误当语义标签使用，会让后续样式和语义都变得别扭。

### 3) 结果与影响

- 对性能/稳定性/可维护性的影响：语义写对了，样式和脚本都更容易挂上去；语义写乱了，后续只能靠补丁式属性和脚本弥补。

## 相似概念对比

| 维度 | `strong` / `em` | `b` / `i` | 结论 |
|---|---|---|---|
| 语义 | 强调重要性和语气 | 更偏视觉表现 | 要强调含义时优先前者 |
| 可访问性 | 辅助技术更容易识别重点 | 语义价值弱 | 不能简单互换 |
| 样式关系 | 样式可再覆盖 | 默认是视觉表现 | 语义和样式应分层 |
| 适用场景 | 重点提醒、语气强调 | 纯视觉微调 | 先语义，后样式 |

## 代码

```html
<article>
  <header>
    <h1>性能复盘</h1>
    <p>发表于 <time datetime="2026-03-06">2026 年 3 月 6 日</time></p>
  </header>

  <section>
    <h2>问题现象</h2>
    <p>首屏出现明显白屏，用户投诉量上升。</p>
  </section>

  <p>这次变更的 <strong>直接影响</strong> 是转化率下降。</p>
</article>
```

### 示例说明

- 这段代码解决什么问题：把文章、章节、标题、时间和强调关系用原生语义标签直接表达出来。
- 关键语句或参数含义：`article` 表示独立内容；`section` 表示章节；`time` 表示时间语义；`strong` 表示重点。
- 常见错误写法：整页都用 `div`，标题不分层，强调语义全部写成粗体样式。

## 真实场景示例

- 场景描述：内容页迁移到新模板后，SEO 流量下滑，读屏用户反馈页面结构读起来很乱。
- 问题表现：页面只有视觉层级，没有稳定结构；标题、列表、引用和表格都靠 `div` 模拟。
- 处理方案：补语义结构标签，整理标题层级，替换掉只负责样式的伪语义写法。
- 验证方式：检查页面大纲、读屏朗读顺序、搜索摘要展示和内容编辑维护成本。

## 常见误区

- 误区：语义化只影响 SEO（Search Engine Optimization）。
- 正解：它同样影响可访问性、协作维护和组件边界。
- 误区：页面好看就说明结构写对了。
- 正解：视觉对了不等于语义对了，读屏器和搜索系统不会看你的设计稿。
- 误区：ARIA（Accessible Rich Internet Applications）能替代所有原生语义标签。
- 正解：ARIA 是补充，原生标签仍然是第一选择。

## 关联知识点

- 依赖：文档声明与 head 配置。
- 影响：表单与原生交互控件、媒体与图形、兼容策略。
- 对比：结构语义和视觉语义。
- 组合：标题层级、列表结构、表格结构、引用和注音。

## 相关面试题

1. 什么是语义化HTML？为什么重要？
- 标准答案：语义化 HTML 指的是用具有明确含义的标签来描述内容结构，而不是只用无语义容器去堆页面。机制上它能帮助浏览器、搜索系统和辅助技术理解页面。场景上内容站、后台系统、组件库都能从语义化里直接受益。
- 考察点：是否能把语义化说到工程价值，而不是只说“利于 SEO”。

2. 对 HTML 语义化的理解
- 标准答案：核心不是“换一批新标签”，而是让标签和内容职责对应起来。机制上标题、导航、文章、列表、表格、时间、引用都有各自语义。场景上语义写准了，后续读屏、样式覆盖和组件抽象都会更顺。
- 考察点：是否真正理解“标签 = 内容意图”。

3. 什么是HTML的语义化标签？列举一些例子
- 标准答案：语义化标签就是名字本身就能说明用途的标签，比如 `header`、`nav`、`main`、`article`、`section`、`aside`、`footer`、`time`、`figure`、`figcaption`。机制上这些标签会让文档结构更清晰。场景上文章页、列表页、详情页都该优先用它们，而不是全部交给 `div`。
- 考察点：是否能列出代表项并说出用途。

4. 常用的HTML标签有哪些？
- 标准答案：可以按类别回答。文本标签常见有 `h1` 到 `h6`、`p`、`span`、`strong`、`em`；结构标签有 `div`、`header`、`nav`、`main`、`section`、`article`、`footer`；列表标签有 `ul`、`ol`、`li`、`dl`、`dt`、`dd`；表格标签有 `table`、`thead`、`tbody`、`tr`、`th`、`td`；媒体和链接有 `a`、`img`、`audio`、`video`。场景上答题时最好按类别分组，不要一口气散着背。
- 考察点：是否具备基础标签分类能力。

5. HTML中的块级元素和行内元素有什么区别？
- 标准答案：块级元素默认独占一行，常见如 `div`、`p`、`h1` 到 `h6`、`ul`、`table`；行内元素默认在一行内排布，常见如 `span`、`a`、`strong`、`em`、`img`。机制上它们的默认显示行为不同，但这只是默认行为，不是 CSS（Cascading Style Sheets）层面永远不可变的物理定律。场景上面试时要说明“这是默认语义和默认布局表现”，别把它说成绝对规则。
- 考察点：是否知道标签默认行为和 CSS 显示模型的边界。

6. 行内元素有哪些？块级元素有哪些？ 空(void)元素有那些？
- 标准答案：行内元素常见有 `a`、`span`、`strong`、`em`、`b`、`i`、`img`、`label`；块级元素常见有 `div`、`p`、`h1` 到 `h6`、`ul`、`ol`、`li`、`table`、`section`、`article`、`header`、`footer`；空元素常见有 `br`、`hr`、`img`、`input`、`meta`、`link`、`source`、`area`、`base`、`col`、`embed`、`track`、`wbr`。机制上空元素没有子内容和结束标签。场景上标签分类答题要给清单，不要只说“有很多”。
- 考察点：是否具备准确列举能力。

7. title 与 h1 的区别、b 与 strong 的区别、i 与 em 的区别？
- 标准答案：`title` 是文档标题，常用于标签页、搜索摘要和分享信息；`h1` 是正文主标题，属于页面结构。`strong` 和 `em` 表示语义强调，`b` 和 `i` 更偏视觉呈现。场景上页面通常既要有 `title`，也要有 `h1`，而文字强调应优先用 `strong` / `em`。
- 考察点：是否能区分结构语义、文本语义和视觉样式。

8. HTML表格的基本结构是什么？
- 标准答案：表格基础结构是 `table` 包 `thead`、`tbody`、`tfoot`，内部再用 `tr` 表示行，`th` 表示表头单元格，`td` 表示数据单元格。机制上浏览器和读屏器会按这个结构理解行列关系。场景上展示结构化数据时应该用原生表格，而不是纯 `div` 模拟。
- 考察点：是否知道表格的结构语义和适用场景。

9. HTML中的列表有哪些类型？
- 标准答案：常见有无序列表 `ul`、有序列表 `ol`、描述列表 `dl`，配套标签分别是 `li` 或 `dt` / `dd`。机制上列表标签会明确表达“并列项”“顺序项”“术语解释”三种不同结构。场景上导航、步骤说明、术语表都该选对应列表，而不是靠段落硬排。
- 考察点：是否知道列表是结构语义，不只是默认小圆点样式。

10. HTML中的pre标签有什么作用？
- 标准答案：`pre` 用于展示预格式化文本，会保留空格、换行和缩进。机制上浏览器不会像普通段落那样折叠这些空白。场景上代码块、日志片段、命令输出和 ASCII（American Standard Code for Information Interchange）图形都适合用它。
- 考察点：是否理解预格式化文本的真实用途。

11. HTML中的abbr标签有什么用途？
- 标准答案：`abbr` 用来标记缩写词或首字母缩略词，通常配合 `title` 给出完整解释。机制上浏览器和辅助技术可以据此识别缩写含义。场景上专业术语、国家缩写和组织简称适合这样标记。
- 考察点：是否知道缩写标签的语义价值，而不只是一个下划线样式。

12. HTML中的kbd、samp、var标签分别用于什么？
- 标准答案：`kbd` 表示用户输入或键盘按键，`samp` 表示程序输出，`var` 表示变量名。机制上它们都属于文本语义标签，用来表达内容角色。场景上技术文档、教程和命令行示例里很常见。
- 考察点：是否能区分“输入”“输出”“变量”三种文本角色。

13. HTML中的cite标签如何使用？
- 标准答案：`cite` 用于标记作品标题或引用来源，比如书名、论文、电影、站点名。机制上它表达的是“引用出处”而不是普通强调。场景上书评、文章引用和参考来源说明都能用它。
- 考察点：是否知道引用语义而不是把它当斜体样式。

14. HTML5中的article和section标签有什么区别？
- 标准答案：`article` 表示一块可以独立分发、独立理解的内容，比如文章、评论、卡片；`section` 表示文档中的章节或主题分组。机制上前者更强调独立性，后者更强调“归属在一个更大的内容里”。场景上博客正文适合 `article`，正文内部的小节更适合 `section`。
- 考察点：是否理解“独立内容”和“章节分组”的边界。

15. HTML中的ruby、rt、rp标签用于什么？
- 标准答案：它们用于东亚文字注音或注释。机制上 `ruby` 是容器，`rt` 是注音内容，`rp` 是不支持注音时的回退括号。场景上中文拼音、日文假名和古文注释都可以使用。
- 考察点：是否知道这些标签不是冷门装饰，而是明确的语言辅助能力。

16. HTML中的wbr标签有什么作用？
- 标准答案：`wbr` 表示“这里可以断行”，但不是强制断行。机制上浏览器会在需要换行时把它当成候选断点。场景上长 URL、长变量名和技术文档里的超长单词很适合加 `wbr`。
- 考察点：是否理解它和 `word-break` 这类 CSS 规则的差异。

17. 什么是HTML实体？常用的有哪些？
- 标准答案：结论是 HTML 实体用于在文档中表示保留字符或特殊符号。机制上浏览器解析实体编码后再渲染对应字符，如 `&lt;`、`&gt;`、`&amp;`、`&nbsp;`、`&quot;`。场景上展示代码片段、保留空格或特殊符号时必须使用实体避免解析冲突。
- 考察点：是否能说明实体的用途和代表项。

18. HTML5中的details和summary标签是什么？
- 标准答案：结论是 `details` 和 `summary` 提供原生折叠面板能力。机制上 `summary` 作为可点击标题控制 `details` 的展开状态。场景上 FAQ、帮助说明和次要内容收纳可直接使用该声明式能力。
- 考察点：是否了解其交互语义和适用场景。
