## 概念定位

构建流水线与产物拆分决定了“代码如何变成可分发资产”，核心关注编译速度、产物体积、缓存友好性与可追踪性。

该节点是性能预算和发布稳定性的共同支点。

## 工作机制

1. 构建阶段完成语法转换、依赖解析、代码分割、压缩与指纹生成。
2. 通过 entry/chunk 策略把首屏关键代码和低频代码拆开。
3. 利用 Tree Shaking（摇树优化）和动态导入降低首包体积。
4. 产物需带内容哈希（content hash）支持长期缓存。
5. 生成构建元数据（sourcemap、资产清单、版本号）用于排障与回滚。

## 关键知识点

### 必须掌握

- bundle/chunk/module 语义要分清：module 是源文件单位，chunk 是运行时加载单位。
- 公共依赖抽离要平衡复用与首屏请求数量。
- 构建速度慢要分清是转译瓶颈还是插件瓶颈。
- Source Map（源码映射）环境策略要区分生产和内部排障。
- 构建优化不仅看体积，还要看解析/执行成本。
- 按业务路由拆包比按技术层盲拆更有效。

### 选型与取舍

- 过细拆包减少首包体积，但增加请求数量与调度成本。
- 合并包减少请求，但可能放大初始解析执行开销。
- 激进压缩可降体积，但构建耗时与调试难度会上升。

## 对页面性能与稳定性的影响

- 节点稳定时，前端工程化 的链路可预测性会明显提升，发布和排障成本更可控。
- 节点失稳时，问题会向上下游扩散，常表现为性能回退、交互异常或交付风险叠加。
- 该节点优化应和上下游协同验证，避免“局部优化把成本转移给后续节点”。

## 观测与排障

### 核心指标

- 构建时长（全量/增量）
- 首包体积与总 JS 体积
- chunk 数量与重复依赖比例
- 线上 Source Map 命中率（排障可用性）

### 排障顺序

1. 先看产物分析报告，定位超大 chunk 与重复依赖。
2. 检查拆包策略是否与路由访问模式匹配。
3. 检查插件链路耗时，识别低收益高耗时插件。
4. 验证摇树是否生效（ESM 导出结构是否可分析）。
5. 回归评估：构建速度、首屏指标、缓存命中率。

### 常见故障场景

- “构建越来越慢”：常见是插件链叠加和依赖膨胀。
- “首包突然变大”：常见是大库进入主入口。
- “分包后体验反而差”：常见是拆得过细导致瀑布请求。
- “线上报错难定位”：常见是 Source Map 策略缺失。

## 常见误区

- 只追求体积最小，忽略执行成本和请求时序。
- 长期不做构建报告审计。
- 构建策略与发布缓存策略脱节。

## 深入拆解

### 拆包策略常见模式

- Route-based Splitting：按路由拆分，优先保障首屏路径。
- Vendor Splitting：稳定第三方依赖抽离，提升缓存复用。
- Component-level Lazy Load：低频大组件延迟加载。

### 构建耗时优化优先级

1. 先排查最慢环节：转译、类型检查、压缩、插件。
2. 再做并行与缓存：持久化缓存、增量构建。
3. 最后做架构调整：依赖瘦身、模块边界优化。

### 常见体积优化失败原因

- 引入 CommonJS 包导致摇树收益下降。
- polyfill 注入策略过宽。
- 重复依赖版本并存。

## 题库深挖补充

### 高频对比与边界（基于题库）

- 构建优化看三件事：速度、体积、可追踪，不是只看包大小。
- 质量门禁应分级阻断：P0 直接阻断，P1 评审放行，P2 持续治理。
- 缓存发布要做入口与静态资源差异化策略，避免版本错配。
- CI/CD 需要幂等和审计，失败重试不能污染环境状态。
- 灰度与监控联动是发布安全关键，不可分离。
- 回滚预案要可演练，不是“有命令就算有能力”。

### 线上排障剧本

1. 先定位问题发生在 build、verify、deploy、release 哪一段。
2. 检查门禁日志与放行记录，确认是否越权放行。
3. 检查缓存策略与发布顺序是否一致。
4. 灰度异常先止损，再做根因定位。
5. 回滚后验证核心链路并进入复盘。

### 高频追问补充（附答案）

1. 为什么说发布成功不等于上线成功？

真实流量下仍可能暴露性能或稳定性问题。

2. 灰度阶段最关键看什么？

错误率、核心性能分位数和关键业务转化。

3. 回滚后还要做什么？

确认恢复、记录时间线、沉淀治理规则。

## 进阶深挖（第二轮）

### 构建策略分层

- 入口层：确保首屏关键路径最短。
- 共享层：稳定依赖抽离提升缓存复用。
- 业务层：按路由与访问频率拆分。
- 低频层：按需动态加载。

### 构建回归清单

1. 首包体积与关键路由体积是否超预算。
2. chunk 重复依赖是否异常增加。
3. Source Map 和版本映射是否可追踪。
4. 构建耗时变化是否在预期窗口。

### 追问扩展

1. 为什么拆包后有时反而慢？

过度拆分会增加请求数量和调度开销，尤其在弱网更明显。

2. Tree Shaking 失效如何快速定位？

先看依赖是否为静态可分析模块，再查副作用声明与导出方式。

## 面试补充

### 30 秒说明

构建优化的目标不是“打包更小”单一目标，而是“构建效率、运行性能、排障可追踪”三者平衡。

### 高频追问

1. 拆包为什么会影响运行时性能？

因为拆包改变请求数量和执行时序，过度拆分会带来额外调度开销。

2. Tree Shaking 失效常见原因是什么？

模块导出不可静态分析、存在副作用或 CommonJS 混用。

3. 如何判断一个 chunk 是否该拆？

看访问频率、首屏关键性和缓存复用价值，而不是只看体积。

4. 构建优化和发布策略有什么关系？

产物指纹、分包策略和缓存 TTL 要联动，否则会出现命中率下降。
