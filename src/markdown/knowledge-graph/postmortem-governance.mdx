## 概念定位

复盘沉淀与规则固化节点把事故经验转成流程和工具规则，目标是“同类问题不再重复发生”。

这是团队工程能力持续增长的关键闭环。

## 工作机制

1. 复盘先还原事实时间线，区分事实、判断和假设。
2. 根因分析采用多层追问（如 5 Why）定位系统性问题。
3. 改进项分级：立刻修复、流程改造、工具化门禁。
4. 每个改进项绑定负责人、截止时间、验收标准。
5. 定期回看改进项完成度与复发率。

## 关键知识点

### 必须掌握

- 复盘目标是改系统，不是找责任人背锅。
- 没有量化验收标准的改进项会失效。
- 高频故障要优先工具化，减少人为依赖。
- 复盘结论要同步到文档、模板、门禁三处。
- 复盘输入应包含监控数据、日志、发布记录。
- 治理有效性的核心指标是复发率下降。

### 选型与取舍

- 深度复盘耗时高，但可显著降低长期事故成本。
- 只做轻量复盘推进快，但问题容易反复。
- 工具化改造投入大，但长期收益稳定。

## 对页面性能与稳定性的影响

- 节点稳定时，前端工程化 的链路可预测性会明显提升，发布和排障成本更可控。
- 节点失稳时，问题会向上下游扩散，常表现为性能回退、交互异常或交付风险叠加。
- 该节点优化应和上下游协同验证，避免“局部优化把成本转移给后续节点”。

## 观测与排障

### 核心指标

- 同类故障复发率
- 复盘改进项完成率
- 复盘到规则落地时长
- 门禁覆盖率提升

### 排障顺序

1. 先收集完整证据，避免回忆偏差。
2. 识别直接原因和系统性根因。
3. 确定改进项优先级和治理路径。
4. 跟踪执行并验证效果。
5. 将高价值经验纳入团队训练材料。

### 常见故障场景

- “同类事故反复发生”：常见是复盘只停留在文档层。
- “改进项没人跟进”：常见是责任和验收不清。
- “复盘会开很多但价值低”：常见是没有数据证据链。
- “工具规则和实际脱节”：常见是改进项未回归验证。

## 常见误区

- 把复盘当形式化会议。
- 只写“加强沟通”这类不可执行结论。
- 不追踪改进项效果。

## 题库深挖补充

### 高频对比与边界（基于题库）

- 需求冻结的核心是边界和验收口径，不是禁止变更。
- 仓库分层要定义依赖方向，避免“能 import 就 import”。
- 依赖治理要覆盖安全漏洞、体积增长、升级节奏三条线。
- 配置中心要支持审计与回滚，避免环境漂移。
- 复盘输出必须可执行：负责人、截止日期、验收标准缺一不可。
- 面试常问“工程化价值”，答案应覆盖效率、稳定性、可追踪性。

### 线上排障剧本

1. 先判断故障是流程缺失还是实现缺陷。
2. 检查需求/架构/依赖/配置是否存在边界越权。
3. 拉出变更时间线和责任链路。
4. 给出短期止血 + 中期治理动作。
5. 把结论沉淀为规则并跟踪复发率。

### 高频追问补充（附答案）

1. 为什么需求评审要带非功能指标？

因为性能、安全、可观测会影响架构路径，后置补救成本高。

2. 依赖治理最容易被忽视的点？

间接依赖和供应链风险，经常不在日常视野内。

3. 配置漂移为什么危险？

同代码在不同环境表现不同，事故定位会显著变慢。

## 进阶追问（补充）

1. 为什么工程化问题常在高峰期集中暴露？

因为流量放大后，缓存错配、发布窗口、告警噪声和回滚能力短板会被同时放大。

2. 门禁越严越好吗？

不是。要按风险分级阻断，既拦高风险问题，也要控制交付节奏。

3. 如何判断一次事故复盘是否有效？

看同类问题复发率是否下降，以及改进项是否转化为自动化规则。

## 案例追问加深（第三轮）

### 真实故障片段

- 故障表现：发布流水线显示通过，但上线后核心指标快速恶化。
- 排查结论：门禁覆盖了代码质量，却没有覆盖发布后观测与缓存一致性。
- 治理动作：把灰度阈值、缓存一致性检查和回滚演练纳入同一发布清单。

### 追问补充

1. 为什么工程化问题常在业务高峰暴露？

高峰流量会放大流程短板，平时隐性的风险会在短时间集中显现。

2. 如何衡量工程化治理是否真的有效？

看变更失败率、恢复时长和同类事故复发率是否持续下降。

## 面试补充

### 30 秒说明

复盘治理的本质是把一次事故变成长期能力。没有规则固化，复盘价值会快速衰减。

### 高频追问

1. 高质量复盘最关键的产出是什么？

可执行、可验证、可追踪的改进项，而不是结论口号。

2. 为什么复盘要强调系统性根因？

否则只修表象问题，系统缺陷仍会触发类似事故。

3. 如何判断复盘是否有效？

看同类故障复发率和改进项落地率是否持续下降。

4. 复盘结果如何落地到工程流程？

同步到门禁规则、发布流程和团队规范文档。
