## 概念定位

源站响应节点决定内容是否正确、是否可缓存、是否可分块传输，是请求链路与渲染链路的分界点。

响应头与响应体策略直接影响后续解析性能和缓存效果。

## 工作机制

1. 源站接收请求并执行业务处理。
2. 返回状态码、响应头和响应体。
3. 通过缓存头声明可缓存性与协商策略。
4. 通过压缩和分块传输优化传输效率。
5. 异常时返回可降级响应并记录 trace。

## 关键知识点

### 必须掌握

- 状态码要和业务语义一致，便于前端分支处理。
- Cache-Control/ETag/Last-Modified 组合决定缓存命中行为。
- 响应体过大应考虑分片、懒加载或流式输出。
- 压缩策略要平衡 CPU 和带宽。
- 错误响应也应可观测且可定位。
- 源站 SLA 会直接影响前端可用性。

### 选型与取舍

- 强一致响应新鲜但缓存收益低。
- 缓存友好响应快，但需要失效治理。
- 流式输出首字节快，但消费复杂度更高。

## 对页面性能与稳定性的影响

- 节点稳定时，Knowledge Graph 主线 的网络与渲染链路更可预测，性能治理收益更稳定。
- 节点失稳时，问题会向后续阶段扩散，常表现为首屏变慢、交互卡顿或版本一致性风险。
- 优化动作应结合上下游联动验证，避免局部优化把成本转移到其他节点。

## 观测与排障

### 核心指标

- 源站响应时间
- 状态码分布
- 压缩比与传输体积
- 缓存头合规率

### 排障顺序

1. 先看状态码和响应头是否符合预期。
2. 检查响应体大小和传输方式。
3. 检查缓存声明与前端行为一致性。
4. 检查慢请求是否集中在特定业务分支。
5. 补充链路追踪字段进行复现。

### 常见故障场景

- 接口 200 但缓存不命中。
- 响应体过大导致首屏慢。
- 源站波动导致页面间歇失败。
- 错误响应无法定位根因。

## 常见误区

- 只关注业务字段，不重视响应头。
- 错误响应缺少上下文。
- 缓存声明与内容版本策略脱节。

## 题库深挖补充

### 高频对比与边界（基于题库）

- GET 偏读取、可缓存；POST 偏创建；PUT 常用于整体替换；PATCH 常用于局部更新。
- 幂等性是重试策略前提：GET/PUT/DELETE 通常幂等，POST 默认不幂等。
- 206（Partial Content）对应分段传输，常用于视频与断点续传。
- 307/308 与 302/301 的关键区别是是否保留原方法语义。
- 304 与缓存协商绑定，不能简单理解为“服务器没处理”。
- HTTP/1.1 Keep-Alive 与 HTTP/2 多路复用的优化点不在同一层。

### 线上排障剧本

1. 先按状态码族（2xx/3xx/4xx/5xx）分层定位。
2. 对关键请求检查方法语义与重试策略是否匹配。
3. 检查跨域请求中 OPTIONS 预检是否异常放大。
4. 检查重定向是否改变方法导致副作用。
5. 拉通前后端日志确认 traceId 全链路一致。

### 高频追问补充（附答案）

1. 为什么 302 可能引发业务副作用？

部分客户端在重定向时可能改变方法语义，导致行为偏差。

2. 304 的核心价值是什么？

复用本地副本，减少下载体积，但仍有请求往返成本。

3. 预检请求什么时候最容易失控？

自定义头过多、跨域策略复杂且无收敛时。

## 进阶追问（补充）

1. 如果网络指标正常但用户仍感知慢，优先看哪一层？

优先看主线程任务和渲染阶段（Layout/Paint/Composite）是否拥塞，网络快不代表交互快。

2. 缓存策略和发布策略为什么必须一起设计？

缓存决定资源复用路径，发布决定资源版本切换；两者脱节会出现版本错配。

3. 如何验证一次链路优化不是偶然收益？

至少做对照组、分位数对比，并覆盖冷缓存与弱网样本。

## 案例追问加深（第三轮）

### 真实故障片段

- 故障表现：同一版本在不同地区首屏耗时差异明显，且刷新方式不同结果不同。
- 排查结论：网络链路、缓存路径和渲染调度叠加导致体验分化，单点优化无法彻底解决。
- 治理动作：建立链路分层观察面板，按导航类型、缓存命中层和渲染阶段联合分析。

### 追问补充

1. 为什么链路问题经常不是单点根因？

Knowledge Graph 本身是多阶段因果链，一个阶段的波动会在后续阶段被放大。

2. 如何快速缩小排障范围？

先固定触发方式和样本维度，再按网络、缓存、渲染三层逐步排除。

## 面试补充

### 30 秒说明

源站响应节点不是单纯返回数据，而是通过状态、头部和传输策略共同决定前端体验上限。

### 高频追问

1. 为什么响应头这么重要？

它决定缓存、压缩、安全和后续处理分支。

2. 200 一定表示体验好么？

不一定，响应体过大或头部策略错误仍会导致慢和错配。

3. 何时考虑流式响应？

大体量内容或需要尽快呈现首段内容时。

4. 如何让错误响应更可排障？

返回可识别错误码并附带 traceId。
