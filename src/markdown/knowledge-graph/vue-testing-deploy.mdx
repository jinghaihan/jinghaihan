## 概念定位

测试、构建与发布策略节点把 Vue 代码质量与交付稳定性连成闭环，核心是关键路径回归和发布可回退。

它决定优化成果是否能稳定落地到生产环境。

## 工作机制

1. 测试分层：组件单测、页面集成测试、关键流程端到端测试。
2. 构建阶段接入类型检查、打包分析和性能预算校验。
3. 发布阶段采用灰度放量和回滚预案。
4. 失败样本自动归档，便于追踪回归根因。
5. 把发布事故和缺陷复盘回流到测试策略。

## 关键知识点

### 必须掌握

- 测试重点是关键路径而非平均覆盖。
- 快测和全量回归应分离执行，兼顾速度和质量。
- UI 回归应覆盖主题、分辨率和关键交互。
- 发布前健康检查要覆盖核心页面可达性。
- 回滚演练要定期执行。
- 测试数据和环境要可重复。

### 选型与取舍

- 全量回归最稳，但耗时高。
- 快测效率高，但覆盖不足。
- 高自动化投入大，但长期收益显著。

## 对页面性能与稳定性的影响

- 节点稳定时，Vue 工作流 的链路可预测性会明显提升，发布和排障成本更可控。
- 节点失稳时，问题会向上下游扩散，常表现为性能回退、交互异常或交付风险叠加。
- 该节点优化应和上下游协同验证，避免局部优化把成本转移给后续节点。

## 观测与排障

### 核心指标

- 回归通过率
- 发布成功率
- 线上缺陷逃逸率
- 回滚触发率

### 排障顺序

1. 先定位失败层级：单测、集成、E2E 还是发布环节。
2. 检查测试环境与生产差异。
3. 检查构建产物和配置注入一致性。
4. 按预案回滚并验证恢复。
5. 将根因映射到新增测试用例。

### 常见故障场景

- 本地通过但线上失败。
- 测试覆盖高但关键流程仍漏测。
- 灰度正常全量异常。
- 回滚流程执行慢。

## 常见误区

- 只看覆盖率，不看关键路径。
- 测试策略与复盘脱节。
- 发布成功即结束，不做上线观察。

## 题库深挖补充

### 高频对比与边界（基于题库）

- Vue 架构与组件通信要优先保证单向数据流和职责边界。
- props/emits 与 provide/inject 的适用场景不同，不能混用代替状态管理。
- v-model 在自定义组件本质是 modelValue + update:modelValue 协议。
- 测试策略要覆盖组件契约、路由流程、发布回归三层。
- 团队治理要把高频问题沉淀成 lint/脚手架/评审清单。
- 面试常问“Vue 项目怎么落地规范”，重点是规则可执行而非文档数量。

### 线上排障剧本

1. 先确认问题属于通信边界、架构边界还是发布回归缺口。
2. 检查 props 透传深度和事件命名语义。
3. 检查是否存在隐式共享状态导致耦合。
4. 回归关键页面与关键交互流程。
5. 复盘后把问题映射到团队规则。

### 高频追问补充（附答案）

1. provide/inject 能替代 Pinia 吗？

不能完全替代。它更适合稳定上下文，不适合复杂共享状态治理。

2. 为什么通信协议要显式命名？

可追踪、可调试、可复盘，避免隐式副作用。

3. Vue 规范治理最有效抓手是什么？

自动化规则 + 评审清单 + 复盘闭环。

## 进阶追问（补充）

1. Vue 性能排障为什么不能只看组件层？

因为请求竞态、状态边界和调度批处理会共同影响组件表现。

2. 什么时候应该优先改架构而不是继续微优化？

当问题持续跨页面复发，且根因是边界设计不清时，应先做结构治理。

3. 如何防止 Vue 项目后期通信失控？

统一通信协议、状态分层和评审规则，并让规则可自动检查。

## 案例追问加深（第三轮）

### 真实故障片段

- 故障表现：功能正确但更新抖动明显，复杂页面偶发状态错乱。
- 排查结论：响应式更新时机、组件通信边界和状态层策略没有统一约束。
- 治理动作：统一调度时机规范、通信协议和状态分层规则，并纳入评审门禁。

### 追问补充

1. 为什么 Vue 项目后期常出现“能跑但难改”？

边界规则缺失导致耦合积累，修改一个点会牵动多个链路。

2. 如何把 Vue 优化变成可持续能力？

把高频问题转成自动化规则和发布检查，不依赖个人经验记忆。

## 面试补充

### 30 秒说明

Vue 测试与发布节点的目标是“让改动可验证、发布可控、故障可回退”，而不是单纯追求通过率。

### 高频追问

1. 为什么要分层测试？

不同层覆盖不同风险，分层能在效率和质量间平衡。

2. 发布前最关键的检查是什么？

关键路径可用性、配置正确性和回滚能力。

3. E2E 用例是不是越多越好？

不是，应聚焦高价值主流程，避免维护成本失控。

4. 测试如何和复盘形成闭环？

把线上事故根因转成稳定复现用例并纳入门禁。
