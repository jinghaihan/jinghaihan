## 概念定义

- 本质定义：该节点聚焦 发布质量可验证可回滚。
- 解决问题：把高频面试问题转成可复述、可落地的机制说明。
- 边界说明：不替代其他节点的专属机制域。

## 核心机制拆解

### 1) 机制/流程

1. 明确输入状态和触发条件。
2. 进入框架调度或编译阶段。
3. 执行更新并产出结果。
4. 用回归验证结果是否稳定。

### 2) 关键条件与触发点

- 条件 A：组件边界和数据来源必须清晰。
- 条件 B：触发时机与副作用清理要成对出现。

### 3) 结果与影响

- 对性能/稳定性/可维护性的影响：机制清晰时可预测，机制混乱时会出现渲染回退、状态错乱或协作成本上升。

## 相似概念对比

| 维度 | SPA | MPA | 结论 |
|---|---|---|---|
| 定义 | 关注单点能力 | 关注对照能力 | 职责边界不同 |
| 触发时机 | 偏局部触发 | 偏链路触发 | 需要按场景选择 |
| 性能影响 | 局部可见 | 全链路可见 | 不可只看单点 |
| 适用场景 | 快速落地 | 长期治理 | 建议组合使用 |

## 代码

```ts
import { describe, expect, it } from 'vitest'
import { mount } from '@vue/test-utils'
import Counter from './Counter.vue'

describe('Counter', () => {
  it('click should increment', async () => {
    const wrapper = mount(Counter)
    await wrapper.get('button').trigger('click')
    expect(wrapper.text()).toContain('1')
  })
})
```

### 示例说明

- 这段代码解决什么问题：验证组件关键交互在发布前可自动回归。
- 关键语句或参数含义：`trigger` 模拟真实事件，`expect` 断言可见结果。
- 常见错误写法：只测工具函数不测页面关键路径，发布后仍会回归。

## 真实场景示例

- 场景描述：核心页面迭代中出现跨组件状态与渲染时序问题。
- 问题表现：功能可用但偶发抖动、错渲染或发布回退。
- 处理方案：先收敛边界，再做最小修复，最后补自动化验证。
- 验证方式：通过关键路径回归、性能指标和线上监控确认生效。

## 常见误区

- 误区：只记 API 就能稳定解决问题。
- 正解：必须同时说明机制、条件和场景。

## 相关面试题

1. Vue 单页应用与多页应用的区别
- 标准答案：结论是 SPA 首次加载重、后续切换快；MPA 每次导航都会请求新文档。机制上 SPA 依赖前端路由和客户端渲染，MPA 依赖服务端路由分发页面。场景上后台系统偏 SPA，内容站或 SEO 强诉求常用 MPA/SSR。
- 考察点：是否能从定义、触发时机、性能影响和适用场景四个维度做对比。

2. assets 和 static 的区别
- 标准答案：结论是 `assets` 会被构建工具处理（哈希、压缩、依赖图），`static/public` 通常原样拷贝。机制上前者可参与缓存控制和 tree-shaking，后者路径稳定。场景上业务图片走 assets，站点图标/robots 走 static/public。
- 考察点：是否能从定义、触发时机、性能影响和适用场景四个维度做对比。

3. 对 SSR 的理解
- 标准答案：结论是 SSR 在服务端先产出 HTML，再由客户端接管交互。机制上它改善首屏可见与 SEO，但引入水合、同构和服务端资源成本。场景上内容密集或搜索流量依赖强的页面优先 SSR。
- 考察点：是否能给出明确结论，并补充机制和业务场景。

4. 对 SPA 单页面的理解，它的优缺点分别是什么？
- 标准答案：结论是 SPA 优点是交互流畅、前后端分离清晰；缺点是首屏和 SEO 压力更大。机制上所有路由共享同一运行时，切换无需整页刷新。场景上可通过懒加载、预渲染、SSR 弥补其短板。
- 考察点：是否能从定义、触发时机、性能影响和适用场景四个维度做对比。
