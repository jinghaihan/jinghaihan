## 概念定位

状态管理节点关注 Context、Redux、Zustand 等方案的边界与取舍，核心是共享状态一致性与更新粒度控制。

这个节点决定复杂应用在规模扩张时是否还能保持可预测更新。

## 工作机制

1. 先按数据类型分层：服务端状态、UI 状态、会话状态。
2. Context 用于低频全局上下文，共享业务状态可用专用 store。
3. 订阅粒度要细化，避免全局更新触发全树重渲染。
4. 异步副作用统一在 middleware 或 action 层处理。
5. 状态变更要可追踪，可回放，可审计。

## 关键知识点

### 必须掌握

- Context 不是全局状态管理银弹。
- store 设计要围绕业务域而非技术模块。
- selector 稳定性直接影响渲染性能。
- 服务端状态应考虑专门方案（如 query 缓存）。
- 状态不可变更新有助于变更追踪。
- 大型应用需要明确状态生命周期策略。

### 选型与取舍

- 统一大 store 管理集中，但改动影响面大。
- 多 store 解耦高，但一致性治理更复杂。
- 强追踪能力提升调试效率，但实现成本更高。

## 对页面性能与稳定性的影响

- 节点稳定时，React 工作流 的链路可预测性会明显提升，发布和排障成本更可控。
- 节点失稳时，问题会向上下游扩散，常表现为性能回退、交互异常或交付风险叠加。
- 该节点优化应和上下游协同验证，避免局部优化把成本转移给后续节点。

## 观测与排障

### 核心指标

- 全局状态更新频次
- 无效重渲染比例
- 状态相关缺陷数
- 状态调试定位时长

### 排障顺序

1. 先确认是状态同步问题还是渲染订阅问题。
2. 检查 selector 是否过宽。
3. 检查状态边界是否过于全局。
4. 分析异步 action 的竞态与幂等性。
5. 优化订阅粒度并回归。

### 常见故障场景

- 一次状态更新导致大量组件刷新。
- 异步 action 竞态导致状态回退。
- 全局 store 逐渐臃肿难维护。
- 排障时无法还原状态演进。

## 常见误区

- 把所有数据都放一个全局 store。
- 忽略 selector 粒度。
- 状态变更不可追踪。

## 深入拆解

### 常见选型建议

- Context：主题、语言、权限等低频全局上下文。
- Redux Toolkit：复杂业务流、可追踪 action 历史。
- Zustand：轻量共享状态、低样板代码。

### 订阅粒度控制

1. selector 只取必要字段。
2. 避免大对象整体订阅。
3. 高频更新状态独立 store。

### 服务端状态建议

- 服务端状态优先交给查询库治理，避免混入本地 UI 状态。

## 题库深挖补充

### 高频对比与边界（基于题库）

- useEffect 关键是依赖完整和清理完整，省依赖是高风险短路。
- Context 适合低频上下文，不适合高频大状态共享。
- 路由与数据获取策略要处理预取命中率和取消竞态。
- 性能优化应先看 Profiler 热点，再决定 memo/useMemo/useCallback。
- 状态管理选型要区分本地 UI 状态与服务端状态。
- 面试常问“Fiber/并发有什么价值”，实战落点是更好的交互优先级调度。

### 线上排障剧本

1. 先确认瓶颈是副作用重跑、状态扩散还是渲染热点。
2. 检查 effect 依赖和异步取消机制。
3. 检查 selector 粒度与订阅范围。
4. 检查路由切换期间数据请求竞态。
5. 用对照组验证优化前后 INP 和重渲染分布。

### 高频追问补充（附答案）

1. 为什么 useEffect 容易出事故？

依赖漏写、清理缺失和异步竞态会叠加放大问题。

2. React.memo 为什么不是万能优化？

它也有比较与维护成本，需命中稳定输入场景才有收益。

3. 状态管理何时该拆 store？

当更新频率和业务边界差异明显时，应拆分避免更新扩散。

## 进阶深挖（第二轮）

### 状态分层建议

- Server State：接口数据与缓存失效策略。
- Client Domain State：业务共享状态。
- UI Ephemeral State：弹窗、面板、输入临时态。

### 订阅粒度优化

1. selector 只返回必要字段。
2. 避免返回新对象导致无效比较。
3. 高频状态独立存储避免扩散。

### 追问扩展

1. 为什么 Context 在大应用里常变慢？

更新粒度粗会触发更大范围订阅刷新。

2. 何时应把状态从全局下放到局部？

当状态只服务单一路径且共享价值低时，下放可显著收敛更新范围。

## 面试补充

### 30 秒说明

React 状态管理的关键是“边界分层 + 粒度控制 + 可追踪变更”，不是选哪个库本身。

### 高频追问

1. Context 为什么不适合所有状态？

高频更新场景会导致订阅范围过大，引发性能问题。

2. Redux/Zustand 这类 store 核心价值是什么？

提供可预测状态流和更细粒度订阅能力。

3. 如何避免 store 膨胀？

按业务域拆分并限制单 store 职责。

4. 状态管理如何支持排障？

记录 action 与状态快照，支持时间线回放。
