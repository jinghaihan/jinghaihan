## 概念定义

- 本质定义：事件循环定义了宏任务、微任务和渲染机会的执行顺序。
- 解决问题：解决异步日志顺序、交互卡顿和渲染延迟的根因定位问题。
- 边界说明：不负责 Promise API 语法细节和请求重试策略。

## 核心机制拆解

### 1) 机制/流程

1. 执行一个宏任务。
2. 清空微任务队列。
3. 浏览器获得渲染机会。
4. 进入下一轮宏任务。

### 2) 关键条件与触发点

- 条件 A：微任务链过长会推迟渲染和输入反馈。
- 条件 B：长时间同步任务会造成主线程阻塞，直接影响交互指标。

### 3) 结果与影响

- 对性能的影响：任务切分合理可降低长任务数量。
- 对稳定性的影响：理解队列顺序可减少异步时序 bug。

## 相似概念对比

| 维度 | 宏任务 | 微任务 | 结论 |
|---|---|---|---|
| 常见来源 | `setTimeout`、I/O | `Promise.then`、`queueMicrotask` | 微任务优先清空 |
| 执行时机 | 每轮主任务 | 当前宏任务末尾 | 顺序差异关键 |
| 对渲染影响 | 过长会阻塞渲染 | 过多会饿死渲染 | 都要控量 |
| 适用场景 | 调度分片与延后任务 | 链式状态收敛 | 不应滥用 |

## 代码

```ts
console.log('A')
setTimeout(() => console.log('B'), 0)
Promise.resolve().then(() => console.log('C'))
console.log('D')

// 输出顺序：A D C B
```

### 示例说明

- 示例体现了“同步 -> 微任务 -> 宏任务”的基本顺序。
- `Promise.then` 属于微任务，优先于 `setTimeout`。
- 常见错误是把定时器回调误判为“比 Promise 更早执行”。

## 相关面试题

1. await 到底在等啥？
   - 标准答案：`await` 会等待右侧表达式转成 Promise 后的 settled 结果。若是非 Promise 值，会被包装为已完成 Promise，再在微任务阶段恢复执行。
   - 考察点：是否理解 `await` 与微任务关系。

2. setTimeout、Promise、Async/Await 的区别
   - 标准答案：`setTimeout` 基于宏任务调度，Promise 回调是微任务，`async/await` 是 Promise 语法糖。执行顺序由事件循环决定，不是“谁先写谁先执行”。
   - 考察点：是否能用队列模型解释顺序。

3. 异步编程的实现方式？
   - 标准答案：从回调、Promise 到 `async/await`，核心演进是可组合性和错误处理能力。现代代码以 Promise 与 `async/await` 为主，再结合取消和并发控制。
   - 考察点：是否能说明演进原因。
```ts
function requestWithCallback(cb: (data: string) => void) {
  setTimeout(() => cb('callback ok'), 200)
}

function requestWithPromise() {
  return new Promise<string>(resolve => {
    setTimeout(() => resolve('promise ok'), 200)
  })
}

async function run() {
  requestWithCallback((msg) => {
    console.log(msg)
  })
  console.log(await requestWithPromise())
}
```
