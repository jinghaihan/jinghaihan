## 概念定位

事件循环（Event Loop）决定脚本任务、微任务与渲染时机如何协作。它是解释“为什么页面卡、为什么顺序错乱”的第一原则。

该节点直接影响 INP、Total Blocking Time（总阻塞时长）和复杂交互稳定性。

## 工作机制

1. 主线程执行一个宏任务（Macro Task）后，会清空微任务（Micro Task）队列，再进入下一轮。
2. 渲染通常发生在宏任务间隙，微任务过多会推迟渲染。
3. `Promise.then`、`queueMicrotask` 属于微任务，`setTimeout`、I/O 回调属于宏任务。
4. 高频微任务链会导致“逻辑完成很快但页面不刷新”的错觉。
5. 把重计算拆分为可中断块，或下沉到 Web Worker（网页工作线程）减少主线程拥塞。

## 关键知识点

### 必须掌握

- 面试常考“事件循环日志输出顺序”，本质是任务队列优先级。
- 微任务不是越多越好，滥用会阻塞渲染。
- 框架调度（如 Vue nextTick、React 调度）都构建在事件循环之上。
- 长任务通常来自同步计算、序列化/反序列化、大批量 DOM 操作。
- 把任务按优先级分层：用户输入 > 可见区更新 > 低优先级后台任务。
- 性能优化要看“任务切片”而不是只看总耗时。

### 选型与取舍

- 过度切片会增加调度开销，切片粒度要结合设备性能。
- 大量微任务可降低延迟但可能恶化渲染时机。
- 同步计算简单直观，但高峰时会直接劣化交互体验。

## 对页面性能与稳定性的影响

- 节点稳定时，前端基础（HTML/CSS/JavaScript） 的链路可预测性会明显提升，发布和排障成本更可控。
- 节点失稳时，问题会向上下游扩散，常表现为性能回退、交互异常或交付风险叠加。
- 该节点优化应和上下游协同验证，避免“局部优化把成本转移给后续节点”。

## 观测与排障

### 核心指标

- INP（Interaction to Next Paint，交互到下一次绘制）
- TBT（Total Blocking Time，总阻塞时长）
- Long Task 计数与最长持续时间
- 微任务队列峰值长度

### 排障顺序

1. 先从性能录制里找长任务分布和触发源。
2. 确认拥塞来自脚本执行还是布局绘制。
3. 检查是否存在连续微任务链阻塞渲染。
4. 对重任务做切片、降频或下沉 Worker。
5. 回归验证：输入响应、动画流畅度、低端机表现。

### 常见故障场景

- “点击后延迟明显”：常见是点击回调触发大计算同步执行。
- “数据更新了但界面没及时刷新”：常见是微任务链过长。
- “CPU 占用高且掉帧”：常见是任务切片不足。
- “偶发顺序 bug”：常见是误判宏/微任务执行顺序。

## 常见误区

- 把事件循环仅当成面试题，不用于实战排障。
- 只看函数耗时，不看任务排队与阻塞关系。
- 把所有异步都塞微任务，忽略渲染机会。

## 深入拆解

### 事件循环时序示意

1. 执行当前宏任务（例如一次 click 回调）。
2. 清空微任务队列（Promise 回调、queueMicrotask）。
3. 浏览器获得渲染机会（若主线程空闲）。
4. 进入下一轮宏任务（setTimeout、MessageChannel 等）。

这个顺序解释了为什么微任务写太多会卡渲染。

### 长任务治理策略

- 大计算切片：把 200ms 连续计算拆成多个小片段。
- 优先级调度：用户输入任务优先，后台统计任务降级。
- 线程下沉：复杂解析放到 Worker，主线程只做最小渲染工作。
- 批处理更新：多次状态变更合并后再提交渲染。

### 排障中的关键判断

- 如果 Long Task 明显高：优先查同步计算和序列化。
- 如果微任务堆积：检查 Promise 链是否递归触发。
- 如果渲染迟迟不发生：确认是否被任务队列长期占满。

## 题库深挖补充

### 高频对比与边界（基于题库）

- 一轮事件循环通常是：执行宏任务 -> 清空微任务 -> 进入渲染机会。
- Promise.then/queueMicrotask 属于微任务，setTimeout 属于宏任务。
- 微任务过多会延后渲染，导致“数据到了但 UI 不刷新”的体感问题。
- Long Task（长任务）通常来自大计算、频繁序列化、批量 DOM 操作。
- Web Worker 适合下沉 CPU 密集任务，不可直接操作 DOM。
- 面试高频是输出顺序题，实战高频是 INP/TBT 过高排障。

### 线上排障剧本

1. 先在性能面板定位长任务分布。
2. 检查微任务链是否递归堆积。
3. 检查是否存在高频事件中的同步重计算。
4. 把重任务切片或下沉 Worker。
5. 观察 INP 与主线程占用是否改善。

### 高频追问补充（附答案）

1. 为什么微任务会影响渲染？

因为每轮宏任务后会先清空微任务队列，渲染要等待。

2. setTimeout(fn, 0) 为什么不是立即执行？

它只是尽快入宏任务队列，仍要排队。

3. 如何快速判断是否事件循环拥塞？

看长任务数量、微任务堆积和交互延迟指标。

## 进阶深挖（第二轮）

### 调度 API 选择原则

- 用户输入链路：优先保证可中断和快速响应，避免长同步任务。
- 可见区更新：与渲染节拍对齐，避免连续微任务压满队列。
- 后台统计与日志：放低优先级，必要时合并批量上报。
- 重计算任务：拆片或下沉 Worker，避免长期占用主线程。

### 常见隐性拥塞点

1. 连续 Promise 链中每步都触发状态更新。
2. 输入事件回调里做 JSON 大对象转换。
3. 一次点击触发多个并行请求回调同时落在主线程。

### 追问扩展

1. 为什么有些页面“看起来没做什么”却 INP 很差？

常见是隐性任务堆积，例如日志、埋点、序列化与小任务风暴，不一定在业务主逻辑里。

2. MessageChannel 什么时候比 setTimeout 更合适？

当你需要更稳定的任务切片节奏而不是受定时器最小间隔影响时。

## 面试补充

### 30 秒说明

事件循环优化不是“让代码更异步”，而是“让关键交互先得到主线程时间片”。

### 高频追问

1. 为什么微任务会影响渲染？

因为每轮宏任务后会先清空微任务队列，微任务过多会推迟下一次绘制。

2. 如何快速判断是否有长任务问题？

看 Performance 面板中超过 50ms 的任务块并定位调用栈。

3. `setTimeout(fn,0)` 为什么不是立刻执行？

它只是把任务放入宏任务队列，仍需等待当前任务和前序队列完成。

4. Worker 适合处理什么任务？

CPU 密集型计算、解析和转换等不依赖 DOM 的任务。
